3. 토큰 매니지먼트 ⑥ — headroom · caveman · LSP

세 가지가 각기 다른 토큰 스트림을 줄인다. 중복이 아니라 적층(stack).

도구 줄이는 대상 광고 절감 설치 비고
headroom API 페이로드(grep·diff 등)를 프록시에서 압축 47~92% (중앙값 54%) pip install --user headroom → headroom wrap claude RTK 번들 포함
caveman Claude 자체 출력(프롬프트·응답)을 telegraphic하게 50~75% claude plugin marketplace add JuliusBrussee/caveman → claude plugin install caveman@caveman /caveman lite\|full\|ultra
LSP 코드 탐색을 본문→좌표로 축소 정의 43× · 참조 2.1× (실측) 언어서버 설치 + tweakcc --apply + /plugin install Piebald-AI 마켓플레이스
codesight 세션 시작 구조 파악을 압축 맵 1개로 세션당 6.8× (실측 ≈27,915토큰) npx codesight --init + --mcp LSP와 상호보완

설치 링크(검증 완료): - headroom: https://github.com/chopratejas/headroom - caveman: https://github.com/JuliusBrussee/caveman - rtk(번들): https://github.com/rtk-ai/rtk - LSP 마켓플레이스: https://github.com/Piebald-AI/claude-code-lsps - codesight: https://github.com/Houseofmvps/codesight - 통합 설치 스택(참고): https://github.com/sgaabdu4/claude-code-tips

caveman 변형: caveman-commit(50자 이내 커밋), caveman-review(한 줄 PR 코멘트), caveman-compress(CLAUDE.md 압축, 입력 ~46% 절감).

LSP 적용 기준: - 파일 20개 미만 소규모 → grep 기본으로 충분 (LSP 불필요) - 중대형/리팩토링 → codesight(세션 시작 구조) + LSP(정의·참조 추적) 병행 - 다중/미지원 언어 → cclsp, 심볼 편집·메모리 필요 시 → Serena(--context claude-code)

주의

⚠️ 절감폭 현실 주의: 한 독립 실측(614M 토큰 코퍼스)에서 headroom+rtk+caveman 합산 실효 절감은 약 3.7%에 그쳤다. 매 턴 컨텍스트가 저렴한 cache_read로 재전송되고, 비용 대부분이 cache_create·output이라 이들 도구가 손대지 않기 때문. 즉 광고 수치는 "특정 스트림 단독 최선값"이며, 실효 절감은 워크로드(셸·grep·리뷰 비중)에 크게 의존한다. LSP/codesight는 탐색이 많은 코드베이스에서 별도로 큰 효과(실측 6~43×). ⚠️ 보안 주의: headroom·rtk·caveman은 코드·프롬프트·명령 출력을 읽는다. 현재는 local-first이나 단일 오염 릴리스가 위험이 될 수 있으므로 버전 고정·출처 검증 권장.

4. SDD 킷 ⑦ — GSD (Get Shit Done)

"Context Rot(컨텍스트 부패)"를 막는 경량 스펙 주도 개발 시스템. 상태를 파일로 외부화하고 각 플랜을 fresh context에서 실행.

  • repo: https://github.com/gsd-build/get-shit-done (★45k, MIT)
  • 설치: npx get-shit-done-cc --global (전역) 또는 --local (현재 프로젝트)

워크플로우:

/gsd:new-project     → 아이디어를 상세 스펙으로 확장
/gsd:create-roadmap  → 마일스톤 기반 로드맵
/gsd:plan-phase      → 페이즈별 XML 최적화 플랜
/gsd:execute-plan    → 원자적 커밋 단위 구현
/gsd:verify          → 명시적 완료 기준 검증
  • Quick 모드: /gsd:quick "다크모드 토글 추가" (소규모 태스크, --research·--full·--validate 옵션)
  • 차별점: XML 최적화 플랜 포맷 — <task><files><action><verify><done> 으로 완료 조건을 구조화
참고

대안/비교: 공식 spec-kit(specify-cli)도 가능. spec-kit은 Claude Code에서 Skills 모드로 설치되어 /speckit-*(하이픈) 표기. GSD는 더 가벼운 단일 마일스톤용, spec-kit은 품질 게이트(clarify·checklist·analyze)가 풍부. API 포탈 규모면 spec-kit으로 계약 단위 명세 → GSD quick으로 소규모 변경 혼용 권장. 연결 포인트: GSD <verify><done> 슬롯에 ⑧의 "기능별 합격 기준"을 그대로 넣으면, 에이전트가 게이트웨이 검증을 건너뛸 수 없다.

5. 게이트웨이 검증 ⑧ — 인증·인가·API키·라우팅·레이트리미트

게이트웨이의 다섯 기능은 비즈니스 로직이 아니라 교차 관심사(cross-cutting)라, 단위 테스트로는 "실제 구동"을 검증할 수 없다. 백엔드를 스텁으로 세우고 게이트웨이를 통째로 띄워 요청을 흘려보내는 컴포넌트/통합 테스트가 핵심이며, Walking Skeleton(전 구간 1줄 관통)과 계약 테스트(스텁≈현실 보증)로 완성한다.

5-1. 용어 정리 (요청 표현 → 정확한 개념)

요청 표현 정확한 개념 게이트웨이에서 쓰임
목업 테스트 Mock (행위 검증) 게이트웨이가 인증서버를 "호출했는지" 확인
스텁 테스트 Stub (정해진 응답 반환) WireMock 백엔드가 200/500 반환
스켈레톤 테스트 Walking Skeleton (얇은 전구간) 클라이언트→게이트웨이→백엔드 1시나리오 관통
참고

테스트 더블 5종(Meszaros): Dummy(채우기)·Stub(정해진 응답)·Spy(호출 기록)·Mock(기대값+행위검증)·Fake(동작하는 경량 구현). 이 중 Mock만 행위 검증을 강제. Fake는 인메모리 DB처럼 동작하나 프로덕션엔 부적합.

5-2. 기능별 테스트 매핑

게이트웨이 기능 검증 대상 테스트 더블 / 도구 합격 기준(SC) 예
라우팅 경로·헤더에 따라 올바른 백엔드로 WireMock 스텁 백엔드 + WebTestClient 요청이 stub A로 감 · route attr 일치
인증 유효/만료/위조 토큰 Stub 토큰 또는 Testcontainers Keycloak 유효=200, 무효/만료=401
인가 역할·스코프 기반 role 클레임 담긴 Stub 토큰 권한 없음=403
API 키 발급 발급→검증 흐름 실제 발급 엔드포인트 + Testcontainers store 발급 키 통과, 폐기 키 차단
레이트 리미트 N회 초과 시 차단 Testcontainers Redis + 반복 호출 한도 내 200, 초과 429 + Retry-After
Walking Skeleton 전 구간 관통 위 전부를 얇게 1시나리오 토큰→라우팅→백엔드 200 한 번 통과

5-3. 테스트 피라미드 (게이트웨이용)

        e2e (walking skeleton) — 최소 1~2개, 전구간 관통
      contract (Spring Cloud Contract/Pact) — 스텁=현실 보증
    component/integration — 게이트웨이+WireMock+Testcontainers (가장 두껍게)
  unit — 필터·토큰파서 (Mockito stub)
  • 단위: 커스텀 필터·토큰 파서를 Mockito stub으로 격리
  • 컴포넌트/통합(가장 중요): 게이트웨이 전체 컨텍스트 + WireMock 백엔드 + Testcontainers Redis. WebTestClient/RestAssured로 호출. 라우팅은 GATEWAY_ROUTE_ATTR를 응답 헤더로 노출해 검증
  • 계약(Contract): 스텁의 함정 방지 — "스텁이 현실을 반영 못하면 통과해도 실제 깨짐". 머지 승인 전 스텁≈현실 주기 검증
  • e2e/skeleton: 끝에서 끝까지 동작하는 미완성 골격을 만든 뒤 점진 발전

5-4. 권장 도구 스택 (Java/Spring)

목적 도구 비고
백엔드 스텁 WireMock (@AutoConfigureWireMock(port=0) 또는 Testcontainers WireMock 모듈) 라우팅·헤더 변조 검증 표준
게이트웨이 호출 WebTestClient (reactive SCG) / MockMvc (MVC) route attr 검증 가능
인프라 의존성 Testcontainers — Redis(레이트리미트)·Keycloak(인증)·PostgreSQL(API키 저장) 실제 컨테이너 통합 검증
계약 테스트 Spring Cloud Contract(spring-cloud-starter-contract-stub-runner) 또는 Pact WireMock 부트스트랩 단순화
