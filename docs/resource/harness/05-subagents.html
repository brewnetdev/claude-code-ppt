Level 5 · Context Hygiene & Second Eyes

서브에이전트와 검증 루프 — 컨텍스트 위생과 제2의 눈

하네스 최상위 레벨의 원칙 두 가지: 큰 조사는 격리된 컨텍스트에 시키고(위생), 일한 에이전트가 스스로를 채점하게 두지 않는다(검증). 그리고 이 검증을 "증거"로 종결시키는 방법.

5-1. 용어 정의

서브에이전트 (Subagent)
격리된 컨텍스트 윈도우에서 자체 에이전트 루프를 돌고 요약만 반환하는 행위자. Task 도구로 생성되며, .claude/agents/<name>.md로 정의를 재사용할 수 있다. 쉬운 비유: 조사 나갔다가 요약 한 장만 들고 돌아오는 인턴. 인턴이 뒤진 서류 더미는 내 책상(메인 컨텍스트)에 올라오지 않는다.
컨텍스트 위생 (Context Hygiene)
다시 볼 일 없는 대량 자료가 메인 컨텍스트를 침수시키지 않게 유지하는 규율. 1강 대전제("차면 저하된다")의 실행 전술.
검증 서브에이전트 (Verification Subagent)
신선한(fresh) 컨텍스트의 모델이 작업 결과를 반박 시도하는 패턴 — 일한 에이전트가 스스로 채점하지 않게 만든다.
Test-time Compute
추론 시점에 컴퓨트를 더 투입해 품질을 올리는 접근. 여기서는 "별도 컨텍스트 윈도우가 결과를 실제로 좋게 만든다 — 한 에이전트가 만든 버그를 같은 모델의 다른 에이전트가 찾는다"로 구체화된다.
에이전트 팀 / git worktree
여러 에이전트가 병렬로 작업하는 구성. worktree는 같은 레포의 브랜치별 독립 작업 폴더로, 에이전트 간 파일 충돌을 격리한다.

용어 근거: 공식 Best Practices(검증 서브에이전트·에이전트 팀), DEV Community(Task 도구·요약 반환), shanraisshan(test-time compute·worktree)

5-2. 이론 — 두 개의 실패 모드와 하나의 해법

실패 모드 1: 컨텍스트 오염

"레이트 리밋이 어디서 강제되는지" 한 문장을 답하기 위해 15개 파일을 읽으면, 그 15개 파일은 답이 나온 뒤에도 컨텍스트에 남아 이후 모든 추론의 예산을 잠식한다. 서브에이전트가 파고들어 한 문단만 건네면 메인 세션은 다시 볼 일 없는 자료로 침수되지 않는다 — "문제에 컴퓨트를 더 쓰고 싶으면 'use subagents'라고 말하라"가 커뮤니티 규칙인 이유. (DEV Community, shanraisshan)

실패 모드 2: 자기 채점 편향

구현한 에이전트의 컨텍스트에는 "이 코드가 왜 옳은지"의 서사가 이미 쌓여 있다 — 사람의 "내가 짠 코드는 내 눈에 맞아 보인다"와 같은 구조다. 공식 처방: 검증 서브에이전트 또는 자기 발견을 재검사하는 동적 워크플로는 신선한 모델이 결과를 반박하게 하여, 일한 에이전트가 채점자가 되지 않게 한다. 별도 컨텍스트는 실제로 결과를 개선한다 — 같은 모델이라도 컨텍스트가 다르면 버그를 서로 찾아낸다. (공식, shanraisshan)

5-3. 적용 전 vs 적용 후 — 목격되는 현상

분리 전분리 후
대량 탐색 후 메인 세션이 무거워져 후반 추론 품질이 급락 — "세션 오래 쓰면 멍청해진다"는 체감.조사 결과가 요약으로만 유입 — 긴 세션에서도 메인 컨텍스트가 가볍게 유지됨.
"다 됐습니다"라는 자기 보고를 사람이 일일이 재검증 — 검증이 사람의 병목.구현–검증–수정 루프가 사람 없이 회전. 사람의 일은 기록된 발견의 스팟 체크로 축소. (공식: 에이전트 팀이 이 루프를 여러 과제에 걸쳐 유지)
구현자가 낙관 보고 → 프로덕션에서 미구현 요구사항 발견.신선한 눈이 PLAN.md와 diff를 대조 — 미구현·범위 밖 변경이 ship 전에 표면화.
완료 판정 = 에이전트의 단언("테스트 통과했습니다").완료 판정 = 증거(테스트 출력·실행 명령·반환값·스크린샷). 증거 검토가 검증 재실행보다 빠르고, 지켜보지 않은 세션에도 통한다. (공식)

5-4. 예제 — 검증 서브에이전트 정의와 호출

사용 방법: 리뷰어 에이전트를 파일로 정의해두고, 구현이 끝나면 메인 세션에서 리뷰를 위임한다.

# .claude/agents/plan-reviewer.md
---
name: plan-reviewer
description: 구현 diff 를 PLAN.md 와 대조 검증할 때 사용.
---
너는 검증 전담 리뷰어다. 주어진 diff 를 PLAN.md 와 대조하여:
1. 모든 요구사항이 구현되었는지
2. 명시된 엣지 케이스에 테스트가 있는지
3. 과제 범위 밖의 변경이 없는지
만 검사한다. 스타일 취향은 보고하지 않는다.
정확성 또는 명시된 요구사항에 영향을 주는 갭만 보고하라.

# 메인 세션에서 (공식 문서의 프롬프트 패턴)
"서브에이전트를 사용해 rate limiter diff 를 PLAN.md 와 대조 리뷰하라.
모든 요구사항이 구현됐는지, 나열된 엣지 케이스에 테스트가 있는지,
과제 범위 밖에서 바뀐 것이 없는지 확인하고,
스타일 취향이 아니라 갭만 보고하라."

5-5. 실증 — 리뷰어의 과보고: 공식 문서가 경고하는 함정

검증 루프를 도입한 팀이 다음으로 겪는 현상은 공식 문서에 그대로 기록되어 있다: "갭을 찾으라고 프롬프트된 리뷰어는 작업이 건전할 때도 보통 무언가를 보고한다 — 그것이 요청받은 일이기 때문이다. 모든 발견을 좇으면 과잉 설계로 이어진다: 불필요한 추상 레이어, 방어 코드, 일어날 수 없는 케이스에 대한 테스트." 처방도 명시적이다: 리뷰어에게 정확성 또는 명시된 요구사항에 영향을 주는 갭만 표시하게 하고, 나머지는 선택 사항으로 취급하라.

이 실증의 의미: 검증자는 공짜가 아니다 — 검증자의 판정 기준 자체가 설계 대상이다. 이 통찰은 7강에서 그대로 재등장한다: LLM-as-Judge도 "채점하라면 무언가를 지적"하며, 루브릭을 좁고 검증 가능하게 제한하지 않으면 평가가 개발을 왜곡한다.

근거: 공식 Best Practices (과보고 관찰과 처방 원문)

5-6. 확장 — 병렬화와 무인 실행

  • 에이전트 팀: 장시간 자율 실행에서는 팀이 구현–검증 루프를 여러 과제에 걸쳐 유지하고, 사람은 기록된 발견을 스팟 체크한다. (공식)
  • tmux + git worktree: 병렬 에이전트마다 독립 작업 폴더를 줘 파일 충돌을 격리한다. (shanraisshan, Habib Mrad)
  • 스킬과의 접점: 스킬에 context: fork를 지정하면 스킬 실행 자체가 격리 서브에이전트에서 돌아, 메인은 최종 결과만 본다(3강 참조).
  • 무인 실행의 완성: 검증 루프(이 강) + Stop 훅 종료 조건(4강) + 증거 원칙이 합쳐질 때 "지켜보지 않은 세션이 올바르게 끝나는" 상태가 된다 — 6강 완성형의 정의로 이어진다.

5-7. 안티패턴 3가지

안티패턴왜 실패하는가교정
리뷰어 지적 전부 수용과보고는 구조적 성질 — 좇으면 과잉 설계(5-5 실증)"정확성·명시 요구사항 갭만" 제한, 나머지는 선택
구현자에게 자기 검증 요청자기 서사가 쌓인 컨텍스트는 반박을 못 한다신선한 컨텍스트의 별도 에이전트로 분리
서브에이전트 결과를 원문 그대로 회수격리의 목적(위생) 자체가 무효화됨요약·결론만 반환하도록 에이전트 정의에 명시
7강 예고 — 리뷰어에서 Judge로

"신선한 눈이 루브릭으로 갭만 보고한다"는 이 구조는 EDD의 LLM-as-Judge와 정확히 동형이다. 차이는 하나 — EDD는 이 판정을 점수화하고, 임계값·골든셋·인간 일치율 검증으로 관리한다. 5강까지 온 팀은 Judge의 프로토타입을 이미 운영 중인 셈이다.

5-8. 체크리스트

  • 대량 파일 탐색이 서브에이전트로 오프로드되는가
  • 구현 에이전트와 검증 에이전트가 분리되어 있는가
  • 리뷰어에게 "갭만, 취향 제외" 제한이 걸려 있는가
  • 완료 판정이 단언이 아니라 증거로 이뤄지는가
  • 병렬 에이전트가 worktree 등으로 작업 공간을 격리하는가
  • 서브에이전트가 요약만 반환하도록 정의되어 있는가
