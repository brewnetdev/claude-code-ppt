훅 — 요청(request)을 보증(guarantee)으로

하네스 설계에서 가장 중요한 한 문장: "CLAUDE.md와 스킬의 지시는 요청이지 보증이 아니다. 반드시 일어나야 하는 일은 훅에 속한다." 이 강은 그 이유(이론), 결과(실증), 방법(구현), 그리고 무인 실행이라는 확장까지 다룬다.

4-1. 용어 정의

훅 (Hook)
에이전트 루프 밖에서 라이프사이클 이벤트에 발화하는 이벤트 핸들러. 셸 명령·HTTP 요청·프롬프트·서브에이전트를 실행할 수 있다. 쉬운 비유: "테스트 돌려주세요"라고 말하는 것(요청)과 테스트가 깨지면 커밋 자체가 안 되는 CI(보증)의 차이. 훅이 후자다.
라이프사이클 이벤트 (Lifecycle Event)
훅이 발화하는 시점들. 대표적으로 PreToolUse(도구 호출 직전 — 차단 가능), PostToolUse(호출 직후 — 후처리), SessionStart(세션 시작), Stop(에이전트가 멈추려는 순간 — 종료 조건 검사).
확률적 준수 vs 결정적 강제
프롬프트·CLAUDE.md의 지시는 대체로 지켜진다(확률적) — 컨텍스트가 차면 누락될 수 있다(1강 대전제). 훅·설정은 모델의 의지와 무관하게 실행된다(결정적).
무인 실행 (Unattended Run)
사람이 지켜보지 않는 장시간 자율 실행. 공식 문서 표현으로, 프롬프트 버전 검증은 아무 과제에나 통하지만 /goal과 Stop 훅 버전이야말로 무인 실행이 사람 없이 올바르게 끝나게 해주는 것이다.

용어 근거: DEV Community(이벤트·실행 종류), mcp.directory(루프 밖 핸들러), 공식 Best Practices(/goal·Stop 훅과 무인 실행)

4-2. 이론 — 왜 "요청"은 반드시 새는가

이 구분이 필요한 근본 이유는 1강의 대전제로 되돌아간다: 컨텍스트가 차면 성능이 저하된다. CLAUDE.md의 규칙은 세션 초반엔 잘 지켜지다가, 긴 세션에서 컨텍스트가 차오르면 주의 경쟁에서 밀려 누락된다 — 즉 요청의 준수율은 세션 길이의 감소 함수다. 규칙 위반이 "가끔, 긴 세션에서, 하필 중요한 순간에" 발생하는 패턴은 버그가 아니라 확률적 준수의 구조적 성질이다. 그래서 실무 결론은 단호하다: "CLAUDE.md와 스킬의 지시는 요청이지 보증이 아니다. 반드시 일어나야 하는 것은 훅에 속한다. 가드레일은 훅에 넣고, 그 외 전부는 정중한 제안이다." 근거: 공식(컨텍스트 저하 대전제), DEV Community(요청/보증 구분 원문)

4-3. 적용 전 vs 적용 후 — 목격되는 현상

훅 도입 전도입 후
"이번엔 포맷을 안 했네", "또 위험한 명령을 시도했네" — 규칙 준수가 세션 길이와 그날 컨디션에 좌우.포맷 안 된 파일·차단 대상 명령이 구조적으로 존재 불가능한 범주가 생김. 규칙 리뷰가 "지켰나"에서 "규칙 자체가 옳은가"로 승격.
긴 세션 후반부에 품질 사고 집중 — 원인 추적이 "왜 이번엔 무시했지?"라는 미궁으로.보증 층 위반은 0건이므로, 남는 사고는 전부 요청 층 설계 문제로 좁혀짐 — 디버깅 표면이 절반으로.
무인 실행을 맡기면 "다 했습니다"라며 미완성 상태로 조기 종료.Stop 훅이 종료 조건(목표 달성 여부)을 검사 — 무인 실행이 사람 없이 올바르게 끝나는 상태에 도달. (공식)
수정 후 테스트를 "나중에 몰아서" → 깨진 상태 위에 작업이 누적.PostToolUse 훅의 빠른 테스트 패스로 깨짐이 발생 즉시 표면화 — 결함의 발견 지연이 0에 수렴.

4-4. 대표 용례 — 이벤트별 정리

이벤트용례보증하는 것
PostToolUse파일 쓰기마다 포매터 자동 실행"포맷 지켜줘"(요청) → 포맷 안 된 파일이 존재할 수 없음(보증)
PostToolUse수정 후 빠른 테스트 패스 실행깨진 상태로 다음 단계 진행 불가
PreToolUse명백히 위험한 셸 명령 사전 차단실수·오판이 실행 전에 걸러짐
Stop종료 조건(/goal 달성) 검사, 미달 시 계속미완성 조기 종료 불가 — 무인 실행의 핵심 부품
SessionStart환경 상태 주입(브랜치·마이그레이션 상태 요약)세션이 낡은 가정으로 시작하지 않음

용례 근거: DEV Community(포맷·테스트·차단 3종), 공식(Stop 훅과 무인 실행)

4-5. 예제 — 자동 포맷 + 위험 명령 차단기

사용 방법: .claude/settings.json의 hooks에 등록한다. 이후 에이전트가 어떤 파일을 수정하든 포매터가 모델의 의지와 무관하게 실행되고, 위험 명령은 실행 전에 걸러진다.

// .claude/settings.json 내 hooks (예시 — 스키마는 설치 버전 공식 문서로 확인)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command",
            "command": "pnpm prettier --write \"$CLAUDE_FILE_PATHS\"" }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command",
            "command": "python3 .claude/hooks/block_dangerous.py" }
        ]
      }
    ]
  }
}

# .claude/hooks/block_dangerous.py — 위험 명령 사전 차단기
import json, re, sys

BLOCK_PATTERNS = [
    r"rm\s+-rf\s+/",          # 루트 삭제
    r"git\s+push\s+--force",   # 강제 푸시
    r"curl[^|]*\|\s*(ba)?sh",  # 원격 스크립트 파이프 실행
]

payload = json.load(sys.stdin)                      # 훅 입력(JSON)
cmd = payload.get("tool_input", {}).get("command", "")

for pat in BLOCK_PATTERNS:
    if re.search(pat, cmd):
        print(f"차단됨: 패턴 '{pat}' 매칭", file=sys.stderr)
        sys.exit(2)   # 비정상 종료 = 도구 호출 차단 신호
sys.exit(0)

훅 입출력 규약(종료 코드 의미, payload 스키마)은 버전에 따라 다를 수 있으므로 설치 버전 공식 문서 대조 필수. 원칙은 불변: 차단 로직은 모델 밖의 코드로 존재한다.

4-6. 실증 — 훅 생태계의 성숙

훅은 이론적 기능이 아니라 이미 전용 생태계가 형성된 프리미티브다. GitHub 트렌딩 1위를 기록한 베스트 프랙티스 저장소는 훅 구현을 별도 전용 레포로 분리해 배포하고 있으며(작업 중 이벤트를 소리로 알리는 커스텀 훅 사운드까지 포함), 에이전트 팀·자기 진화 루프(Ralph Wiggum 패턴) 같은 고급 패턴이 훅을 기반 부품으로 사용한다. 즉 훅은 "옵션 기능"이 아니라 고급 워크플로 전체가 딛고 서는 바닥이다.

근거: shanraisshan/claude-code-best-practice (전용 훅 레포·훅 사운드·고급 패턴 구현 디렉토리)

4-7. 배치 지도 — 어디에 무엇을 넣을지 최종 정리

계층성격넣을 것
CLAUDE.md (1강)항상 로드되는 요청짧고 안정적인 컨벤션, 완료의 정의
스킬 (3강)조건부 로드되는 요청작업 유형별 긴 노하우, Gotchas
settings.json (2강)결정적 경계권한 allow/ask/deny, 하네스 강제 설정
훅 (4강)결정적 행동반드시 실행/차단되어야 하는 것 전부 + 무인 실행의 종료 조건

4-8. 안티패턴 3가지

안티패턴왜 실패하는가교정
보증이 필요한 규칙을 CLAUDE.md에 "NEVER" 문장으로확률적 준수는 긴 세션에서 반드시 샌다 (4-2 이론)판별 질문(2강): 어기면 곤란한가? → 훅/설정으로
무거운 훅 남발모든 편집마다 전체 테스트를 돌리면 루프가 기어감 — 보증에도 지연 비용이 있다PostToolUse엔 빠른 검사만, 전체 검증은 커맨드/CI로
훅 실패를 조용히 삼킴보증 층이 침묵하면 요청 층보다 위험 — "지켜지고 있다"는 거짓 안심훅 실패는 반드시 표면화(비정상 종료 + 로그)
7강 예고 — 훅 자리에 평가를 꽂으면

훅의 구조(이벤트 → 검사 → 통과/차단)는 EDD의 blocking eval과 동형이다. 차이는 검사 내용뿐 — 정규식 패턴 대신 골든셋 점수와 임계값을 넣으면, 그것이 7강의 평가 게이트다. 4강까지 온 팀은 EDD의 실행 인프라를 이미 가진 셈이다.

4-9. 체크리스트

  • "반드시 일어나야 하는 일" 목록이 존재하고, 전부 훅/설정으로 구현되어 있는가
  • CLAUDE.md에 "NEVER/반드시" 문장이 남아있지 않은가
  • 파일 수정 → 자동 포맷/빠른 테스트 훅이 걸려 있는가
  • 위험 명령 차단기가 모델 밖 코드로 존재하는가
  • 무인 실행에 Stop 훅 기반 종료 조건이 있는가
  • 훅 실패가 침묵하지 않고 표면화되는가
