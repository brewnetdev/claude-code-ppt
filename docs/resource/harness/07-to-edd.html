Beyond the Harness · Evaluation-Driven Development

하네스 이후 — EDD로 가는 방향성

완성형 하네스는 "올바르게 실행"한다. EDD는 거기에 "얼마나 잘하는지 측정하고, 그 숫자로 결정"을 얹는다. 이 강은 (1) 하네스가 못 하는 것의 근거, (2) 도입 전후의 실증, (3) 6강까지의 부품이 EDD 부품으로 승격되는 경로를 다룬다.

7-1. 용어 정의 (EDD 최소 세트)

Eval (평가)
LLM 시스템의 출력 품질을 체계적·반복적으로 측정하는 절차와 점수. 쉬운 비유: 매번 같은 심사표로 채점하는 고정 심사위원단 — 기분 따라 바뀌는 손님 리뷰(바이브 체크)의 반대.
골든 데이터셋
입력·정답(기준) 쌍의 영구 컬렉션. 6강 PLAN.md가 1회용 대조표라면, 골든셋은 재사용되는 대조표다.
LLM-as-Judge
별도 LLM이 명시적 루브릭으로 출력을 채점하는 패턴. 5강의 검증 서브에이전트에 점수·임계값·인간 일치율 검증을 부여한 것.
평가 게이트 / Blocking eval
CI에서 점수 미달 시 머지를 차단하는 게이트(오프라인), 런타임 크리티컬 패스에서 출력을 차단·재시도·폴백시키는 평가(온라인). 4강 훅의 "검사 내용"이 점수로 바뀐 것.

전체 용어·이론·통계(pass@k vs pass^k, 오차 범위)는 별도 문서 「EDD 완전 가이드 (10장)」 1·2·7장에서 상세히 다룬다. 이 강은 하네스와의 접점에 집중한다.

7-2. 근거 — 하네스만으로 안 되는 것 4가지와 EDD의 해결 장치

하네스의 한계EDD의 해결 장치근거
① 판정이 정성적. review(5강)는 "갭 있음/없음"만 낸다 — 개선·회귀의 정도를 말하지 못한다. 루브릭 점수 + 임계값. Judge–인간 일치율로 채점 자체를 교정. Anthropic Docs — 성공 기준을 측정 가능하게, 채점 방법 선택 원칙
② 추이를 모른다. 게이트는 이번 변경만 본다 — "지난달보다 나아졌는가"에 답할 데이터가 없다. 골든셋 고정 + 매 변경 재채점 → regression delta 시계열. Anthropic, Demystifying evals — 지표가 감(guesswork)을 대체
③ 도구·프레임워크는 모델 수준 평가에 부분적으로 갇힘. 프롬프트-응답 정확도 중심이라, 컨텍스트 엔진·메모리·가드레일을 포함한 컴파운드 시스템의 실패(도구 오용·메모리 오염)가 미탐지로 남는다. 파이프라인(프롬프트·중간 결과·출력)과 아티팩트(목표·메모리·도구·가드레일) 전체를 평가 범위로 삼는 시스템 수준 평가 + 실패 트리아지(실패 모드 분류 → 최초 결정 지점에 표적 수정). Xia et al., EDDOps (arXiv:2411.13768) — 기존 도구의 partial support 지적, 시스템 수준·연속 평가, 트리아지 방법론
④ 배포 후가 무방비. 하네스의 게이트는 배포 전에 끝난다 — 입력 분포 변화·드리프트는 아무도 안 잰다. 온라인 모니터링 eval(샘플링 채점) + 런타임 blocking eval(가드레일). 동일 논문 — 오프라인+온라인 연속 평가를 일급 기능으로

7-3. 적용 전 vs 적용 후 — 목격되는 현상 (Anthropic 실전 보고)

Eval 없이 (완성형 하네스만으로)EDD 도입 후
반응적 루프: 하나를 고치면 다른 게 깨지고, 진짜 회귀인지 노이즈인지 구분하지 못한 채 수렁에 빠짐.개발 가속: 실패가 테스트 케이스가 되고, 테스트 케이스가 회귀를 막고, 지표가 감을 대체 — 가치가 복리로 쌓임.
프로덕션에서만 발견: 사용자 불만 → 수동 재현 → 수정 → "다른 게 안 깨졌기를 기도".배포 전 검출: 변경을 수백 시나리오에 먼저 돌리고 점수 델타로 판단.
"느낌이 나빠졌다" 논쟁: 팀원마다 체감이 달라 행동으로 옮길 수 없음.공동의 오를 언덕: "느낌"이 측정 항목으로 바뀌고 팀 전체가 같은 지표를 봄 — 단, eval을 부속물이 아닌 핵심 컴포넌트로 다룰 때만.

근거(전/후 전체): Anthropic, Demystifying evals for AI agents

7-4. 실증 사례 — 실제 팀의 전환 기록

Descript (수동 채점 → LLM grader 진화): 영상 편집 에이전트에 성공적 워크플로의 3차원 — "망가뜨리지 말 것 / 시킨 것을 할 것 / 잘할 것" — 으로 eval을 설계. 수동 채점에서 시작해 제품팀이 기준을 정의한 LLM grader로 진화했고, 주기적 인간 교정을 유지하며 현재 품질 벤치마킹용·회귀 테스트용 이중 스위트를 상시 운영한다. 5강 리뷰어의 "루브릭 좁히기 + 인간 재교정"이 조직 프로세스가 된 형태.

Bolt (대규모 서비스 후 3개월 구축): 이미 널리 쓰이는 에이전트를 가진 상태에서 뒤늦게 시작해, 3개월 만에 정적 분석으로 출력 채점 + 브라우저 에이전트로 생성 앱 실제 테스트 + 지시 이행은 LLM judge 판정이라는 3중 채점 시스템을 구축했다. 코드형 grader / 실행 검증 / Judge를 섞는 이 구성은 "채점자를 사려 깊게 설계하고 여러 유형을 결합하라"는 원칙의 실물이다.

시작 규모의 실증: 수백 케이스가 필요하다는 오해와 달리, 실제 실패에서 추출한 20~50개 단순 과제로 충분히 시작한다 — 초기 변경은 효과 크기가 커서 작은 표본으로 신호가 잡히며, 프로덕션 중이라면 버그 트래커·CS 큐가 최고 소재다. 과제는 "두 전문가가 동일하게 채점 가능한" 명세로 다듬는다.

근거: Anthropic, Demystifying evals(Descript·Bolt·grader 결합 원칙), Inkeep 해설(20~50건·동일 채점 원칙)

7-5. 부품 승격표 — 6강까지의 자산이 EDD의 무엇이 되는가

7-6. 승격 예제 — /pre-ship 커맨드를 CI 평가 게이트로

사용 방법: 3강 커맨드의 점검을 골든셋 기반 평가로 바꾸고 PR마다 자동 실행되게 한다. 이제 프롬프트·CLAUDE.md·스킬의 변경도 코드와 동일한 게이트를 통과해야 한다 — 1강의 "CLAUDE.md 변경도 결국 프롬프트 변경"이라는 복선의 회수.

# .github/workflows/eval-gate.yml
name: eval-gate
on: [pull_request]

jobs:
  evals:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt deepeval
      - run: deepeval test run tests/evals/   # 임계값 미달 시 exit 1 → 머지 차단
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

7-7. 방향성 로드맵 — EDD 성숙도 LV0→5 (요약)

레벨상태와 목격 현상다음 행동
LV0 바이브 체크채팅창에서 몇 개 돌려보고 배포 — 7-3 표의 "적용 전" 현상 전부실패 20~50건으로 골든셋 v1
LV1 골든셋+수동 채점"3번·7번 케이스가 깨졌다"는 대화가 처음 가능. 채점이 병목규칙화 가능한 항목부터 코드 grader 자동화
LV2 자동 채점실험 속도 급상승. "정확하지만 불친절한 답"이 여전히 샘루브릭 명문화 → Judge 도입 (인간 일치율 검증 먼저)
LV3 Judge+CI 게이트배포 전 검출 정착. Judge 편향의 자기강화 리스크 등장 (arXiv:2504.19076)Judge 재교정 프로세스화, 런타임 확장
LV4 런타임 가드레일드리프트가 지표로 잡힘. 단 "달았다≠막힌다" — 인젝션 공격 대상 가드레일 평균 탐지율 58.57%, 차단 성공 37.26% 미만 실측 (TRIAD)가드레일 자체를 공격셋으로 정기 평가
LV5 완전한 EDD 루프실패→분류→최초 결정 지점 수정→골든셋 환류가 규정으로 회전. Descript의 이중 스위트가 실례eval 스위트 자체의 포화 관리·갱신

레벨별 상세 실증·분석은 별도 문서 「EDD 완전 가이드 (10장)」 6장 참조.

7-8. 마지막 주의 — EDD에서도 하네스의 교훈은 유효하다

  • Judge는 5강 리뷰어와 같은 병을 앓는다: 채점하라면 무언가를 지적한다. LLM 판정이 개발과 평가 양쪽에 쓰이면 편향이 자기강화되어 거짓 성공을 가리킬 수 있다 — 루브릭은 좁고 검증 가능하게, 인간 일치율은 정기 재측정. (arXiv:2504.19076)
  • 점수도 증거가 필요하다: "85%→87%"는 표본 크기·오차 범위 없이는 판정 근거가 아니다 — eval은 노이즈 속의 측정 과학이다. (Anthropic, statistical approach)
  • 일찍 시작하되 완벽을 기다리지 마라: eval은 개발 초기에 기대 행동을 명시적으로 인코딩하는 데 특히 유용하고, Bolt처럼 늦게 시작해도 3개월이면 구축된다 — 그러나 그 사이의 반응적 루프 비용은 돌려받지 못한다. (Anthropic)
  • 문제는 모델에게 충분히 어려워야 한다: 프론티어 모델의 SWE-bench Verified가 1년 만에 40%→80%+로 뛰며 6개월 전 eval이 포화되는 시대 — 일괄 100%는 축하가 아니라 문항 교체 신호다. (Inkeep 해설)
시리즈 결론

CLAUDE.md 한 파일(1강)에서 시작한 여정의 종착지: 하네스는 에이전트가 "올바르게 일하게" 만들고, EDD는 그 일이 "얼마나 좋아지고 있는지"를 숫자로 관리한다. 순서는 정해져 있다 — 하네스 레벨을 건너뛴 EDD는 측정할 파이프라인이 없고, EDD 없는 완성형 하네스는 성장을 감에 맡긴다. 그리고 7-5의 승격표가 보여주듯, 성실하게 쌓은 하네스는 EDD의 부품을 이미 절반 이상 갖고 있다.

7-9. 체크리스트

  • review 판정에 루브릭·점수·임계값이 부여되어 있는가
  • PLAN.md류의 검증 목록이 골든셋으로 영속화되고 있는가
  • 프롬프트·CLAUDE.md·스킬 변경이 CI 평가 게이트를 통과하는가
  • Judge와 인간 라벨의 일치율을 측정한 적이 있는가
  • 운영 실패가 골든셋으로 환류되는 경로가 있는가
  • 일괄 100% 문항(포화)을 교체하는 갱신 주기가 있는가
