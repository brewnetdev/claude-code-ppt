목차 / 완성형 · 6강
Complete Harness · Converged Pattern

완성형 하네스 — research → plan → execute → review → ship

레벨 1~5의 부품이 갖춰지면 하나의 게이트 파이프라인으로 조립된다. 이 패턴은 누가 설계한 게 아니라 여러 방법론이 독립적으로 수렴한 결과이며, 그 수렴 자체가 가장 강한 실증이다.

6-1. 용어 정의

바이브 코딩 (Vibe Coding)
계획도 리뷰도 테스트도 없는 원샷 프롬프팅 — "설명하고 배포". 프로토타입에는 통하지만 프로덕션 규모에서 무너진다. 쉬운 비유: 설계도 없이 감으로 짓는 창고. 헛간까지는 서지만 3층부터 기운다.
에이전틱 엔지니어링 (Agentic Engineering)
research→plan→execute→review→ship 루프로 에이전트를 오케스트레이션하고, 사람이 타이피스트가 아니라 각 게이트의 감독자로 서는 방식. Karpathy 등이 채택하며 시대를 명명한 용어.
게이트 (Gate)
단계 전환마다 명시된 통과 조건. 승인(plan), 훅 녹색(execute), 갭 0건(review), 증거 확인(ship)처럼 조건이 명시되어야 게이트다 — "대충 봤고 괜찮아 보임"은 게이트가 아니다.
수직 슬라이스 / Tracer Bullet
DB+서비스+UI 등 전 레이어를 관통하는 얇은 조각 단위로 단계를 자르는 방식. AI는 기본적으로 수평 페이징(DB 단계 → API 단계 → 프런트 단계)을 택해 엔드투엔드 피드백이 마지막까지 지연되는데, 수직 슬라이스가 이를 교정한다.

용어 근거: mcp.directory(vibe coding·agentic engineering·게이트 프리미티브), shanraisshan(수직 슬라이스와 AI의 수평 페이징 기본값)

6-2. 실증 — "수렴"이라는 증거

이 5단계 패턴의 가장 강한 근거는 특정 회사의 주장이 아니라 독립 재발명의 수렴이다. Superpowers, gstack, BMAD-METHOD, OpenSpec, Spec Kit 등 서로 무관하게 만들어진 워크플로 방법론들이 각자 도달한 답이 동일했다: research → plan → execute → review → ship, 각 게이트에 사람의 감독. 2025년의 질문("AI가 동작하는 코드를 쓸 수 있는가")이 2026년의 질문("프로덕션 코드를 배포하도록 AI를 어떻게 오케스트레이션하는가")으로 바뀌면서, 답이 이 패턴으로 모인 것이다.

거시 지표도 이 전환을 뒷받침한다: 코딩 에이전트 활동이 있는 GitHub 프로젝트 비중은 2025년 말 이후 2배 이상이 됐고, Claude Code 사용자는 주당 평균 20시간을 이 도구에 쓴다 — "가끔 쓰는 보조"가 아니라 상시 파이프라인이 된 규모이며, 상시 파이프라인에는 구조(게이트)가 필요해진다.

근거: mcp.directory(방법론 수렴 목록·시대 전환), Anthropic Research(GitHub 2배·주당 20시간)

6-3. 이론 — "셋업 ↔ 주의"의 거래

공식 문서의 표현: "각 단계는 셋업을 지불하고 주의(attention)를 되사는 거래다." 프롬프트로 매번 지시하는 검증은 오늘 아무 과제에나 통하지만 사람의 주의를 계속 소비하고, /goal과 Stop 훅으로 구축한 검증은 셋업 비용이 들지만 무인 실행이 사람 없이 올바르게 끝나게 해준다. 완성형 하네스란 이 거래를 5단계 전부에서 완료한 상태다 — 매 단계의 감독이 "상시 주시"에서 "게이트 순간의 확인"으로 압축된다. 근거: 공식 Best Practices

주의 예산의 물리적 근거(참고): 커뮤니티 관찰로 컨텍스트 사용 ~40%부터 성능 저하 밴드가 시작되고, 1M 컨텍스트 모델도 ~300–400k 토큰 이후 품질이 떨어진다는 보고가 있다(공식 수치 아님). 단계 분리는 각 단계의 컨텍스트를 이 밴드 아래로 유지하는 효과도 갖는다. (mcp.directory 용어집)

6-4. 적용 전 vs 적용 후 — 목격되는 현상

바이브 코딩 (파이프라인 없음)완성형 하네스
수백 줄 구현 후 "이게 아닌데" — 잘못된 문제를 푼 코드가 주기적으로 발생.plan 게이트에서 방향 오류가 코드 0줄 시점에 걸러짐.
수평 페이징(DB→API→UI)으로 진행 — 엔드투엔드 동작 확인이 마지막 단계까지 지연.수직 슬라이스로 매 단계 끝에 관통 동작 확인 — 통합 리스크가 앞당겨 소진됨.
사람이 에이전트를 상시 주시 — "지켜보는 자동화"라는 모순.감독이 게이트 순간으로 압축 — 무인 실행 + 스팟 체크 체제.
완료 선언이 단언 기반 — 프로덕션에서 미구현 발견.review 갭 0건 + ship 증거 확인의 이중 게이트 통과분만 반영.

6-5. 단계별 통과 조건 — 완성형의 명세

단계산출물게이트 통과 조건동원 부품
research조사 요약 (한 문단~한 페이지)메인 컨텍스트가 원자료로 오염되지 않았는가서브에이전트(5강)
planPLAN.md — 요구사항·엣지 케이스·범위사람 승인. 계획이 검증 가능한 문장인가플랜 모드(2강)
executediff + 통과하는 테스트훅 전부 녹색 (포맷·빠른 테스트·차단기)훅(4강), 커맨드(3강)
review갭 목록 (또는 "갭 없음")정확성 갭 0건. 취향 지적은 게이트 아님검증 서브에이전트(5강)
ship증거 묶음 (테스트 출력·명령 반환값)사람이 증거를 확인. 단언만으로 통과 불가증거 원칙(5강), ask 권한(2강)

6-6. 예제 — 계획을 검증 가능한 문장으로 쓰는 법

사용 방법: plan의 산출물이 아래 형태여야 review가 기계적으로 대조할 수 있다 — 모호한 계획은 모호한 검증을 낳는다. 실무 보조 패턴 두 가지: (1) 최소 스펙에서 시작해 모델이 나를 인터뷰하게 만들어 모호성을 표면화한 뒤 새 세션에서 스펙을 실행시키기, (2) "불명확한 것은 시작 전에 무엇이든 물어라"를 비자명 과제의 기본 지시로 삼기 — 좋은 에이전트는 200줄을 잘못된 길로 쓰기 전에 모호성(어떤 토큰 라이브러리? 리프레시 토큰은?)을 먼저 드러낸다. 근거: shanraisshan(인터뷰 패턴), DEV Community(사전 질문 지시)

# PLAN.md (요구사항이 곧 검증 항목이 되도록 쓴다)

## 목표
API 레이트 리미터 추가 (IP당 분당 60회)

## 요구사항 (review 단계가 이 목록과 diff 를 대조한다)
- [ ] 60회 초과 시 429 + Retry-After 헤더 반환
- [ ] 한도는 env RATE_LIMIT_PER_MIN 으로 설정 가능 (기본 60)
- [ ] 화이트리스트 IP(설정값)는 제한 제외

## 엣지 케이스 (각각 테스트 필수)
- [ ] 정확히 60번째 요청은 통과, 61번째는 429
- [ ] 윈도우 경계(59초→61초)에서 카운터 리셋
- [ ] X-Forwarded-For 스푸핑 시 신뢰 프록시만 인정

## 범위 밖 (review 가 "범위 밖 변경"으로 잡아야 할 것)
- 인증 로직, 로깅 포맷, 기존 미들웨어 순서

6-7. 안티패턴 3가지

안티패턴왜 실패하는가교정
게이트 없는 5단계단계 이름만 있고 통과 조건이 없으면 바이브 코딩의 5막 연극일 뿐단계마다 조건을 문장으로 명시 (6-5 표)
수평 페이징 방치AI의 기본값 — 통합 리스크가 마지막 단계에 몰림plan에서 수직 슬라이스로 단계 재단
모호한 PLAN.mdreview가 대조할 기준이 없어 게이트가 형식화요구사항을 체크박스 문장으로, 범위 밖을 명시
완성형의 남은 구멍 — 7강의 존재 이유

이 파이프라인이 돌면 "지켜보지 않아도 올바르게 끝나는" 상태에 도달한다. 그러나 구멍이 하나 남는다: review의 판정이 여전히 정성적(갭 있음/없음)이고, 시간에 따른 품질 추이를 아무도 측정하지 않는다. "지난달보다 나아졌는가?"에 숫자로 답할 수 없는 하네스는, 완성형이어도 성장은 감에 의존한다. 이 구멍을 메우는 것이 7강 — EDD다.

6-8. 체크리스트

  • 5단계가 모두 존재하고, 단계 전환마다 명시적 통과 조건이 있는가
  • PLAN.md의 요구사항이 review가 기계적으로 대조 가능한 문장인가
  • 단계가 수직 슬라이스로 잘려 매 단계 끝에 관통 동작을 확인하는가
  • review→execute 되돌림 루프가 사람 개입 없이 도는가
  • ship이 단언이 아니라 증거로만 통과되는가
  • "지난달보다 나아졌는가"에 숫자로 답할 수 있는가 (없다면 → 7강)
