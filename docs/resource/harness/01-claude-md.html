Level 1 · Persistent Context

CLAUDE.md — 하네스 최고의 레버리지 포인트

단 하나의 마크다운 파일이 왜 하네스 전체에서 투자 대비 효과가 가장 큰가, 왜 대부분의 팀이 이 파일을 망치는가, 그리고 그것을 어떻게 측정 가능한 사실로 확인하는가.

1-1. 용어 정의

CLAUDE.md
프로젝트 루트(또는 홈 디렉토리)에 두는 마크다운 파일로, 매 세션 시작 시 자동 로드되는 영속 메모리. 프로젝트 컨벤션·명령어·영속 컨텍스트를 담는다. 쉬운 비유: 기억이 매일 리셋되는 신입의 책상에 붙여둔 "1페이지 온보딩 문서". 출근(세션 시작)할 때마다 반드시 읽고 시작한다.
컨텍스트 윈도우 (Context Window)
모델이 한 번에 "볼 수 있는" 텍스트 총량. 시스템 지시, CLAUDE.md, 대화, 읽은 파일, 도구 출력이 전부 이 예산을 나눠 쓴다. 쉬운 비유: 책상의 넓이. 자료를 올릴수록 정작 작업할 공간이 줄어든다.
메모리 계층 (Memory Hierarchy)
전역(사용자 홈) CLAUDE.md → 프로젝트 루트 CLAUDE.md → 하위 디렉토리 CLAUDE.md가 위치에 따라 자동 병합되는 구조. "전역 원칙 + 로컬 제약"을 만든다.
.claude/rules/*.md
paths: glob이 달린 규칙 파일. 해당 경로의 파일을 만질 때만 lazy 로드되어, "항상 로드"인 CLAUDE.md의 예산 문제를 우회한다.

용어 근거: 공식 Best Practices(CLAUDE.md 역할), Habib Mrad(디렉토리 병합), mcp.directory(rules + paths glob)

1-2. 이론 — 왜 이 파일이 "최고 레버리지"이면서 동시에 "최다 오남용"인가

공식 베스트 프랙티스의 대전제는 단 하나다: "대부분의 모범 사례는 '컨텍스트 윈도우는 빨리 차고, 차면 성능이 저하된다'는 단일 제약에 기반한다." CLAUDE.md는 이 제약의 양날이다. (공식)

  • 레버리지인 이유: 이 파일의 모든 줄은 앞으로의 모든 세션, 모든 프롬프트에 곱해진다. 컨벤션 한 줄을 잘 쓰면 수천 번의 반복 설명이 사라진다. 실무 정리들이 "레포에서 가장 레버리지가 높은 파일"이라 부르는 이유다.
  • 오남용되는 이유: 같은 곱셈이 비용에도 적용된다. 이 파일은 매 세션 컨텍스트 예산을 선점하며, 실제 프롬프트가 받을 주의(attention)와 경쟁한다. 스타일 가이드 전문을 밀어 넣은 400줄짜리 CLAUDE.md는 부분적으로 무시된다 — 길수록 오히려 덜 지켜지는 역설이 발생한다.

근거: DEV Community ("highest-leverage file... and the most commonly abused", 400줄 파일의 부분 무시 관찰, ~60줄 경험칙)

1-3. 적용 전 vs 적용 후 — 목격되는 현상

좋은 CLAUDE.md 없이 (또는 비대한 채로)간결한 CLAUDE.md 정착 후
세션마다 "우리는 pnpm 씁니다", "테스트는 이렇게 돌려요"를 반복 설명. 새 세션 = 신입 재온보딩.세션 시작 즉시 프로젝트 문법으로 대화 시작. 빌드·테스트 명령을 틀리는 일이 사라짐.
같은 유형의 실수(마이그레이션 파일 수정, CJS import)가 세션을 바꿔가며 재발.실수 → 규칙 1줄 추가 → 이후 전 세션에서 예방. 실패가 자산으로 전환되는 첫 경험.
400줄 규칙 파일인데도 규칙이 안 지켜져 "모델이 멍청하다"는 결론으로 빠짐.규칙 수가 줄었는데 준수율이 올라감 — 문제가 모델이 아니라 예산 배분이었음을 확인.
"완료"의 기준이 세션·사람마다 달라 결과물 품질이 널뜀."완료의 정의"가 파일에 있어 에이전트의 자기 판정 기준이 고정됨 — 5강 검증, 7강 평가의 씨앗.

이 표의 전/후 대조는 1-2절에 인용된 출처들의 관찰(반복 설명 제거, 부분 무시, 안정 규칙 원칙)을 현상 단위로 재구성한 것이다.

1-4. 실증 — 왜 이 파일의 품질이 결과를 좌우하는가: 40만 세션 데이터

Anthropic이 2025-10~2026-04의 약 40만 Claude Code 세션(약 23.5만 명)을 프라이버시 보존 방식으로 분석한 연구의 핵심 발견: 전형적 세션에서 사람이 계획 결정(what)의 대부분을, Claude가 실행 결정(how)의 대부분을 내린다. 그리고 사람이 도메인 전문성을 많이 가져올수록 지시 1건당 Claude가 해내는 작업량이 커지고, 성공은 코딩 훈련 여부가 아니라 풀려는 문제를 얼마나 이해하는가로 결정된다.

CLAUDE.md와의 연결: 이 파일은 바로 그 "도메인 전문성·문제 이해"를 세션에 주입하는 공식 통로다. 컨벤션·완료의 정의·건드리면 안 되는 것을 담은 CLAUDE.md는 모든 세션의 시작점에서 사람의 이해를 대리 전달하며, 연구가 보여준 "이해가 많을수록 에이전트 산출이 커진다"의 곱셈기를 상시화한다.

근거: Anthropic, How Claude Code is used in practice (40만 세션·what/how 분업·전문성 증폭·성공 결정 요인)

1-5. 무엇을 넣고, 무엇을 빼는가 — 계층 배치의 원칙

넣는다 (안정적 규칙)뺀다 (다른 계층으로)
빌드·테스트·실행 명령어 (pnpm test, make dev)스타일 가이드 전문 → 린터/포매터 + 훅(4강)이 강제
코드 컨벤션의 핵심 몇 줄 (예: "ESM 사용, CJS 금지")조건부로만 필요한 긴 지식 → 스킬의 점진적 공개(3강)
"완료의 정의(definition of done)"반드시 강제돼야 하는 것 → settings.json·훅(2·4강). 예: 커밋 어트리뷰션은 "NEVER 추가"라는 문장보다 attribution.commit: "" 설정 한 줄이 결정적
절대 건드리면 안 되는 것 (마이그레이션, 시크릿 경로)자주 바뀌는 임시 컨텍스트 → 세션 프롬프트로
특정 경로에서만 유효한 규칙 → 하위 CLAUDE.md 또는 rules/*.md의 paths glob—

근거: DEV Community(안정 규칙만), shanraisshan(결정적 처리 가능한 것을 문장으로 적지 말라, ≤200줄 규칙)

줄 수 기준은 공식 제한이 아니라 경험칙이다 — 한 정리는 ~60줄, 다른 정리는 ≤200줄. 공통 원칙: "짧게, 안정적인 것만."

1-6. 예제 — 좋은 CLAUDE.md 전문

# CLAUDE.md

## 명령어
- 개발 서버: `pnpm dev`
- 테스트: `pnpm test` (커밋 전 반드시 통과)
- 린트+포맷: `pnpm check`

## 컨벤션
- TypeScript strict. `any` 금지, 불가피하면 `// why-any:` 주석.
- 모듈은 ESM. CommonJS `require` 금지.
- API 핸들러는 src/api/ 에만. 비즈니스 로직은 src/core/ 로 분리.

## 완료의 정의
- 새 코드에는 테스트가 있다. `pnpm test` 녹색.
- 스키마 변경 시 prisma/migrations 에 마이그레이션 포함.

## 건드리지 말 것
- prisma/migrations/ 의 기존 파일 (새 파일 추가만 허용)
- .env* — 읽지도, 출력하지도 말 것

# .claude/rules/api-layer.md — 경로 조건부 규칙 (해당 경로 작업 시에만 로드)
---
paths: ["src/api/**"]
---
- 모든 핸들러는 zod 스키마로 입력 검증 후 core 호출만 한다.
- 응답 포맷은 { data, error } 봉투를 벗어나지 않는다.

사용 방법: 루트 CLAUDE.md는 git에 커밋해 팀 전체가 공유하고, 경로 종속 규칙은 rules로 분리해 예산을 아낀다. 디렉토리별 CLAUDE.md 자동 병합으로 "전역 원칙 + 로컬 제약" 계층을 만들 수 있다 — 시니어 엔지니어의 사고 구조와 같다. (Habib Mrad)

1-7. 운영 — 이 파일은 살아있는 문서다

  • 모델에게 개선을 시켜라: 커뮤니티 모범 사례 저장소는 "자기 프로젝트에서 Claude에게 팁 목록을 보여주고 CLAUDE.md 재구성을 제안하게 하라"를 명시적 워크플로로 권한다. (shanraisshan)
  • 자동 생성으로 시작할 수도 있다: 프로젝트 구조·기술 스택·개발 패턴을 분석해 CLAUDE.md를 생성하는 커스텀 커맨드(/custom-init)를 두는 팀도 있다 — 단, 생성물은 1-5절 기준으로 반드시 다이어트한다. (awattar)
  • 실수 2회 = 규칙 후보: 같은 실수가 두 번 나오면 교정 1줄이 이 파일 후보다. 반드시 일어나야 하는 수준이면 4강의 훅으로 승격한다.

1-8. 안티패턴 3가지

안티패턴왜 실패하는가교정
스타일 가이드 덤핑400줄 파일은 실제 프롬프트와 주의를 경쟁하다 부분 무시됨 — 규칙이 많을수록 덜 지켜짐린터가 잡을 수 있는 건 전부 린터+훅으로, 파일엔 핵심만
"NEVER ..." 문장 의존문장은 요청이지 보증이 아님(4강). 결정적으로 처리 가능한 것을 확률에 맡기는 셈settings.json / 훅으로 이관
임시 컨텍스트 축적"이번 스프린트는 X 중"류가 쌓여 파일이 낡고 비대해짐임시 사항은 세션 프롬프트로, 파일엔 안정 규칙만

1-9. 체크리스트

  • CLAUDE.md가 프로젝트 루트에 있고 git에 커밋되어 있는가
  • 200줄 미만이며 안정적 규칙만 담고 있는가
  • 빌드·테스트 명령어와 "완료의 정의"가 명시되어 있는가
  • 경로 종속 규칙이 rules/*.md의 paths glob으로 분리되어 있는가
  • "NEVER/반드시" 문장이 남아있지 않은가 (→ 2·4강 이관 후보)
  • 최근 1개월 내 실수→규칙 반영이 일어난 적 있는가 (살아있는 문서인가)
