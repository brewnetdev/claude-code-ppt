Level 2 · Programmable Safety

권한 모델과 settings.json — 프로그래머블 안전

"부탁"과 "강제"를 구분하는 것이 하네스 설계의 두 번째 계단이다. 보수적 권한 모델은 마찰이 아니라 프로그래밍 가능한 안전장치이며, 경계가 명시되는 순간 자율성은 오히려 커진다.

2-1. 용어 정의

권한 모델 (Permission Model)
에이전트의 각 도구 호출을 allow(무확인 실행) / ask(사람 확인 후 실행) / deny(무조건 차단)로 분류하는 규칙 체계. 쉬운 비유: 신입의 사원증 권한. 사무실 출입은 자유(allow), 서버실은 동행 필수(ask), 금고는 접근 불가(deny).
settings.json
.claude/settings.json에 두는 하네스 설정 파일. 권한 규칙, 모델 선택, 어트리뷰션 등 하네스가 결정적으로 강제하는 행동을 담으며, 레포에 체크인해 팀이 공유한다.
플랜 모드 (Plan Mode)
에이전트가 곧장 실행하지 않고 계획을 제안한 뒤 승인을 기다리는 모드. 권한 모델의 "시간 축" 버전 — 공간(어떤 도구)이 아니라 시점(실행 전)에 게이트를 세운다.
비가역 작업 (Irreversible Action)
실행 후 되돌리기 어렵거나 불가능한 작업 — 원격 push, 배포, 삭제, 외부 발송. 권한 분류의 1차 기준이 된다.
YOLO 모드
확인 프롬프트를 전부 건너뛰는 실행(--dangerously-skip-permissions). 학습·개인 프로젝트에서 유용하지만, 단계별 확인을 우회하므로 공유 레포·프로덕션 코드베이스에서는 확인을 유지해 감독을 지키라는 것이 실무 권고다. 근거: lowtouch.ai 셋업 가이드

2-2. 이론 — 왜 명시적 경계가 생산성을 "올리는가"

Claude Code의 보수적 권한 모델은 흔히 마찰로 오해되지만 실제로는 프로그래머블 안전이다. 핵심 논리는 역설적이다: 에이전트 생산성은 안전 경계가 암묵적으로 가정되는 대신 명시적으로 정의될 때 극적으로 증가한다. (Habib Mrad)

  • 경계가 없으면 사람은 두 나쁜 균형 중 하나를 고른다 — (a) 모든 호출을 확인하다 지쳐 에이전트를 안 쓰거나, (b) 겁나서 자율 실행 자체를 안 맡긴다.
  • 경계가 있으면 allow 영역 안에서는 확인 없이 전속력으로 달리고, 위험 신호만 사람에게 올라온다. 확인의 총량은 줄고 확인의 밀도(위험 대비 확인 비율)는 올라간다.
  • 시간 축의 같은 원리가 플랜 모드다: 곧장 코딩에 들어가게 두면 "잘못된 문제를 푸는 코드"가 나온다 — 리서치·계획을 구현에서 분리하고 계획 승인 후 실행하게 하라는 것이 공식 권고다. (공식 Best Practices)

2-3. 적용 전 vs 적용 후 — 목격되는 현상

권한 경계 명시 전명시 후
매 도구 호출마다 확인 팝업 → 확인 피로 → "그냥 다 승인"이 습관화되어 확인이 무의미해짐.안전 작업은 무확인 통과, 확인 요청은 진짜 위험 신호로만 — 확인 1건의 정보 가치가 회복됨.
반대 극단: 겁나서 자율 실행을 못 맡기고 에이전트를 "자동완성"으로만 사용.allow 영역 안에서 장시간 자율 실행을 맡길 수 있게 됨 — 40만 세션 연구가 보여준 "사람은 what, 에이전트는 how" 분업이 안정 작동. (Anthropic Research)
팀원마다 로컬 권한 설정이 달라, 같은 프롬프트가 사람마다 다른 위험 프로파일로 실행됨.체크인된 settings.json으로 팀 전원의 하네스가 동일 경계 공유 — 스코프 지정 allow/ask가 프로젝트 수준 설정의 가장 유용한 메커니즘으로 꼽히는 이유. (awattar)
계획 없이 곧장 구현 → 수백 줄을 쓴 뒤에야 방향이 틀렸음을 발견.플랜 모드로 탐색·계획을 분리, 승인된 계획만 실행 — "잘못된 문제를 푸는 코드"가 사전 차단. (공식)

2-4. 예제 — 프로젝트 settings.json

사용 방법: .claude/settings.json을 레포에 체크인한다. 이후 팀 누구의 세션이든 같은 경계 안에서 돈다.

// .claude/settings.json (예시 — 키 이름은 설치 버전 공식 문서로 확인)
{
  "permissions": {
    "allow": [
      "Bash(pnpm test:*)",       // 테스트는 자유롭게
      "Bash(pnpm check)",
      "Read(src/**)"
    ],
    "ask": [
      "Bash(git push:*)",        // 원격 반영은 사람 확인
      "Bash(pnpm add:*)"         // 의존성 추가도 확인
    ],
    "deny": [
      "Read(.env*)",             // 시크릿 접근 원천 차단
      "Bash(rm -rf:*)"
    ]
  }
}

설정 스키마는 버전에 따라 다를 수 있으므로 반드시 설치 버전 공식 문서와 대조. 핵심은 스키마 암기가 아니라 "부탁이 아닌 규칙으로 경계를 긋는다"는 원칙이다.

2-5. 판별법 — "결정적으로 처리 가능한 것"을 찾아내기

커뮤니티 팁 모음의 원칙: 하네스가 강제할 수 있는 행동(어트리뷰션·권한·모델 선택)은 settings.json으로 처리하라 — 커밋 어트리뷰션을 CLAUDE.md에 "NEVER add Co-Authored-By"라고 적는 것보다 attribution.commit: "" 설정이 결정적이다. (shanraisshan)

판별 질문 2개 (순서대로)

Q1. 이 규칙이 어겨지면 곤란한가, 아쉬운가? — 아쉬운 정도면 CLAUDE.md(요청)로 충분. 곤란하면 Q2로.

Q2. 하네스 설정으로 표현 가능한가? — 가능(권한·어트리뷰션·모델)하면 settings.json. 불가능(임의 로직 필요)하면 4강의 훅.

2-6. 실증 — 분업 데이터가 말해주는 것

40만 세션 분석에서 확인된 분업 — 사람이 계획 결정의 대부분, 에이전트가 실행 결정의 대부분 — 은 권한 설계의 데이터 기반 청사진이다: 사람이 유지해야 할 결정(what: 무엇을 만들지, 무엇을 원격에 반영할지)은 ask 이상으로, 에이전트에게 넘어간 결정(how: 어떤 파일을 읽고 어떤 테스트를 돌릴지)은 allow로 배치하는 것이 데이터가 보여주는 자연 분업과 일치한다. 연구는 또한 도메인 전문가가 오류·오해에서 더 잘 회복함을 보여주는데, ask 게이트는 바로 그 "사람의 회복 개입 지점"을 제도화한 것이다.

근거: Anthropic, How Claude Code is used in practice

2-7. 안티패턴 3가지

안티패턴왜 실패하는가교정
전부 ask확인 피로 → 무지성 승인 → 확인의 정보 가치 0안전 작업을 적극적으로 allow로 내려 확인 밀도를 회복
공유 레포에서 YOLO 상시화단계별 감독이 사라져 비가역 실수가 그대로 반영됨YOLO는 학습·개인 프로젝트 한정, 공유 코드베이스는 확인 유지 (lowtouch.ai)
개인 로컬 설정 방치팀원마다 다른 위험 프로파일 — 사고가 "누구 세션이었냐"에 좌우settings.json 체크인으로 경계를 코드처럼 리뷰

2-8. 체크리스트

  • .claude/settings.json이 레포에 체크인되어 팀이 공유하는가
  • 시크릿(.env 등) 접근이 deny로 원천 차단되어 있는가
  • 비가역 작업(push·배포·삭제)이 ask 이상으로 분류되어 있는가
  • "곤란한 규칙"이 CLAUDE.md 문장이 아니라 설정으로 존재하는가 (판별 질문 2개 통과)
  • 비자명한 작업에서 플랜 모드(계획 승인 → 실행)가 기본인가
  • YOLO 모드 사용 범위가 개인·학습 프로젝트로 한정되어 있는가
