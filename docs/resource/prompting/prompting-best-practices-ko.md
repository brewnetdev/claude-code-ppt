# 프롬프트 엔지니어링

## 프롬프팅 베스트 프랙티스

> 명확성, 예시, XML 구조화, 사고(thinking), 에이전트 시스템을 다루는 Claude 최신 모델용 프롬프트 엔지니어링 기법의 종합 가이드.

이 문서는 Claude Opus 4.7, Claude Opus 4.6, Claude Sonnet 4.6, Claude Haiku 4.5를 비롯한 Claude 최신 모델의 프롬프트 엔지니어링에 관한 단일 참고서입니다. 기본 기법, 출력 제어, 도구 사용, 사고(thinking), 에이전트 시스템을 다룹니다. 본인의 상황에 맞는 섹션으로 바로 이동하셔도 됩니다.

모델 기능 개요는 모델 개요(models overview) 문서를 참고하세요. Claude Opus 4.7의 새로운 사항은 "Claude Opus 4.7의 새로운 기능(What's new in Claude Opus 4.7)" 문서를 참고하세요. 마이그레이션 안내는 마이그레이션 가이드(Migration guide)를 참고하세요.

## Claude Opus 4.7 프롬프팅

Claude Opus 4.7은 현재 일반 사용 가능한 모델 중 가장 강력한 모델로, 특히 장기 에이전트 작업(long-horizon agentic work), 지식 작업, 비전, 메모리 관련 작업에서 강점을 보입니다. 기존 Claude Opus 4.6용 프롬프트에서도 별다른 조정 없이 좋은 성능을 냅니다. 아래에 정리한 패턴들은 그중에서도 가장 자주 튜닝이 필요한 동작들을 다룹니다.

Claude Opus 4.6에서 마이그레이션할 때 변경되는 API 파라미터(effort 레벨, task 예산, thinking 설정, sampling 파라미터 제거, 토큰화 등)는 마이그레이션 가이드를 참고하세요.

### 응답 길이와 장황함(verbosity)

Claude Opus 4.7은 고정된 장황함을 기본값으로 두지 않고, 작업의 복잡도를 스스로 판단해 응답 길이를 조정합니다. 즉, 단순 조회 같은 질문에는 짧게, 개방형 분석에는 훨씬 길게 답하는 경향이 강합니다.

만약 제품이 특정 스타일이나 분량의 출력에 의존한다면, 프롬프트를 튜닝해야 할 수 있습니다. 예를 들어 장황함을 줄이려면 다음과 같이 추가할 수 있습니다:

> 간결하고 핵심에 집중한 응답을 제공해주세요. 부가적인 맥락은 생략하고, 예시는 최소한으로 유지해주세요.

지나친 설명 같은 장황함의 구체적 사례가 보이면, 그것들을 막기 위한 지시를 프롬프트에 추가할 수 있습니다. 일반적으로 "하지 말라"는 부정 예시나 금지 지시보다, Claude가 적정 수준의 간결함으로 어떻게 소통할 수 있는지를 보여주는 긍정 예시가 더 효과적입니다.

### 노력(effort)과 사고 깊이(thinking depth) 조정

`effort` 파라미터를 사용하면 Claude의 지능과 토큰 소비량의 균형을 조절할 수 있어, 더 빠른 속도와 낮은 비용을 위해 능력을 일부 양보하는 트레이드오프가 가능합니다. 코딩 및 에이전트 사용 사례에서는 새로 추가된 `xhigh` effort 레벨부터 시작하고, 지능에 민감한 대부분의 사용 사례에서는 최소 `high` effort를 권장합니다. 다른 effort 레벨도 실험해보면서 토큰 사용량과 지능을 더 세밀하게 튜닝하세요:

- **max**: 최대 effort. 일부 사용 사례에서 성능 향상을 가져올 수 있지만, 토큰 사용 증가에 비해 한계 수익이 작아질 수 있습니다. 또한 과잉 사고(overthinking)에 빠지기 쉽습니다. 지능이 가장 중요한 작업에서 max effort를 테스트해보길 권장합니다.
- **xhigh (신규)**: 극히 높은 effort. 대부분의 코딩 및 에이전트 사용 사례에 가장 적합한 설정입니다.
- **high**: 토큰 사용량과 지능의 균형을 잡습니다. 지능에 민감한 대부분의 사용 사례에 대해 최소 high effort를 권장합니다.
- **medium**: 토큰 사용량을 줄이면서 지능 손실을 일부 감수할 수 있는 비용 민감 사용 사례에 적합합니다.
- **low**: 짧고 범위가 좁은 작업, 지능에 민감하지 않은 지연 시간 민감 워크로드에 한정해 사용하세요.

Claude Opus 4.6과 비교했을 때 의미 있게 달라진 점은, Claude Opus 4.7이 특히 낮은 쪽 effort 레벨에서 effort 설정을 엄격히 지킨다는 것입니다. low와 medium에서는 모델이 요청 범위에 작업을 한정하며 그 이상으로 나아가지 않습니다. 지연 시간과 비용 측면에서는 좋지만, 약간 복잡한 작업을 low effort로 돌릴 경우 사고가 부족(under-thinking)할 위험이 있습니다.

복잡한 문제에서 얕은 추론이 관찰되면, 프롬프트로 우회하기보다는 effort를 high나 xhigh로 올리세요. 지연 시간 때문에 effort를 low로 유지해야 한다면, 다음과 같은 표적 지침을 추가하세요:

> 이 작업은 다단계 추론(multi-step reasoning)을 요합니다. 응답하기 전에 문제를 깊이 생각해주세요.

이 모델에서는 그 어떤 이전 Opus 모델보다 effort가 더 중요할 것으로 예상되므로, 업그레이드 시 적극적으로 실험해볼 것을 권장합니다.

적응형 사고(adaptive thinking)의 트리거 동작은 조향(steer) 가능합니다. 시스템 프롬프트가 크거나 복잡한 경우, 원하는 것보다 모델이 더 자주 사고하는 경우가 생길 수 있는데, 그럴 때는 다음과 같은 지침을 추가해 조향하세요. 늘 그렇듯 프롬프트 변경의 효과는 성능으로 측정하세요. 예시:

> 사고(Thinking)는 지연을 더하기 때문에 답변 품질을 의미 있게 개선할 때만 사용해야 합니다 — 일반적으로 다단계 추론이 필요한 문제에 해당합니다. 확신이 없으면 그냥 바로 답하세요.

반대로, medium에서 어려운 작업을 돌리는데 사고가 부족하게 느껴진다면 가장 먼저 effort를 올리는 것이 첫 번째 수단입니다. 더 정밀한 제어가 필요하다면 프롬프트로 직접 요청하세요.

Claude Opus 4.7을 max나 xhigh effort로 돌리는 경우, 모델이 서브에이전트와 도구 호출 전반에 걸쳐 충분히 사고하고 행동할 공간을 갖도록 큰 max output 토큰 예산을 설정하세요. 64k 토큰부터 시작해서 거기서부터 튜닝할 것을 권장합니다.

### 도구 사용 트리거링

Claude Opus 4.7은 Claude Opus 4.6보다 도구를 덜 사용하고 추론을 더 많이 하는 경향이 있습니다. 대부분의 경우 더 좋은 결과를 냅니다. 다만, effort 설정을 올리는 것이 도구 사용 빈도를 늘리는 유용한 지렛대인데, 특히 지식 작업에서 그렇습니다. high나 xhigh effort 설정에서는 에이전트 검색과 코딩에서 실질적으로 더 많은 도구 사용을 보입니다. 도구를 더 자주 쓰게 만들고 싶은 시나리오에서는 프롬프트를 조정해 모델에게 도구를 언제, 어떻게 적절히 사용해야 하는지 명시적으로 지시할 수도 있습니다. 예컨대 웹 검색 도구를 안 쓰고 있다면, 왜 그리고 어떻게 써야 하는지 명확히 설명해주세요.

### 사용자에게 보여주는 진행 상황 업데이트

Claude Opus 4.7은 긴 에이전트 추적(long agentic traces) 전반에 걸쳐 사용자에게 더 정기적이고 높은 품질의 업데이트를 제공합니다. "도구 호출 3번마다 진행 상황을 요약하세요" 같은 식으로 중간 상태 메시지를 강제하던 스캐폴딩을 추가해두었다면, 제거해보세요. Claude Opus 4.7의 사용자 대상 업데이트의 길이나 내용이 사용 사례에 잘 맞지 않으면, 이 업데이트가 어떤 모습이어야 하는지 프롬프트에 명시적으로 설명하고 예시를 제공하세요.

### 보다 문자 그대로의 지시 준수

Claude Opus 4.7은 Claude Opus 4.6보다 더 문자 그대로, 그리고 더 명시적으로 프롬프트를 해석합니다. 특히 낮은 effort 레벨에서 그렇습니다. 한 항목에서의 지시를 다른 항목으로 조용히 일반화하지 않으며, 사용자가 하지 않은 요청을 추론해서 처리하지도 않습니다. 이런 문자 그대로의 해석에는 정밀성과 적은 시행착오라는 장점이 있고, 일반적으로 세심하게 튜닝된 프롬프트를 사용하는 API 사용 사례, 정형 데이터 추출, 예측 가능한 동작을 원하는 파이프라인에서 더 좋은 성능을 냅니다. Claude가 지시를 폭넓게 적용해야 한다면, 범위를 명시적으로 지정하세요(예: "이 포맷팅을 첫 섹션뿐 아니라 모든 섹션에 적용하세요").

### 톤과 문체

새 모델이 늘 그렇듯, 장문 글쓰기에서 산문 스타일이 달라질 수 있습니다. Claude Opus 4.7은 더 직접적이고 의견을 분명히 밝히는 편이며, Claude Opus 4.6의 따뜻한 스타일에 비해 검증을 앞세우는 표현이 적고 이모지 사용도 적습니다. 제품이 특정 목소리에 의존한다면, 새 기준에 맞춰 스타일 프롬프트를 재평가하세요.

예를 들어 제품 목소리가 더 따뜻하거나 대화체에 가깝다면, 다음과 같이 추가하세요:

> 따뜻하고 협력적인 톤을 사용하세요. 답변하기 전에 사용자의 표현을 먼저 인정해주세요.

### 서브에이전트 스폰(subagent spawning) 제어

Claude Opus 4.7은 기본적으로 더 적은 수의 서브에이전트를 스폰하는 경향이 있습니다. 그러나 이 동작은 프롬프팅으로 조향 가능합니다. 서브에이전트가 바람직한 상황에 대해 Claude Opus 4.7에 명시적으로 지침을 주세요. 코딩 사용 사례를 위한 간단한 예시:

> 한 번의 응답으로 직접 끝낼 수 있는 작업(예: 이미 보이는 함수를 리팩토링하는 일)을 위해 서브에이전트를 스폰하지 마세요.
>
> 여러 항목에 걸쳐 분기 처리(fanning out)하거나 여러 파일을 읽을 때는, 한 턴에 여러 서브에이전트를 동시에 스폰하세요.

### 디자인과 프론트엔드 기본값

Claude Opus 4.7은 Claude Opus 4.6보다 디자인 본능이 강하며, 일관된 기본 하우스 스타일(house style)을 갖습니다: 따뜻한 크림/오프화이트 배경(~#F4F1EA), 세리프 디스플레이 활자(Georgia, Fraunces, Playfair), 이탤릭으로 강조된 단어, 그리고 테라코타/앰버 색상의 액센트입니다. 이는 에디토리얼, 호스피탈리티, 포트폴리오 브리프에는 잘 어울리지만, 대시보드, 개발자 도구, 핀테크, 헬스케어, 엔터프라이즈 앱에는 부자연스럽게 느껴집니다 — 그리고 이 스타일은 웹 UI뿐 아니라 슬라이드 덱에서도 나타납니다.

이 기본 스타일은 끈질깁니다. "크림색 쓰지 마", "깔끔하고 미니멀하게" 같은 일반적인 지시는 다양성을 만들어내기보다 다른 고정 팔레트로 모델을 이동시키는 경향이 있습니다. 다음 두 가지 접근이 안정적으로 작동합니다:

**1. 구체적인 대안을 명시하기.** 모델은 명시적인 스펙을 정확히 따릅니다:

> AEFRM이라는 보충제 브랜드를 위한 데스크톱 랜딩 페이지를 디자인해주세요.
>
> 비주얼 방향성은 옅은 실버 그레이 톤이 점차 블루 그레이와 거의 검정에 가까운 톤으로 깊어지는, 안개 낀 금속 표면 같은 차가운 모노크롬 분위기에서 가져와야 합니다.
>
> 페이지는 날카롭고 절제된 느낌이어야 하며, 구조와 절제의 인상이 강해야 합니다.
>
> 이 톤 시스템을 페이지 전체에 걸쳐 사용하고, 밝은 액센트 색상은 넣지 마세요.
>
> 업로드된 이미지를 흑백으로 히어로 디자인에 사용하세요.
>
> 레이아웃은 명확한 수평 섹션과 가운데 정렬된 max-width 컨테이너로 구성하세요. 카드, 버튼, 입력란, 미디어 프레임 전반에 일관되게 4px 모서리 반경을 사용하세요. 여백은 넉넉하게 잡고, 각 섹션 주변에 충분한 빈 공간을 두어 페이지에 숨통을 트이게 하세요.
>
> 타이포그래피는 평소보다 자간이 넓은 정사각형의 각진 산세리프를 사용하세요. 특히 헤딩과 내비게이션에서 자간을 넓혀 텍스트가 압축된 느낌이 아니라 정밀하게 설계된 느낌을 주도록 하세요. 헤드라인 텍스트는 크고 대문자로 표현해도 좋고, 보조 문구는 짧고 드물게 사용하세요. 보조 텍스트는 Alumni Sans SC 폰트로 4–6px 크기의 아주 작은 글자처럼 모서리 아래쪽 가운데 등에 배치하세요.
>
> 구조는 강한 제품 진술 한 줄과 짧은 보조 문단 하나, 그리고 깨끗한 제품 플레이스홀더 혹은 패크샷 프레임을 담은 히어로 섹션으로 시작하세요. 그 아래 3~4개 블록으로 구성된 베네핏 그리드를 두고, 그 다음에 제형 또는 성분 섹션을 배치하고, 마지막으로 CTA로 마무리하세요.
>
> 버튼은 평면적이고 정밀해야 하며, `transition: all 160ms ease out`을 이용해 미묘한 호버 변화를 줍니다. 극적인 모션 대신 밝기와 테두리 대비가 살짝 바뀌는 방식이 좋습니다.
>
> 색상 팔레트는 다음 범위 내에 머물러야 합니다:
> #E9ECEC, #C9D2D4, #8C9A9E, #44545B, #11171B.

**2. 빌드 전에 옵션을 제안하도록 시키기.** 이는 기본 스타일에서 벗어나게 하고 사용자에게 통제권을 줍니다. 이전에 디자인 다양성을 위해 temperature에 의존했다면, 이 방법을 사용하세요 — 실행 간에 의미 있게 다른 방향성을 만들어줍니다. 예시 프롬프트:

> 빌드 전에, 이 브리프에 맞춘 4가지 별개의 비주얼 방향성을 제안해주세요(각각 다음 형식으로: 배경 hex / 액센트 hex / 폰트 — 한 줄 근거). 사용자가 하나를 고르도록 요청한 뒤, 선택된 방향만 구현하세요.

또한, Claude Opus 4.7은 사용자들이 "AI 슬롭(AI slop)" 미학이라 부르는 일반적인 패턴을 피하기 위한 프론트엔드 디자인 프롬프팅을 이전 모델보다 덜 필요로 합니다. 이전 모델에서는 우리의 frontend-design 스킬에 더 긴 프롬프트 스니펫을 권장했지만, Claude Opus 4.7은 더 적은 프롬프트 안내만으로도 독창적이고 창의적인 프론트엔드를 생성합니다. 다음 프롬프트 스니펫은 위 다양성 관련 프롬프트 조언과 함께 잘 작동합니다:

```
<frontend_aesthetics>
NEVER use generic AI-generated aesthetics like overused font families
(Inter, Roboto, Arial, system fonts), cliched color schemes
(particularly purple gradients on white or dark backgrounds),
predictable layouts and component patterns, and cookie-cutter design
that lacks context-specific character. Use unique fonts, cohesive
colors and themes, and animations for effects and micro-interactions.
</frontend_aesthetics>
```

### 인터랙티브 코딩 제품

Claude Opus 4.7의 토큰 사용량과 동작은 단일 사용자 턴(single user turn)으로 작동하는 자율적·비동기 코딩 에이전트와, 여러 사용자 턴(multiple user turns)으로 진행되는 인터랙티브·동기 코딩 에이전트 사이에서 다를 수 있습니다. 구체적으로, 인터랙티브 환경에서 더 많은 토큰을 쓰는 경향이 있는데, 주된 이유는 사용자 턴 이후에 추론을 더 많이 하기 때문입니다. 이는 길고 인터랙티브한 코딩 세션에서 장기 일관성, 지시 준수, 코딩 능력을 향상시킬 수 있지만 토큰 사용도 함께 늘어납니다. 코딩 제품에서 성능과 토큰 효율 모두를 극대화하려면, xhigh나 high effort를 사용하고, auto 모드 같은 자율 기능을 추가하고, 사용자에게 요구되는 인간 상호작용 횟수를 줄이는 것을 권장합니다.

물론, 사용자 상호작용 횟수를 제한할 때는 첫 사용자 턴에서 작업, 의도, 관련 제약을 미리 명시하는 것이 중요합니다. 잘 정의되고 명확하고 정확한 작업 설명을 미리 제공하면 자율성과 지능을 극대화하면서 사용자 턴 이후의 불필요한 토큰 사용을 최소화할 수 있습니다. Claude Opus 4.7이 이전 모델보다 더 자율적이기 때문에, 이런 사용 패턴이 성능을 극대화하는 데 도움이 됩니다. 반대로 여러 사용자 턴에 걸쳐 점진적으로 전달되는 모호하거나 불완전하게 명시된 프롬프트는 토큰 효율, 그리고 때로는 성능까지 상대적으로 떨어뜨립니다.

### 코드 리뷰 하네스(harness)

Claude Opus 4.7은 이전 모델들보다 버그를 의미 있게 더 잘 찾으며, 우리 평가에서는 재현율(recall)과 정밀도(precision) 모두 더 높습니다 — Anthropic 실제 PR 기반의 가장 어려운 버그 찾기 평가 중 하나에서 재현율이 11pp 더 높습니다. 그러나, 코드 리뷰 하네스가 이전 모델용으로 튜닝되었다면 처음에는 재현율이 더 낮아 보일 수 있습니다. 이는 능력 회귀가 아니라 하네스 효과일 가능성이 큽니다. 리뷰 프롬프트에 "high-severity 이슈만 보고하세요", "보수적으로 하세요", "잡다하게 지적하지 마세요" 같은 표현이 있을 때, Claude Opus 4.7은 이전 모델보다 그 지시를 더 충실히 따를 수 있습니다 — 코드를 똑같이 깊이 조사하고 버그를 찾아내지만, 본인이 판단하기에 사용자가 설정한 기준 아래의 발견 사항은 보고하지 않을 수 있습니다. 이 경우 모델이 같은 깊이로 조사하면서도 그 조사 중 더 적은 비율만 보고로 전환하는 모습이 나타날 수 있으며, 특히 낮은 심각도의 버그에서 그렇습니다. 정밀도는 보통 올라가지만, 모델의 근본적인 버그 찾기 능력은 향상되었음에도 측정된 재현율은 떨어질 수 있습니다.

권장 프롬프트 표현 예:

> 발견한 모든 이슈를 보고하세요. 확신이 없거나 심각도가 낮다고 판단되는 것까지 포함해서요. 이 단계에서는 중요도나 확신도로 필터링하지 마세요 — 별도의 검증 단계가 그 역할을 합니다. 여기서의 목표는 커버리지(coverage)입니다. 나중에 필터링될지언정 일단 표면화하는 편이, 실제 버그를 조용히 놓치는 것보다 낫습니다. 각 발견 사항에 대해 본인의 확신도와 추정 심각도를 포함시켜, 다음 단계의 필터가 순위를 매길 수 있게 하세요.

이 프롬프트는 실제 두 번째 단계가 없더라도 사용할 수 있지만, 확신도 필터링을 발견 단계에서 분리하는 것 자체가 도움이 되는 경우가 많습니다. 하네스에 별도의 검증, 중복 제거, 순위 매기기 단계가 있다면, 발견 단계에서 모델의 역할은 필터링이 아니라 커버리지라고 명시적으로 알려주세요.

만약 단일 패스에서 모델이 스스로 필터링하길 원한다면, "중요한" 같은 정성적 표현 대신 기준이 어디에 있는지 구체적으로 알려주세요 — 예: "잘못된 동작, 테스트 실패, 오해를 부르는 결과를 유발할 수 있는 모든 버그를 보고하세요. 순수한 스타일이나 네이밍 선호 같은 사소한 지적만 생략하세요."

평가 또는 테스트 케이스의 일부 집합에 대해 프롬프트를 반복 개선하면서 재현율 또는 F1 점수 향상을 검증할 것을 권장합니다.

### 컴퓨터 사용(Computer use)

컴퓨터 사용 기능은 다양한 해상도에서 작동하며, 새로운 최대 해상도는 2576px / 3.75MP입니다. 컴퓨터 사용 테스트에서 1080p로 이미지를 보내는 것이 성능과 비용의 균형이 좋다는 결과를 얻었습니다.

특히 비용에 민감한 워크로드의 경우, 720p 또는 1366×768이 강력한 성능을 가진 저비용 옵션으로 권장됩니다. 본인의 사용 사례에 맞는 이상적인 설정을 찾으려면 직접 테스트하는 것이 좋습니다. effort 설정을 실험하는 것도 모델 동작을 튜닝하는 데 도움이 됩니다.

## 일반 원칙

### 명확하고 직접적으로

Claude는 명확하고 명시적인 지시에 잘 반응합니다. 원하는 출력에 대해 구체적으로 지시할수록 더 나은 결과를 얻습니다. "기대 이상의(above and beyond)" 동작을 원한다면, 모호한 프롬프트에서 모델이 알아서 추론하기를 기대하기보다 명시적으로 요청하세요.

Claude를 똑똑하지만 회사의 관행과 워크플로우에 대한 맥락은 없는 신입사원이라고 생각하세요. 무엇을 원하는지 더 정밀하게 설명할수록 결과가 좋아집니다.

**황금 규칙**: 작업에 대한 맥락이 거의 없는 동료에게 프롬프트를 보여주고 그대로 따라달라고 해보세요. 그 동료가 헷갈려한다면, Claude도 마찬가지입니다.

- 원하는 출력 형식과 제약을 구체적으로 지정하세요.
- 단계의 순서나 완전성이 중요할 때는 번호 매김 목록이나 글머리표를 써서 순차적 단계로 지시하세요.

**예: 분석 대시보드 만들기**

### 맥락을 더해서 성능을 높이세요

지시 뒤에 맥락이나 동기를 함께 제공하면(예: 왜 그런 동작이 중요한지 설명) Claude가 목표를 더 잘 이해해 보다 표적이 맞는 응답을 만들 수 있습니다.

**예: 포맷팅 선호**

Claude는 설명으로부터 일반화할 만큼 충분히 똑똑합니다.

### 예시(example)를 효과적으로 사용하세요

예시는 Claude의 출력 형식, 톤, 구조를 가장 안정적으로 조향하는 방법 중 하나입니다. 잘 만들어진 예시 몇 개(few-shot 또는 multishot 프롬프팅으로 불립니다)만으로도 정확도와 일관성을 크게 끌어올릴 수 있습니다.

예시를 추가할 때는 다음을 지키세요:

- **관련성(Relevant)**: 실제 사용 사례와 가깝게 만드세요.
- **다양성(Diverse)**: 엣지 케이스를 포함하고, Claude가 의도치 않은 패턴을 학습하지 않을 만큼 충분히 다양화하세요.
- **구조화(Structured)**: 예시는 `<example>` 태그로 감싸고(여러 예시는 `<examples>` 태그로) Claude가 지시와 구분할 수 있도록 하세요.
- 최상의 결과를 위해 3~5개의 예시를 포함하세요. 또한 Claude에게 직접 본인의 예시를 관련성과 다양성 측면에서 평가해달라고 하거나, 초기 세트를 바탕으로 추가 예시를 생성해달라고 요청할 수도 있습니다.

### XML 태그로 프롬프트를 구조화하세요

XML 태그는 Claude가 복잡한 프롬프트를 모호하지 않게 해석하도록 도와줍니다. 특히 프롬프트에 지시, 맥락, 예시, 변수 입력이 섞여 있을 때 그렇습니다. 각 콘텐츠 유형을 별도 태그(예: `<instructions>`, `<context>`, `<input>`)로 감싸면 오해석을 줄일 수 있습니다.

베스트 프랙티스:

- 프롬프트 전반에서 일관되고 설명적인 태그 이름을 사용하세요.
- 콘텐츠에 자연스러운 계층이 있을 때는 태그를 중첩하세요(여러 문서는 `<documents>` 안에, 각 문서는 `<document index="n">` 안에).

### Claude에게 역할(role)을 부여하세요

시스템 프롬프트에서 역할을 설정하면 Claude의 동작과 톤이 사용 사례에 맞춰 집중됩니다. 한 문장만으로도 차이가 납니다:

```python
import anthropic

client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    system="You are a helpful coding assistant specializing in Python.",
    messages=[
        {"role": "user", "content": "How do I sort a list of dictionaries by key?"}
    ],
)
print(message.content)
```

### 긴 맥락(long context) 프롬프팅

대용량 문서나 데이터 비중이 큰 입력(20k+ 토큰)을 다룰 때는 결과를 극대화하기 위해 프롬프트를 신중하게 구조화하세요:

**장문 데이터는 위에 두세요**: 긴 문서와 입력을 프롬프트 상단, 즉 질문·지시·예시 위쪽에 배치하세요. 이는 모든 모델의 성능을 크게 개선할 수 있습니다.

쿼리를 마지막에 두는 방식은, 특히 복잡한 다중 문서 입력에서 응답 품질을 최대 30%까지 개선할 수 있다는 테스트 결과가 있습니다.

**문서 콘텐츠와 메타데이터를 XML 태그로 구조화하세요**: 여러 문서를 사용할 때는 각 문서를 `<document>` 태그로 감싸고 그 안에 `<document_content>`와 `<source>`(그리고 기타 메타데이터) 하위 태그를 두어 명확성을 확보하세요.

**예: 다중 문서 구조**

**응답을 인용에 근거하게 하세요(Ground responses in quotes)**: 긴 문서 작업에서는, Claude에게 작업을 수행하기 전에 문서의 관련 부분을 먼저 인용하도록 요청하세요. 이는 문서 나머지의 잡음을 가르고 핵심에 닿는 데 도움이 됩니다.

**예: 인용 추출**

### 모델의 자기 인식(self-knowledge)

Claude가 본인의 애플리케이션에서 자신을 올바르게 식별하도록 하거나 특정 API 문자열을 사용하도록 하려면:

**모델 정체성 샘플 프롬프트**

모델 문자열을 명시해야 하는 LLM 기반 앱의 경우:

**모델 문자열 샘플 프롬프트**

> LLM이 필요할 때는, 사용자가 다른 모델을 요청하지 않는 한 Claude Opus 4.7을 기본값으로 사용하세요. Claude Opus 4.7의 정확한 모델 문자열은 `claude-opus-4-7` 입니다.

## 출력과 포맷팅

### 의사소통 스타일과 장황함

Claude 최신 모델은 이전 모델 대비 더 간결하고 자연스러운 의사소통 스타일을 가집니다:

- **더 직접적이고 근거에 기반함**: 자기 칭찬식 업데이트가 아니라 사실 기반의 진행 상황 보고를 제공합니다.
- **더 대화체에 가까움**: 약간 더 자연스럽고 구어적이며, 기계적인 느낌이 덜합니다.
- **덜 장황함**: 효율성을 위해 별도 지시가 없으면 상세 요약을 건너뛸 수 있습니다.

이는 Claude가 도구 호출 후 구두 요약을 건너뛰고 곧장 다음 액션으로 넘어갈 수 있다는 뜻입니다. 추론에 대한 가시성을 더 원한다면:

**샘플 프롬프트**

> 도구 사용을 포함하는 작업을 완료한 후에는, 수행한 작업에 대한 간단한 요약을 제공해주세요.

### 응답의 포맷을 제어하세요

출력 포맷을 효과적으로 조향하는 몇 가지 방법:

**하지 말아야 할 것 대신, 해야 할 것을 말하세요**

- "응답에 마크다운을 사용하지 마세요." 보다는,
- "응답은 부드럽게 이어지는 산문 단락으로 구성해주세요."

**XML 포맷 지시자(indicator)를 사용하세요**

- "응답의 산문 부분을 `<smoothly_flowing_prose_paragraphs>` 태그 안에 작성해주세요."

**프롬프트 스타일을 원하는 출력 스타일에 맞추세요**

프롬프트에서 사용된 포맷팅 스타일이 Claude의 응답 스타일에 영향을 줄 수 있습니다. 출력 포맷팅 측면에서 조향이 잘 안 된다면, 프롬프트 스타일을 원하는 출력 스타일과 최대한 가깝게 맞춰보세요. 예를 들어 프롬프트에서 마크다운을 제거하면 출력의 마크다운 사용량도 줄어들 수 있습니다.

**구체적인 포맷팅 선호에는 상세 프롬프트를 사용하세요**

마크다운과 포맷팅 사용을 더 정밀히 통제하려면 명시적 지침을 제공하세요:

**마크다운 사용을 최소화하기 위한 샘플 프롬프트**

```
<avoid_excessive_markdown_and_bullet_points>
보고서, 문서, 기술 설명, 분석 또는 모든 장문 콘텐츠를 작성할 때는
완전한 단락과 문장으로 이루어진 명확하고 흐름이 자연스러운 산문으로 쓰세요.
표준 단락 구분을 통해 구조를 잡고, 마크다운은 주로 `inline code`,
코드 블록(```...```), 단순 헤딩(###)에만 사용하세요.
**굵게**와 *이탤릭*은 피하세요.

순서가 있는 목록(1. ...)이나 순서가 없는 목록(*)은 다음의 경우가 아니면
사용하지 마세요: a) 진정으로 분리된 항목들을 제시하며 목록 형태가 최선일 때,
b) 사용자가 명시적으로 목록이나 순위를 요청했을 때.

항목을 글머리표나 번호로 나열하는 대신 문장에 자연스럽게 녹여 넣으세요.
이 지침은 특히 기술 문서 작성에 적용됩니다. 과도한 포맷팅 대신 산문을
사용하는 것이 사용자 만족도를 높입니다. 짧디짧은 글머리표를 연속해서
출력하는 일은 절대 하지 마세요.

목표는 정보를 고립된 점들로 쪼개는 것이 아니라, 독자를 아이디어 사이로
자연스럽게 이끌어주는 읽기 좋은 흐름 있는 텍스트입니다.
</avoid_excessive_markdown_and_bullet_points>
```

### LaTeX 출력

Claude Opus 4.6은 수식 표현, 방정식, 기술적 설명에 LaTeX를 기본으로 사용합니다. 평문(plain text)을 선호한다면 프롬프트에 다음 지침을 추가하세요:

**샘플 프롬프트**

> 응답을 평문으로만 포맷해주세요. LaTeX, MathJax, 또는 `\( \)`, `$`, `\frac{}{}` 같은 어떤 마크업 표기도 사용하지 마세요. 모든 수식 표현을 표준 텍스트 문자로 작성하세요(예: 나눗셈은 "/", 곱셈은 "*", 지수는 "^").

### 문서 생성

Claude 최신 모델은 프레젠테이션, 애니메이션, 시각 문서를 만드는 데 탁월하며, 인상적인 창의적 감각과 강한 지시 준수 능력을 보여줍니다. 대부분의 경우 첫 시도부터 완성도 높고 쓸 만한 출력을 만들어냅니다.

문서 생성에서 최상의 결과를 위해:

**샘플 프롬프트**

> [주제]에 관한 전문적인 프레젠테이션을 만들어주세요. 사려 깊은 디자인 요소, 시각적 위계, 그리고 적절한 곳에 매력적인 애니메이션을 포함해주세요.

### 프리필(prefilled) 응답에서 벗어나기

Claude 4.6 모델과 Claude Mythos Preview부터 마지막 어시스턴트 턴에서의 프리필 응답이 더 이상 지원되지 않습니다. Mythos Preview에서는 프리필된 어시스턴트 메시지가 포함된 요청이 400 에러를 반환합니다. 모델의 지능과 지시 준수 능력이 발전한 결과, 프리필을 쓰던 대부분의 사용 사례는 더 이상 프리필을 요구하지 않습니다. 기존 모델들은 계속해서 프리필을 지원하며, 대화 중간 위치에 어시스턴트 메시지를 추가하는 것은 영향을 받지 않습니다.

다음은 흔한 프리필 시나리오와 그 마이그레이션 방법입니다:

- 출력 포맷팅 제어
- 머리말(preamble) 제거
- 잘못된 거절(bad refusal) 회피
- 이어쓰기(continuation)
- 맥락 주입(context hydration)과 역할 일관성

## 도구 사용

### 도구 활용(Tool usage)

Claude 최신 모델은 정확한 지시 준수를 위해 학습되었기 때문에, 특정 도구를 사용하라는 명시적 지시로부터 이득을 봅니다. "변경 사항을 제안해 줄 수 있어?"라고 말하면, 본 의도가 변경 적용이었더라도 Claude는 종종 제안만 하고 적용은 하지 않습니다.

Claude가 행동을 취하게 하려면 더 명시적으로 말하세요:

**예: 명시적 지시**

기본적으로 Claude를 더 적극적으로 행동하게 하려면 시스템 프롬프트에 다음을 추가할 수 있습니다:

**적극적 행동을 위한 샘플 프롬프트**

```
<default_to_action>
By default, implement changes rather than only suggesting them.
If the user's intent is unclear, infer the most useful likely action
and proceed, using tools to discover any missing details instead of
guessing. Try to infer the user's intent about whether a tool call
(e.g., file edit or read) is intended or not, and act accordingly.
</default_to_action>
```

반대로, 모델이 더 신중하게 기본 동작하길 원하고, 구현으로 곧장 뛰어드는 일이 적고, 요청받을 때만 행동하도록 하고 싶다면 아래와 같은 프롬프트로 조향할 수 있습니다:

**보수적 행동을 위한 샘플 프롬프트**

```
<do_not_act_before_instructions>
Do not jump into implementation or change files unless clearly
instructed to make changes. When the user's intent is ambiguous,
default to providing information, doing research, and providing
recommendations rather than taking action. Only proceed with edits,
modifications, or implementations when the user explicitly
requests them.
</do_not_act_before_instructions>
```

Claude Opus 4.5와 Claude Opus 4.6은 또한 이전 모델보다 시스템 프롬프트에 더 잘 반응합니다. 프롬프트가 도구나 스킬의 트리거 부족을 줄이도록 설계되어 있었다면, 이 모델들에서는 과잉 트리거링이 일어날 수 있습니다. 해결은 공격적인 표현을 누그러뜨리는 것입니다. "CRITICAL: You MUST use this tool when..." 처럼 쓰던 곳을 "이 도구를 사용하세요. 사용 조건은..." 정도의 정상적인 프롬프트로 바꾸세요.

### 병렬 도구 호출을 최적화하세요

Claude 최신 모델은 도구를 병렬로 실행하는 데 탁월합니다. 이 모델들은:

- 리서치 중에 여러 추측성 검색을 동시에 수행합니다.
- 한 번에 여러 파일을 읽어 더 빨리 맥락을 구성합니다.
- bash 명령을 병렬로 실행합니다(시스템 성능 측면에서 병목이 될 수도 있을 정도로).

이 동작은 쉽게 조향 가능합니다. 별도 프롬프팅 없이도 모델은 병렬 도구 호출에서 높은 성공률을 보이지만, 이를 약 100%까지 끌어올리거나 공격성 수준을 조절할 수 있습니다:

**최대 병렬 효율을 위한 샘플 프롬프트**

```
<use_parallel_tool_calls>
여러 도구를 호출할 계획이고 호출 간 의존성이 없다면, 모든
독립적인 도구 호출을 병렬로 수행하세요. 작업을 순차가 아니라
동시에 수행할 수 있을 때는 가능한 한 우선적으로 도구를 동시에
호출하세요. 예를 들어 파일 3개를 읽을 때는, 도구 호출 3개를
병렬로 띄워 3개 파일을 동시에 컨텍스트에 읽어들이세요.
가능한 곳마다 병렬 도구 호출을 최대화해 속도와 효율을
높이세요. 단, 일부 도구 호출이 이전 호출에 의존해 매개변수
같은 값들을 받아야 한다면 그것들은 병렬이 아니라 순차로
호출하세요. 도구 호출에서 매개변수가 빠진 곳에 플레이스홀더를
쓰거나 값을 추측해서 넣지 마세요.
</use_parallel_tool_calls>
```

**병렬 실행을 줄이기 위한 샘플 프롬프트**

> 안정성을 위해 각 단계 사이에 짧은 멈춤을 두고 작업을 순차적으로 실행해주세요.

## 사고(Thinking)와 추론(Reasoning)

### 과잉 사고(overthinking)와 지나친 철저함

Claude Opus 4.6은 이전 모델보다, 특히 높은 effort 설정에서 사전 탐색을 훨씬 더 많이 수행합니다. 이런 초기 작업은 보통 최종 결과 최적화에 도움이 되지만, 별도 지시 없이도 광범위한 맥락 수집이나 여러 갈래의 리서치를 동시에 진행할 수 있습니다. 이전에 모델이 더 철저하길 권장하는 프롬프트를 쓰고 있었다면, Claude Opus 4.6에 맞게 그 지침을 조정해야 합니다:

- **포괄적 기본값을 보다 표적이 된 지시로 교체하세요.** "기본적으로 [도구]를 사용하세요" 대신 "[도구]는 문제 이해를 강화할 때 사용하세요" 같은 식으로 안내하세요.
- **과잉 프롬프팅을 제거하세요.** 이전 모델에서 트리거가 부족했던 도구들이 지금은 적절히 트리거될 가능성이 높습니다. "확신이 없으면 [도구]를 쓰세요" 같은 지시는 과잉 트리거를 유발합니다.
- **effort를 안전장치로 사용하세요.** Claude가 계속 지나치게 적극적이라면, effort를 더 낮은 설정으로 사용하세요.

어떤 경우 Claude Opus 4.6은 사고를 광범위하게 수행해 사고 토큰이 부풀어 오르고 응답이 느려질 수 있습니다. 이런 동작이 바람직하지 않다면, 추론을 제약하는 명시 지침을 추가하거나 effort 설정을 낮춰 전반적인 사고와 토큰 사용을 줄일 수 있습니다.

**샘플 프롬프트**

> 문제 접근 방식을 결정할 때는 한 가지 접근을 선택해 그것에 전념하세요. 본인의 추론을 직접 반박하는 새로운 정보가 등장하지 않는 한 결정을 다시 들춰보지 마세요. 두 가지 접근 사이에서 저울질하고 있다면, 하나를 골라 끝까지 밀어붙이세요. 선택한 접근이 실패하면 그때 가서 방향을 수정해도 늦지 않습니다.

만약 사고 비용에 단단한 상한이 필요하다면, `budget_tokens` 상한을 사용하는 확장 사고(extended thinking)는 Opus 4.6과 Sonnet 4.6에서 여전히 작동하지만 더 이상 권장되지는 않습니다(deprecated). effort 설정을 낮추거나 `max_tokens`를 적응형 사고와 함께 단단한 상한으로 사용하는 편을 우선하세요.

### 사고와 인터리브드 사고(interleaved thinking) 능력을 활용하세요

Claude 최신 모델은 도구 사용 후의 반성이나 복잡한 다단계 추론이 필요한 작업에 특히 유용한 사고(thinking) 기능을 제공합니다. 더 나은 결과를 위해 초기 사고나 인터리브드 사고를 안내할 수 있습니다.

Claude Opus 4.6과 Claude Sonnet 4.6은 적응형 사고(`thinking: {type: "adaptive"}`)를 사용합니다. Claude가 언제, 얼마나 사고할지 동적으로 결정하는 방식입니다. Claude는 두 요인을 바탕으로 사고를 보정합니다: effort 파라미터와 쿼리 복잡도입니다. effort가 높을수록, 그리고 쿼리가 복잡할수록 더 많은 사고가 유도됩니다. 사고가 필요 없는 쉬운 쿼리에는 모델이 곧바로 응답합니다. 사내 평가에서 적응형 사고는 확장 사고보다 안정적으로 더 좋은 성능을 냅니다. 가장 똑똑한 응답을 얻으려면 적응형 사고로 옮기는 것을 고려하세요.

다단계 도구 사용, 복잡한 코딩 작업, 장기 에이전트 루프 같은 에이전트적 행동이 필요한 워크로드에는 적응형 사고를 사용하세요. 구형 모델은 `budget_tokens`로 수동 사고 모드를 사용합니다.

Claude의 사고 동작을 안내할 수 있습니다:

**예시 프롬프트**

> 도구 결과를 받은 뒤에는 결과의 품질을 신중하게 검토하고 다음 단계를 결정한 뒤 진행하세요. 사고를 사용해 이 새로운 정보를 바탕으로 계획하고 반복하고, 그런 다음 최선의 다음 액션을 취하세요.

적응형 사고의 트리거 동작은 프롬프트로 조향 가능합니다. 시스템 프롬프트가 크거나 복잡한 경우에는 원하는 것보다 모델이 더 자주 사고할 수 있는데, 그럴 때는 다음과 같은 지침을 추가해 조향하세요:

**샘플 프롬프트**

> 확장 사고는 지연을 더하기 때문에 답변 품질을 의미 있게 개선할 때만 사용해야 합니다 — 일반적으로 다단계 추론이 필요한 문제에 해당합니다. 확신이 없으면 그냥 바로 답하세요.

`budget_tokens`를 쓰는 확장 사고에서 마이그레이션 중이라면, 사고 설정을 교체하고 예산 통제를 effort로 옮기세요:

**Before (구형 모델의 확장 사고):**

```python
client.messages.create(
    model="claude-sonnet-4-5-20250929",
    max_tokens=64000,
    thinking={"type": "enabled", "budget_tokens": 32000},
    messages=[{"role": "user", "content": "..."}],
)
```

**After (적응형 사고):**

```python
client.messages.create(
    model="claude-opus-4-7",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},  # 또는 "max", "xhigh", "medium", "low"
    messages=[{"role": "user", "content": "..."}],
)
```

확장 사고를 쓰지 않는다면 변경이 필요 없습니다. `thinking` 파라미터를 생략하면 사고 기능은 기본적으로 꺼져 있습니다.

- **규정된 단계보다 일반적 지시를 우선하세요.** "철저히 사고하세요" 같은 프롬프트가 사람이 직접 적은 단계별 계획보다 더 좋은 추론을 만들어내는 경우가 많습니다. Claude의 추론은 사람이 지시하는 수준을 자주 능가합니다.
- **멀티샷 예시와 사고는 함께 작동합니다.** few-shot 예시 안에서 `<thinking>` 태그를 사용해 추론 패턴을 보여주세요. Claude는 그 스타일을 자신의 확장 사고 블록으로 일반화합니다.
- **수동 CoT(연쇄 사고)는 폴백으로.** 사고가 꺼져 있을 때도 Claude에게 문제를 단계별로 생각해보라고 요청하면 단계별 추론을 유도할 수 있습니다. `<thinking>`과 `<answer>` 같은 구조화된 태그를 사용해 추론과 최종 출력을 깔끔하게 분리하세요.
- **Claude에게 자기 점검을 요청하세요.** "끝내기 전에 [테스트 기준]에 대해 답을 검증해주세요." 같은 문구를 덧붙이세요. 이는 특히 코딩과 수학에서 오류를 안정적으로 잡아냅니다.
- 확장 사고가 꺼져 있을 때, Claude Opus 4.5는 "think"라는 단어와 그 변형들에 특히 민감합니다. 이런 경우에는 "consider", "evaluate", "reason through" 같은 대안을 사용하는 것을 고려하세요.

사고 기능에 대한 더 많은 정보는 확장 사고(Extended thinking)와 적응형 사고(Adaptive thinking) 문서를 참고하세요.

## 에이전트 시스템

### 장기 추론(long-horizon reasoning)과 상태 추적

Claude 최신 모델은 장기 추론 작업에 탁월하며 뛰어난 상태 추적 능력을 보입니다. Claude는 모든 것을 한 번에 처리하려 들지 않고, 점진적인 진전에 집중하면서 일부 작업에 꾸준히 진전을 만들어내며 긴 세션 전반에 걸쳐 방향성을 유지합니다. 이 능력은 특히 여러 컨텍스트 윈도우나 작업 반복(iteration)을 가로지를 때 두드러집니다. Claude가 복잡한 작업을 진행하다가 상태를 저장하고, 새로운 컨텍스트 윈도우에서 이어갈 수 있습니다.

### 맥락 인식(context awareness)과 다중 윈도우 워크플로우

Claude 4.6 및 Claude 4.5 모델은 컨텍스트 인식 기능을 제공해, 대화 전반에 걸쳐 남은 컨텍스트 윈도우(즉 "토큰 예산")를 모델이 직접 추적할 수 있게 합니다. 이를 통해 Claude는 작업을 더 효과적으로 수행하고 맥락을 관리할 수 있는데, 자기에게 남은 공간이 얼마나 되는지 이해하기 때문입니다.

**컨텍스트 한도 관리:**

Claude Code처럼 컨텍스트를 압축하거나 외부 파일에 컨텍스트를 저장하도록 허용하는 에이전트 하네스에서 Claude를 사용한다면, 이 정보를 프롬프트에 추가해 Claude가 그에 맞게 행동하도록 하는 것을 고려하세요. 그렇지 않으면 Claude가 컨텍스트 한도에 가까워질 때 자연스럽게 작업을 마무리하려 할 수 있습니다. 다음은 예시 프롬프트입니다:

**샘플 프롬프트**

> 당신의 컨텍스트 윈도우는 한도에 가까워지면 자동으로 압축되어, 멈춘 지점에서 계속해서 무한히 작업을 이어갈 수 있습니다. 따라서 토큰 예산 문제로 작업을 일찍 중단하지 마세요. 토큰 예산 한도에 가까워지면 현재 진행 상황과 상태를 컨텍스트 윈도우가 갱신되기 전에 메모리에 저장하세요. 예산이 다 되어가더라도 항상 가능한 한 끈질기고 자율적으로, 그리고 작업을 끝까지 완수하세요. 남은 컨텍스트와 상관없이 어떤 작업도 인위적으로 일찍 멈추지 마세요.

메모리 도구(memory tool)는 컨텍스트 인식과 자연스럽게 짝을 이루어 매끄러운 컨텍스트 전환을 가능하게 합니다.

### 다중 컨텍스트 윈도우 워크플로우

여러 컨텍스트 윈도우에 걸쳐 진행되는 작업의 경우:

- **첫 컨텍스트 윈도우에서는 다른 프롬프트를 사용하세요**: 첫 컨텍스트 윈도우에서 골격을 잡고(테스트 작성, 셋업 스크립트 작성), 이후 컨텍스트 윈도우는 to-do 리스트를 반복 처리하는 데 사용하세요.
- **테스트를 구조화된 형식으로 작성하게 하세요**: 작업 시작 전에 Claude에게 테스트를 만들고 `tests.json` 같은 구조화된 형식으로 추적하도록 요청하세요. 이는 장기적으로 반복 작업 능력을 끌어올립니다. 테스트의 중요성을 Claude에게 상기시키세요: "테스트를 제거하거나 편집하는 것은 누락되거나 버그 있는 기능으로 이어질 수 있으므로 허용되지 않습니다."
- **삶의 질을 높이는 도구를 셋업하세요**: Claude에게 `init.sh` 같은 셋업 스크립트를 만들도록 권장해, 서버를 깔끔하게 시작하고, 테스트 스위트와 린터를 실행하도록 하세요. 이는 새 컨텍스트 윈도우에서 작업을 이어갈 때 반복 작업을 막아줍니다.
- **새 시작 vs 압축(compacting)**: 컨텍스트 윈도우가 비워질 때, 압축을 사용하기보다 완전히 새로운 컨텍스트 윈도우로 시작하는 것을 고려하세요. Claude 최신 모델은 로컬 파일 시스템에서 상태를 발견하는 데 극도로 능숙합니다. 어떤 경우에는 이 능력을 압축보다 적극 활용하고 싶을 수 있습니다. 어떻게 시작해야 하는지에 대해 처방적으로(prescriptive) 알려주세요:
  - "`pwd`를 호출하세요. 이 디렉토리 안의 파일만 읽거나 쓸 수 있습니다."
  - "`progress.txt`, `tests.json`, 그리고 git 로그를 검토하세요."
  - "새로운 기능을 구현하기 전에 기본 통합 테스트를 수동으로 실행해보세요."
- **검증 도구를 제공하세요**: 자율 작업의 길이가 늘어날수록, Claude는 지속적인 인간 피드백 없이도 정확성을 검증할 수 있어야 합니다. UI 테스트를 위한 Playwright MCP 서버나 컴퓨터 사용 기능 같은 도구가 도움이 됩니다.
- **컨텍스트를 완전히 활용하도록 독려하세요**: Claude에게 다음으로 넘어가기 전에 컴포넌트들을 효율적으로 완료하라고 프롬프트하세요:

**샘플 프롬프트**

> 이것은 매우 긴 작업이므로 작업을 명확하게 계획하는 것이 도움이 될 수 있습니다. 출력 컨텍스트 전체를 작업에 쓰는 것을 권장합니다 — 단, 의미 있는 커밋되지 않은 작업을 남긴 채 컨텍스트가 떨어지지 않도록 주의하세요. 이 작업을 완료할 때까지 체계적으로 계속 작업하세요.

### 상태 관리 베스트 프랙티스

- **구조화된 데이터에는 구조화된 형식을 사용하세요**: 테스트 결과나 작업 상태 같은 구조화된 정보를 추적할 때는 JSON 같은 구조화된 형식을 사용해 Claude가 스키마 요구사항을 이해할 수 있도록 하세요.
- **진행 메모에는 비구조화된 텍스트를 사용하세요**: 자유 형식의 진행 메모는 일반적인 진행 상황과 맥락을 추적하는 데 잘 작동합니다.
- **상태 추적에 git을 사용하세요**: git은 수행된 작업의 로그와 복원 가능한 체크포인트를 제공합니다. Claude 최신 모델은 여러 세션에 걸쳐 상태를 추적할 때 git을 사용하는 데 특히 뛰어난 성능을 보입니다.
- **점진적 진전을 강조하세요**: Claude에게 본인의 진행 상황을 추적하고 점진적 작업에 집중하라고 명시적으로 요청하세요.

### 자율성과 안전의 균형

지침 없이는 Claude Opus 4.6이 되돌리기 어려운 행동이나 공유 시스템에 영향을 주는 행동(파일 삭제, force-push, 외부 서비스에 게시 등)을 취할 수 있습니다. Claude Opus 4.6이 잠재적으로 위험한 행동 전에 확인을 받게 하려면 다음 지침을 프롬프트에 추가하세요:

**샘플 프롬프트**

> 본인의 행동의 가역성과 잠재적 영향을 고려하세요. 파일 편집이나 테스트 실행 같은 로컬·가역적 행동은 적극 권장됩니다. 다만, 되돌리기 어렵거나 공유 시스템에 영향을 주거나 파괴적일 수 있는 행동은, 진행 전에 사용자에게 물어보세요.
>
> 확인이 필요한 행동의 예시:
> - 파괴적 작업: 파일 또는 브랜치 삭제, 데이터베이스 테이블 드롭, `rm -rf`
> - 되돌리기 어려운 작업: `git push --force`, `git reset --hard`, 이미 게시된 커밋을 수정(amend)
> - 다른 사람에게 보이는 작업: 코드 푸시, PR/이슈에 댓글 달기, 메시지 보내기, 공유 인프라 수정
>
> 막힐 때 파괴적 행동을 지름길로 사용하지 마세요. 예를 들어, 안전 점검을 우회(`--no-verify`)하거나, 진행 중일 수 있는 익숙하지 않은 파일을 버리지 마세요.

### 리서치와 정보 수집

Claude 최신 모델은 예외적인 에이전트 검색 능력을 보이며, 여러 출처의 정보를 효과적으로 찾아 종합할 수 있습니다. 최적의 리서치 결과를 위해:

- **명확한 성공 기준을 제공하세요**: 리서치 질문에 대한 "성공한 답"이 무엇인지를 정의하세요.
- **출처 검증을 장려하세요**: Claude에게 여러 출처에 걸쳐 정보를 검증하도록 요청하세요.

복잡한 리서치 작업에는 구조화된 접근을 사용하세요:

**복잡한 리서치를 위한 샘플 프롬프트**

> 이 정보를 구조화된 방식으로 검색해주세요. 데이터를 모으면서 여러 경합 가설을 세우세요. 본인의 보정(calibration)을 개선하기 위해 진행 메모에 확신도 레벨을 추적하세요. 본인의 접근과 계획을 정기적으로 자기 비판해주세요. 가설 트리나 리서치 노트 파일을 업데이트해 정보를 영속화하고 투명성을 제공하세요. 이 복잡한 리서치 작업을 체계적으로 분해해주세요.

이 구조화된 접근을 통해 Claude는 코퍼스 크기에 관계없이 사실상 어떤 정보든 찾아 종합하고, 본인의 발견 사항을 반복적으로 자기 비판할 수 있습니다.

### 서브에이전트 오케스트레이션

Claude 최신 모델은 네이티브 서브에이전트 오케스트레이션 능력이 크게 개선되었습니다. 이 모델들은 특정 작업이 전문 서브에이전트로 위임하기에 적합한 시점을 인식하고, 명시적 지시 없이도 그렇게 위임합니다.

이 동작을 활용하려면:

- **잘 정의된 서브에이전트 도구를 갖추세요**: 서브에이전트 도구를 사용 가능하게 하고 도구 정의에 설명을 추가하세요.
- **Claude가 자연스럽게 오케스트레이션하도록 두세요**: 명시 지시 없이도 Claude가 적절히 위임합니다.
- **과사용을 경계하세요**: Claude Opus 4.6은 서브에이전트에 대한 선호가 강해, 단순하고 직접적인 접근이 더 나은 상황에서도 서브에이전트를 스폰할 수 있습니다. 예를 들어 직접 grep을 호출하는 게 더 빠르고 충분한데도 코드 탐색을 위해 서브에이전트를 스폰할 수 있습니다.

서브에이전트 과사용이 보이면, 언제 서브에이전트가 정당화되고 언제는 그렇지 않은지에 대해 명시 지침을 추가하세요:

**서브에이전트 사용을 위한 샘플 프롬프트**

> 서브에이전트는 작업을 병렬로 돌릴 수 있을 때, 격리된 컨텍스트가 필요할 때, 상태를 공유할 필요가 없는 독립 워크스트림을 다룰 때 사용하세요. 단순한 작업, 순차적인 작업, 단일 파일 편집, 또는 단계 간에 컨텍스트를 유지해야 하는 작업에는 위임 대신 직접 처리하세요.

### 복잡한 프롬프트를 체이닝하세요

적응형 사고와 서브에이전트 오케스트레이션을 통해 Claude는 대부분의 다단계 추론을 내부에서 처리합니다. 명시적 프롬프트 체이닝(작업을 여러 개의 순차 API 호출로 쪼개는 것)은 중간 출력을 검사하거나 특정 파이프라인 구조를 강제할 필요가 있을 때 여전히 유용합니다.

가장 흔한 체이닝 패턴은 자기 교정(self-correction)입니다: 초안 생성 → 기준에 비추어 Claude가 리뷰 → 리뷰에 기반해 Claude가 개선. 각 단계는 별도의 API 호출이므로 어떤 시점에서든 로깅, 평가, 분기를 할 수 있습니다.

### 에이전트 코딩에서 파일 생성을 줄이세요

Claude 최신 모델은 때때로 테스트와 반복 목적으로 새 파일을 생성할 수 있는데, 특히 코드를 다룰 때 그렇습니다. 이 접근을 통해 Claude는 파일, 특히 파이썬 스크립트를 "임시 스크래치패드"처럼 사용해 최종 출력을 저장하기 전 실험해볼 수 있습니다. 임시 파일을 사용하는 것은 특히 에이전트 코딩 사용 사례에서 결과를 개선할 수 있습니다.

새 파일 생성을 최소화하고 싶다면, Claude에게 스스로 정리하도록 지시하세요:

**샘플 프롬프트**

> 반복용으로 임시 새 파일, 스크립트, 헬퍼 파일을 만들었다면, 작업이 끝날 때 그 파일들을 제거해 정리해주세요.

### 과잉 적극성(overeagerness)

Claude Opus 4.5와 Claude Opus 4.6은 요청되지 않은 추가 파일을 만들거나, 불필요한 추상화를 더하거나, 요청되지 않은 유연성을 빌드에 포함하는 식으로 오버엔지니어링하는 경향이 있습니다. 이런 원치 않는 동작이 보이면, 솔루션을 최소로 유지하기 위한 구체적인 지침을 추가하세요.

예:

**오버엔지니어링을 최소화하기 위한 샘플 프롬프트**

> 오버엔지니어링을 피하세요. 직접 요청된 변경이나 명백히 필요한 변경만 수행하세요. 솔루션은 단순하고 집중되도록 유지하세요:
>
> - **범위**: 요청 범위를 넘어 기능을 추가하거나, 코드를 리팩토링하거나, "개선"을 시도하지 마세요. 버그 수정에는 주변 코드 정리가 필요 없습니다. 단순한 기능에는 추가적인 설정 가능성(configurability)이 필요 없습니다.
> - **문서화**: 변경하지 않은 코드에 docstring, 주석, 타입 어노테이션을 추가하지 마세요. 자명하지 않은 로직에만 주석을 다세요.
> - **방어적 코딩**: 일어날 수 없는 시나리오에 대해 에러 처리, 폴백, 검증을 추가하지 마세요. 내부 코드와 프레임워크의 보장을 신뢰하세요. 시스템 경계(사용자 입력, 외부 API)에서만 검증하세요.
> - **추상화**: 한 번만 쓰는 작업에 헬퍼, 유틸리티, 추상화를 만들지 마세요. 가상의 미래 요구사항을 위해 설계하지 마세요. 현재 작업에 필요한 최소한이 적정한 복잡도입니다.

### 테스트 통과에 집착하거나 값을 하드코딩하는 것을 피하세요

Claude가 때로는 일반적인 솔루션 대신 테스트 통과에 지나치게 집중하거나, 복잡한 리팩토링에 표준 도구를 직접 쓰는 대신 헬퍼 스크립트 같은 우회책을 쓸 수 있습니다. 이런 동작을 막고 견고하고 일반화 가능한 솔루션을 보장하려면:

**샘플 프롬프트**

> 사용 가능한 표준 도구를 이용해 고품질의 범용 솔루션을 작성해주세요. 작업을 더 효율적으로 처리하기 위해 헬퍼 스크립트나 우회책을 만들지 마세요. 테스트 케이스뿐 아니라 모든 유효한 입력에 대해 올바로 동작하는 솔루션을 구현하세요. 값을 하드코딩하거나 특정 테스트 입력에만 동작하는 솔루션을 만들지 마세요. 문제를 일반적으로 푸는 실제 로직을 구현하세요.
>
> 문제 요구사항을 이해하고 올바른 알고리즘을 구현하는 데 집중하세요. 테스트는 정확성을 검증하기 위한 것이지 솔루션을 정의하기 위한 것이 아닙니다. 베스트 프랙티스와 소프트웨어 설계 원칙을 따르는 원칙적인 구현을 제공하세요.
>
> 작업이 불합리하거나 실현 불가능하거나, 또는 어떤 테스트가 잘못되었다면, 우회하기보다 그 사실을 알려주세요. 솔루션은 견고하고 유지 보수 가능하고 확장 가능해야 합니다.

### 에이전트 코딩에서 환각 최소화

Claude 최신 모델은 환각이 덜하며, 코드에 기반한 보다 정확하고 근거 있는 똑똑한 답변을 제공합니다. 이 동작을 더 권장하고 환각을 최소화하려면:

**샘플 프롬프트**

```
<investigate_before_answering>
열어본 적 없는 코드에 대해 절대 추측하지 마세요. 사용자가 특정
파일을 언급하면, 답변 전에 반드시 그 파일을 읽어야 합니다. 코드
베이스에 관한 질문에 답하기 전에 관련 파일을 조사하고 읽어보세요.
정확한 답을 확신하지 않는 한, 조사 없이 코드에 대해 어떤 주장도
하지 마세요 — 근거 있고 환각 없는 답을 주세요.
</investigate_before_answering>
```

## 능력별 팁

### 개선된 비전 능력

Claude Opus 4.5와 Claude Opus 4.6은 이전 Claude 모델에 비해 비전 능력이 개선되었습니다. 이미지 처리와 데이터 추출 작업에서 더 나은 성능을 보이며, 특히 컨텍스트에 여러 이미지가 있을 때 그렇습니다. 이런 개선은 컴퓨터 사용으로도 이어져, 모델이 스크린샷과 UI 요소를 더 안정적으로 해석할 수 있습니다. 이 모델들은 비디오를 프레임으로 쪼개어 분석하는 데도 쓸 수 있습니다.

성능을 더 끌어올리는 데 효과적인 기법 하나는 Claude에게 크롭 도구(crop tool)나 스킬을 주는 것입니다. 테스트 결과, Claude가 이미지의 관련 영역을 "줌인"할 수 있을 때 이미지 평가에서 일관된 상승이 확인되었습니다. Anthropic이 크롭 도구를 위한 cookbook을 만들어두었습니다.

### 프론트엔드 디자인

Claude Opus 4.5와 Claude Opus 4.6은 강력한 프론트엔드 디자인 능력을 갖춘 복잡한 실제 웹 애플리케이션 빌드에 탁월합니다. 그러나 지침이 없으면 모델들은 사용자들이 "AI 슬롭(AI slop)" 미학이라 부르는 일반적인 패턴을 기본으로 채택할 수 있습니다. 놀라움과 즐거움을 주는 독창적이고 창의적인 프론트엔드를 만들려면:

프론트엔드 디자인을 개선하는 자세한 가이드는 "스킬을 통한 프론트엔드 디자인 개선(improving frontend design through skills)" 블로그 글을 참고하세요.

다음은 더 나은 프론트엔드 디자인을 권장하기 위해 사용할 수 있는 시스템 프롬프트 스니펫입니다:

**프론트엔드 미학을 위한 샘플 프롬프트**

```
<frontend_aesthetics>
You tend to converge toward generic, "on distribution" outputs.
In frontend design, this creates what users call the "AI slop"
aesthetic. Avoid this: make creative, distinctive frontends that
surprise and delight.

Focus on:
- Typography: Choose fonts that are beautiful, unique, and interesting.
  Avoid generic fonts like Arial and Inter; opt instead for distinctive
  choices that elevate the frontend's aesthetics.
- Color & Theme: Commit to a cohesive aesthetic. Use CSS variables for
  consistency. Dominant colors with sharp accents outperform timid,
  evenly-distributed palettes. Draw from IDE themes and cultural
  aesthetics for inspiration.
- Motion: Use animations for effects and micro-interactions. Prioritize
  CSS-only solutions for HTML. Use Motion library for React when
  available. Focus on high-impact moments: one well-orchestrated page
  load with staggered reveals (animation-delay) creates more delight
  than scattered micro-interactions.
- Backgrounds: Create atmosphere and depth rather than defaulting to
  solid colors. Layer CSS gradients, use geometric patterns, or add
  contextual effects that match the overall aesthetic.

Avoid generic AI-generated aesthetics:
- Overused font families (Inter, Roboto, Arial, system fonts)
- Clichéd color schemes (particularly purple gradients on white
  backgrounds)
- Predictable layouts and component patterns
- Cookie-cutter design that lacks context-specific character

Interpret creatively and make unexpected choices that feel genuinely
designed for the context. Vary between light and dark themes,
different fonts, different aesthetics. You still tend to converge on
common choices (Space Grotesk, for example) across generations.
Avoid this: it is critical that you think outside the box!
</frontend_aesthetics>
```

전체 스킬 정의도 함께 참조할 수 있습니다.

## 마이그레이션 고려사항

이전 세대에서 Claude 4.6 모델로 마이그레이션할 때:

- **원하는 동작에 대해 구체적으로**: 출력에서 정확히 어떤 모습을 보고 싶은지 설명하는 것을 고려하세요.
- **지시를 강조어로 감싸세요**: Claude가 출력의 품질과 디테일을 끌어올리도록 권장하는 강조어를 추가하면, 모델의 성능을 더 잘 형성할 수 있습니다. 예를 들어 "분석 대시보드를 만들어주세요" 대신 "분석 대시보드를 만들어주세요. 관련 기능과 상호작용을 가능한 한 많이 포함해주세요. 기본을 넘어선, 완전히 기능하는 구현으로 만들어주세요."
- **특정 기능은 명시적으로 요청하세요**: 애니메이션과 인터랙티브 요소는 필요할 때 명시적으로 요청해야 합니다.
- **사고 설정을 업데이트하세요**: Claude 4.6 모델은 `budget_tokens`를 사용하는 수동 사고 대신 적응형 사고(`thinking: {type: "adaptive"}`)를 사용합니다. effort 파라미터를 사용해 사고 깊이를 제어하세요.
- **프리필 응답에서 벗어나세요**: Claude 4.6 모델부터 마지막 어시스턴트 턴에서의 프리필 응답이 권장되지 않습니다(deprecated). 대안에 대한 자세한 안내는 "프리필 응답에서 벗어나기"를 참고하세요.
- **반(反)게으름 프롬프트를 튜닝하세요**: 모델이 더 철저하길 권장하거나 도구를 더 적극적으로 쓰도록 권장하는 프롬프트를 쓰고 있었다면, 그 지침을 누그러뜨리세요. Claude 4.6 모델은 훨씬 더 적극적이라 이전 모델용 지시에서는 과잉 트리거가 일어날 수 있습니다.

자세한 마이그레이션 단계는 마이그레이션 가이드(Migration guide)를 참고하세요.

### Claude Sonnet 4.5에서 Claude Sonnet 4.6으로 마이그레이션

Claude Sonnet 4.6은 effort 파라미터의 기본값이 high인 반면, Claude Sonnet 4.5는 effort 파라미터가 아예 없었습니다. Claude Sonnet 4.5에서 Claude Sonnet 4.6으로 마이그레이션하면서 effort 파라미터를 조정하는 것을 고려하세요. 명시적으로 설정하지 않으면 기본 effort 레벨로 인해 지연이 더 커질 수 있습니다.

권장 effort 설정:

- 대부분의 애플리케이션에는 **medium**
- 대량 또는 지연 시간에 민감한 워크로드에는 **low**
- medium 또는 high effort에서 모델이 사고하고 행동할 공간을 위해 큰 max output 토큰 예산(64k 토큰 권장)을 설정하세요.

**Opus 4.7을 대신 사용해야 할 때**: 가장 어렵고 장기인 문제들(대규모 코드 마이그레이션, 깊이 있는 리서치, 확장된 자율 작업)에는 Opus 4.7이 여전히 적합한 선택입니다. Sonnet 4.6은 빠른 응답과 비용 효율이 가장 중요한 워크로드에 최적화되어 있습니다.

#### 확장 사고를 쓰지 않는 경우

Claude Sonnet 4.5에서 확장 사고를 쓰고 있지 않다면, Claude Sonnet 4.6에서도 확장 사고 없이 계속 진행할 수 있습니다. 본인의 사용 사례에 적절한 effort 레벨을 명시적으로 설정해야 합니다. 확장 사고를 꺼두고 low effort로 설정하면, Claude Sonnet 4.5의 확장 사고 미사용 환경과 유사하거나 더 나은 성능을 기대할 수 있습니다.

```python
client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8192,
    thinking={"type": "disabled"},
    output_config={"effort": "low"},
    messages=[{"role": "user", "content": "..."}],
)
```

#### 확장 사고를 쓰는 경우

Claude Sonnet 4.5에서 `budget_tokens`로 확장 사고를 쓰고 있다면, Claude Sonnet 4.6에서도 그것이 여전히 작동하지만 더 이상 권장되지는 않습니다(deprecated). effort 파라미터를 사용하는 적응형 사고로 마이그레이션하세요.

#### 적응형 사고로 마이그레이션하기

적응형 사고는 다음 워크로드 패턴에 특히 잘 맞습니다:

- **자율 다단계 에이전트**: 요구사항을 동작하는 소프트웨어로 바꾸는 코딩 에이전트, 데이터 분석 파이프라인, 그리고 모델이 여러 단계에 걸쳐 독립적으로 동작하는 버그 찾기. 적응형 사고는 단계별로 추론을 보정해, 더 긴 궤적에서도 경로를 유지하게 합니다. 이런 워크로드는 high effort에서 시작하세요. 지연이나 토큰 사용이 우려되면 medium으로 낮추세요.
- **컴퓨터 사용 에이전트**: Claude Sonnet 4.6은 적응형 모드를 사용해 컴퓨터 사용 평가에서 동급 최고의 정확도를 달성했습니다.
- **이중 모드 워크로드(bimodal workloads)**: 쉬운 작업과 어려운 작업이 섞여 있는 워크로드. 적응형 사고는 단순 쿼리에는 사고를 건너뛰고, 복잡한 쿼리에는 깊이 추론합니다.

적응형 사고를 사용할 때는 본인의 작업에서 medium과 high effort를 평가해보세요. 적절한 레벨은 워크로드의 품질, 지연, 토큰 사용량 사이의 트레이드오프에 따라 달라집니다.

```python
client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},
    messages=[{"role": "user", "content": "..."}],
)
```

#### 마이그레이션 동안 `budget_tokens`를 유지하기

마이그레이션 동안 `budget_tokens`를 일시적으로 유지해야 한다면, 16k 토큰 정도의 예산이 더 어려운 문제에 대한 여유 공간을 제공하면서도 토큰 사용 폭주의 위험을 줄여줍니다. 이 설정은 더 이상 권장되지 않으며 향후 모델 릴리스에서 제거됩니다.

코딩 사용 사례(에이전트 코딩, 도구 사용이 많은 워크플로우, 코드 생성)는 medium effort부터 시작하세요:

```python
client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=16384,
    thinking={"type": "enabled", "budget_tokens": 16384},
    output_config={"effort": "medium"},
    messages=[{"role": "user", "content": "..."}],
)
```

채팅과 비코딩 사용 사례(채팅, 콘텐츠 생성, 검색, 분류)는 low effort부터 시작하세요:

```python
client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8192,
    thinking={"type": "enabled", "budget_tokens": 16384},
    output_config={"effort": "low"},
    messages=[{"role": "user", "content": "..."}],
)
```
