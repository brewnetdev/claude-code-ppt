# 빌트인 명령어 레퍼런스

## 개요

Claude Code CLI에 내장된 67개 빌트인 명령어를 13개 카테고리로 정리합니다.

**표 범례:**
- **별칭**: 동일 동작을 하는 대체 명령어
- **인수**: `[선택]` = 생략 가능, `<필수>` = 반드시 입력
- **(숨김)**: `/help`에 표시되지 않는 명령어

## 주요 명령어 심화 설명

### `/btw` — 메인 대화를 방해하지 않는 사이드 질문

`/btw`는 진행 중인 작업 흐름을 중단하지 않고 별도 질문을 처리할 때 사용합니다.

**동작 방식:**
- **Fork 모델** 활용: 현재 대화의 컨텍스트를 상속한 새 서브세션에서 실행
- 질문에 대한 답변은 현재 대화 히스토리에 **포함되지 않음**
- 따라서 `/btw` 응답이 이후 메인 대화의 AI 동작에 영향을 주지 않음

```
사용 예시:
/btw 파이썬 딕셔너리와 JSON 객체의 차이는?
```

**적합한 상황:**
- 구현 중 문득 생긴 언어/기술 관련 궁금증
- 메인 컨텍스트를 오염시키고 싶지 않은 일회성 조회
- 작업 흐름 유지가 중요한 장기 세션

### `/insights` — 세션 분석 리포트

`/insights`는 현재 세션의 사용 패턴을 분석하여 구조화된 리포트를 생성합니다.

**리포트 포함 항목:**
- 도구별 호출 횟수 및 비율 (Read/Write/Bash/Edit 등)
- 토큰 소비 추이 및 집중 구간
- 작업 유형 분류 (코딩/질문/리뷰 등)
- 생산성 지표: 응답당 평균 도구 사용 수, 승인율

**활용 방법:**
```
/insights
```

장기 작업 세션 후 실행하여 자신의 Claude 활용 패턴을 파악하고, 도구 허용/제한 설정이나 `CLAUDE.md` 지시사항을 개선하는 데 참고합니다.

### `/batch` — 병렬 격리 에이전트 대규모 실행

`/batch`는 단일 요청으로 대규모 변경 작업을 병렬 처리합니다.

**실행 흐름:**

```
1. /batch <지시사항> 입력
2. 리서치 단계: 코드베이스 분석, 작업 범위 파악
3. 계획 단계: 5~30개 독립 서브태스크로 분해
4. 병렬 실행: 각 서브태스크가 격리된 git worktree에서 독립 실행
5. 완료: 각 에이전트가 별도 브랜치에 커밋 + PR 생성
```

**격리 실행의 의미:**
- 각 에이전트는 `isolation: "worktree"` 옵션으로 독립된 git worktree에서 작동
- 에이전트 간 파일 충돌 없음
- 실패한 에이전트가 다른 에이전트에 영향 없음

**적합한 작업:**
```
/batch 모든 컴포넌트에서 deprecated API 호출을 신규 API로 마이그레이션
/batch test coverage가 0%인 파일들에 단위 테스트 추가
/batch 전체 코드베이스에서 ESLint 오류 수정
```

> `largeContextWindowEnabled` 기능 플래그가 활성화된 환경에서만 사용 가능합니다.

### `/compact` — 컨텍스트 압축과 연속성 유지

`/compact`는 컨텍스트 윈도우가 가득 찰 때 대화를 요약하여 공간을 확보합니다.

**기본 동작:**
```
/compact
```
Claude가 자동으로 대화 전체를 요약하고, 요약본을 새 컨텍스트로 교체합니다.

**커스텀 요약 지시사항:**
```
/compact 핵심 코드 구조와 미해결 버그만 유지해줘
/compact API 엔드포인트 스펙과 인증 로직 중심으로 요약
```

**PostCompact Hook 연동:**
`hooks.json`에서 `PostCompact` 이벤트를 정의하면 `/compact` 완료 직후 자동으로 특정 파일 로드, 메모 작성 등의 후처리 작업을 수행할 수 있습니다.

```json
{
  "hooks": {
    "PostCompact": [{
      "hooks": [{
        "type": "command",
        "command": "echo '컨텍스트 압축 완료' >> .claude/compact-log.txt"
      }]
    }]
  }
}
```

### `/branch` — 대화 분기점 생성

`/branch`(별칭: `/fork`)는 현재 대화 상태를 복사하여 독립된 분기를 만듭니다.

**활용 시나리오:**
- 두 가지 접근법 중 어느 쪽이 나은지 실험할 때
- 위험한 변경을 시도하기 전 현재 상태 보존
- A/B 비교: 다른 모델, 다른 프롬프트 전략 테스트

```
현재 대화: 리팩터링 진행 중
  ├─ /branch approach-a → 함수형 스타일 시도
  └─ /branch approach-b → 객체지향 스타일 시도
```

분기된 대화는 `/resume` 명령어로 목록에서 선택하여 돌아올 수 있습니다.

### `/loop` — 반복 실행 모드

`/loop`는 지정한 간격마다 동일한 프롬프트를 자동 재실행합니다.

**기본 동작:**
```
/loop                           # 10분 간격, 대화형 프롬프트 입력
/loop 5m 새 PR 확인 후 리뷰     # 5분 간격으로 반복
/loop 30s 빌드 상태 체크         # 30초 간격
```

**간격 형식:**
- `s` — 초 (예: `30s`)
- `m` — 분 (예: `5m`)
- `h` — 시간 (예: `1h`)

**적합한 사용 사례:**
- 외부 이벤트 폴링 (새 이슈, PR, 빌드 결과)
- 단계적 변환 작업 (파일 100개를 10분마다 10개씩)
- 장기 모니터링 세션 (배포 후 에러 로그 감시)

> `/loop`와 Stop Hook을 함께 사용하면 Ralph Wiggum 패턴처럼 완전한 자율 반복 루프를 구성할 수 있습니다. 상세 내용은 [../10-고급-기능/01-Ralph-Wiggum-루프.md](../10-고급-기능/01-Ralph-Wiggum-루프.md)를 참고하세요.
