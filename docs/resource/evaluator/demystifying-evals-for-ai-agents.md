## 30초 요약

에이전트가 프로토타입을 넘어 **프로덕션·스케일** 단계에 들어가면 "감(感)으로 만들기"가 무너진다. 그때 필요한 것이 자동화된 **평가(eval)**다. 핵심은 다섯 가지다. ① 채점기는 코드형·모델형·사람형을 **섞어** 쓴다. ② 비결정성은 `pass@k`(한 번이라도 성공)와 `pass^k`(매번 성공) 두 지표로 관리한다. ③ 완벽한 평가셋을 기다리지 말고 **실패 사례 20~50개**로 지금 시작한다. ④ 경로가 아니라 **결과**를 채점하고, **트랜스크립트를 직접 읽는다.** ⑤ 어떤 단일 평가도 모든 결함을 못 잡으니 여러 겹으로 막는 **스위스 치즈 모델**로 접근한다. 결론: "평가를 나중 일이 아니라 핵심 부품으로 취급할 때에만 가치가 복리로 쌓인다."

---

## 1. 서론

좋은 평가는 팀이 AI 에이전트를 **자신 있게 출시**하도록 돕고, "터지면 고친다" 식의 반응형 디버깅 루프를 예방한다. 이 글은 이전 글 *Building effective agents*를 이어받으며, 에이전트의 특성 — 자율성(autonomy), 지능(intelligence), 유연성(flexibility) — 이 오히려 단일 턴(single-turn) 시스템보다 평가를 더 어렵게 만든다고 짚는다.

---

## 2. 평가의 구조 (The Structure of an Evaluation)

### 핵심 정의

- **Evaluation("eval")**: AI에 입력을 주고, 채점 로직(grading logic)을 적용해 성공을 측정하는 테스트.
- **Single-turn 평가**: 프롬프트 → 응답 → 채점의 단순한 흐름.
- **Multi-turn 평가**: 더 복잡하며, 역량이 발전할수록 점점 흔해진다.
- **Agent 평가**: 가장 복잡. 에이전트가 여러 턴에 걸쳐 도구를 쓰고, 상태를 바꾸며, 적응한다.

### 필수 용어

- **Task / problem / test case**: 정의된 입력과 성공 기준을 가진 하나의 테스트.
- **Trial**: 한 태스크에 대한 한 번의 시도. 모델 출력의 변동성 때문에 여러 trial을 돌린다.
- **Grader(채점기)**: 에이전트 성능에 점수를 매기는 로직. 한 태스크에 여러 grader와 여러 assertion("checks")을 둘 수 있다.
- **Transcript / trace / trajectory**: 출력·도구 호출·추론·중간 결과·모든 API 상호작용을 포함한 완전한 기록.
- **Outcome(결과)**: trial 종료 시점의 최종 환경 상태 (예: DB에 항공권 예약이 존재함).
- **Evaluation harness(평가 하네스)**: 평가를 엔드투엔드로 실행하는 인프라. 지시·도구 제공, 태스크 동시 실행, 단계 기록, 채점, 결과 집계를 담당.
- **Agent harness / scaffold**: 모델이 에이전트로 동작하도록 만드는 시스템. 입력 처리·도구 호출 오케스트레이션·결과 반환 (예: Claude Code, Agent SDK).
- **Evaluation suite(평가 스위트)**: 하나의 큰 목표를 공유하며 특정 역량·행동을 측정하는 태스크 모음 (고객지원 예: 환불·취소·에스컬레이션).

## 3. 왜 평가를 만드는가? (Why Build Evaluations?)

팀은 초기에 수동 테스트, 도그푸딩(dogfooding), 직관으로 나아가지만 **스케일에서 한계점**에 부딪힌다. 사용자가 품질 저하를 보고해도, 팀은 추측 말고는 변경을 검증할 방법이 없다. "불평을 기다린다 → 수동 재현 → 버그 수정 → 회귀 없기를 기대"하는 반응형 디버깅은 평가 없이는 실패한다. 회귀(regression)와 노이즈를 구분할 수도, 출시 전 수백 개 시나리오에 대해 변경을 검증할 수도 없기 때문이다.

### 실제 사례

- **Claude Code**: 사내·외부 사용자 피드백으로 빠르게 반복 → 좁은 범위의 eval(간결성, 파일 편집) 추가 → 복잡한 행동(과잉 엔지니어링)까지 확장. eval이 문제를 식별하고 개선을 이끌며 리서치-제품 협업의 초점이 됐다. 프로덕션 모니터링·A/B·유저 리서치와 결합해 스케일에서 지속 개선을 지원한다.
- **Descript**: 영상 편집 에이전트를 세 축으로 평가 — ① 망가뜨리지 말 것 ② 시킨 걸 할 것 ③ 잘 할 것. 수동 채점 → 제품팀 기준의 LLM 채점 + 주기적 사람 보정으로 진화. 지금은 벤치마킹용·회귀 테스트용 두 스위트를 운영.
- **Bolt**: 에이전트가 널리 쓰이기 시작한 뒤 eval 구축. 3개월 만에 정적 분석 + 브라우저 에이전트 테스트로 앱을 검증하는 시스템을 만들었고, 지시 준수 행동은 LLM 심판으로 판정.

### 시점별 이점

- **초기 개발**: 제품팀이 "성공"을 명세하도록 강제.
- **스케일**: 일관된 품질 기준을 유지.
- **모델 업그레이드**: eval이 있으면 모델 강점을 파악하고 프롬프트를 튜닝해 **며칠** 만에 교체 (없으면 **몇 주**의 수동 테스트).

일단 eval이 생기면 정적 태스크 뱅크 위에서 지연(latency)·토큰 사용량·태스크당 비용·에러율을 추적하는 베이스라인/회귀 테스트가 된다. eval은 제품과 리서치 사이에서 **가장 대역폭 높은 소통 채널**이 되어, 연구자가 최적화할 지표를 정의한다.

## 4. AI 에이전트를 어떻게 평가하는가

### 4-1. 채점기의 종류

**코드형 채점기 (Code-Based Graders)**

| 방법 | 강점 | 약점 |
|---|---|---|
| 문자열 매칭(정확·regex·fuzzy), 이진 테스트(fail-to-pass·pass-to-pass), 정적 분석(lint·type·security), 결과 검증, 도구 호출 검증(사용 도구·파라미터), 트랜스크립트 분석(턴 수·토큰) | 빠르고 저렴, 객관적, 재현 가능, 디버깅 쉬움, 특정 조건 검증 | 기대 패턴과 정확히 안 맞는 유효한 변형에 뻣뻣함, 뉘앙스 부족, 주관적 태스크에 한계 |

**모델형 채점기 (Model-Based Graders)**

| 방법 | 강점 | 약점 |
|---|---|---|
| 루브릭 기반 점수, 자연어 assertion, 쌍대 비교(pairwise), 참조 기반 평가, 다중 심판 합의(multi-judge consensus) | 유연·확장, 뉘앙스 포착, 개방형·자유형 출력 처리 | 비결정적, 코드보다 비쌈, 정확도 위해 사람 채점과 보정 필요 |

**사람형 채점기 (Human Graders)**

| 방법 | 강점 | 약점 |
|---|---|---|
| SME(전문가) 리뷰, 크라우드소싱 판단, 스팟체크 샘플링, A/B 테스트, 평가자 간 일치도(inter-annotator agreement) | 정답지(gold) 수준 품질, 전문가/사용자 판단과 일치, 모델 채점기 보정에 사용 | 비싸고 느림, 스케일에서 전문가 필요 |

채점 방식은 **가중치형**(합산 점수가 임계값 도달), **이진형**(모든 grader 통과 필수), **하이브리드**로 나뉜다.

### 4-2. 역량 평가 vs 회귀 평가

- **역량/품질 평가(Capability/Quality Evals)**: "이 에이전트가 무엇을 잘하나?" 낮은 통과율에서 시작해, 아직 못 푸는 태스크를 목표로 점진 개선.
- **회귀 평가(Regression Evals)**: "예전에 하던 걸 여전히 하나?" 거의 100% 통과율을 유지. 하락은 무언가 깨졌다는 신호.

출시·최적화 후, 통과율 높은 역량 eval은 회귀 스위트로 **"졸업(graduate)"**해 상시 실행되며 드리프트를 감지한다. "이거 할 수 있나?"가 "이거 여전히 안정적으로 되나?"로 바뀐다.

### 4-3. 코딩 에이전트 평가

코드를 작성·테스트·디버깅하고, 코드베이스를 탐색하며 명령을 실행한다. 소프트웨어 평가는 "코드가 돌고 테스트를 통과하는가?"로 명확하기 때문에 **결정적(deterministic) 채점기가 적합**하다.

**벤치마크**
- **SWE-bench Verified**: 인기 파이썬 저장소의 GitHub 이슈를 주고, 테스트 스위트 실행으로 채점. 기존 테스트를 깨지 않고 실패 테스트를 고쳐야 통과. LLM은 1년 만에 40% → 80%+로 발전.
- **Terminal-Bench**: 리눅스 커널 소스 빌드, ML 모델 훈련 등 엔드투엔드 기술 태스크.

결과 테스트를 넘어 트랜스크립트도 채점한다 — 코드 품질 휴리스틱으로 더 깊이 평가하고, 명확한 루브릭의 모델 채점기로 도구 호출·사용자 상호작용을 판정.

**코딩 에이전트 eval 예시 (YAML)**

```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token
```

> 참고: 이 예시는 채점기 전체 범위를 보여주기 위한 것. 실무에서는 보통 정확성엔 unit test, 품질엔 LLM 루브릭을 쓰고 필요 시 추가 grader를 더한다.

### 4-4. 대화형 에이전트 평가

고객지원·영업·코칭 같은 도메인에서 상호작용한다. 상태를 유지하고, 도구를 쓰고, 대화 중 액션을 취한다. 전통 챗봇과 달리 멀티턴으로 동작하며, **상호작용의 질 자체를 평가**해야 하는 것이 난제.

검증 가능한 최종 상태 + 태스크 완수·상호작용 품질을 담는 루브릭을 쓴다. 종종 **사용자를 시뮬레이션하는 두 번째 LLM**이 필요하다 (Anthropic의 얼라인먼트 감사 에이전트가 적대적 스트레스 테스트에 사용).

성공은 다차원적 — 티켓 해결(state check), 10턴 이내 종료(transcript 제약), 적절한 톤(LLM 루브릭).

**벤치마크**
- **τ-Bench**: 여러 도메인(리테일 지원, 항공 예약)의 멀티턴 상호작용.
- **τ2-Bench**: 후속. 한 모델이 사용자 페르소나를 연기하고 에이전트가 현실 시나리오를 처리.

**대화형 에이전트 eval 예시 (환불 태스크)**

```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics:
      - n_turns
      - n_toolcalls
      - n_total_tokens
  - type: latency
    metrics:
      - time_to_first_token
      - output_tokens_per_sec
      - time_to_last_token
```

### 4-5. 리서치 에이전트 평가

정보를 수집·종합·분석해 답변이나 리포트를 낸다. 코딩의 이진 unit test와 달리, 리서치 품질은 태스크에 상대적이다. "포괄적", "출처가 탄탄한", "정확한"의 기준이 시장 조사·인수 실사·과학 리포트마다 다르다.

**난점**: 전문가들도 종합의 완결성에 대해 이견이 있고, 참조 콘텐츠가 바뀌면 ground truth가 이동하며, 길고 개방적인 출력일수록 오류 여지가 크다.

**벤치마크**
- **BrowseComp**: 열린 웹에서 바늘 찾기 — 검증은 쉽지만 풀기는 어려운 질문.

**전략**: 채점기를 조합한다. **Groundedness check**(주장이 검색 출처로 뒷받침되는지), **Coverage check**(좋은 답이 반드시 포함해야 할 핵심 사실), **Source quality check**(먼저 검색된 게 아니라 권위 있는 출처인지). 객관적으로 정답이 있는 질문("X사 3분기 매출?")은 exact match가 통한다. LLM은 근거 없는 주장과 누락을 잡아내되, 종합의 일관성·완결성은 별도로 검증한다. 주관적 특성상 LLM 루브릭은 **전문가 판단과 자주 보정**해야 한다.

### 4-6. 컴퓨터 사용 에이전트(Computer Use Agents)

API나 코드가 아니라 사람과 같은 인터페이스 — 스크린샷·마우스 클릭·키보드·스크롤 — 로 상호작용한다. 디자인 툴부터 레거시 엔터프라이즈 SW까지 모든 GUI 앱을 쓴다. 실제/샌드박스 환경에서 에이전트를 돌려 **의도한 결과를 달성했는지** 확인해야 한다.

**벤치마크**
- **WebArena**: URL·페이지 상태 체크로 브라우저 탐색을 검증하고, 백엔드 상태 검증으로 데이터 변경을 확인(확인 페이지가 아니라 DB에 주문이 실제로 들어갔는지).
- **OSWorld**: 전체 OS 제어. 평가 스크립트가 파일 시스템 상태·앱 설정·DB 내용·UI 요소 속성 등 다양한 완료 산출물을 점검.

브라우저 에이전트는 **토큰 효율 vs 지연**의 균형을 맞춘다. DOM 기반은 빠르지만 토큰을 많이 먹고, 스크린샷은 느리지만 토큰 효율적. 예: 위키백과 요약은 DOM 텍스트 추출이 유리, 아마존에서 새 노트북 케이스 찾기는 스크린샷이 유리(전체 DOM 추출은 토큰 과다). Claude for Chrome은 맥락에 맞는 도구를 골랐는지 확인하는 eval을 개발.

### 4-7. 평가의 비결정성 (Non-Determinism)

에이전트 행동은 실행마다 달라 해석이 복잡하다. 각 태스크는 고유 성공률을 갖고(한 태스크 90%, 다른 태스크 50%), 이번에 통과한 태스크가 다음엔 실패할 수 있다. 목표는 에이전트가 **얼마나 자주**(trial 비율) 성공하는지를 측정하는 것.

**두 핵심 지표**

**pass@k** — k번 시도 중 적어도 한 번 정답을 낼 확률. k가 커지면 값이 **상승**(시도 많을수록 성공 확률↑).
- 공식: `1 − (1 − success_rate)^k`
- 50% pass@1 = 첫 시도에 절반의 태스크를 성공.
- 코딩(첫 시도 해법)이나 한 번의 성공이면 충분한 경우에 유용.

**pass^k** — k번 trial이 **모두** 성공할 확률. k가 커지면 값이 **하락**(더 많은 시도에서 일관성을 요구하는 게 더 어려움).
- 공식: `success_rate^k`
- 시도당 75% 성공률, 3회: `(0.75)³ ≈ 42%`
- 매번 신뢰할 수 있는 동작이 필요한 고객 대면 에이전트에 중요.

> 다이어그램상 발산: k=1에선 두 지표가 시도당 성공률과 같지만, k=10에선 pass@k가 100%에 근접하는 동안 pass^k는 0%로 떨어진다. 제품 요구가 지표 선택을 결정한다 — 한 번의 성공이 중요하면 pass@k, 일관성이 필수면 pass^k.

## 5. 제로에서 원까지: 훌륭한 에이전트 평가 로드맵

### Step 0 — 일찍 시작하라
수백 개 태스크가 있어야 한다는 생각으로 미루지 마라. 실제 실패에서 뽑은 **간단한 태스크 20~50개**면 충분히 강력하게 출발한다. 초기 변경은 효과가 크고 뚜렷해 작은 표본으로도 잡힌다. 성숙한 에이전트일수록 더 크고 어려운 eval이 필요. 처음엔 80/20으로 접근하라. **미룰수록 eval 만들기는 더 어려워진다** — 초기엔 제품 요구가 자연스레 테스트 케이스가 되지만, 늦으면 라이브 시스템에서 성공을 역설계해야 한다.

### Step 1 — 이미 수동으로 테스트하는 것에서 시작하라
릴리스 전마다 돌리는 수동 체크와 흔한 최종 사용자 태스크로 시작. 프로덕션이라면 버그 트래커와 지원 큐를 본다. 사용자 보고 실패를 테스트 케이스로 전환하면 스위트가 실제 사용을 반영하고, 영향도로 우선순위를 매기면 중요한 곳에 노력이 집중된다.

### Step 2 — 참조 해법이 있는 모호하지 않은 태스크를 써라
좋은 태스크 품질 = **전문가 2명이 독립적으로 같은 합/불 판정**에 도달. 스스로 풀 수 있는가? 아니라면 다듬어야 한다. 태스크 명세의 모호함은 곧 지표의 노이즈가 된다.

각 태스크는 지시를 따르는 에이전트라면 통과 가능해야 한다. (미묘한 예: 파일 경로 없이 스크립트를 짜라고 하면서 테스트는 특정 경로를 가정 → 에이전트가 잘못 없이 실패.) grader는 태스크가 명시한 것만 확인해야 하며, 모호한 명세로 실패시켜선 안 된다.

프론티어 모델이 pass@100에서 0%(여러 trial)라면 **에이전트가 무능한 게 아니라 태스크가 깨진 신호** — 명세와 grader를 재점검하라. 각 태스크에 **참조 해법**(모든 grader를 통과하는 검증된 출력)을 만들어 풀이 가능성과 grader 설정을 동시에 증명한다.

### Step 3 — 균형 잡힌 문제 집합을 만들어라
행동이 **일어나야 하는** 케이스와 **일어나면 안 되는** 케이스를 모두 테스트한다. 한쪽으로 치우친 eval은 한쪽으로 치우친 최적화를 낳는다. 클래스 불균형을 피하라.

**Claude.ai 웹검색 예시**: 검색이 필요 없을 때 막으면서(예: 애플 창업자) 필요할 땐 보존하는 것(예: 날씨)이 과제. 언더트리거 vs 오버트리거의 균형을 잡기 위해 프롬프트와 eval을 여러 번 다듬었고, 새 문제가 생길 때마다 커버리지를 계속 확장.

### Step 4 — 안정적 환경의 견고한 eval 하네스를 구축하라
eval 속 에이전트가 프로덕션과 거의 동일하게 작동하고, 환경이 추가 노이즈를 넣지 않아야 한다. **각 trial은 깨끗한 환경에서 시작.** 불필요한 공유 상태(남은 파일, 캐시, 자원 고갈)는 인프라 불안정으로 인한 상관 실패를 부른다. 공유 상태는 성능을 인위적으로 부풀린다(예: Claude가 이전 trial의 git 히스토리를 참고). 같은 환경적 제약(CPU 메모리 부족)으로 여러 trial이 실패하면 서로 독립적이지 않아 결과가 신뢰할 수 없게 된다.

### Step 5 — 채점기를 신중히 설계하라
가능하면 결정적, 필요하면 LLM, 검증엔 사람을 신중히. 흔한 본능은 에이전트가 **아주 특정한 단계 순서**를 따랐는지 확인하는 것 — 너무 경직됐다. 에이전트는 설계자가 예상 못한 유효한 접근을 자주 찾는다. **경로가 아니라 산출물(결과)을 채점**하고 창의성을 벌하지 마라.

다중 구성 태스크엔 **부분 점수(partial credit)**를 둔다. 문제를 정확히 파악하고 고객을 확인했지만 환불 처리에 실패한 지원 에이전트는 즉시 실패보다 의미 있게 낫다 — 이 연속성을 반영하라.

모델 채점은 정확도 검증을 위한 신중한 반복이 필요. LLM-as-judge를 사람 전문가와 밀접히 보정하고, LLM에 **탈출구**(정보가 부족하면 "Unknown" 반환)를 주며, 차원별로 나눈 명확한 루브릭으로 각 차원을 **분리된 LLM 심판**으로 채점하면 할루시네이션을 피할 수 있다.

일부 eval은 성능이 좋아도 낮은 점수를 내는 **미묘한 실패 모드**(채점 버그, 하네스 제약, 모호함)가 있다.

- **CORE-Bench 사례**: Opus 4.5가 처음 42%. 연구자가 문제들을 발견 — "96.124991…"을 기대하며 "96.12"를 벌한 경직 채점, 모호한 명세, 재현 불가능한 확률적 태스크. 버그 수정 + 덜 제약된 스캐폴드 사용 후 **95%로 급등**.
- **METR time horizon 벤치마크 사례**: 명시된 임계값까지 최적화하라고 하면서 채점은 그걸 **초과**해야 통과하도록 잘못 설정 → 지시를 따른 Claude를 벌하고, 명시 목표를 무시한 모델이 더 높은 점수를 받는 역설.

grader는 우회·해킹에 강해야 한다. **통과하려면 실제로 풀어야지, 의도치 않은 허점을 악용해선 안 된다.**

### Step 6 — 트랜스크립트를 확인하라
많은 trial 트랜스크립트와 점수를 읽기 전엔 grader가 잘 작동하는지 알 수 없다. Anthropic은 트랜스크립트 뷰어에 투자하고 정기적으로 읽는다. 태스크가 실패하면 트랜스크립트는 에이전트가 진짜 틀렸는지, 아니면 grader가 유효한 해법을 반려했는지를 드러낸다. **실패는 공정해 보여야 한다** — 무엇을 왜 틀렸는지 명확해야. 점수가 안 오를 때 그게 에이전트 성능 탓임을 확신하려면 트랜스크립트를 읽어야 한다.

### Step 7 — 역량 평가의 포화(Saturation)를 감시하라
100% 점수는 회귀는 추적하지만 개선 신호는 못 준다. **Eval saturation** = 풀 수 있는 모든 태스크를 통과해 개선 여지가 사라진 상태.

- **SWE-Bench Verified 사례**: 올해 30%에서 시작해 프론티어 모델이 80%+로 포화에 접근. 포화에 가까울수록 진전이 느려지고(가장 어려운 것만 남음), 큰 역량 향상이 작은 점수 증가로 보여 결과가 오해를 부른다.
- **Qodo 코드 리뷰 사례**: 원샷 코딩 eval이 길고 복잡한 태스크의 이득을 못 잡아 처음엔 Opus 4.5에 시큰둥. 새 **에이전틱 eval 프레임워크**를 개발해 진전을 명확히 파악.

원칙: **누군가 eval 내부를 파고 트랜스크립트를 읽기 전까지 점수를 액면 그대로 믿지 마라.** 불공정 채점·모호한 태스크·유효 해법 감점·모델을 제약하는 하네스는 eval 개정 신호다.

### Step 8 — 개방적 기여와 유지보수로 스위트를 장기적으로 건강하게
eval 스위트는 지속적 관심과 명확한 소유권이 필요한 **살아있는 산출물**이다. Anthropic이 실험한 결과 가장 효과적인 방식: **전담 eval 팀이 핵심 인프라를 소유**하고, 도메인 전문가·제품팀이 대부분의 태스크를 기여하고 평가를 실행.

AI 제품팀에게 eval 소유·반복은 **단위 테스트 유지만큼 일상**이어야 한다. 팀은 "동작하는" 기능이 암묵적 기대를 못 채워 몇 주를 낭비하곤 하는데, 잘 설계된 eval이 이를 조기에 드러낸다.

**Eval-driven development(평가 주도 개발)**를 실천하라 — 에이전트가 아직 못하는 계획된 역량을 **먼저 eval로 정의**하고, 잘 될 때까지 반복. 오늘은 "그럭저럭" 만들되 몇 달 뒤 모델 역량에 베팅하는 경우가 많은데, 낮은 점수에서 시작하는 역량 eval이 이 베팅을 가시화한다. 새 모델이 나오면 스위트를 빠르게 돌려 어떤 베팅이 성공했는지 확인. 제품 요구·사용자에 가장 가까운 사람(PM·CSM·영업)이 Claude Code로 eval 태스크를 PR로 기여하게 하면 더 좋다.

## 6. 평가는 다른 방법들과 어떻게 맞물리는가

자동 평가는 프로덕션 배포·사용자 영향 없이 수천 개 태스크로 에이전트를 검증하는 **여러 이해 방법 중 하나**일 뿐이다. 완전한 그림은 프로덕션 모니터링, 사용자 피드백, A/B 테스트, 수동 트랜스크립트 리뷰, 체계적 사람 평가를 포함한다.

### 방법 비교표

| 방법 | 장점 | 단점 |
|---|---|---|
| **자동 평가** (실제 사용자 없이 프로그램적으로 테스트) | 빠른 반복, 완전 재현 가능, 사용자 영향 없음, 배포 없이 대규모 시나리오 테스트 | 초기 구축 투자 필요, 제품/모델 진화에 맞춘 유지보수 필요, 실사용과 안 맞으면 거짓 확신 위험 |
| **프로덕션 모니터링** (라이브 시스템의 지표·에러 추적) | 대규모 실사용 행동 노출, 합성 eval이 놓친 이슈 포착, 실제 성능의 ground truth | 반응형(문제가 사용자에 먼저 도달), 신호가 노이즈일 수 있음, 계측 투자 필요, 채점 ground truth 부재 |
| **A/B 테스트** (실제 트래픽으로 변형 비교) | 실제 사용자 결과(리텐션·완수율) 측정, 교란 통제, 확장·체계적 | 유의성 도달까지 며칠~몇 주로 느림, 배포된 변경만 테스트, 트랜스크립트 리뷰 없이는 "왜" 신호 약함 |
| **사용자 피드백** (thumbs-down, 버그 리포트 등) | 예상 못한 문제 노출, 실제 사람 사례 동반, 제품 목표와 상관 | 희소·자기선택적, 심각 이슈로 치우침, 사용자가 이유를 잘 설명 안 함, 비자동·사용자가 문제를 겪어야 함(부정적 영향) |
| **수동 트랜스크립트 리뷰** (사람이 대화를 읽음) | 실패 모드 직관 형성, 자동 체크가 놓친 미묘한 품질 이슈 포착, "좋음" 보정 | 시간 집약적·확장 안 됨, 커버리지 불균일, 리뷰어 피로/편차, 보통 정성 신호만 |
| **체계적 사람 연구** (훈련된 평가자의 구조화 채점) | 다수 평가자의 정답지 수준 판단, 주관·모호 태스크 처리, 모델 채점기 개선 신호 | 상대적으로 비싸고 느림, 자주 돌리기 어려움, 평가자 간 이견 조율 필요, 복잡 도메인(법률·금융·의료)엔 전문가 필요 |

**개발 단계별 매핑**: 자동 평가는 출시 전·CI/CD의 1차 방어선(모든 에이전트 변경·모델 업그레이드마다 실행). 프로덕션 모니터링은 출시 후 분포 드리프트·예상 못한 실패를 감지. A/B는 충분한 트래픽으로 중요한 변경을 검증. 사용자 피드백·트랜스크립트 리뷰는 상시 실천(피드백 상시 분류, 주간 트랜스크립트 샘플링). 체계적 사람 연구는 LLM 채점기 보정이나 주관적 출력 평가용으로 아껴 쓴다.

**스위스 치즈 모델(Swiss Cheese Model)**: 안전공학처럼, **어떤 단일 평가 층도 모든 이슈를 잡지 못한다.** 여러 방법을 겹치면 한 층이 놓친 실패를 다른 층이 잡는다.

## 7. 결론

eval 없는 팀은 반응형 루프에 갇힌다 — 한 실패를 고치면 다른 실패가 생기고, 회귀와 노이즈를 구분 못한다. 일찍 투자한 팀은 정반대다 — 실패가 테스트 케이스가 되며 개발이 가속되고, 테스트가 회귀를 막고, 지표가 추측을 대체한다. eval은 오를 언덕을 명확히 주어 "에이전트가 나빠진 것 같다"를 실행 가능한 신호로 바꾼다. **가치는 eval을 나중 일이 아니라 핵심 부품으로 취급할 때에만 복리로 쌓인다.**

패턴은 에이전트 유형마다 다르지만 기본은 불변이다: 완벽한 스위트를 기다리지 말고 일찍 시작하라 · 관찰된 실패에서 현실적 태스크를 뽑아라 · 모호하지 않고 견고한 성공 기준을 정의하라 · 여러 유형을 조합해 채점기를 신중히 설계하라 · 문제를 모델에게 충분히 어렵게 하라 · 신호 대 노이즈를 개선하며 반복하라 · **트랜스크립트를 읽어라!**

AI 에이전트 평가는 아직 초기의 빠르게 진화하는 분야다. 에이전트가 더 긴 태스크를 다루고, 멀티 에이전트로 협업하며, 더 주관적인 일을 맡을수록 적응 기법도 따라올 것이다.

## 부록: Eval 프레임워크

인프라를 처음부터 짜지 않아도 되게 돕는 오픈소스·상용 프레임워크들. 선택은 에이전트 유형, 기존 스택, 오프라인 평가/프로덕션 관측 중 무엇이 필요한지에 달렸다.

- **Harbor**: 컨테이너 환경에서 에이전트를 대규모 trial로 실행하는 인프라. 태스크·grader를 정의하는 표준 포맷. Terminal-Bench 2.0 등 인기 벤치마크가 Harbor 레지스트리로 배포돼 커스텀 스위트와 함께 돌리기 쉽다.
- **Braintrust**: 오프라인 평가 + 프로덕션 관측 + 실험 추적 결합. `autoevals` 라이브러리에 factuality·relevance 등 사전 제작 스코어러 포함.
- **LangSmith**: 트레이싱, 오프라인/온라인 평가, 데이터셋 관리. LangChain 생태계와 긴밀 통합.
- **Langfuse**: 유사 기능의 셀프호스트 오픈소스 대안. 데이터 레지던시 요건이 있는 팀에 적합.
- **Arize**: Phoenix(LLM 트레이싱·디버깅·오프라인/온라인 평가 오픈소스) + AX(Phoenix를 확장한 SaaS, 스케일·최적화·모니터링).

> 많은 팀이 여러 도구를 섞거나 커스텀 프레임워크를 만든다. 프레임워크는 진전을 가속하고 표준화하지만, **결국 그 위를 흐르는 eval 태스크만큼만 좋다.** 워크플로에 맞는 프레임워크를 고르고, 에너지는 **고품질 테스트 케이스와 grader를 반복 개선**하는 데 쏟는 것이 최선이다.
