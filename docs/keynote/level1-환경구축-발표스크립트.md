### S1. 커버 — LEVEL 1: 환경 구축과 클로드 코드 기본 설정 ⏱ ~45초
> 안녕하세요. **『클로드 코드 마스터』** 저자, 코드빌런입니다. 「초보자를 위한 클로드 코드 한방에 해결하기」에 오신 걸 환영합니다.
> 이번 강의는 **Level 1, '환경 구축과 클로드 코드 기본 설정'** 입니다. 여기서 우리는 클로드 코드를 내 컴퓨터에 설치하고, 터미널과 Git이라는 두 가지 기본 도구를 익힌 다음, effort나 model, 기본 설정들을 점검 해보고 난 뒤에 클로드 코드와 첫 대화를 나눠서 직접 'Hello, World!'를 출력하는 데까지 가 볼 겁니다. 우리가 나누는 대화를 입력하는 과정을 프롬프팅이라고 하고, 모델이 결과물을 리서치한 뒤 사고과정을 반복하는 루프와 도구를 호출하는 과정인 툴 콜링을 거쳐 출력하는 과정을 응답이라고 하는데, 이 하나의 과정을 turn 이라고 표현합니다. 턴, 세션, 그리고 컨텍스트로 이어지는게 개발 과정인데, 이 부분에 대해서 설명을 드려야, 공통적인 기본 지식을 이해하고 있다 라고 말씀 드릴 수 있습니다.이런 이해가 바탕이 되어야 나머지 설명들도 쉽게 접근할 수 있고, 듣는 수강자분들 입장에서도 기본적인 지식들이 공고히 쌓여서 내공이 되고, 어느 순간 깨달음의 순간이 오게 될 텐데 이때 가장 중요한게 기본 지식들이 얼마나 탄탄히 쌓여있는가? 가 영향을 미칩니다. 그래서 약속드리는 부분은 이 강의는 비전공자, 개발자가 아닌 분들도 따라올 수 있게 쉽게귑게 설명드리려고 하고, 이해가 안되는 부분은 채팅방이나 커뮤니티 (run-ai.kr)에서  질문을 남겨주시거나 인프런 게시판에 문의 주시면 답변을 드리도록 하겠습니다. 또 기본 과정이기 때문에 이번 Level1 수업은 **터미널을 한 번도 안 써 본 분**을 기준으로 만들었습니다. 처음 보는 용어가 나오면 그때그때 풀어서 설명할 예정이고, 간략하게나마 히스토리나 유래에 대해서도 언급하고 넘여가려고 합니다. 동영상 강의 특성상 장황하게 늘리면 오히려 독이 되니까 최대한 짧게 핵심만 언급하고 넘어갈 예정이고, 강의 자체는 자세히, 초보자도 이해할 수 있게 풀어서 설명할 예정이니, 모른다고 멈추지 마시고 끝까지 따라오시면 됩니다.

### S2. 오프닝 — 무엇을·왜·어떻게 배우나 ⏱ ~70초
> 본격적으로 들어가기 전에 이 레벨의 큰 그림을 '왜·무엇을·어떻게' 세 가지로 짚겠습니다.
> 먼저 **왜** 배우느냐. 클로드 코드는 흔히 생각하는 '웹 채팅 AI'나 'IDE 안의 플러그인'이 아니라, **터미널에서 직접 도는 AI 코딩 에이전트**입니다. 여기서 '에이전트(agent)'란, 내가 시키면 스스로 파일을 읽고 고치고 명령을 실행하는 '대리 작업자'라는 뜻이에요. LLM이랑은 다른 개념입니다. 이 에이전트를 검색엔진처럼 쓰시는 분들도 계시는데, 뭐 잘못된건 아닌데, 10분의 1도 활용하지 못하는거 아닐까 싶습니다. 진가는 자율성, 스스로 실행하는 능력이고 특히나 코덱스와 클로드 코드는 개발에 특화된 터미널 베이스의 도구입니다. 그래서 터미널·Git·권한 같은 기초 환경이 약하면, 뒤에 나올 CLAUDE.md나 Skill, Hook 같은 고급 기능이 전부 헛돕니다. 그래서 1단계, 사람으로 치면 기초 체력을 단단히 다지는 과정이라고 이해하시면 됩니다.
> **무엇을** 배우느냐. 다섯 단계입니다. 1.1 터미널과 Git허브, 1.2 클로드 코드 설치와 모델 선택, 1.3 개발환경과 권한 모드, 1.4 설정 파일, 1.5는 대망의 첫 세션이에요. 모든 프로그래밍 언어의 기본은 Hello World를 출력해보는게 밈, 필수 코스? 라고 할 수 있습니다. 이 헬로 월드는 옛날에 벨 연구소의 B언어 튜토리얼 메모에서 처음 등장했다고 하구여, 현재 프로그래밍 언어의 시작점이 The C Programming Language의 첫 예제로 실리면서 모든 언어의 입문서에 쓰이는 관례(컨벤션) 라고 보시면 됩니다. 이 헬로 월드를 출력해야 개발의 시작, 개발자의 입문 과정 첫걸음을 띄었다고 할 수 있습니다. 
기본적으로 타입스크립트, 파이썬으로 하나씩 작성해서 실습해볼 예정입니다. 스프링은 설정이 살짝 어렵기 때문에 초보 과정에서는 제외하고 이 두개 언어만 실습해보도록 하겠습니다. 

> **어떻게** 배우느냐. 무작정 외우지 않습니다. 명령어를 **어원으로 이해**합니다. 예를 들어 `ls`는 LiSt(목록), `rm`은 ReMove(제거), `pwd`는 Print Working Directory(현재 위치 출력)의 줄임말이에요. 이렇게 뜻을 알고 → 직접 쳐 보고 → 실습으로 내 것으로 만드는 흐름입니다.
> 이번 레벨의 **체크포인트** 한 줄은 이겁니다. "설치와 Git·터미널을 이해하고, 클로드 코드로 Python과 TypeScript로 첫 Hello, World!를 출력한다."

개발 툴도 물론 배우지만 기본적으로 깃과 터미널 환경이 중요하기 때문에 강의 시간을 좀 더 할애하는 것이라고 이해하시면 되고, 개발툴에서 어떤 버튼을 눌러야 한다, 이 기능은 뭐다 이런 개념도 중요하지만 결국은 클로드 코드, 코덱스, 제미나이나 안티그래비티, 커서등의 에이전트 기반 코딩 툴들은 에이전트를 어떻게 세팅하고 호출하는가? 프롬프팅을 어떻게 해야 하는가? 컨텍스트를 어떻게 핸들링 해야 하는가? 이런 방법론을 배워야 하나를 깨닫게 되면 다른 에이전트들도 그대로 적용할 수 있습니다. 개발툴은 계속 업데이트가 될 거라서 개발툴 기준으로만 설명하면 일년도 안되서 쓸모없는 지식이 될 가능성이 큽니다.
클로드 코드만 해도 지난 26년 5개월간 업데이트 된 기능들이 과거 25년 1년간 업데이트 된 내용보다 많습니다. 앞으로 더 많아 질것이구요. 
그럼 툴만 배워서는 답이 없고 강의도 계속 업데이트 해야 합니다. 이 과정은 에이전트를 어떻게 다루느냐, 에이전트 특성이 어떻게 되고 어떤 방식으로 동작하느냐, 전체 과정 마무리가 되면 클로드 코드 동작 과정을 한번 더 정리할 텐데, 물론 중간중간 클로드 코드의 동작 과정을 설명을 드릴 텐데, 아무튼 클로드 코드 딥다이브를 통해 뜯어보면서 에이전트 내부의 동작 방식을 이해하면서 도구가 바뀌어도 그대로 활용할 수 있는 근간 지식들을 배우는데 초점을 두었습니다. 이론, 따라하기, 실습과정 이런 싸이클을 배우면서 내것으로 온전히 체득하고 각 과정 마다 실습을 통해 과정에서 배운 중요 내용들을 다시 한번 되새김해서 공고히 하는 단계를 반복해 나가면서 유레카 모먼트라고 하죠? 아.. 이건 이렇게 동작하는구나, 이렇게 써야 맞는거구나 이런 번득이는 이해의 순간들을 빠르게 경험할 수 있도록 도와드릴 예정입니다.

## PART 01 — 1.1 터미널과 Git 명령어의 이해

### S4. 섹션 표지 · PART 01 ⏱ ~12초
> 첫 번째 파트, 터미널과 Git입니다. 이 챕터에서 배우는 건 딱 두 가지예요. **하나, 터미널 기본 명령어. 둘, GitHub 가입부터 Git 명령어까지.** 클로드 코드가 일하는 '작업장'을 먼저 정리하는 단계라고 보시면 됩니다.

### S5. 1.1.1 Windows 터미널 설정 — WSL 설치 ⏱ ~60초
> 윈도우 사용자부터 챙기겠습니다. 윈도우는 맥·리눅스와 명령어가 조금 다른데, **WSL**을 깔면 윈도우 안에서 리눅스를 그대로 쓸 수 있습니다. WSL은 'Windows Subsystem for Linux', 즉 '윈도우 안에 들어가는 리눅스 환경'이라는 뜻이에요. 이걸 깔면 맥 사용자와 똑같은 명령어를 쓸 수 있어 강의 따라오기가 훨씬 쉬워집니다.
> 순서는 네 단계입니다. ① 작업 관리자(Ctrl+Shift+Esc) 성능 탭에서 '가상화: 사용'을 확인하고, 꺼져 있으면 BIOS에서 켭니다. ② PowerShell을 **관리자 권한**으로 열고 `wsl --install` 한 줄을 칩니다. ③ 재부팅한 뒤 Ubuntu를 처음 실행해 사용자 이름과 비밀번호를 정합니다. 이때 **비밀번호는 화면에 아무것도 안 보이는 게 정상**이에요. 그냥 치고 Enter 누르시면 됩니다. ④ `sudo apt update && sudo apt upgrade -y`로 최신화하면 끝입니다.
> 윈도우용 클로드 코드 본 설치는 1.2에서 다시 자세히 다루니, 지금은 'WSL부터 깐다'만 기억하세요.

### S6. 1.1.2 Mac 터미널 설정 — iTerm2 ⏱ ~55초
> 맥 사용자는 기본 '터미널' 앱도 되지만, **iTerm2**라는 무료 터미널을 권합니다. 탭·분할·검색·자동완성이 다 되는 더 강력한 터미널이에요.
> 설치는 iterm2.com에서 받거나, 패키지 관리자 Homebrew가 있다면 `brew install --cask iterm2` 한 줄이면 됩니다. 설치 후 두 가지만 켜 두세요. 첫째, **Unlimited scrollback** — 긴 빌드 로그나 테스트 출력을 끝까지 스크롤해서 볼 수 있게 해 줍니다. 둘째, **Natural Text Editing** — Option+화살표로 단어 단위 이동, Cmd+화살표로 줄 처음·끝 이동이 가능해져, 다른 맥 앱과 똑같은 감각으로 커서를 움직일 수 있어요.
> 더 예쁘게 쓰고 싶으면 Powerlevel10k 테마도 선택입니다만, 필수는 아닙니다.

### S7. 1.1.3 분류별 명령어 (1/3) — 어원으로 외우기 ⏱ ~60초
> 이제 터미널 명령어인데, 외우지 말고 **어원**으로 이해합시다. 1970년대 유닉스 시절에 한 글자라도 줄이려고 만든 약어들이거든요.
> 위치·이동은 `pwd`(Print Working Directory, 내 위치 출력), `ls`(LiSt, 폴더 내용 보기), `cd`(Change Directory, 폴더 이동). 생성은 `mkdir`(MaKe DIRectory, 새 폴더), `touch`(빈 파일 만들기). 복사·이동·삭제는 `cp`(CoPy), `mv`(MoVe·이름변경 겸용), `rm`(ReMove, **영구 삭제** — 휴지통 안 거칩니다, 주의!). 내용 보기는 `cat`(conCATenate, 전체), `less`(페이지 단위), `head`·`tail`(앞·뒷부분). 검색은 `grep`(패턴 검색), `find`(파일 찾기), `man`(MANual, 매뉴얼). 그리고 `sudo`(SuperUser DO, 관리자 권한으로 실행).
> 다 외울 필요 없어요. '약어에는 다 영어 뜻이 있다'는 감각만 가지시면, 처음 보는 명령어도 덜 무섭습니다.

### S8. 1.1.3 분류별 명령어 (2/3) — 실전 예시 ⏱ ~40초
> 방금 명령어를 실제로 치면 이렇게 됩니다. `pwd`를 치면 내 위치가 나오고, `ls`로 폴더 안을 봅니다. `cd Documents`로 폴더에 들어가고, `cd ..`는 한 단계 위로, `cd ~`는 홈 폴더로 돌아갑니다. `mkdir -p a/b/c`는 중간 폴더까지 한 번에 만들고, `grep "error" server.log`는 로그에서 error만 찾아 줍니다. `tail -f server.log`는 로그가 쌓이는 걸 **실시간으로** 보여 줘서, 서버 띄워 놓고 지켜볼 때 자주 씁니다.

### S9. 1.1.3 옵션 플래그 (3/3) — `-r` · `-f` · `-a` ⏱ ~55초
> 명령어 뒤에 붙는 `-` 한 글자, 이걸 '옵션' 또는 '플래그(flag)'라고 합니다. 이것도 다 영어 약어예요. `-r`은 Recursive(폴더 안쪽까지), `-f`는 Force(묻지 말고 강제로), `-a`는 All(전부). 그리고 옵션은 **붙여 쓸 수 있습니다**. `ls -l -a`와 `ls -la`는 같은 뜻이에요.
> 여기서 가장 조심할 명령이 `rm -rf`입니다. 분해하면 'Remove(지워라) + Recursive(폴더 안쪽까지) + Force(묻지 말고)', 즉 **"묻지도 따지지도 말고 폴더 통째로 다 지워라"** 입니다. 휴지통도 안 거쳐서 복구가 안 돼요. 그래서 실수 방지로 `~/.zshrc`에 `alias rm='rm -i'`를 넣어 두면, 지울 때마다 한 번 물어봐 줍니다.
> 참고로 클로드 코드는 이런 위험 명령을 **권한 모드와 Hook으로 막을 수 있는데**, 그건 1.3과 1.4에서 다룹니다.

### S10. 1.1.4 터미널 단축키 (1/3) ⏱ ~20초
> 단축키도 약어로 외웁니다. `Cmd+F`는 Find(터미널 출력 안에서 검색), `Cmd+T`는 Tab(새 탭), `Cmd+N`은 New(새 창). 손이 마우스로 안 가도 되게 해 주는 것들이에요.

### S11. 1.1.4 터미널 단축키 (2/3) ⏱ ~55초
> 여기가 진짜 실무에서 손에 익혀야 할 단축키입니다. 커서 이동은 `Ctrl+A`(Ahead, 줄 맨 앞), `Ctrl+E`(End, 줄 맨 끝). 삭제는 `Ctrl+U`(커서 앞 전부), `Ctrl+K`(Kill, 커서부터 끝까지), `Ctrl+W`(Word, 앞 단어 하나).
> 그리고 꼭 기억할 두 개. **`Ctrl+C`는 Cancel — 실행 중인 명령을 강제로 멈추는 비상 정지 버튼**입니다. 뭔가 멈추지 않고 돌면 이걸 누르세요. `Ctrl+L`은 cLear, 화면만 깨끗이 지웁니다(기록은 남아요). `Ctrl+R`은 Reverse search, 예전에 친 명령어를 거꾸로 검색해 줘서 위쪽 화살표 연타보다 훨씬 빠릅니다.

### S12. 1.1.4 단축키 (3/3) — 실전 워크플로우 ⏱ ~45초
> 실제 상황으로 묶어 볼게요. 긴 명령어를 치다 앞부분에 오타를 발견하면 `Ctrl+A`로 줄 맨 앞으로 점프해서 고칩니다. 예전에 친 `git push`를 다시 쓰고 싶으면 `Ctrl+R` 누르고 'git push'만 쳐서 찾습니다. 개발 서버가 무한 루프에 빠지거나 멈추면 `Ctrl+C`로 강제 종료합니다. iTerm2에서 Natural Text Editing을 켰다면 Option+화살표로 단어 단위, Cmd+화살표로 줄 처음·끝 이동까지 됩니다.

### S13. 1.1.5 GitHub 가입 · Repository · PAT 발급 ⏱ ~65초
> 이제 **GitHub**입니다. GitHub은 '코드를 저장하고 공유하는 클라우드 창고'라고 보시면 돼요. 여기서 새 용어 두 개가 나옵니다. **Repository(리포지토리)**는 '프로젝트 하나가 들어가는 저장소(폴더)'고, **PAT(Personal Access Token)**는 '비밀번호 대신 쓰는 개인 접근 열쇠'입니다.
> 순서는 네 단계예요. ① github.com에서 Sign up으로 가입하고, 보안을 위해 2단계 인증(2FA)을 켭니다. ② New repository로 첫 저장소를 만듭니다. 이때 .gitignore 템플릿과 라이선스(MIT 권장)를 같이 고를 수 있어요. ③ Settings → Developer settings에서 **PAT를 발급**합니다. 만료일은 90일을 권하고, **발급 직후 바로 복사해 두세요. 다시 못 봅니다.** ④ 한 번 입력한 인증 정보는 OS 키체인에 저장돼서, 그다음부터는 자동으로 처리됩니다.

### S14. 1.1.6 Git 기본 사이클 (1/5) — 명령어 표 ⏱ ~60초
> Git의 핵심 명령을 한 바퀴 돌려 보겠습니다. Git은 '코드의 변경 이력을 기록하는 타임머신'이라고 생각하세요.
> `git clone`은 원격 저장소를 내 컴퓨터로 복제(처음 프로젝트 받기). `git checkout`은 다른 작업 갈래(브랜치)로 이동. `git fetch`는 원격에 뭐가 바뀌었는지 '확인만'(내 코드는 안 건드림). `git pull`은 그 변경을 실제로 '내려받아 합치기'. `git commit`은 작업 내용을 저장하는 **세이브 포인트** — 게임 저장하듯 체크포인트를 찍는 겁니다. `git merge`는 두 갈래를 합치기, `git push`는 내 작업을 원격 창고로 올리기. 이 일곱 개가 매일 쓰는 기본기입니다.

### S15. 1.1.6 Git 최초 사이클 — 터미널 실습 ⏱ ~55초
> 실제 흐름을 보겠습니다. 시작은 두 경로예요. 이미 있는 저장소를 받으면 `git clone <주소>`, 빈 폴더에서 새로 시작하면 `git init`으로 'Git 저장소로 초기화'한 뒤 `git remote add origin <주소>`로 원격을 연결합니다. 여기서 `origin`은 '내 기본 원격 저장소의 별명'이에요.
> 그다음은 매번 반복되는 4단계입니다. `git status`로 뭐가 바뀌었나 보고 → `git add .`로 변경을 담고(스테이징) → `git commit -m "메시지"`로 세이브하고 → `git push origin main`으로 원격에 올립니다. 남의 변경을 받을 땐 `git pull`이고요. 이 리듬이 몸에 배면 Git의 절반은 끝난 겁니다.

### S16. 1.1.6 Git (2/5) — init·clone·remote·branch ⏱ ~50초
> 저장소와 브랜치 명령을 정리합니다. `git init`은 현재 폴더를 Git 저장소로 만들고, `git remote add origin <주소>`로 원격을 등록, `git remote -v`로 등록된 원격을 확인합니다.
> **브랜치(branch)**는 '본 줄기에서 갈라져 나온 작업 가지'예요. 기능을 따로 만들다가 검증되면 본 줄기에 합치는 식이죠. `git branch`로 목록을 보고(별표가 현재 위치), `git branch <이름>`으로 새 가지를 만들고, `git branch -d`로 지웁니다. 브랜치를 제대로 쓰는 협업 흐름은 **Level 4 실전 워크플로우**에서 더 깊게 다룹니다.

### S17. 1.1.6 Git (2/5) — checkout·switch ⏱ ~45초
> 가지를 옮겨 타는 명령입니다. 전통적으로는 `git checkout <브랜치>`를 썼는데, 이게 '브랜치 이동'과 '파일 되돌리기'를 둘 다 해서 헷갈렸어요. 그래서 Git 2.23 버전부터 **`git switch`**가 새로 나왔습니다. `git switch main`은 브랜치 이동, `git switch -c <이름>`은 새 브랜치를 만들면서 바로 이동. 요즘은 이동할 땐 `switch`를 쓰는 걸 권합니다.

### S18. 1.1.6 Git (3/5) — add·commit·status·diff ⏱ ~50초
> 커밋할 때마다 반복되는 4단계입니다. `git add <파일>`로 저장할 변경을 고르는 걸 **스테이징(staging)**이라고 해요. '커밋에 담을 것들을 장바구니에 담는다'고 생각하면 됩니다. `git add .`은 전부 담기. `git commit -m "메시지"`로 세이브하고, `--amend`는 방금 한 커밋 메시지를 고칠 때(단, push 전에만 안전). `git status`는 매번 먼저 확인하는 습관, `git diff`는 정확히 어떤 줄이 바뀌었는지 보여 줍니다.

### S19. 1.1.6 Git (3/5) — 변경→커밋 흐름 실습 ⏱ ~35초
> 흐름을 그대로 따라가면, `git status`로 바뀐 파일을 보고 → `git diff`로 내용 확인 → `git add -p`로 변경을 조각(청크) 단위로 골라 담고 → `git commit`으로 저장 → `git log --oneline --graph`로 결과를 한눈에 확인합니다. 여기서 `git add -p`의 `-p`는 부분(patch) 단위로 골라 담게 해 주는 옵션이라, 한 커밋에 한 가지 변경만 깔끔하게 담고 싶을 때 유용합니다.

### S20. 1.1.6 Git (4/5) — fetch·pull·push ⏱ ~45초
> 원격과 동기화하는 3종 세트입니다. **`git fetch`는 가져오기만(합치지 않음) — 가장 안전**합니다. **`git pull`은 가져와서 자동으로 합치기**(fetch + merge). 협업할 땐 이력이 깔끔해지는 `git pull --rebase`를 권하기도 합니다. **`git push`는 내 커밋을 원격으로 보내기**. 첫 push 때 `git push -u origin <브랜치>`로 추적을 걸어 두면, 그다음부터는 그냥 `git push`만 쳐도 됩니다.

### S21. 1.1.6 Git (5/5) — merge·tag ⏱ ~55초
> 마무리 단계, 합치기와 버전 표시입니다. `git merge <브랜치>`로 가지를 현재 줄기에 합칩니다. 합치다 같은 줄을 양쪽이 고쳤으면 **충돌(conflict)**이 나는데, 충돌 표시를 정리하고 `git add` 후 `git commit`하면 해결됩니다. 잘못됐으면 `git merge --abort`로 합치기를 통째로 취소할 수 있어요.
> **태그(tag)**는 'v0.1.0' 같은 **버전 이름표**입니다. 배포할 때 "이 시점이 1.0 릴리스다" 하고 표시하는 거예요. `git tag -a v0.1.0 -m "MVP 릴리스"`로 만들고, `git push origin --tags`로 원격에 올립니다.

### S22. 1.1.6 Git (5/5) — merge 실습 흐름 ⏱ ~40초
> 실제 합치기는 보통 이 순서예요. ① 먼저 로컬 main을 최신으로 당겨 옵니다(`git pull`). ② 내 기능 브랜치로 가서 main을 흡수합니다(`git merge main`). 여기서 충돌이 나면 미리 해결해 두는 거죠. ③ 다시 main으로 돌아가 기능 브랜치를 합칩니다. 이미 정리됐으니 깔끔하게 들어갑니다. **"충돌은 내 브랜치에서 미리 처리하고, main은 마지막에 깔끔하게 합친다"** — 이 순서가 실무 안전 패턴입니다.

### S23. 섹션 표지 · PART 02 ⏱ ~12초

### S24. 1.2 Claude Code 개요 ⏱ ~60초
> 클로드 코드가 뭔지 한 문장으로 정리하면, **Anthropic이 만든 '터미널 네이티브 AI 코딩 에이전트'**입니다. '터미널 네이티브'란 브라우저 챗봇이나 IDE 플러그인이 아니라 터미널에서 직접 돈다는 뜻이고요. 그래서 내 컴퓨터의 파일을 직접 읽고 쓰고, 셸 명령을 실행하고, Git을 다룹니다.
> 동시에 VS Code·IntelliJ 같은 **IDE 플러그인**으로도 쓸 수 있고, 웹·모바일과도 연동됩니다. 그리고 **CLAUDE.md**라는 파일로 프로젝트의 규칙·맥락을 세션이 바뀌어도 기억하게 만들 수 있어요. 또 **MCP(Model Context Protocol)**라는 표준으로 GitHub 같은 외부 서비스와 연결됩니다. MCP는 'AI가 외부 도구와 대화하는 공통 규격'인데, 깊은 내용은 **Level 3**에서 다룹니다. 지금은 'AI가 직접 손발을 움직이는 도구'라는 감만 잡으세요.

### S25. 1.2.1 macOS — Homebrew 설치 ⏱ ~45초
> 맥 설치부터 갑니다. 먼저 사전 준비물 두 가지, Node.js와 Git이 필요해요. **Node.js(노드)**는 자바스크립트 실행 환경인데, 클로드 코드 같은 도구가 도는 바탕입니다. v22 버전을 권합니다.
> 그리고 **Homebrew(홈브루)**를 깝니다. Homebrew는 '맥용 앱 설치 관리자'예요. 윈도우의 '설치 마법사'를 명령어 한 줄로 대신한다고 보시면 됩니다. 화면의 `curl` 한 줄을 붙여넣어 설치하고, `brew --version`으로 확인한 뒤 `brew install node`로 Node를 깔면 됩니다.

### S26. 1.2.1 macOS — Git 설치 ⏱ ~40초
> Git이 아직 없다면 두 가지 방법이 있어요. `xcode-select --install`로 애플이 주는 개발 도구 묶음(Git 포함)을 깔거나, Homebrew로 `brew install git` 하면 됩니다.
> 설치 후 **딱 한 번** 본인 정보를 등록하세요. `git config --global user.name`과 `user.email`이에요. 이건 커밋에 "누가 작업했는지" 이름표를 붙이는 설정이라, 안 해 두면 나중에 커밋할 때 잔소리를 듣습니다.

### S27. 1.2.1 macOS — 설치 스크립트 (자동 경로) ⏱ ~45초
> 자, 클로드 코드 본 설치입니다. 맥·리눅스는 이 한 줄이면 끝나요. `curl -fsSL https://claude.ai/install.sh | bash`. 이건 'Anthropic 공식 서버에서 설치 스크립트를 받아 바로 실행하라'는 뜻입니다.
> 설치 후 `claude --version`을 쳤을 때 버전(예: 2.1.11)이 안 나오면, 셸 설정을 다시 읽어 주면 됩니다. zsh 쓰면 `source ~/.zshrc`, bash 쓰면 `source ~/.bashrc`예요. 또는 터미널을 껐다 켜도 됩니다.

### S28. 1.2.1 macOS — PATH 강제 설정 ⏱ ~55초
> 그런데 가끔 `claude`를 쳐도 "명령을 못 찾겠다"고 나올 때가 있어요. 이건 **PATH** 문제입니다. PATH는 '컴퓨터가 명령어 프로그램을 찾으러 돌아다니는 폴더 목록'이에요. 클로드 코드가 깔린 폴더가 이 목록에 없으면 못 찾는 거죠.
> 해결은 그 폴더(`~/.local/bin`)를 목록에 직접 추가하는 겁니다. `echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc`로 한 줄 넣고, `source ~/.zshrc`로 적용한 뒤 `claude --version`으로 확인하면 됩니다. 설치 스크립트가 이걸 자동으로 못 잡았을 때만 쓰는 비상 처치예요.

### S29. 1.2.2 Windows — PowerShell 공식 스크립트 ⏱ ~40초
> 윈도우는 두 가지 길이 있습니다. 먼저 윈도우 네이티브 설치예요. **PowerShell을 관리자 권한으로** 열고 `irm https://claude.ai/install.ps1 | iex` 한 줄을 칩니다. `irm`은 다운로드, `iex`는 실행이라고 보시면 돼요.
> 설치가 끝나면 'Location'에 경로가 표시되는데, 이 경로를 복사해 두세요. 다음 슬라이드에서 PATH에 등록할 때 씁니다. 그다음 **새** PowerShell 창에서 `claude --version`으로 확인합니다.

### S30. 1.2.2 Windows — 환경 변수 Path 설정 ⏱ ~45초
> 윈도우에서도 `claude`를 못 찾으면 PATH에 직접 등록해야 합니다. 마우스로 하는 방법이에요. `Win+R`을 누르고 `sysdm.cpl`을 입력 → [고급] 탭 → [환경 변수] → 사용자 변수에서 Path 선택 후 [편집] → [새로 만들기]에 아까 복사한 경로를 붙여넣고 확인. 그리고 **새** PowerShell 창에서 `claude --version`으로 확인합니다. 창을 새로 열어야 적용된다는 점만 기억하세요.

### S31. 1.2.2 Windows — Path 설정 화면 ⏱ ~12초
> (화면) 방금 말씀드린 환경 변수 창입니다. 빨간 박스 순서대로 따라가시면 돼요. 이런 GUI 설정이 번거롭다면, 다음에 소개할 **WSL** 방식이 오히려 더 깔끔합니다.

### S32. 1.2.2 Windows — WSL2로 설치 (개요) ⏱ ~55초
> 두 번째 길이자 더 권하는 방식, **WSL2**입니다. 윈도우 안에 리눅스를 넣고 그 안에서 클로드 코드를 까는 거예요. 명령어가 맥·리눅스와 똑같아져서 강의 따라오기 훨씬 편합니다.
> 사전 조건만 짚을게요. 윈도우 10(1809 이상) 또는 11이 필요하고, 메모리는 개발용으로 8GB 이상을 권합니다. 그리고 **유료 플랜(Pro·Max·Team·Enterprise)이 필요**합니다 — 무료 Claude.ai 플랜으로는 클로드 코드를 못 써요. 네이티브 설치 방식은 Node.js 없이 독립 실행되니 그것도 부담이 적습니다.

### S33. 1.2.2 Windows — WSL2 설치 명령 ⏱ ~35초
> 설치는 관리자 PowerShell에서 `wsl --install` 한 줄, 그리고 재부팅 한 번이면 됩니다. 이걸로 WSL2와 Ubuntu가 같이 깔려요. 깔린 뒤 `wsl -l -v`로 확인했을 때 **VERSION이 2**여야 합니다. 혹시 1로 나오면 `wsl --set-default-version 2`로 바꿔 주세요. 이미 깔려 있으면 `wsl --update`로 최신화합니다.

### S34. 1.2.2 Windows — Ubuntu 초기 설정 ⏱ ~40초
> 재부팅 후 Ubuntu가 처음 뜨면, **리눅스용** 사용자 이름과 비밀번호를 새로 만듭니다. 이건 윈도우 계정과는 무관하고, 앞으로 `sudo`(관리자 권한) 명령 쓸 때 입력할 비밀번호예요. 앞에서처럼 비밀번호는 화면에 안 보이는 게 정상입니다.
> 이후 진입은 Windows Terminal의 Ubuntu 탭으로 합니다. 그리고 설치 전에 `sudo apt update && sudo apt upgrade -y`로 패키지를 최신화해 두세요.

### S35. 1.2.2 Windows — WSL 안에서 설치 (핵심) ⏱ ~50초
> 여기가 윈도우 WSL 설치의 핵심입니다. **WSL 안의 Ubuntu는 윈도우와 별개인 독립 리눅스**라서, 앞서 윈도우 네이티브용으로 쓰던 `install.ps1`은 여기서 쓰면 안 됩니다. 반드시 **Ubuntu 터미널 안에서** 리눅스용 스크립트로 깔아야 해요.
> 명령은 맥과 똑같습니다. `curl -fsSL https://claude.ai/install.sh | bash`. 설치되면 `~/.local/bin/claude`에 들어가고, 자동 업데이트도 지원합니다. 참고로 `npm install -g` 방식도 되지만, **`sudo npm install -g`는 금지**예요 — 권한 꼬임의 원인이 됩니다.

### S36. 1.2.2 Windows — WSL PATH 설정 ⏱ ~40초
> WSL에서도 PATH는 대부분 자동으로 잡힙니다. 다만 설치 직후 같은 창에서는 아직 갱신이 안 돼 "command not found"가 날 수 있는데, 터미널을 새로 열거나 `source ~/.bashrc` 하면 됩니다.
> 설치 로그에 "~/.local/bin is not in your PATH" 경고가 보이면, `echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc` 한 줄을 직접 넣어 주세요. 큰따옴표 안에서는 `~` 대신 `$HOME`을 쓰는 게 더 안전합니다.

### S37. 1.2.2 Windows — WSL 검증 & 개발 팁 ⏱ ~50초
> 설치가 잘 됐는지 점검하는 단계입니다. `which claude`로 위치를 보고, `claude --version`으로 버전을 확인하고, **`claude doctor`**로 설치·설정을 자동 진단합니다. 그리고 `claude`로 첫 실행하면 브라우저가 열리며 로그인하라고 합니다.
> 중요한 팁 하나. WSL에서는 코드를 윈도우 쪽 경로(`/mnt/c/...`)가 아니라 **리눅스 네이티브 경로(`~/`)**에 두세요. 윈도우 마운트 경로는 입출력이 느려서 작업이 답답해집니다. VS Code는 'WSL 확장'을 깔고 'Open Folder in WSL'로 열면 됩니다.

## PART 03 — 1.2.3~1.2.5 모델 선택 · Thinking · 컨텍스트

### S38. 섹션 표지 · PART 03 ⏱ ~12초
> 세 번째 파트는 **모델 선택과 Thinking 모드, 그리고 컨텍스트 윈도우**입니다. 같은 클로드 코드라도 어떤 '두뇌(모델)'를 쓰느냐에 따라 속도·비용·똑똑함이 달라집니다. 그 고르는 기준을 배웁니다.

### S39. 1.2.3 모델 3계열 — Opus·Sonnet·Haiku ⏱ ~50초
> 클로드에는 세 계열의 모델이 있어요. 사람 역할에 빗대면 이해가 쉽습니다. **Opus(오퍼스)는 시니어 아키텍트** — 가장 똑똑하지만 비싸고 조금 느립니다. 복잡한 설계·분석에 씁니다. **Sonnet(소넷)은 실무 개발자** — 성능과 비용의 균형이 좋아 일상 개발의 기본값입니다. **Haiku(하이쿠)는 민첩한 팀원** — 가장 빠르고 싸서, 간단한 작업이나 여러 일을 동시에 돌리는 서브에이전트에 적합합니다. 작업 성격에 맞춰 골라 쓰는 게 핵심이에요.

### S40. 1.2.3 2026년 5월 기준 모델 비교 ⏱ ~55초
> 숫자로 비교해 볼게요. **Opus 4.8**은 모델 ID가 `claude-opus-4-8`, 최대 출력 128K 토큰, 컨텍스트 100만(1M) 토큰입니다. **Sonnet 4.6**은 `claude-sonnet-4-6`, 출력 64K, 컨텍스트도 1M. **Haiku 4.5**는 `claude-haiku-4-5-20251001`, 출력 64K, 컨텍스트는 20만(200K)입니다.
> 비용은 입력·출력으로 나뉘는데, 출력 기준 Opus가 100만 토큰당 25달러, Sonnet 15달러, Haiku 5달러로 차이가 큽니다. 속도는 Haiku가 가장 빠르고요. **'토큰(token)'**이란 AI가 글을 처리하는 최소 단위인데, 대략 영어 한 단어가 1~2토큰, 한글은 글자당 더 많이 잡아먹습니다. 자세한 비용 절약은 **Level 6**에서 다룹니다.

### S41. 1.2.3 모델 출시 타임라인 & 특징 ⏱ ~60초
> 모델이 어떻게 발전해 왔는지 짧게 짚을게요. 2025년 9월 Sonnet 4.5, 10월 Haiku 4.5, 11월 Opus 4.5가 나왔고, 2026년 들어 Opus 4.6에서 **Adaptive Thinking**이 도입됐습니다. 그리고 4.7을 거쳐 **2026년 5월 28일 Opus 4.8이 현재 최신 플래그십**으로 나왔어요.
> Opus 4.8은 코딩 벤치마크 SWE-bench Verified에서 88.6%를 기록했고, 이전 버전보다 코드 결함을 놓치는 비율이 크게 줄고 '정직성'이 강화됐습니다 — 즉 모르는 걸 아는 척 단정하는 경향이 줄었다는 뜻이에요. Sonnet 4.6은 '예전 Opus급 성능을 Sonnet 가격에' 주는 모델이라 일상 작업의 기본 추천이고, Haiku 4.5는 경량 모델 중 최고 수준의 코딩 성능으로 서브에이전트·대규모 처리에 적합합니다.

### S42. 1.2.3 요금제 비교 — 5종 ⏱ ~45초
> 요금제는 다섯 가지예요. **Pro는 월 20달러**로 일반적인 사용에 충분하고, **Max는 100달러(Pro의 5배), 200달러(20배)**로 사용량이 많을 때, **Team은 1인당 30달러**로 팀 협업, **Enterprise는 별도 문의**입니다. 처음 시작하시는 분께 드리는 조언은 단순합니다. **일단 Pro로 시작하세요.** 쓰다 보면 본인 사용량이 보이고, 그때 올려도 늦지 않습니다.

### S43. 1.2.3 로그인 방식 — 구독 vs API 콘솔 ⏱ ~45초
> 클로드 코드를 처음 켜면 로그인 방식 두 가지 중 하나를 고릅니다. 하나는 **Claude.ai 구독** — 월 정액제라 사용량이 그 안에 포함됩니다. 다른 하나는 **Anthropic Console(API)** — 쓴 만큼 내는 종량제고, 미리 크레딧을 충전해야 해요. API는 자동화 파이프라인이나 개발용입니다.
> 입문자라면 고민할 것 없습니다. **월 정액 Pro로 시작**해서 사용량을 보고, 개인 사이드 프로젝트 정도면 Max 100도 괜찮습니다.

### S44. 1.2.3 용도별 모델 선택 가이드 ⏱ ~40초
> 그래서 언제 뭘 쓰느냐. 정리하면 이렇습니다. **복잡한 설계·대규모 리팩터링·레거시 분석은 Opus 4.8.** 여기서 '리팩터링'은 '동작은 그대로 두고 코드 구조를 더 깔끔하게 다시 짜는 것'이에요. **일반적인 기능 구현·버그 수정·테스트 작성은 Sonnet 4.6**(기본 추천). **간단한 수정·구문 오류·프로토타이핑은 Haiku 4.5.** 헷갈리면 그냥 Sonnet으로 시작하시면 됩니다.

### S45. 1.2.3 하이브리드 전략 & 모델 전환 (1/2) ⏱ ~50초
> 똑똑한 사용법은 한 작업 안에서도 모델을 **섞어 쓰는 것**입니다. 예를 들어 ① Opus로 설계를 분석하고, ② Sonnet으로 기능을 구현하고, ③ Haiku로 빠르게 프로토타입을 찍고, ④ 다시 Opus로 최종 리뷰를 하는 식이죠. 비싼 모델은 머리 쓸 때만 쓰는 겁니다.
> 세션 도중 바꾸는 건 `/model` 명령이에요. `/model Sonnet`처럼 치면 바뀝니다. 특히 **`/model opusplan`**이 유용한데, 계획 짤 땐 Opus, 실행할 땐 Sonnet으로 자동 전환해 줍니다.

### S46. 1.2.3 모델 전환 (2/2) — 플래그·환경변수 ⏱ ~50초
> 모델을 바꾸는 방법은 세 가지입니다. ① 세션 중엔 방금 본 `/model`. ② 시작할 때 `claude --model sonnet`처럼 플래그로. ③ 아예 기본값을 고정하려면 `~/.zshrc`에 `export ANTHROPIC_MODEL="claude-sonnet-4-6"`처럼 **환경 변수**로 박아 둡니다.
> 우선순위가 중요한데, **세션 중 `/model` 명령 > 시작 플래그 > 환경 변수 > 설정 파일** 순입니다. 즉 그때그때 친 명령이 가장 셉니다. 의사결정은 단순해요. "복잡한가? → 그렇다면 Opus, 보통이면 Sonnet, 간단하면 Haiku."

### S47. 1.2.4 Extended Thinking — 추론 강도 조절 ⏱ ~55초
> **Extended Thinking(확장 사고)**은 답하기 전에 AI가 머릿속으로 더 오래 고민하게 하는 모드입니다. 사람으로 치면 '즉답' 대신 '잠깐 따져 보고 답하기'예요. 여러 접근법을 비교하고 트레이드오프를 검토한 뒤 답을 냅니다.
> 차이를 보면, 일반 모드는 즉시·빠르게 답하고 토큰도 적게 씁니다. Thinking 모드는 느리지만 복잡한 설계·디버깅에 강해요. 다만 **비용 주의**가 있습니다. 출력 비용은 '생각한 토큰 + 실제 답변 토큰'을 합쳐서 매겨집니다. 생각 1만 + 답변 2천이면 1만 2천 토큰이 과금되는 거죠. 그래서 간단한 일엔 끄고, 어려운 일에만 켜는 게 요령입니다.

### S48. 1.2.4 모델별 지원 Effort 레벨 ⏱ ~40초
> 생각의 '강도'를 **effort 레벨**로 조절합니다. Opus 4.8은 low·medium·high·xhigh·max까지 지원하고 **기본값은 high**입니다. Sonnet 4.6과 Opus 4.6은 low·medium·high·max를 지원해요.
> 두 가지 팁. 프롬프트에 **`ultrathink`**라는 키워드를 넣으면 그 턴에서만 더 깊이 생각하라는 힌트를 줍니다(레벨 자체를 바꾸진 않아요). 그리고 Opus 4.8로 코딩·에이전트 작업을 할 땐 기본 high보다 **xhigh**를 권합니다.

### S49. 1.2.4 Thinking 활성화 (1/2) — 단축키·/config·/effort ⏱ ~45초
> Thinking 모드 켜는 방법은 네 가지인데 먼저 세 개. ① **단축키 토글** — 맥은 `Option+T`, 윈도우·리눅스는 `Alt+T`입니다. 누르면 켜고 끄고가 바뀌고, 현재 상태는 화면 하단에 표시돼요. ② **`/config`** — 설정 메뉴에서 Thinking Mode를 켭니다. ③ **`/effort`** — low·medium·high 중에 직접 고릅니다. 상황에 맞게 손쉽게 조절하시면 됩니다.

### S50. 1.2.4 Thinking 활성화 (2/2) — 환경 변수 ⏱ ~50초
> 네 번째는 **환경 변수로 기본값 고정**입니다. `~/.zshrc`에 `export CLAUDE_CODE_EFFORT_LEVEL=xhigh`처럼 적어 두면 매번 안 바꿔도 돼요. 깊은 추론은 xhigh, 일반은 high, 비용 아끼려면 medium입니다.
> 한 가지 주의. 예전에 쓰던 `MAX_THINKING_TOKENS`(생각 토큰 상한) 설정은 **Opus 4.8에서는 무시됩니다.** Adaptive Thinking으로 바뀌었기 때문이에요. 그러니 4.8에서는 `CLAUDE_CODE_EFFORT_LEVEL`로 제어하세요. 옛 방식이 꼭 필요한 특수한 경우(Opus 4.6·Sonnet 4.6)에만 별도 설정으로 살릴 수 있습니다.

### S51. 1.2.5 컨텍스트 윈도우 (개념) ⏱ ~50초
> **컨텍스트 윈도우(context window)**는 'AI가 한 번에 머릿속에 올려 둘 수 있는 최대 글의 양'입니다. 사람으로 치면 '한 번에 펼쳐 놓고 볼 수 있는 책상 크기'예요. 이게 크면 큰 코드베이스 전체나 긴 로그를 한꺼번에 다룰 수 있습니다.
> Opus 4.8과 Sonnet 4.6은 100만(1M) 토큰 — 대략 책 2,500페이지 분량입니다. Haiku 4.5는 20만(200K) 토큰, 약 500페이지고요. 큰 코드베이스 분석, 긴 대화 유지, 대형 로그 분석에 직접 영향을 줍니다.

### S52. 1.2.5 컨텍스트 윈도우 (명령) ⏱ ~40초
> 1M 컨텍스트를 명시적으로 켜려면 `/model sonnet[1m]` 또는 `/model opus[1m]`처럼 모델 뒤에 `[1m]`을 붙입니다. 그리고 지금 컨텍스트를 얼마나 쓰고 있는지는 **`/context`**로 시각화해서 보고, **`/status`**로 모델과 사용량을 자세히 확인합니다. 이 모니터링 습관이 나중에 토큰 비용을 아끼는 출발점이에요. 컨텍스트 관리는 1.5에서, 더 깊게는 **Level 6**에서 다룹니다.

## PART 04 — 1.3 개발환경 설정

### S53. 섹션 표지 · PART 04 ⏱ ~12초
> 네 번째 파트, 개발환경 설정입니다. 이 챕터에서 배우는 건 **iTerm2와 셸 최적화, VS Code·IntelliJ 같은 IDE 연동, 그리고 권한 모드**예요. 클로드 코드를 매일 편하게 쓰기 위한 작업 공간을 꾸미는 단계입니다.

### S54. 1.3.2 iTerm2 + tmux ⏱ ~45초
> 앞서 본 iTerm2 권장 설정을 다시 짚습니다. **Unlimited scrollback**(긴 출력 끝까지 보기)과 **Natural Text Editing**(맥 감각의 커서 이동) 두 가지를 켜 두세요.

## PART 05 — 1.4 핵심 설정 파일

### S65. 섹션 표지 · PART 05 ⏱ ~12초
> 다섯 번째 파트, 핵심 설정 파일입니다. 이 챕터에서 배우는 건 **`~/.claude/` 디렉토리 구조, settings.json, 환경 변수, 그리고 권한 모드**예요. 클로드 코드의 행동을 내가 원하는 대로 길들이는 단계입니다.

### S66. 1.4.1 ~/.claude/ 디렉토리 구조 ⏱ ~60초
> 클로드 코드 설정은 두 군데에 삽니다. **전역**은 `~/.claude/`(모든 프로젝트에 적용), **프로젝트별**은 각 프로젝트 폴더 안의 `.claude/`예요. 여기서 `~`는 '내 홈 폴더'를 뜻하는 기호입니다.
> 각 폴더 안엔 `settings.json`(설정), `CLAUDE.md`(메모리·규칙), `agents/`(서브에이전트), `commands/`(나만의 슬래시 커맨드)가 들어갑니다. 그리고 핵심 규칙 하나. **우선순위는 '프로젝트 settings.local > 프로젝트 settings > 전역 settings.local > 전역 settings'** 순입니다. 즉 더 구체적이고 가까운 설정이 이깁니다. `settings.local.json`은 나만 쓰는 로컬 오버라이드라 Git에 안 올리는 걸 권해요.

### S67. 1.4.2 settings.json — 팀 공유용 설정 ⏱ ~55초
> `settings.json`은 **Git에 커밋해서 팀 전체가 공유하는** 설정 파일입니다. 예시를 보면, `model`로 기본 모델을, `effortLevel`로 사고 강도를 정하고, `permissions`로 권한을 제어해요.
> 권한 부분이 핵심인데 두 종류입니다. **`allow`**는 '묻지 않고 자동 허용할 작업'(예: `npm run *`, `git status`), **`deny`**는 '절대 막을 작업'(예: `.env` 파일 읽기, `git push`, `rm -rf`, `sudo`)입니다. 이렇게 적어 두면 팀원 모두가 같은 안전장치를 갖고 시작하게 됩니다.

### S68. 1.4.2 DB 위험 명령 차단 — Hook 패턴 ⏱ ~60초
> 여기서 **Hook(훅)**이라는 강력한 개념이 나옵니다. Hook은 '특정 동작 직전·직후에 자동으로 끼어들어 검사·차단하는 갈고리'예요. 예를 들어 `PreToolUse`(도구 실행 직전) Hook으로, 명령에 `DROP`·`TRUNCATE`·`DELETE FROM` 같은 위험한 DB 명령이 들어 있으면 실행을 막아 버릴 수 있습니다.
> 두 가지를 강조할게요. 첫째, 이런 금지 사항은 **CLAUDE.md에도 글로 명시**해 두세요 — AI가 규칙으로 인식합니다. 둘째, **`deny`는 항상 `allow`보다 우선**합니다. 막는 규칙이 늘 이겨요. Hook의 본격적인 작성법은 **Level 3**에서 다룹니다. 지금은 "위험 명령을 시스템 차원에서 막을 수 있다"는 것만 기억하세요.

### S69. 1.4.2 settings.json 주요 키 ⏱ ~40초
> `settings.json`에서 자주 쓰는 키를 정리하면, `model`(기본 모델), `effortLevel`(사고 강도), `permissions.defaultMode`(기본 권한 모드), `permissions.allow`·`permissions.deny`(자동 허용·거부 목록), `env`(환경 변수)입니다. 다시 한번, **deny가 allow보다 항상 우선**한다는 원칙만 머릿속에 박아 두세요. 안전의 핵심입니다.

### S70. 1.4.3 환경 변수 ⏱ ~55초
> **환경 변수(environment variable)**는 '컴퓨터나 프로그램이 참고하는 전역 설정값'입니다. 설정 경로는 세 가지예요. 셸 설정 파일에 `export`, settings.json의 `env` 블록, 그리고 CI/CD(자동화 파이프라인)입니다.
> 자주 쓰는 것들을 보면, `ANTHROPIC_API_KEY`(API 키), `ANTHROPIC_MODEL`(기본 모델), `CLAUDE_CODE_EFFORT_LEVEL`(사고 강도), `MAX_THINKING_TOKENS`(생각 토큰 상한 — 단, 최신 Opus에선 무시) 등이 있어요. 다 외울 건 없고, "모델·사고강도·API키 같은 걸 변수로 고정할 수 있다"는 정도면 충분합니다.

### S71. 1.4.3 환경 변수 설정 (1/2) — 셸·settings ⏱ ~40초
> 설정 방법을 구체적으로 볼게요. ① **셸 프로파일**(내 개인 전역 설정) — `~/.zshrc`에 `export ANTHROPIC_MODEL="claude-sonnet-4-6"`처럼 적습니다. ② **settings.json의 env 블록**(팀 공유 가능) — JSON 형식으로 `CLAUDE_CODE_EFFORT_LEVEL`, `MAX_THINKING_TOKENS` 등을 넣습니다. 개인용이면 셸, 팀 공유면 settings.json — 이렇게 용도를 나눠 쓰시면 됩니다.

### S72. 1.4.3 환경 변수 설정 (2/2) — CI/CD ⏱ ~40초
> ③ 세 번째는 **CI/CD 파이프라인** 패턴입니다. CI/CD는 '코드를 자동으로 검사·배포하는 자동화 라인'을 뜻해요. 여기선 사람이 없으니 명령 한 줄에 환경 변수와 옵션을 다 붙여서 돌립니다. 예를 들어 `claude -p "테스트 실행하고 결과 요약해줘"`에 `--permission-mode acceptEdits`, `--output-format json`, `--max-budget-usd 2`(비용 상한 2달러) 같은 걸 함께 줍니다. 이런 자동화의 실전은 **Level 7·8**에서 다룹니다.

### S73. 1.3.3 권한 모드 4종 비교 ⏱ ~60초
> 이제 중요한 **권한 모드**입니다. 클로드 코드가 '어디까지 알아서 하고, 어디서 나한테 물어볼지'를 정하는 설정이에요. 네 종류입니다. **default** — 파일 읽기만 자유, 나머지는 매번 물어봄(가장 안전, 시작점). **acceptEdits** — 파일 읽기·편집까지 자동 승인(실무의 90%가 여기). **dontAsk** — 미리 허락한 것만 하고 나머진 조용히 거부(자동화·CI 전용). **bypassPermissions** — 전부 검사 없이 실행(격리된 컨테이너·VM 전용, 위험).
> **`Shift+Tab`**으로 default ↔ acceptEdits를 오갈 수 있어요. 안전한 default에서 시작해 필요하면 acceptEdits로 올리는 게 정석입니다.

### S74. 1.3.3 default · acceptEdits — 일상 작업 ⏱ ~55초
> 일상에서 쓰는 두 모드를 자세히 볼게요. **default**는 파일 읽기만 자유롭고 편집·명령 실행은 매번 승인을 받습니다. 안전하지만 흐름이 자주 끊겨요. 그래서 시작점으로만 쓰고 acceptEdits로 넘어가는 걸 권합니다.
> **acceptEdits**는 파일 읽기·편집을 자동 승인합니다. mkdir·touch 같은 일반 파일 명령도 작업 폴더 안에선 자동이고요. 단, 일반 Bash 실행 명령은 여전히 물어봅니다. 화면 하단에 '⏵⏵ accept edits on'으로 표시돼요. **안전 수칙**: 쓰기 전에 반드시 git 커밋으로 체크포인트를 찍고, `git push`나 `rm -rf` 같은 건 deny에 넣어 두세요.

### S75. 1.3.3 Git checkpoint 후 복구 ⏱ ~45초
> 방금 말한 '체크포인트' 패턴을 실제로 보여드릴게요. 핵심은 단순합니다. ① 작업 전에 `git add . && git commit -m "checkpoint"`로 현재 상태를 저장합니다. ② acceptEdits로 클로드 코드에게 일을 시킵니다. ③ 결과가 마음에 들면 그대로 커밋. ④ **마음에 안 들면 `git reset --hard HEAD`로 체크포인트로 즉시 되돌립니다.** 이 안전망이 있으면 AI에게 과감하게 맡길 수 있어요. "되돌릴 수 있다"는 확신이 생산성을 만듭니다.

### S76. 1.3.3 (심화) plan 모드 & auto 모드 ⏱ ~55초
> 심화 두 모드입니다. **plan 모드** — 코드 편집(Edit·Write)은 막고, 파일 읽기와 탐색만 허용합니다. 즉 클로드가 코드베이스를 분석해 **계획만 세우고 실제 변경은 안 해요.** `Shift+Tab` 두 번으로 들어가거나, 한 번만 적용하려면 `/plan` 뒤에 요청을 적습니다. 큰 작업 전에 "먼저 계획부터 보자" 할 때 아주 유용합니다.
> **auto 모드** — 모든 작업을 백그라운드 안전성 검사와 함께 자동 실행합니다. 사람이 매번 승인하는 대신 분류기가 의도와 행동의 일치를 판단해요. `claude --enable-auto-mode`로 켜며, 주로 Team·Enterprise·API용입니다.

### S77. 1.3.3 dontAsk · bypassPermissions ⏱ ~55초
> 잠긴 두 모드입니다. **dontAsk** — allow 목록에 없는 모든 도구를 자동 거부합니다. 거부할 때 **아무 메시지 없이 조용히** 막아서 디버깅이 어려워요. 그래서 헤드리스 자동화(`claude -p`)나 CI/CD 전용입니다.
> **bypassPermissions** — 권한 확인을 통째로 건너뛰고 완전 자율로 실행합니다. 가장 위험해요. 다만 안전장치로, `.git`·`.claude`·`.vscode` 같은 설정 디렉토리에 쓰는 건 이 모드에서도 여전히 확인을 받습니다. `claude --dangerously-skip-permissions`로 켜는데, 이름 그대로 'dangerously(위험하게)'가 붙어 있죠. **반드시 격리된 컨테이너나 VM에서만** 쓰세요.

## PART 06 — 1.5 첫 세션: Hello World부터 단축키까지

### S78. 섹션 표지 · PART 06 ⏱ ~12초
> 마지막 파트, 드디어 첫 세션입니다. 이 챕터에서 배우는 건 **Python·TypeScript로 Hello World 만들기, 턴·세션·컨텍스트 개념, 대화형 vs 단발 모드, 특수 문법(@·!·#), 단축키, 그리고 Claude Code on the Web**이에요. 지금까지 다진 기초를 실제로 써먹는 단계입니다.

### S79. 1.5.1 Hello World — Python ⏱ ~50초
> 자, 첫 작품을 만들어 봅시다. 폴더를 만들고(`mkdir claude-code-project`) 들어간 뒤(`cd`), `claude`를 실행합니다. 첫 실행이면 테마 선택 → 로그인 → 프로젝트 초기화가 차례로 진행돼요.
> 그리고 그냥 한국어로 부탁하면 됩니다. **"Hello World를 출력하는 Python 프로그램을 작성해줘."** 파일을 실행할 권한을 물어보면 1번('이번 한 번만')을 고릅니다. 그다음 터미널에서 `python3 hello.py`를 치면 화면에 'Hello, World!'가 찍힙니다. 축하합니다 — 방금 클로드 코드로 첫 프로그램을 만든 거예요.

### S80. 1.5.1 Hello World — TypeScript & 에러 대처 ⏱ ~60초
> 이번엔 조금 더 나아가 봅니다. **"HTML과 TypeScript로 사용자가 입력한 값을 화면에 출력하는 프로그램을 작성해줘."** 만들어진 index.html을 우클릭 → 'Open with Live Server'로 브라우저에서 띄웁니다.
> 여기서 가장 중요한 실무 팁. **에러가 나면 그 에러 메시지를 그대로 복사해서 채팅창에 붙여넣으세요.** 그러면 클로드 코드가 알아서 원인과 해결책을 제시합니다. 이게 클로드 코드의 핵심 사용법이에요.
> 또 하나 꼭 알아 둘 것. **AI는 같은 질문에도 매번 조금씩 다른 코드를 냅니다.** 이걸 '비결정적 특성'이라고 해요. 그래서 성공의 기준은 '코드가 똑같이 나오느냐'가 아니라 **'제대로 동작하느냐'**입니다. 이 비결정성을 테스트와 명세로 다스리는 방법은 **Level 2~3**에서 본격적으로 배웁니다.

### S81. 1.5.1 Accept mode 팝업 화면 ⏱ ~12초
> (화면) 처음 실행할 때 뜨는 권한 선택 팝업입니다. 세 가지 선택지가 나오는데, 다음 슬라이드에서 각각이 뭘 뜻하는지 설명할게요.

### S82. 1.5.1 Accept mode 팝업 — 3가지 선택지 ⏱ ~40초
> 그 팝업의 세 선택지입니다. **1번 Auto-accept** — 모든 작업을 승인 없이 즉시 실행. **2번 Manually approve edits** — 변경마다 수동 승인. **3번 Plan mode** — 계획만 세우고 실행은 나중에 결정. 입문자라면 **처음엔 2번**으로 시작해 클로드가 뭘 하는지 보면서 익히는 걸 권합니다. 익숙해지면 1번으로 속도를 올리시면 돼요.

### S83. 1.5.2 턴·세션·컨텍스트 (개념) ⏱ ~60초
> 동작 원리를 이해하려면 꼭 구분해야 할 세 단어입니다. **턴(Turn)**은 '내 메시지 하나 + 클로드 응답 하나'의 한 쌍이에요. "테스트 코드 추가해줘" 하고 답을 받는 한 번의 주고받음입니다. **세션(Session)**은 'claude를 실행해서 종료할 때까지'의 단위로, 여러 턴이 쌓입니다. IDE를 열어 작업하고 닫는 것과 비슷해요. **컨텍스트(Context)**는 '매 순간 모델이 실제로 보고 있는 전체 글'입니다. 시스템 프롬프트 + 도구 정의 + CLAUDE.md + 그동안 쌓인 턴 전부예요.
> 포함 관계로 외우세요. **턴 ⊂ 세션 ⊂ 컨텍스트.** 턴이 모여 세션이 되고, 세션 안의 누적된 턴과 정적 요소가 합쳐져 컨텍스트가 됩니다.

### S84. 1.5.2 컨텍스트의 구조 ⏱ ~65초
> 컨텍스트 윈도우 안에 뭐가 들어가는지 뜯어볼게요. 크게 둘로 나뉩니다. **정적(STATIC) 부분**은 세션 시작 때 한 번 로드돼요 — 시스템 프롬프트(약 11K, Anthropic 비공개·수정 불가), 도구 정의(약 13K), 그리고 **CLAUDE.md(약 5K) — 여기가 우리가 영향력을 행사할 수 있는 핵심**입니다. **동적(SESSION) 부분**은 매 턴 쌓여서 사용량이 점점 늘어요.
> 그래서 사용률이 70%에 도달하면 **`/compact`**로 대화를 요약·압축하고, 95%를 넘거나 주제가 완전히 바뀌면 **`/clear`**로 초기화합니다. 그리고 `/context`로 어디가 토큰을 잡아먹는지 실시간으로 확인하죠. 팁: 큰 파일은 head·grep으로 잘라 읽고, 작업을 끝낼 땐 HANDOFF 문서로 인계하면 다음 세션이 매끄럽습니다. 이 토큰 관리는 **Level 6**의 핵심 주제예요.

### S85. 1.5.2 컨텍스트 관련 명령어 ⏱ ~55초
> 컨텍스트를 다루는 명령어를 정리하겠습니다. **`/context`** — 지금 사용량 시각화(수시로, 70% 넘으면 압축 고려). **`/clear`** — 대화 히스토리 비우기(주제가 완전히 바뀔 때). **`/compact`** — 히스토리를 요약본으로 압축(같은 작업 이어가며 공간만 확보). **`/memory`** — CLAUDE.md를 편집기로 열기(발견한 규칙을 영구 저장). **`/init`** — 프로젝트 분석 후 CLAUDE.md 초안 생성(새 프로젝트에 한 번). **`--continue` / `/resume`** — 이전 세션 이어가기. **`/status`** — 모델·세션 정보. **`/cost`** — 누적 토큰·비용 확인. 이 명령들이 일상 작업의 손발이 됩니다.

### S86. 1.5.2 휘발성 — 무엇이 사라지고 남는가 ⏱ ~50초
> 이걸 모르면 "어제 작업이 왜 사라졌지?" 하게 됩니다. 핵심 세 가지예요. **🔥 휘발** — 컨텍스트(쌓인 턴과 도구 결과)는 세션이 끝나면 **전부 사라집니다.** 다음 세션으로 자동으로 안 넘어가요. **📌 보존** — 단, **CLAUDE.md만은 다음 세션에서도 자동으로 다시 로드됩니다.** 그래서 "어제 정한 규칙"은 거기 적어 둬야 살아남아요. **🔗 이어가기** — `--continue`, `/resume`, `/compact`, 그리고 HANDOFF 문서 — 이 네 가지로 세션을 연결할 수 있습니다.

### S87. 1.5.3 대화형 vs 단발 모드 (1/2) ⏱ ~50초
> 클로드 코드를 쓰는 두 가지 방식입니다. **대화형(Interactive) 모드**는 그냥 `claude`로 실행해 계속 주고받는 방식이에요. **단발 모드(Print, `-p`)**는 한 번 묻고 답만 받고 끝나는 방식입니다. `claude -p "이 프로젝트 구조를 설명해줘"`처럼요.
> `-p` 모드의 진짜 강점은 **파이프(`|`)**와 결합할 때입니다. 파이프는 '앞 명령의 출력을 뒤 명령의 입력으로 넘기는 연결관'이에요. 예를 들어 `cat error.log | claude -p "이 에러 원인 분석해줘"`처럼, 에러 로그를 통째로 넘겨 분석시킬 수 있습니다.

### S88. 1.5.3 대화형 vs 단발 모드 (2/2) ⏱ ~45초
> 두 모드를 비교하면 이렇습니다. **대화형**은 컨텍스트가 유지되고 반복 수정 작업에 좋지만 자동화엔 부적합해요. **단발(`-p`)**은 컨텍스트는 안 남지만 자동화·파이프라인·파이프 입력에 강합니다. `--output-format json`을 붙이면 결과를 JSON으로 받아 다른 프로그램이 처리하게 할 수도 있어요. 정리하면, **사람이 대화하며 작업할 땐 대화형, 스크립트로 자동 처리할 땐 `-p`** 입니다.

### S89. 1.5.4 특수 문법 (1/2) — @ ⏱ ~50초
> 프롬프트 입력창에서 쓰는 특수 기호 세 개를 배웁니다. 먼저 **`@`(파일·디렉토리 참조)**예요. `@src/utils/auth.js 이 파일 로직 설명해줘`처럼 쓰면 그 파일을 대화에 콕 집어 넣습니다. `@src/components/`처럼 폴더를 통째로 넣을 수도 있고, `@`를 친 뒤 Tab을 누르면 파일 목록이 자동완성됩니다. 단, `@`로 넣은 파일은 컨텍스트(토큰)를 소비한다는 점은 기억해 두세요.

### S90. 1.5.4 특수 문법 (2/2) — ! · # ⏱ ~55초
> 나머지 두 개입니다. **`!`(Bash 명령 직접 실행)** — `! git status`, `! npm test`처럼 느낌표 뒤에 명령을 쓰면 클로드를 거치지 않고 터미널 명령이 바로 실행됩니다. **토큰을 안 쓴다**는 게 장점이에요. **`#`(CLAUDE.md에 기록)** — `# 이 프로젝트는 TypeScript strict 모드를 쓴다`처럼 샵 뒤에 적으면 그 내용이 CLAUDE.md에 규칙으로 저장됩니다. 정리하면, **`@`는 파일 넣기, `!`는 명령 실행, `#`은 규칙 기록**입니다.

### S91. 1.5.5 핵심 단축키 9종 ⏱ ~55초
> 첫 세션에서 꼭 필요한 단축키 아홉 개입니다. 가장 중요한 건 **`Esc`** — 클로드가 답하는 도중 멈출 때 씁니다. **`Ctrl+C`는 세션 자체를 종료**하니, 응답만 멈추려면 반드시 Esc예요. **`Esc`를 두 번** 누르면 이전 메시지로 돌아가 잘못 보낸 걸 수정할 수 있습니다. **`Shift+Tab`**은 권한 모드 전환, **`Option+T`**(맥)/`Alt+T`는 Thinking 토글, **`Ctrl+L`**은 화면 정리, **위·아래 화살표**는 명령 히스토리, **`Ctrl+R`**은 히스토리 검색, **`Option+Enter`**는 줄바꿈입니다. "멈출 땐 Ctrl+C 말고 Esc" — 이거 하나만 확실히 하세요.

### S92. 1.5.5 실전 단축키 활용 패턴 4종 ⏱ ~45초
> 단축키를 상황별로 묶어 볼게요. **상황 1**: 클로드가 엉뚱한 방향으로 갈 때 → `Esc`로 멈추고 `Shift+Tab`으로 Plan 모드로 바꿔 전략을 다시 짭니다. **상황 2**: 메시지를 잘못 보냈을 때 → `Esc+Esc`로 이전 메시지로 가서 수정 후 재전송. **상황 3**: 간단한 작업을 빠르게 → `Shift+Tab`으로 acceptEdits 진입. **상황 4**: 복잡한 추론이 필요할 때 → `Option+T`로 Thinking 켜고 effort를 high로. 이 네 패턴이 일상의 90%를 커버합니다.

### S93. 1.5.6 클라우드 세션 (1/2) ⏱ ~55초
> 마지막 주제, **Claude Code on the Web**입니다. 두 가지가 있는데 먼저 **클라우드 세션**이에요. 이건 **Anthropic이 관리하는 클라우드 VM에서** 작업이 도는 방식입니다. 각 세션은 격리된 가상 컴퓨터에서 GitHub 저장소를 복제해 실행돼요.
> 브라우저로 claude.ai/code에 접속하거나, CLI에서 `claude --remote "인증 버그를 src/auth/login.ts에서 수정해줘"`로 시작합니다. 단, **반드시 git push를 먼저** 해 둬야 클라우드가 최신 코드를 받습니다. 클라우드에서 한 작업을 로컬로 가져올 땐 `claude --teleport`를 씁니다.

### S94. 1.5.6 클라우드 세션 (2/2) — 모니터링 ⏱ ~30초
> 클라우드 세션을 지켜보는 명령입니다. **`/tasks`** — 백그라운드에서 도는 클라우드 세션 전체 목록을 보여 주고, 목록에서 `t` 키를 누르면 바로 그 세션을 로컬로 가져옵니다(teleport). **`/mobile`** — QR 코드를 띄워서 모바일 앱으로 진행 상황을 모니터링할 수 있어요. 출근길에 폰으로 작업을 확인하는 식이죠.

### S95. 1.5.6 Remote Control ⏱ ~55초
> 두 번째는 **Remote Control(원격 제어)**입니다. 앞의 클라우드 세션과 헷갈리기 쉬운데 핵심이 달라요. Remote Control은 **세션이 내 로컬 컴퓨터에서 계속 돌고**, 폰이나 브라우저는 그 로컬 세션을 들여다보는 '창'일 뿐입니다. 로컬 터미널에서 `claude remote-control`로 켜고, 스페이스바를 누르면 QR 코드가 떠서 모바일 앱과 빠르게 연결됩니다.
> 단, 한 가지 주의. **2026년 4월 기준 Remote Control은 아직 리서치 프리뷰(시험) 단계**고 일부 버그가 보고돼 있어서, 프로덕션 핫픽스 같은 중요한 작업엔 신중하게 쓰세요.

### S96. 1.5.6 클라우드 세션 vs Remote Control ⏱ ~50초
> 둘을 표로 비교하며 정리하겠습니다. **실행 위치** — 클라우드 세션은 Anthropic 클라우드, Remote Control은 내 로컬 머신. **로컬 파일·MCP·커스텀 설정 접근** — 클라우드 세션은 다 안 되고, Remote Control은 다 됩니다(내 컴퓨터에서 도니까요). **로컬 프로세스를 끄면** — 클라우드 세션은 그대로 살아 있고, Remote Control은 같이 종료됩니다. 지원 플랜도 달라요. 정리하면, **'완전히 맡기고 컴퓨터 꺼도 되는 건 클라우드 세션, 내 환경 그대로 폰으로 들여다보는 건 Remote Control'**입니다.
