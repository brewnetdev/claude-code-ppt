ABOUTcarve-harness 한눈에

carve-harness는 코드베이스를 분석해 그 프로젝트에 맞는 스킬·훅·서브에이전트를 대화형으로 선택 설치하는 하네스 엔지니어링 CLI입니다. 이 가이드의 /carve-workflow 스킬은 v1.7.0(2026-06-15)에 새로 추가된 기능입니다.

핵심 동작 한 줄

carve install → 스택 탐지 → (구성요소 선택) → .claude/에 자산 생성
              → 생성된 검증 훅이 위험 명령을 exit code 2로 결정적 차단

7대 특징

  • 토큰 효율 기본 탑재 — codesight·LSP MCP 자동 등록. grep 대신 정확 탐색.
  • 결정적 안전 — 위험 명령·시크릿 파일을 exit 2로 강제 차단 (권고 아님).
  • 맞춤 선택 설치 — 탐지 → 추천 → 선택. 일괄 없음, 멱등 재설치·클린 제거.
  • anti-slop 게이트 — HTML·SVG·문서 슬롭을 check-slop 린터로 결정적 차단.
  • Squad 9 에이전트 — review · plan · refactor · qa · debug · docs · gitops · audit · evaluator.
  • 자기검증 auditor — 설치 전 secret·과도 권한·훅 주입·셸 문법 스캔, 통과해야 설치.
  • 빌드 0 — .ts 직접 실행 (Node 22.18+ 타입 스트리핑).
이 가이드의 위상

/carve-workflow는 carve-harness v1.7.0의 신규 스킬로, Fable5 모델식 7단계 루프와 최소 출력형식·완료 게이트·에스컬레이션을 구현합니다. 기존 자산(iterate · sprint-contract · squad-evaluator)을 지휘하고, net-new 원칙(불신·결정외화·범위수렴)을 항상-켜짐으로 주입합니다.

PART 1한 문장 정의

workflow는 7단계 절차 루프와 8요소 보고 형식으로 다단계·고모호 작업을 "장기 실행"으로 끌고 가는 메타 스킬입니다. 직접 코드를 만들지 않고 sprint-contract · evaluation-criteria · iterate · squad-* · tdd 같은 기존 자산을 지휘합니다.

쉽게 말해 — "이번 작업은 즉답이 아니라 프로젝트로 다뤄야 한다"고 선언하는 스위치입니다.

스킬을 부르는 방법 — 2가지

① 슬래시 명시 — 채팅 입력에 /carve-workflow 한 줄.

② 자연어 트리거 어구 — 다음 어구 중 하나가 사용자 발화에 포함되면 Claude가 이 스킬을 자동 invoke합니다 (단일 출처: .claude/skills/workflow/SKILL.md의 description 필드):

  • "절차대로 완수해"
  • "목표 분해해서 검증까지"
  • "장기 작업으로 진행"
  • "Fablize"

⚠ 주의 — 이 어구 없이 그냥 작업 내용만 적으면 workflow는 켜지지 않습니다. Claude는 평범한 다단계 요청으로 처리하며, 7단계 게이트와 8요소 보고는 보장되지 않습니다.

PART 27단계 루프 + 3원칙

7단계 (.claude/skills/workflow/SKILL.md 단일 출처)

1.목표 재정의 → 2.작업 분해 → 3.완료 기준 → 4.가정 정리 → 5.실행 → 6.자체 검증 → 7.리스크 + 다음
#단계한 줄
1목표 재정의요청을 측정 가능한 형태로 바꾼다. 모호하면 가정 명시 후 진행, 치명적 모호성은 멈춤.
2작업 분해2단계 이상이면 분해 + 단계별 완료기준·의존성. squad-plan에 위임 가능.
3완료 기준"무엇이 완료의 증거인가"에 먼저 답 — 테스트·실행 결과·로그.
4가정 정리가정·검증법·fallback을 표로 외화. sprint-contract.md의 ## 가정 절.
5실행즉시 실행. 약속만 하고 미루지 않음.
6자체 검증산출물을 실제로 실행·관찰 — 코드는 테스트, 시각물은 렌더, 설정은 적용. iterate·squad-qa에 위임 가능.
7리스크 + 다음남은 불확실성·"검증되지 않음"을 정직 표시, 다음 액션/에스컬레이션 제안.

베이스라인 3원칙 (.claude/CLAUDE.md가 단일 출처)

원칙풀어쓰기실전 적용
Contractual Distrust 계약적 불신 — 입력·출력·미래의 나를 의심하라 경계 4종(정상·매핑·None·변조)을 테스트. "이 함수에 빈 배열이 들어오면?" 같은 질문.
Externalized Decisions 결정의 외화 — 결정에 D1·D2 같은 ID를 붙이고 주석에 이유·비용·탈출구 명시 "왜 zod 대신 valibot?" 같은 결정은 코드 위에 // D3: 주석으로.
Scope Convergence 범위 수렴 — "안 건드릴 것"을 먼저 적어 변경 반경을 좁힌다 sprint-contract의 ## 범위 밖을 계획 전에 채운다.

PART 38요소 출력 형식 + 완료 게이트

작업 보고는 이 8요소로 고정됩니다 (이게 workflow의 시그니처):

[목표] 무엇을 측정 가능한 형태로 완수하려는가
[계획] 어떤 단계로 쪼개 진행하는가
[가정] 무엇을 사실로 가정하고, 어떻게 검증하는가
[실행] 실제로 무엇을 했는가 (약속 아님)
[증거] 실행 결과 — 명령 출력, 파일 diff, 스크린샷
[검증] 자체 검증 결과 — 게이트 통과 여부
[리스크] 남은 불확실성 / "검증 안 된 부분"
[다음] 다음 액션 또는 에스컬레이션

완료 게이트 (evaluation-criteria.md — 100점 만점, PASS ≥ 90)

G는 Gate(게이트), DoD는 Definition of Done(완료 정의), MUST PASS는 "하나라도 0점이면 총점과 무관하게 FAIL"인 결정적 항목입니다.

게이트항목0점 = 즉시 FAIL
G1빌드/타입체크 (npm run build / typecheck exit 0)빌드 실패
G2테스트 통과 (npm run test)테스트 1개라도 fail
G3안전 위반 0건 (파괴 명령·비밀 노출 검증 훅)1건 이상 위반

이 3개는 총점 90/100을 받아도 0이면 즉시 탈락. workflow의 6단계(자체 검증)는 항상 이 3개부터 돕니다.

PART 4언제 쓰고 언제 쓰지 말 것

쓸 때 (네 조건 중 둘 이상 해당)
  • 다단계 — 3개 이상의 독립 작업이 직렬·병렬로 묶임
  • 고모호 — "어떤 게 완료인지"가 시작 시점에 불명
  • 검증 필수 — 테스트·렌더·anti-slop·게이트 중 둘 이상을 만족해야 함
  • 시간 분산 — 한 세션에 안 끝남, handoff로 다음 세션에 인계 가능성
쓰지 말 것
  • 1줄 typo 수정
  • 단발 질문("이 함수가 뭐 해?")
  • 탐색만 하는 작업("이 폴더에 뭐가 있어?")
  • 사용자가 대화를 원하는 상황 — workflow는 완수가 목적

PART 5프로젝트에서 사용 예시

각 예시는 슬래시 + 자연어 트리거 2종을 회전 노출합니다. 자연어 트리거의 전체 4종("절차대로 완수해" · "목표 분해해서 검증까지" · "장기 작업으로 진행" · "Fablize")은 위 PART 1 callout에 정리돼 있으며, 어느 것으로 발화해도 동일한 workflow가 발동됩니다. 7단계 매핑은 PART 2를 참조하세요.

예시 A — Level 8 신규 챕터 덱 작성

방법 1 · 슬래시 명시 
/carve-workflow
Level 8 챕터 덱을 처음부터 작성해서 anti-slop 통과·export viewer까지 가져가줘
방법 2 · "절차대로 완수" 
Level 8 챕터 덱을 처음부터 만드는 걸 절차대로 완수해줘 — 토픽 매핑·다이어그램·page-num·slop 검증·export까지
방법 3 · "장기 작업으로 진행" 
Level 8 챕터 덱 0→완성을 장기 작업으로 진행해줘. md-to-slidedeck → slide-to-image → check-slop → export-html-deck 순으로
작업 핵심
토픽 매핑 → 슬라이드 골격 → 다이어그램 → 임베드 → page-num → slop 검증 → export
호출 자산
md-to-slidedeck · slide-to-image · clean-html · check-slop.mjs · export-html-deck
완료 게이트
anti-slop 0 ERROR · page-num 연속 · typecheck 통과 · viewer 정상

예시 B — 전 덱(Level 1~7) watermark 일괄 적용

방법 1 · 슬래시 명시 
/carve-workflow
전 덱에 brand watermark 입히고 슬라이드 깨짐 없는지 확인해줘
방법 2 · "Fablize" 
Fablize: 전 덱 watermark 적용 — Level 1부터 7까지, 한 덱 적용 후 diff·렌더 확인 다음 덱
방법 3 · "목표 분해해서 검증까지" 
전 덱 watermark 일괄 적용을 목표 분해해서 검증까지 가져가줘. 1덱 파일럿 → 회귀 점검 → 나머지 6덱 순으로
작업 핵심

호출 자산

완료 게이트

예시 C — 다른 챕터 발표 스크립트 일괄 작성

방법 1 · 슬래시 명시 
/carve-workflow

PART 6Anti-pattern (이렇게 쓰면 안 된다)

workflow를 형식만 빌려 쓰면 오히려 해롭습니다. 다음 네 가지 시그널은 즉시 폐기 대상입니다.

Anti-pattern 4종
  1. 약속형 5단계 — [실행]에 "이렇게 진행하겠습니다"만 쓰고 실제 도구 호출이 없는 경우. 5단계는 동사 + 결과여야 합니다.
  2. 검증 우회 — [검증]을 생략하거나 "통과한 것으로 보입니다" 같은 모호한 문장으로 게이트 G1·G2·G3을 건너뛰는 경우. 게이트는 명령 + exit code로 적어야 합니다.
  3. 가정으로 덮기 — 치명적 모호성(스펙 밖 결함, 고정 답 없는 창의 디테일)을 [가정] 표에 적어 밀어붙이는 경우. 에스컬레이션이 정답입니다.
  4. 단발 과적용 — 1줄 typo 수정에 8요소를 채우는 경우. 오버헤드만 발생합니다. 그냥 고치세요.

PART 7마무리 격언

workflow는 새 도구가 아니라 기존 도구를 결합하는 규율입니다.
7단계는 회의실 화이트보드, 8요소는 회의록, G1·G2·G3 게이트는 출입증.

설치된 자산(iterate · tdd · test-gen · squad-plan · squad-evaluator · squad-qa · handoff · check-slop.mjs · export-html-deck · slide-to-image · watermark)이 모두 그대로 있고, workflow는 이들을 언제·어떤 순서로 부를지 결정하는 지휘봉입니다. 7단계의 ①②③이 plan, ④가 contract, ⑤가 execute, ⑥이 evaluate, ⑦이 handoff — 결국 이 프로젝트의 하네스 전체가 7단계에 한 번씩 호출되는 구조입니다.
