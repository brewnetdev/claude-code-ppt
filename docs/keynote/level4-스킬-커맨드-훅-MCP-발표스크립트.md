>
> 이 레벨의 내용을 기억하는 가장 빠른 방법은 비유 세 가지입니다. 강의 내내 이 비유가 반복됩니다.

>
> **훅은 과속 방지턱입니다.** 운전자가 아무리 급해도 물리적으로 속도를 줄이게 만드는 구조물. 모델에게 부탁하는 것이 아니라 도로 설계 자체를 바꾸는 것입니다.
>
> 그리고 네 번째, **MCP는 USB-C 포트입니다.** 어떤 외부 장치든 같은 규격으로 연결합니다.

>

>

## 슬라이드 6 — 4.1.3 커스텀 커맨드 ⏱ ~70초

(화면) .claude/commands/code-review.md 코드 예시 + 스코프 설명

> 커스텀 커맨드의 구조입니다. `.claude/commands/` 아래 마크다운 파일 하나를 만들면 그게 슬래시 커맨드가 됩니다.
>
> 화면의 `/code-review` 파일은 세 부분으로 구성됩니다.
>
> 첫째, frontmatter — 파일 상단의 YAML 블록. `description`에 이 커맨드가 무엇을 하는지 적고, `allowed-tools`에 이 커맨드가 쓸 수 있는 도구만 명시합니다. git 명령과 Read, Grep만 허용했습니다.
>
> 둘째, 본문 — 프롬프트 텍스트입니다. 절차를 단계별로 적어두면 모델이 그대로 따릅니다.
>
> 스코프는 두 가지입니다. `.claude/commands/`는 프로젝트 전용으로 팀 공유가 되고, `~/.claude/commands/`는 모든 프로젝트에서 쓰는 개인 커맨드입니다. 둘 다 있으면 프로젝트 쪽이 우선합니다.
>
> 이 커맨드가 없었다면 코드 리뷰를 요청할 때마다 "git diff 보고, 파일별로 의도·위험·누락 테스트 정리하고, 보안·성능 관점 확인해서, 표 형태로 출력해줘"라는 긴 지시를 매번 반복해야 합니다. 파일 하나로 그 반복이 없어집니다.

👉 진행 팁: `allowed-tools` 제한이 왜 중요한지 짚어주세요. 도구를 열어두면 모델이 예상 외의 동작을 할 수 있습니다.

## 슬라이드 7 — 4.1.4 $ARGUMENTS ⏱ ~110초

(화면) fix-issue.md 와 add-feature.md 코드 예시

> 커맨드에 인자를 받고 싶을 때는 `$ARGUMENTS`를 씁니다.
>
> 그 전에 frontmatter를 잠깐 보겠습니다. 커맨드의 frontmatter는 스킬의 frontmatter와 살짝 다릅니다. 화면에 보이는 `argument-hint` — 커맨드를 칠 때 어떤 인자를 넣어야 하는지 힌트를 보여주는 필드입니다. 이 외에도 `disable-model-invocation`처럼 모델이 이 커맨드를 스스로 호출하지 못하게 막는 필드, `model`, `context` 같은 필드를 추가할 수 있습니다. 이 필드들이 전부 적용된 복잡한 예제는 별도 소스 예제로 제공하겠습니다.
>
> 왼쪽 예시 `fix-issue.md`를 보면, 커맨드 본문 곳곳에 `$ARGUMENTS`가 있습니다. `/fix-issue 12`라고 치면 이 12라는 숫자가 `$ARGUMENTS` 자리마다 치환돼 들어갑니다. `gh issue view 12`가 터미널 명령어로 실행되고, `gh issue view 12 --comments`, 본문의 "이슈 #12"까지 — 여기에도 12, 여기에도 12, 이런 식으로 들어가서 이 하나의 프롬프트가 frontmatter에 명시한 권한으로 실행됩니다.
>
> 작업 흐름은 네 단계입니다. 이슈 본문과 코멘트에서 요구사항을 분석하고, 관련 파일을 탐색하고, 최소 변경안을 제안하고, 마지막으로 **PR 본문 초안을 작성**합니다. `Closes #12`까지 자동으로 들어갑니다. 이슈 번호 하나 받아서 깃허브 이슈 처리부터 PR 초안까지 가는 커맨드입니다.
>
> 오른쪽 `add-feature.md`는 기능 스캐폴딩 커맨드입니다. 스캐폴딩이 뭐냐 — 껍데기, 디렉터리 구조를 만드는 것입니다. `/add-feature user-profile`이라고 치면 커맨드 안에서 먼저 정규식 검증을 합니다. 소문자·숫자·하이픈 조합인지, 길이가 2~40자인지 확인한 뒤에야 다음 작업을 진행합니다.
>
> 검증이 통과되면 기존에 있는 패턴을 참고해서 — `ls src/features/`로 확인하죠 — `mkdir -p`로 인자로 받은 이름의 디렉터리를 만들고, 컴포넌트·서버액션·Drizzle 스키마·Vitest 테스트 파일을 생성하고, 마지막에 생성 결과를 요약해서 보여줍니다.
>
> 포인트는 검증입니다. 잘못된 인자가 들어왔을 때 모델이 무작정 그대로 작업하지 않도록 검증 절차를 커맨드 안에 박아둔 겁니다. `$ARGUMENTS`는 커맨드를 일회성 지시가 아니라 인자를 받는 재사용 가능한 도구로 만드는 핵심 요소입니다.

## 슬라이드 8 — 4.1.5 실전 패턴 3종 ⏱ ~80초

(화면) TDD 사이클 / 자동화 파이프라인 / 컨텍스트 오염 방지 세 블록

> 커스텀 커맨드를 실전에서 조합하는 패턴 세 가지입니다.
>
> 첫째, **TDD 사이클**. `/tdd-red`로 실패하는 테스트만 작성하고, `/tdd-green`으로 그 테스트를 통과하는 최소 구현만 작성하고, `/tdd-refactor`로 구조만 정리합니다. 세 커맨드가 단계를 강제로 분리합니다. Red 없이 Green을 실행하면 순서가 무너집니다.
>
> 둘째, **자동화 파이프라인**. 커밋 전 점검 네 가지 — `/lint-fix`, `/typecheck`, `/test-affected`, `/commit-msg` — 를 매번 순서대로 직접 칠 수도 있습니다. 하지만 그건 자동화가 아니라 암기죠. 자동화의 실체는 **래퍼 커맨드 하나**입니다. `.claude/commands/pre-commit.md` 파일 본문에 이 네 단계를 순서대로 적어 두면, `/pre-commit` 한 줄로 전체가 실행됩니다. 단축키 네 개를 묶은 매크로 키 하나를 만드는 셈입니다. 중간 단계가 실패하면 거기서 멈추고, 고친 뒤 이어갑니다. CI에 가기 전에 로컬에서 같은 순서로 사전 점검이 끝납니다.
>
> 셋째, **컨텍스트 오염 방지**. 긴 세션에 쌓인 시행착오를 `/checkpoint`로 핵심 결정만 저장하고, `/clear`로 컨텍스트를 초기화한 뒤, `/restore-checkpoint`로 저장한 결정만 다시 로드합니다. 누적 오류 없이 깨끗한 컨텍스트로 재개합니다.
>
> 공통 원칙 — 한 커맨드는 한 작업. 작업의 경계를 명확하게 끊는 것이 커맨드 설계의 핵심입니다. 가급적 작은 단위, 하나의 책임을 갖는 체계로 커맨드를 설계하세요. `/clear`를 제외한 나머지는 `.claude/commands/`에 직접 만드는 커스텀 예시 커맨드입니다.
>
> 여기까지가 Slash Commands입니다. 다음은 Agent Skills — 다음 강의에서 이어가겠습니다. 감사합니다.

👉 진행 팁: 4.1 파트를 한 편으로 끊어 녹화한다면 여기가 종료 지점입니다. 이어서 녹화한다면 마지막 인사말은 생략하고 슬라이드 9로 바로 넘어가세요.

## 슬라이드 9 — 4.2 Agent Skills 파트 표지 ⏱ ~70초

(화면) PART 02 표지

> 두 번째 파트, Agent Skills입니다.
>
> 커맨드는 사용자가 직접 치는 단축키였습니다. 스킬은 다릅니다. **모델이 스스로 서랍에서 꺼내 보는 매뉴얼**입니다.
>
> 그런데 "모델이 꺼낸다"가 사용자와 무관하다는 뜻은 아닙니다. 스킬이 발동되는 경로는 세 가지입니다.
>
> 첫째, 커맨드처럼 `/스킬명`으로 직접 칠 수 있습니다. 둘째 — 이게 스킬의 진짜 힘인데 — **자연어 발화로 발동됩니다.** 정확한 스킬 이름을 몰라도 됩니다. "이 PR 리뷰해줘"라고 말하면 모델이 그 발화를 스킬들의 description과 대조해서 code-reviewer 스킬을 꺼냅니다. "커밋 메시지 만들어줘"라고 하면 commit 스킬이, "이 마크다운 슬라이드로 만들어줘"라고 하면 슬라이드 변환 스킬이 발동됩니다. 슬래시를 한 번도 안 쳤는데도요. 셋째, 아무 지시가 없어도 모델이 작업 맥락을 보고 "지금 이 스킬이 필요하다"고 판단하면 스스로 호출합니다.
>
> 정리하면 — 커맨드는 정확한 이름을 외워서 치는 단축키, 스킬은 **말하면 알아듣는 도구**입니다. 이 "알아듣기"를 가능하게 하는 것이 description이고, 잠시 뒤에 자세히 다룹니다.
>
> 파트 02에서는 스킬이 어떻게 동작하는지, Reference와 Action 두 종류의 차이, 공식 카탈로그 활용법, 직접 만드는 방법을 다룹니다.

👉 진행 팁: "슬래시를 한 번도 안 쳤는데도요"에서 잠깐 쉬어주세요. 커맨드와의 대비가 가장 선명해지는 지점입니다.

## 슬라이드 10 — 4.2.1 Skill 자가 호출 개념 ⏱ ~75초

(화면) Command vs Skill 비교 표 + description 경고 박스

> 스킬의 핵심 특성은 **자가 호출**입니다.
>
> 비교 표를 보면 — 슬래시 커맨드는 사용자가 `/명령`을 입력해야 발동됩니다. 스킬은 모델이 description을 읽고 "이 상황에 이 스킬이 필요하다"고 판단했을 때 자동으로 호출됩니다. 커맨드는 발동기, 스킬은 컨텍스트 메뉴입니다.
>
> 여기서 **description이 전부**입니다. 모델은 description만 보고 호출 여부를 결정합니다. "언제 써야 하는지"가 명확히 적혀 있지 않으면 스킬은 절대 호출되지 않습니다.
>
> 앞 표지에서 말한 발화 발동이 바로 이 메커니즘입니다. 사용자가 "이 PR 좀 봐줘"라고 말하면, 모델은 등록된 스킬들의 description을 훑습니다. code-reviewer의 description에 "PR 리뷰 요청 시 사용"이라고 적혀 있으면 — 매칭, 발동. 그래서 description에 트리거가 될 발화를 그대로 적어두는 것이 실전 요령입니다. "리뷰해줘", "코드 봐줘" 같은 사용자가 실제로 할 법한 말을요.
>
> 매뉴얼 비유로 설명하면 — description은 **매뉴얼 표지**입니다. 표지에 "코드 리뷰 시 참조"라고 적혀 있어야 모델이 꺼냅니다. "유용한 스킬입니다"라고만 적혀 있으면 서랍 속에서 영원히 잠자게 됩니다.
>
> 실무에서 "스킬을 등록했는데 모델이 안 쓴다"는 문제의 90%는 description 문제입니다. 사용 시점이 명확하지 않은 description입니다.
>
> 통합 메커니즘 하나 더 — 현재 슬래시 커맨드와 스킬은 동일한 lazy-load 메커니즘 위에서 동작합니다. 신규 작성 시에는 SKILL.md 형식을 권장합니다. 모델의 자가 발견·호출이 가능하기 때문입니다.

👉 진행 팁: "description이 전부"에서 잠깐 멈추고 강조하세요. 이 한 문장이 스킬 설계의 90%입니다.

## 슬라이드 15 — 4.2.4 두 가지 Skill 유형: Reference vs Action ⏱ ~55초

(화면) Reference vs Action 비교 카드 2장 + 판단 기준

> 스킬을 뜯어보기 전에 유형부터 정리합니다. 모든 스킬은 결국 둘 중 하나로 수렴합니다 — 지식을 주는가, 절차를 실행하는가.
>
> **Reference Skill**은 인용형입니다. "X 작업할 때 이 규칙을 참조해." 지식과 컨벤션만 제공하고 도구 호출은 거의 없습니다. allowed-tools가 Read 정도로 끝나고, 바꾸는 것은 모델의 **판단** — 무엇이 옳은 코드인가입니다. 코딩 표준, 용어집, 커밋 컨벤션이 여기 속합니다.
>
> **Action Skill**은 실행형입니다. "X 요청이 오면 Y 절차로 처리해." 절차와 도구 호출까지 하나로 묶습니다. Bash와 Write까지 허용되고, 바꾸는 것은 모델의 **행동** — 무엇을 어떤 순서로 실행하느냐입니다. 스테이징 배포, DB 백업 같은 것들입니다.
>
> 판단 기준은 간단합니다. 읽고 참고만 하면 Reference, 실행까지 하면 Action — **allowed-tools에 Bash나 Write가 들어가는 순간 Action**입니다. 한 스킬이 둘을 섞기도 합니다. 규칙을 제공하면서 마무리 검증 명령까지 실행하는 식이죠.
>
> 이제 두 유형을 실제 SKILL.md 파일로 하나씩 뜯어보겠습니다.

## 슬라이드 24 — 4.3 Hooks 파트 표지 ⏱ ~60초

(화면) PART 03 표지 — 부제: "8개 이벤트 · stdin JSON · 가드레일 훅, 주의사항과 디버깅 — 모델 행동을 결정적으로 통제하는 유일한 수단"

>
> 훅을 걸 수 있는 **8개 이벤트** —를 설명할거구요. 어느 시점에 끼어들 수 있는가. 훅이 정보를 받는 통로인 **stdin JSON**standard input(표준 입력) — 무슨 도구가 무슨 인자로 불렸는지를 훅이 읽어 들이는 방법. 위험 명령·보호 파일을 막는 **가드레일 훅**과 포맷·알림을 자동화하는 이런것들을 살펴보고 여러가지 예제들을 클로드 코드를 통해서 프로젝트 진행할 때 같이 구현해볼 예정입니다.

## 슬라이드 25 — 4.3.1 Hook 8개 이벤트 ⏱ ~95초

(화면) 8개 이벤트 표

> 훅을 걸 수 있는 이벤트 8개입니다. "훅을 건다"는 건 "이 시점에 내 스크립트를 끼워 넣는다"는 뜻이고, 그래서 이벤트를 아는 게 곧 끼어들 수 있는 시점을 아는 겁니다.
>
> **PreToolUse** — 도구 호출 직전. 차단·승인 게이트로 씁니다. 여기서 용어 하나 미리 풀겠습니다 — **exit code(종료 코드)** = 스크립트가 끝나면서 운영체제에 돌려주는 숫자입니다. 0이면 성공, 0이 아니면 실패라는 게 셸의 관례입니다. 훅에서는 이 숫자로 Claude Code에 신호를 보냅니다. PreToolUse 훅이 **exit 2**를 반환하면 그 도구 호출이 막힙니다. 이 "exit 2 = 차단"은 훅 파트 내내 반복되니 지금 기억해두세요.
>
> **PostToolUse** — 도구 호출 직후. 결과 검증이나 자동 포맷, 로깅에 씁니다.
>
> **UserPromptSubmit** — 사용자 메시지 제출 직후. 입력을 정제하거나 라우팅하는 데 씁니다.
>
> **Notification** — 모델이 권한이나 정보를 요청할 때. 알림 발송에 씁니다.
>
> **Stop** — 모델 응답 종료 시. 응답이 끝날 때마다 한 번 실행됩니다.
>
> **SubagentStop** — 서브 에이전트 종료 시.
>
> **PreCompact** — `/compact` 직전. 압축 전 현재 상태를 저장합니다.
>
> **SessionStart** — 세션 시작 시. 핸드오프 파일을 주입하는 데 씁니다.
>
> 여기서 앞으로 계속 나올 용어 하나를 확실히 짚고 갑니다 — **matcher(매처)**입니다. 방금 PreToolUse·PostToolUse에서 "어느 도구에 훅을 걸지"를 정해야 하는데, 그 도구를 골라내는 **필터**가 matcher입니다. settings.json에 정규식으로 적습니다. 예를 들어 `"Edit|Write"`라고 쓰면 세로줄(`|`)이 "또는"이라는 뜻이라 Edit 또는 Write 도구를 부를 때만 훅이 발화하고, `"Bash"`라고 쓰면 Bash에만 걸립니다. 쉽게 말해 "이 훅은 어떤 도구를 감시할래?"에 대한 답입니다.
>
> 그래서 주의사항 — settings.json에서 matcher를 아예 누락하면 해당 이벤트의 **모든** 도구 호출에 훅이 걸립니다. Read나 Grep 같은 읽기 도구까지 전부 가로채서 성능이 눈에 띄게 저하됩니다. matcher는 반드시 지정하세요.

## 슬라이드 27 — 4.3.2 PreCompact + SessionStart 핸드오프 ⏱ ~100초

(화면) settings.json PreCompact·SessionStart 훅 설정 코드

> PreCompact와 SessionStart를 조합한 핸드오프 자동화 패턴입니다.
>
> settings.json에서 PreCompact 이벤트에 `dump-handoff.sh`를 연결하고, SessionStart 이벤트에 `inject-handoff.sh`를 연결합니다. matcher는 각각 "manual"과 "resume"입니다.
>
> 여기서 matcher가 아까와 다르게 쓰인 걸 눈치채셨을 겁니다. 앞 슬라이드에서 matcher는 "Edit|Write" 같은 **도구 이름 필터**라고 했는데, PreCompact·SessionStart는 도구와 엮인 이벤트가 아닙니다. 이럴 때 matcher는 **그 이벤트가 어떤 방식으로 발생했는지**를 거릅니다. PreCompact의 "manual"은 사용자가 손수 `/compact`를 쳤을 때만 — 자동 압축 때는 안 돕니다. SessionStart의 "resume"은 `--resume`으로 세션을 재개했을 때만 발화합니다. 그러니까 matcher는 한 문장으로 "이 훅을 언제 발화시킬지 좁히는 필터"이고, 이벤트 성격에 따라 도구 이름이 되기도, 발생 방식이 되기도 합니다.
>
> 동작 흐름 — 사용자가 수동으로 `/compact`를 호출하면 PreCompact 훅이 발화하고 dump-handoff.sh가 현재 결정과 미해결 사항을 파일로 저장합니다. `--resume`으로 세션을 재개하면 SessionStart 훅이 inject-handoff.sh를 실행해서 저장된 내용을 새 세션에 주입합니다.
>
> 결과적으로 세션이 끊겨도 컨텍스트가 사라지지 않습니다. 안전망입니다.
>
> 적용 시 체크리스트 세 가지 — dump-handoff.sh와 inject-handoff.sh 스크립트 작성, 두 파일에 실행 권한 `chmod +x` 적용, 그리고 `jq` 설치입니다. **jq**는 커맨드라인에서 **JSON(자바스크립트 객체 표기법)**을 다루는 파서로, 이름은 'JSON에 질의(query)한다'에서 왔습니다. 뒤에 보겠지만 훅은 이벤트 정보를 JSON 덩어리로 받는데, 그 안에서 원하는 필드만 뽑아낼 때 jq를 씁니다. (stdin·JSON·jq 셋의 정확한 약자와 동작은 슬라이드 28에서 한 번에 정리합니다.) 이 세 가지 중 하나라도 빠지면 훅이 에러도 없이 **조용히 실패**합니다. 특히 jq는 macOS 기본 설치가 아니라 `brew install jq`가 따로 필요합니다.

👉 진행 팁: 핸드오프 패턴은 Level 8 하네스 설계의 핵심 요소 중 하나입니다. "세션이 바뀌어도 작업 맥락을 이어가는 구조"를 지금 간략하게 체감하는 정도로 소개하면 됩니다.

## 슬라이드 28 — 4.3.3 stdin JSON 페이로드 ⏱ ~95초

(화면) stdin 페이로드 필드 표

> 훅 스크립트가 어떻게 정보를 받는지입니다. 이 슬라이드에서 파트 표지에 적힌 "stdin JSON"의 정체를 낱낱이 풉니다. 세 단어부터 짚고 가겠습니다.
>
> 먼저 **stdin** — **standard input(표준 입력)**의 줄임말입니다. 프로그램이 데이터를 받아 들이는 기본 입력 통로를 말합니다. 우리가 키보드로 치는 것도, 파이프(`|`)로 앞 명령의 출력을 넘기는 것도 전부 이 stdin으로 들어갑니다. 짝이 되는 개념이 **stdout**(standard output, 표준 출력)과 앞에서 본 stderr입니다. 이 셋이 모든 유닉스 프로그램의 기본 입출력 세 통로입니다.
>
> 다음 **JSON** — **JavaScript Object Notation(자바스크립트 객체 표기법)**의 약자입니다. `{ "키": "값" }` 꼴로 데이터를 적는 텍스트 형식입니다. 원래 자바스크립트에서 나왔지만 지금은 언어를 안 가리는 사실상 표준 데이터 교환 포맷입니다. 사람도 읽을 수 있고 기계도 파싱하기 쉬워서 API·설정 파일이 다 이걸 씁니다.
>
> 합치면 — 훅은 환경변수가 아니라 **stdin으로 JSON 한 덩어리**를 받습니다. Claude Code가 "지금 이런 이벤트가 났고, 이런 도구를 이런 인자로 부르려 한다"는 상황을 JSON으로 적어서 훅 스크립트의 stdin에 흘려보내는 겁니다. 이 JSON 덩어리를 **페이로드(payload)** — '실어 나르는 알맹이'라는 뜻 — 라고 부릅니다.
>
> 스크립트에선 `payload=$(cat)`으로 stdin을 통째로 변수에 담습니다. `cat`이 stdin을 그대로 읽어주죠. 그다음 이 JSON에서 원하는 값만 뽑을 때 **jq**를 씁니다. **jq**는 이름 그대로 **JSON에 질의(query)하는** 커맨드라인 도구입니다 — 텍스트를 다루는 `sed`·`awk`의 JSON 판이라고 보면 됩니다. 예를 들어 `echo "$payload" | jq -r '.tool_name'`이라고 하면 페이로드 안에서 tool_name 필드 값 하나만 깔끔히 꺼내줍니다. stdin으로 받고 — jq로 뽑고 — 이 두 박자가 모든 훅 스크립트의 공통 뼈대입니다.
>
> 주요 필드 다섯 개입니다. `session_id` — 현재 세션 식별자. `cwd` — Claude Code 실행 디렉터리. `tool_name` — 호출되려는 도구 이름. `tool_input` — 도구 인자 객체. Bash 훅이면 `{"command": "npm test"}`가 들어옵니다. `tool_response` — PostToolUse에서는 도구 실행 결과가 여기 들어옵니다.
>
> 보호 파일 훅에서 `tool_input.file_path`를 꺼내 검사하는 것, 알림 훅에서 `message` 필드를 꺼내 알림 내용으로 쓰는 것 — 다음 슬라이드들에서 이 필드들이 실제로 어떻게 쓰이는지 보게 됩니다.

## 슬라이드 30 — 4.3.4 Slack Webhook 개설·연동 ⏱ ~110초

(화면) Slack Webhook 개설 4단계 + OS별 환경변수 등록 절차 장표

> 다음은 Slack 알림 훅입니다. 그런데 스크립트를 보기 전에 그 재료인 **Webhook URL**부터 만들어야 합니다. 이게 없으면 스크립트가 메시지를 쏠 곳이 없으니까요. **Webhook(웹훅)** = 어떤 이벤트가 났을 때 미리 등록해둔 URL로 데이터를 자동으로 쏘아 보내는 방식입니다. 우리는 Slack이 발급한 URL 하나에 메시지를 POST하면 지정한 채널에 글이 뜨는 구조를 씁니다.
>
> 개설은 네 단계입니다.
>
> 첫째, 브라우저에서 **api.slack.com/apps** 에 접속해 "Create New App"을 누릅니다. "From scratch"를 선택하고, 앱 이름 — 예를 들어 claude-notify — 을 적고, 알림을 받을 워크스페이스를 선택합니다.
>
> 둘째, 만들어진 앱 설정 화면 왼쪽 메뉴에서 **"Incoming Webhooks"** 를 클릭하고, 상단 토글을 **On**으로 켭니다.
>
> 셋째, 같은 화면 하단의 **"Add New Webhook to Workspace"** 버튼을 누르면 채널 선택 화면이 나옵니다. 알림을 받을 채널 — #dev-alerts 같은 — 을 고르고 허용을 누릅니다.
>
> 넷째, 화면에 생긴 **`https://hooks.slack.com/services/T.../B.../...`** 형태의 URL을 복사합니다. 이게 Webhook URL입니다.
>
> URL을 손에 넣었으면 이제 환경변수로 등록합니다. 등록 위치는 OS마다 다른데, 뒤에 나올 API 키 저장 방법(슬라이드 48)과 **완전히 같은 패턴**이니 여기서 한 번 익혀두면 그대로 재사용됩니다.
>
> **macOS·Linux** — 셸 프로필에 넣습니다. 맥 기본 셸은 zsh라 `~/.zshrc`, 배시면 `~/.bashrc`입니다. `export SLACK_WEBHOOK_URL="복사한 URL"` 한 줄을 추가하고, `source ~/.zshrc`로 현재 창에 반영하거나 새 터미널을 엽니다. 확인은 `echo $SLACK_WEBHOOK_URL`.
>
> **Windows** — `setx SLACK_WEBHOOK_URL "복사한 URL"`로 사용자 환경변수에 등록합니다. 주의: setx는 **새 터미널부터** 적용됩니다. 열어둔 창에서 안 뜬다고 당황하지 마세요. PowerShell에서 값 확인은 `echo $env:SLACK_WEBHOOK_URL`.
>
> **WSL** — WSL 안에서 claude를 돌리면 Windows 쪽 환경변수가 전달되지 않습니다. WSL 리눅스의 `~/.bashrc`에 macOS·Linux와 똑같이 `export`로 따로 넣어야 합니다. 기준은 "claude를 어느 셸에서 띄우느냐"입니다.
>
> 등록했으면 연동 테스트. **macOS·Linux·WSL**은 새 터미널에서 `curl -X POST -H 'Content-type: application/json' --data '{"text":"연동 테스트"}' $SLACK_WEBHOOK_URL` 한 줄. **Windows PowerShell**은 curl 대신 `Invoke-RestMethod -Uri $env:SLACK_WEBHOOK_URL -Method Post -ContentType 'application/json' -Body '{"text":"연동 테스트"}'`. 채널에 "연동 테스트" 메시지가 뜨면 개설·연동 완료입니다.
>
> 마지막으로 보안 — **이 URL 자체가 비밀번호입니다.** URL을 아는 사람은 누구나 그 채널에 메시지를 보낼 수 있습니다. 코드나 settings.json에 하드코딩해서 커밋하면 안 되고, 반드시 환경변수로만 씁니다. 실수로 커밋했다면 Slack 앱 설정에서 그 Webhook을 폐기(revoke)하고 새로 발급하세요.

👉 진행 팁: 이 절차는 말로만 하면 따라오기 어렵습니다. 녹화 시 브라우저로 api.slack.com/apps 화면을 실제로 띄워 네 단계를 클릭해 보여준 뒤, 다음 슬라이드의 스크립트로 넘어가세요.

## 슬라이드 33 — 4.3.5 가드레일: 보호 파일 ⏱ ~65초

(화면) protect-paths.sh 스크립트

> 가드레일 훅입니다. 과속 방지턱 비유에서 가장 직접적인 예입니다.
>
> `protect-paths.sh`는 PreToolUse, matcher "Edit|Write"에 연결합니다. 파일을 쓰거나 수정하는 모든 도구 호출 직전에 이 스크립트가 실행됩니다.
>
> 스크립트가 하는 일은 하나입니다. stdin 페이로드에서 `tool_input.file_path`를 꺼내서 보호 경로 목록과 비교합니다. `.env`, `credentials`, `.git/config` — 이 경로와 일치하면 **stderr**에 사유를 출력하고 **exit 2**를 반환합니다. 여기서 **stderr(표준 에러 출력)** = 프로그램이 정상 결과가 아닌 오류·진단 메시지를 내보내는 별도 출력 채널입니다. 훅에서는 이 stderr가 **모델에게 말을 거는 통로**입니다 — exit 2로 차단할 때 stderr에 적어둔 사유가 차단과 함께 모델에게 전달됩니다. 그래서 모델이 왜 막혔는지 알고 다른 방법을 시도할 수 있습니다. exit 2인데 stderr가 비어 있으면 모델은 이유를 모른 채 막히기만 합니다.
>
> 이게 방지턱입니다. 모델이 아무리 급해도, 사용자 지시가 있어도, .env는 건드릴 수 없습니다. 부탁으로 막는 것이 아니라 구조가 막습니다.

## 슬라이드 42 — 4.4.1 MCP 아키텍처 다이어그램 (Host · Client · Server) ⏱ ~55초

(화면) MCP Host · Client · Server 아키텍처 SVG — USB-C 은유

> 이 한 장에 MCP 구조가 전부 담겨 있습니다. 맨 위 문구 그대로 — **MCP는 AI를 위한 표준 연결 포트**입니다. USB-C처럼 프로토콜 하나로 모든 외부 도구·데이터를 같은 방식으로 연결합니다.
>
> 구조는 3층입니다. **MCP Host** — Claude Code·Claude 앱·기타 AI 도구, 모델이 도는 쪽입니다. **MCP Client** — 호스트 안에서 서버마다 1:1 연결을 유지하는 부분입니다. **MCP Server** — 실제 능력을 제공하는 쪽으로, 화면엔 GitHub(PR·이슈·코드 리뷰), Slack(메시지·채널 알림), Filesystem(로컬 파일 읽기·쓰기) 셋이 꽂혀 있습니다.
>
> 오른쪽 위 GitHub·Slack은 **원격 서비스**, 아래 Filesystem은 **로컬 데이터**입니다. 핵심은 가운데 화살표입니다 — 호스트는 서버가 무엇이든 **같은 MCP 프로토콜**로 대화합니다. 그래서 맨 아래 한 줄이 결론입니다. **서버 하나를 더 꽂으면 능력이 하나 늘어난다 — 호스트 코드는 그대로.** 이게 USB-C의 의미입니다. 다음 슬라이드에서 이 연결을 실제로 어떻게 잇는지, 연결 방식을 보겠습니다.

## 슬라이드 43 — 4.4.1 Transport 2종: http · stdio ⏱ ~75초

(화면) Transport 비교 표 + GitHub MCP settings 예시

> 앞 장이 MCP의 구조였다면, 이 장은 그 호스트-서버 연결을 실제로 어떻게 잇느냐입니다. **Transport(트랜스포트)** = Claude Code가 MCP 서버와 데이터를 주고받는 연결 방식, 즉 통신 통로입니다. 두 가지가 있습니다.
>
> **http** — 원격 HTTP 서버 방식. 권장 방식입니다. 스트리밍과 인증 헤더를 지원합니다. GitHub MCP 서버처럼 외부 서비스에 연결할 때 씁니다.
>
> **stdio** — 로컬 프로세스 방식. npx나 uvx로 실행하는 로컬 도구·CLI 래퍼에 씁니다. 여기서 **npx** = Node 패키지를 따로 설치하지 않고 바로 실행해주는 실행기이고, **uvx** = 그 파이썬 판, uv 도구가 파이썬 패키지를 즉석 실행해주는 명령입니다. 둘 다 "설치 없이 그 자리에서 실행"이라 로컬 MCP 서버를 띄우는 데 자주 쓰입니다.
>
> 화면의 GitHub MCP 설정을 보면 — `.mcp.json`에 type "http", url, headers로 Authorization 토큰을 넣습니다. 이 설정 하나로 Claude가 GitHub 이슈, PR, 코드 검색, 코드 리뷰를 직접 다룰 수 있게 됩니다. 사람이 GitHub 브라우저를 열지 않아도 모델이 직접 대화합니다.
>
> CLI로 추가하는 방법도 있습니다. `claude mcp add --transport http github <url>` 한 줄입니다.

## 슬라이드 44 — 4.4.1 Scope — 저장 위치 + 4단계 진단 ⏱ ~85초

(화면) 다이어그램 2개 — ① Scope별 저장 위치 표 ② MCP 안 잡힐 때 4단계 진단

> MCP 서버의 등록 범위, Scope입니다. 화면에 다이어그램이 둘인데, 위는 **어디에 저장되나**, 아래는 **안 잡힐 때 어떻게 고치나**입니다.
>
> 먼저 저장 위치. 세 가지입니다. **local**(기본값) — `~/.claude.json`의 `projects.<경로>.mcpServers`에 들어가고, **현재 사용자 + 현재 프로젝트만** 봅니다. 개인용이라 git 공유는 안 됩니다. **project** — 프로젝트 루트의 `.mcp.json`입니다. 주의할 점, `.claude/` 안이 아니라 **루트 직속**입니다. 모든 사용자 + 현재 프로젝트가 공유하고, git에 커밋합니다. **user** — `~/.claude.json`의 **최상위 `mcpServers` 키**에 들어가고, 현재 사용자의 모든 프로젝트에서 씁니다.
>
> 여기 함정이 하나 있습니다. `~/.claude.json`은 **user와 local을 둘 다 담는 이중 인격 파일**입니다. 같은 파일인데 — **최상위 `mcpServers`면 user 스코프**, **`projects.<경로>.mcpServers`면 local 스코프**입니다. 이걸 헷갈리면 "분명 등록했는데 왜 안 보이지"가 됩니다.
>
> 그래서 아래 다이어그램이 **MCP가 안 잡힐 때 4단계 진단**입니다. 위에서 아래로 순차 점검하면 90%는 여기서 풀립니다. **① 파일 위치** — `.mcp.json`은 프로젝트 루트, `~/.claude.json`은 홈. 검증 `ls -la ~/.claude.json .mcp.json`. **② Scope 선택** — 의도한 스코프에 저장됐나, `projects.` 키가 정확한가. 검증 `cat ~/.claude.json | jq '.projects | keys'`. **③ 환경변수** — `${VAR}` 보간은 새 셸에서만 반영되니 셸 프로필에 export 후 재시작. 검증 `echo $GITHUB_TOKEN`. **④ 등록 확인** — `/mcp` 명령이나 `claude mcp list`로 서버 목록을 봅니다. 안 보이면 1번부터 다시.
>
> 선택 기준은 단순합니다. 팀 전체가 쓰면 project, 혼자 모든 프로젝트에서 쓰면 user, 이 프로젝트에서만 개인적으로 쓰면 local.

## 슬라이드 47 — 4.4.3 .mcp.json 구문 ⏱ ~50초

(화면) .mcp.json 코드 블록

> 팀과 공유할 때는 커맨드 대신 파일입니다. 프로젝트 루트에 `.mcp.json`을 만들어 git에 커밋하면, 팀원 전체가 같은 MCP 구성을 씁니다. 앞에서 배운 project 스코프가 바로 이 파일입니다.
>
> 구문은 두 갈래입니다. 원격 서버는 `type: http`에 `url`과 `headers`, 로컬 서버는 `command`와 `args`, 필요하면 `env`까지. 화면 예시에 두 방식이 다 들어 있습니다.
>
> 마지막으로 키 관리. 이 파일은 git에 올라가는 파일입니다. API 키를 그대로 적으면 사고입니다. 예시의 `${GITHUB_PAT}`에서 GITHUB_PAT는 GitHub **PAT(Personal Access Token, 개인 액세스 토큰)** — GitHub이 비밀번호 대신 발급해주는 접근 키 — 을 담은 환경변수 이름입니다. 이렇게 `${환경변수이름}` 꼴로 적어두면 Claude Code가 파일을 읽을 때 그 자리를 실제 환경변수 값으로 치환합니다. 키 자체는 셸 환경에 두고 — 맥은 `~/.zshrc`, 윈도우는 `setx`, WSL은 WSL 안 `~/.bashrc` — 파일에는 자리표시자만 남깁니다. 이 원칙만 지키면 됩니다.

## 슬라이드 48 — 4.4.3 API 키 저장 위치: ${ENV} OS별 세팅 ⏱ ~60초

(화면) macOS / Windows 코드블록 2열 + WSL 주의 밴드

> 방금 본 .mcp.json의 ${GITHUB_PAT} 같은 플레이스홀더 — 그 키는 어디에 저장하느냐. 원리는 하나입니다. **claude를 실행한 셸의 환경변수**에서 치환됩니다. .mcp.json 자체는 git에 커밋되는 파일이니까, 키는 절대 여기 쓰지 않고 셸 프로필에만 둡니다.
>
> **macOS** — `~/.zshrc`에 export로 등록하고 `source` 후 claude를 재시작합니다. 더 안전하게 가려면 Keychain에 키를 넣고 `security find-generic-password`로 꺼내 쓰는 한 줄 패턴이 있습니다. 프로필 파일에도 평문이 안 남습니다.
>
> **Windows** — `setx`로 사용자 환경 변수에 등록합니다. 주의할 점 하나: setx는 **새 터미널부터** 적용됩니다. 열려 있던 창에서 안 된다고 당황하지 마세요. GUI로 하려면 시스템 속성 → 고급 → 환경 변수입니다. 확인은 PowerShell에서 `echo $env:GITHUB_PAT`.
>
> **WSL 함정** — WSL에서 claude를 돌리면 Windows 환경변수가 전달되지 않습니다. WSL 쪽 `~/.bashrc`에 따로 export 해야 합니다. 어느 셸에서 claude를 띄우는지가 기준입니다.
>
> 정리하면 — 키는 셸 프로필에, .mcp.json엔 ${} 플레이스홀더만. 커밋 한 번이면 유출입니다.

| **Harness** | 에이전트에서 모델을 뺀 나머지 전부 — 컨텍스트 큐레이션·도구 관리·권한 제어·에러 복구 |
| **하네스** | Harness 한국어 표기. Level 8에서 심화 |
| **defense in depth** | 방어의 깊이 — 가이드·연산적·시스템 프롬프트·추론적 네 층을 겹쳐 한 층이 뚫려도 다음 층이 잡는 하네스 설계 원칙 |
| **plugin** | 스킬·커맨드 묶음 배포 단위 |
| **auto memory** | Claude Code **내장** 메모리(v2.1.59+ 기본 활성). Claude가 스스로 작업 노트를 `~/.claude/projects/<프로젝트>/memory/`에 축적, MEMORY.md 첫 200줄이 매 세션 자동 로드. 설치 불필요, `/memory`로 열람·토글 |
| **MCP Memory 서버** | `@modelcontextprotocol/server-memory` — 지식 그래프(개체·관계·관찰) 저장소. 내장 auto memory와 **별개**, 별도 설치 필요 |

### 내장 memory vs MCP Memory 서버 — 별개 기능 (공식 문서 확인)

- **내장** (`code.claude.com/docs/en/memory`): ① CLAUDE.md — 사용자가 쓰는 영속 지침 ② **auto memory** — v2.1.59+ 기본 활성, Claude가 스스로 마크다운 노트 축적(MEMORY.md 인덱스 + 토픽 파일). 설치 없이 동작. 공식 memory 문서에 MCP 서버 언급 없음 = 통합 아님.
- **MCP `server-memory`**: entity–relation–observation **지식 그래프**를 도구 호출(create_entities·search_nodes 등)로 저장·질의. 별도 설치품.
- **판단 기준**: "기억해줘" 수준 → 내장으로 충분, 설치 불필요. 관계형 질의·Claude Desktop 등 타 MCP 클라이언트와 메모리 공유 필요 → 그때만 MCP 서버 설치.
- 같은 패턴 주의: Filesystem MCP도 내장 Read/Write와 겹쳐 보임 — 차별점은 **허용 디렉토리 제한**(인자로 넘긴 경로 밖 접근 차단).

### `.mcp.json`의 `${ENV}` 키 — OS별 저장 위치

`${GITHUB_PAT}`는 **claude를 실행한 셸의 환경변수**에서 치환된다. 키를 넣는 곳:

| OS | 저장 위치 | 명령 |
|---|---|---|
| macOS/Linux | `~/.zshrc`(또는 `~/.bashrc`) — dotfiles git 관리 시 `~/.env.secrets` 분리(chmod 600) 후 source | `export GITHUB_PAT="ghp_xxxx"` |
| macOS (안전판) | Keychain | `security add-generic-password -s github-pat -w "..."` → rc에서 `export GITHUB_PAT=$(security find-generic-password -s github-pat -w)` |
| Windows | 사용자 환경변수(레지스트리 HKCU\Environment) — **새 터미널부터 적용** | `setx GITHUB_PAT "ghp_xxxx"` 또는 PowerShell `[Environment]::SetEnvironmentVariable(...,"User")` |
| WSL | Windows 아닌 **WSL 안** `~/.bashrc` (두 세계 환경변수 비공유) | Linux와 동일 |

함정: `claude mcp add --env KEY=실제값`은 설정 파일에 리터럴 저장 — 커밋되는 `.mcp.json`엔 `${...}` 자리표시자만.

## 부록 — 커맨드·스킬·훅 설정 요소 (2026-07-10 공식 문서 검증)

> 슬라이드 7(4.1.4)·4.3 Hooks 파트 팩트 근거. 전체 필드 표는 `docs/keynote/level4-frontmatter-레퍼런스.md` 참조.

- **커맨드·스킬 frontmatter는 통합 — 스키마 동일.** `.claude/commands/`는 대체 위치 취급, 신규는 SKILL.md 권장. "살짝 다르다"는 표현은 전형적 사용 필드 차이(커맨드=`argument-hint` 인자 처리, 스킬=`description` 자동 호출)로 말할 것.
- 공식 필드: `name` `description` `when_to_use` `argument-hint` `arguments` `disable-model-invocation` `user-invocable` `allowed-tools` `disallowed-tools` `model` `effort` `context`(fork) `agent` `hooks` `paths` `shell`. — `metadata`/`version`/`license`는 비공식, 강의에 싣지 말 것.
- **훅은 frontmatter 없음 — settings.json JSON only.** 요소: 이벤트 키(PreToolUse·PostToolUse·UserPromptSubmit·SessionStart 등 24종+) > `matcher`/`if` > `type`(command·http·prompt·agent·mcp_tool)+`command`+`timeout`. stdin으로 이벤트 JSON 수신, exit 0=진행 / **exit 2=차단+stderr가 모델 피드백** / 기타=진행+stderr 첫 줄 표시.
- 경계 겹침 한 곳: 스킬 frontmatter의 `hooks` 필드 = 스킬 활성 중에만 도는 스코프 훅.
