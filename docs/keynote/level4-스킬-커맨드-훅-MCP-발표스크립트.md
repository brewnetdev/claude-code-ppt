>
> 그리고 네 번째, **MCP는 USB-C 포트입니다.** 어떤 외부 장치든 같은 규격으로 연결합니다.

## 슬라이드 6 — 4.1.3 커스텀 커맨드 ⏱ ~70초

(화면) .claude/commands/code-review.md 코드 예시 + 스코프 설명

> 커스텀 커맨드의 구조입니다. `.claude/commands/` 아래 마크다운 파일 하나를 만들면 그게 슬래시 커맨드가 됩니다.
>
> 화면의 `/code-review` 파일은 세 부분으로 구성됩니다.
>
> 첫째, frontmatter — 파일 상단의 YAML 블록. `description`에 이 커맨드가 무엇을 하는지 적고, `allowed-tools`에 이 커맨드가 쓸 수 있는 도구만 명시합니다. git 명령과 Read, Grep만 허용했습니다.
>
> 둘째, 본문 — 프롬프트 텍스트입니다. 절차를 단계별로 적어두면 모델이 그대로 따릅니다.
>
> 스코프는 두 가지입니다. `.claude/commands/`는 프로젝트 전용으로 팀 공유가 되고, `~/.claude/commands/`는 모든 프로젝트에서 쓰는 개인 커맨드입니다. 둘 다 있으면 프로젝트 쪽이 우선합니다.
>
> 이 커맨드가 없었다면 코드 리뷰를 요청할 때마다 "git diff 보고, 파일별로 의도·위험·누락 테스트 정리하고, 보안·성능 관점 확인해서, 표 형태로 출력해줘"라는 긴 지시를 매번 반복해야 합니다. 파일 하나로 그 반복이 없어집니다.

👉 진행 팁: `allowed-tools` 제한이 왜 중요한지 짚어주세요. 도구를 열어두면 모델이 예상 외의 동작을 할 수 있습니다.

## 슬라이드 7 — 4.1.4 $ARGUMENTS ⏱ ~70초

(화면) fix-issue.md 와 add-feature.md 코드 예시

> 커맨드에 인자를 받고 싶을 때는 `$ARGUMENTS`를 씁니다.
>
> 왼쪽 예시 `fix-issue.md`를 보면, 커맨드 본문 곳곳에 `$ARGUMENTS`가 있습니다. `/fix-issue 42`라고 치면 `$ARGUMENTS` 자리에 `42`가 치환됩니다. `gh issue view 42`, `gh issue view 42 --comments`, `Closes #42`까지 이슈 번호가 자동으로 들어갑니다.
>
> 오른쪽 `add-feature.md`는 기능 스캐폴딩 커맨드입니다. `/add-feature user-profile`이라고 치면 커맨드 안에서 먼저 정규식 검증을 합니다. 소문자·숫자·하이픈 조합인지, 길이가 2~40자인지 확인한 뒤에 스캐폴딩을 진행합니다.
>
> 잘못된 인자가 들어왔을 때 모델이 무작정 진행하지 않도록 검증 절차를 커맨드 안에 박아둔 겁니다. `$ARGUMENTS`는 커맨드를 일회성 지시가 아니라 재사용 가능한 도구로 만드는 핵심 요소입니다.

## 슬라이드 8 — 4.1.5 실전 패턴 3종 ⏱ ~80초

(화면) TDD 사이클 / 자동화 파이프라인 / 컨텍스트 오염 방지 세 블록

> 커스텀 커맨드를 실전에서 조합하는 패턴 세 가지입니다.
>
> 첫째, **TDD 사이클**. `/tdd-red`로 실패하는 테스트만 작성하고, `/tdd-green`으로 그 테스트를 통과하는 최소 구현만 작성하고, `/tdd-refactor`로 구조만 정리합니다. 세 커맨드가 단계를 강제로 분리합니다. Red 없이 Green을 실행하면 순서가 무너집니다.
>
> 둘째, **자동화 파이프라인**. 커밋 전 점검 네 가지 — `/lint-fix`, `/typecheck`, `/test-affected`, `/commit-msg` — 를 매번 순서대로 직접 칠 수도 있습니다. 하지만 그건 자동화가 아니라 암기죠. 자동화의 실체는 **래퍼 커맨드 하나**입니다. `.claude/commands/pre-commit.md` 파일 본문에 이 네 단계를 순서대로 적어 두면, `/pre-commit` 한 줄로 전체가 실행됩니다. 단축키 네 개를 묶은 매크로 키 하나를 만드는 셈입니다. 중간 단계가 실패하면 거기서 멈추고, 고친 뒤 이어갑니다. CI에 가기 전에 로컬에서 같은 순서로 사전 점검이 끝납니다.
>
> 셋째, **컨텍스트 오염 방지**. 긴 세션에 쌓인 시행착오를 `/checkpoint`로 핵심 결정만 저장하고, `/clear`로 컨텍스트를 초기화한 뒤, `/restore-checkpoint`로 저장한 결정만 다시 로드합니다. 누적 오류 없이 깨끗한 컨텍스트로 재개합니다.
>
> 공통 원칙 — 한 커맨드는 한 작업. 작업의 경계를 명확하게 끊는 것이 커맨드 설계의 핵심입니다. `/clear`를 제외한 나머지는 `.claude/commands/`에 직접 만드는 커스텀 예시 커맨드입니다.

## 슬라이드 10 — 4.2.1 Skill 자가 호출 개념 ⏱ ~75초

(화면) Command vs Skill 비교 표 + description 경고 박스

> 스킬의 핵심 특성은 **자가 호출**입니다.
>
> 비교 표를 보면 — 슬래시 커맨드는 사용자가 `/명령`을 입력해야 발동됩니다. 스킬은 모델이 description을 읽고 "이 상황에 이 스킬이 필요하다"고 판단했을 때 자동으로 호출됩니다. 커맨드는 발동기, 스킬은 컨텍스트 메뉴입니다.
>
> 여기서 **description이 전부**입니다. 모델은 description만 보고 호출 여부를 결정합니다. "언제 써야 하는지"가 명확히 적혀 있지 않으면 스킬은 절대 호출되지 않습니다.
>
> 매뉴얼 비유로 설명하면 — description은 **매뉴얼 표지**입니다. 표지에 "코드 리뷰 시 참조"라고 적혀 있어야 모델이 꺼냅니다. "유용한 스킬입니다"라고만 적혀 있으면 서랍 속에서 영원히 잠자게 됩니다.
>
> 실무에서 "스킬을 등록했는데 모델이 안 쓴다"는 문제의 90%는 description 문제입니다. 사용 시점이 명확하지 않은 description입니다.
>
> 통합 메커니즘 하나 더 — 현재 슬래시 커맨드와 스킬은 동일한 lazy-load 메커니즘 위에서 동작합니다. 신규 작성 시에는 SKILL.md 형식을 권장합니다. 모델의 자가 발견·호출이 가능하기 때문입니다.

👉 진행 팁: "description이 전부"에서 잠깐 멈추고 강조하세요. 이 한 문장이 스킬 설계의 90%입니다.

## 슬라이드 11 — 4.2.1 Skill 기본 구조 ⏱ ~55초

(화면) SKILL.md 기본 템플릿

> 스킬 파일 하나의 구조입니다.
>

## 슬라이드 18 — 4.2.4 Action Skill: deploy-staging ⏱ ~70초

(화면) deploy-staging SKILL.md 전체

> **Action Skill**은 다릅니다. "X 요청이 들어오면 Y 절차로 처리해" — 도구 호출까지 묶음입니다.
>
> `deploy-staging` 스킬을 보겠습니다. description — "사용시점: 사용자가 '스테이징에 배포해줘' 같은 요청을 할 때." allowed-tools는 `Bash(npm:*)`, `Bash(vercel:*)`, Read. npm과 vercel CLI만 열었습니다.
>
> 두 가지 특별한 설정이 있습니다. `disable-model-invocation: true` — 이걸 true로 설정하면 모델이 자동으로 호출하는 것을 막습니다. 사용자가 명시적으로 `/deploy-staging`을 입력해야만 동작합니다. 배포는 의도적으로 실행해야 하는 작업이기 때문입니다. `argument-hint` — 커맨드 입력 시 어떤 인자를 넣어야 하는지 자동완성 힌트를 보여줍니다.
>
> 본문 절차는 다섯 단계입니다. 빌드, 테스트, 태그 생성, 배포, 결과 출력. 모델은 이 절차를 그대로 실행하고 각 단계에서 실패하면 중단합니다.
>
> Reference와 Action의 차이가 명확합니다. Reference는 읽고 규칙대로 작업, Action은 절차를 도구 호출까지 실행하고 결과를 내놓습니다. 같은 매뉴얼이라도 읽기용과 실행용은 설계가 다릅니다.

👉 진행 팁: `disable-model-invocation: true`의 의미를 강조하세요. 배포 같은 위험한 작업은 자동 호출을 막는 것이 안전합니다.

## 슬라이드 24 — 4.3.1 에이전트 = 모델 + 하네스 ⏱ ~70초

(화면) 하네스 4사분면 멘탈 모델

> 훅을 이해하려면 하네스 개념이 먼저입니다.
>
> ThoughtWorks의 Birgitta Böckeler가 정리한 정의입니다. "에이전트 = 모델 + 하네스." 하네스는 에이전트에서 모델을 뺀 나머지 전부 — 컨텍스트 큐레이션, 도구 관리, 권한 제어, 에러 복구.
>
> 화면의 4사분면을 보면 두 축이 있습니다. 가로 축은 Computational(결정론적·CPU·저렴)과 Inferential(비결정론적·LLM·GPU). 세로 축은 피드포워드(행동 전 차단)와 피드백(사후 교정).
>
> 여기서 설계 원칙이 나옵니다. **결정론적으로 처리 가능한 것은 Computational에 맡겨야 합니다.** 파일 보호, 타입 체크, 린트 — 이것들은 LLM에게 부탁할 이유가 없습니다. 스크립트가 더 빠르고 더 확실합니다. 훅이 방지턱인 이유가 이겁니다. **의미적 판단이 필요한 것만 Inferential에 맡깁니다.** 코드 리뷰의 의도 파악, LLM-as-judge 검수가 여기 해당합니다.
>
> 훅은 Computational의 피드포워드 자리입니다. 하네스 심화는 Level 7에서 다룹니다.

> **UserPromptSubmit** — 사용자 메시지 제출 직후. 입력을 정제하거나 라우팅하는 데 씁니다.
>
> **Notification** — 모델이 권한이나 정보를 요청할 때. 알림 발송에 씁니다.
>
> **Stop** — 모델 응답 종료 시. 응답이 끝날 때마다 한 번 실행됩니다.
>
> **SubagentStop** — 서브 에이전트 종료 시.
>
> **PreCompact** — `/compact` 직전. 압축 전 현재 상태를 저장합니다.
>
> **SessionStart** — 세션 시작 시. 핸드오프 파일을 주입하는 데 씁니다.
>
> 주의사항 — settings.json에서 matcher를 누락하면 해당 이벤트의 모든 도구 호출에 훅이 걸립니다. Read나 Grep까지 가로채서 성능이 저하될 수 있습니다. matcher는 반드시 지정하세요.

> 화면의 GitHub MCP 설정을 보면 — `.mcp.json`에 type "http", url, headers로 Authorization 토큰을 넣습니다. 이 설정 하나로 Claude가 GitHub 이슈, PR, 코드 검색, 코드 리뷰를 직접 다룰 수 있게 됩니다. 사람이 GitHub 브라우저를 열지 않아도 모델이 직접 대화합니다.
>
> CLI로 추가하는 방법도 있습니다. `claude mcp add --transport http github <url>` 한 줄입니다.

## 슬라이드 43 — 4.4.1 Scope ⏱ ~50초

(화면) Scope 다이어그램 두 개

> MCP 서버의 등록 범위, Scope입니다.
>
> 세 가지가 있습니다. **local** — 현재 세션에서만 유효합니다. **project** — `.mcp.json`에 등록, 프로젝트 전체에 공유됩니다. **user** — `~/.claude/`에 등록, 해당 사용자의 모든 프로젝트에서 씁니다.
>
> 팀 공용 도구는 project 스코프로 `.mcp.json`에 넣어서 git에 커밋하면 팀 전체가 같은 MCP 서버를 씁니다. 개인 도구는 user 스코프에 넣어서 프로젝트에 영향을 주지 않게 합니다.
>
> 선택 기준은 단순합니다. 팀 전체가 쓰면 project, 혼자 쓰면 user, 일회성이면 local.

## 슬라이드 44 — 4.4.2 개발 필수 MCP 서버 ⏱ ~55초

(화면) 필수 MCP 서버 다이어그램

> 개발에서 자주 쓰이는 MCP 서버들입니다.
>
> **GitHub MCP** — 이슈, PR, 코드 검색, 코드 리뷰 등 GitHub 작업을 Claude가 직접 수행합니다.
>
> **Filesystem MCP** — 파일 시스템 접근. 읽기 전용으로 제한해서 안전하게 사용합니다.
>
> **Postgres/SQLite MCP** — 데이터베이스 조회. 스키마 파악이나 데이터 분석에 씁니다.
>
> **Codesight MCP** — 코드베이스 의미 검색(semantic search). 전체 파일을 읽지 않고 관련 코드를 찾아줍니다.
>
> 주의사항 하나 — MCP 서버가 노출하는 도구 정의가 크면 매 턴 시스템 프롬프트 무게가 증가합니다. 5개 서버에 도구 10개씩이면 수천 토큰입니다. 안 쓰는 서버는 비활성화하는 것이 토큰 비용을 아끼는 방법입니다.

## 슬라이드 45 — 4.5 실습 파트 표지 ⏱ ~45초

(화면) PART 05 표지

> 마지막 파트, 실습입니다.
>
> 지금까지 배운 것을 한번 더 정리합니다.
>
> **슬래시 커맨드는 단축키** — 반복 지시를 `.claude/commands/` 파일 하나로. 자주 치는 긴 프롬프트를 한 단어로 줄입니다.
>

>
