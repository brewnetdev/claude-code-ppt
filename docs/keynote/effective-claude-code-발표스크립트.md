## CHAPTER 0 — 컨텍스트 운용 핵심 정리

### S5. 챕터 0 표지 ⏱ ~12초
> 챕터 0, 컨텍스트 운용 핵심 정리입니다. 본격 실습 전에, 전 과정에서 반복되는 핵심 용어와 명령어를 한 번에 잡고 갑니다. 여기가 흔들리면 뒤가 다 흔들려요.

### S6. 챕터 0 오프닝 — WHY·WHAT·HOW ⏱ ~55초
> **이 챕터에서 배우는 것은** 네 가지입니다. 핵심 3개념(턴·세션·컨텍스트), CLAUDE.md의 위치 계층, 컨텍스트 관리 슬래시 명령어, 그리고 세션 인계 패턴이에요.
> **왜** 먼저 배우냐면, 클로드는 **새 세션마다 'zero knowledge', 즉 아무것도 모르는 백지 상태에서 시작**합니다. 용어와 명령어를 먼저 잡지 않으면 같은 설명을 매번 반복하게 돼요. 컨텍스트 운용의 80%가 여기서 갈립니다.
> **어떻게** 배우냐면, 정의 → 컨텍스트 윈도우 구조 → 휘발성 → 명령어 사용 시점 → 세션 연결 순으로 단계별로 갑니다. 오늘의 체크포인트 한 줄은 이거예요. **"`/clear`와 `/compact`의 차이를 설명할 수 있다."** 가장 헷갈리는 짝이자 분기점입니다.

### S7. CH.0 다룰 영역 한눈에 ⏱ ~20초 ⏩
> 이 챕터에서 다루는 걸 다섯 영역으로 묶었습니다. 핵심 3개념, 자동 주입 문서 CLAUDE.md, 모델이 보는 숨은 시스템 프롬프트, 컨텍스트 관리 명령어 8종, 그리고 보조 문서 패턴(HANDOFF·MEMORY)이에요. 하나씩 보겠습니다.

### S8. PART 1 표지 — 핵심 3개념 ⏱ ~10초
> 첫 파트, 가장 자주 헷갈리는 세 단어 — 턴·세션·컨텍스트입니다. 포함 관계와 휘발성부터 잡고 갑니다.

### S9. CH.0·P1 턴·세션·컨텍스트 정의 ⏱ ~55초
> 세 단어를 정확히 구분하겠습니다. **턴(Turn)**은 '내 메시지 하나 + 클로드 응답 하나'의 한 쌍이에요. "테스트 코드 추가해줘"와 그 답이 한 턴입니다. **세션(Session)**은 'claude 실행부터 종료까지'의 단위로, 여러 턴이 누적되고 고유한 session_id가 붙습니다. IDE를 열어 작업하고 닫는 것과 비슷해요. **컨텍스트(Context)**는 '매 추론 순간 모델이 실제로 보는 토큰 전체'입니다. 시스템 프롬프트 + 도구 정의 + CLAUDE.md + 그동안 쌓인 턴 전부죠.
> 포함 관계로 외우세요. **턴 ⊂ 세션 ⊂ 컨텍스트.** 턴이 모여 세션이 되고, 세션의 누적 턴에 정적 요소가 더해져 컨텍스트가 됩니다.

### S10. CH.0·P1 컨텍스트 윈도우의 구조 ⏱ ~65초
> 컨텍스트 윈도우 안을 뜯어봅시다. **컨텍스트 윈도우는 모델이 한 번에 볼 수 있는 토큰의 총량**이고, Sonnet·Opus는 기본 200K, 확장하면 1M까지 갑니다. 크게 둘로 나뉘어요. **정적(STATIC) 부분**은 세션 시작 때 한 번 로드됩니다 — 시스템 프롬프트(약 11K, Anthropic 비공개·수정 불가), 도구 정의(약 13K), 그리고 **CLAUDE.md(약 5K) — 우리가 영향력을 행사할 수 있는 핵심**이에요. **동적(SESSION) 부분**은 매 턴 쌓여서 T1, T2, T3… 점점 늘어납니다.
> 그래서 70%에 도달하면 **`/compact`로 요약 압축**, 95%를 넘거나 주제가 바뀌면 **`/clear`로 초기화**하고, **`/context`로 어디가 토큰을 잡아먹는지 실시간 확인**합니다. 큰 파일은 잘라 읽고, 종료 직전엔 HANDOFF로 인계 — 이게 컨텍스트 운용의 기본 리듬입니다.

### S11. CH.0·P1 휘발성 ⏱ ~45초
> 이걸 모르면 "어제 작업이 왜 사라졌지?"가 됩니다. 세 가지예요. **휘발** — 컨텍스트(누적 턴 + 도구 결과)는 세션이 끝나면 전부 사라지고, 다음 세션으로 자동으로 안 넘어갑니다. **보존** — 단, **CLAUDE.md만은 다음 세션에서도 자동으로 다시 로드**됩니다. 그래서 "어제 정한 규칙"은 거기 적어 둬야 살아남아요. **이어가기** — `--continue`, `/resume`, `/compact`, 그리고 HANDOFF 문서, 이 네 가지로 세션을 연결할 수 있습니다.

### S12. PART 2 표지 — CLAUDE.md ⏱ ~12초
> 두 번째 파트, CLAUDE.md입니다. 세션이 시작될 때마다 자동으로 컨텍스트에 로드돼서 모델이 세션 내내 보게 되는 **프로젝트 규칙 문서**예요.

### S13. CH.0·P2 CLAUDE.md의 정체와 위치 ⏱ ~60초
> 여기 중요한 오해 하나를 짚겠습니다. CLAUDE.md를 흔히 "시스템 프롬프트"라고 부르지만, **기술적으로는 시스템 프롬프트가 아닙니다.** 세션 시작 시 **첫 user 메시지로 주입되는 메모리**이고, 결과적으로 모델이 매 턴 보게 되는 거예요.
> 위치는 네 가지 스코프(scope, 적용 범위)가 있습니다. **User(전역)** `~/.claude/CLAUDE.md`는 내 모든 프로젝트 공통, **Project** `<repo>/CLAUDE.md`는 이 프로젝트 협업자 전체(Git 커밋 권장), **Local** `CLAUDE.local.md`는 나만(gitignore), **Subfolder**는 해당 폴더 작업 시에만 자동 로드됩니다. 예를들어 프로젝트 구조가 프론트엔드, 백엔드가 같이 공존하는 모노레포 구조이거나 멀티 프로젝트 구조 (API, Batch, Queue Worker 등)일 때 각 폴더 구조 하위에도 메인 클로드 문서가 위치해야 합니다. 우선순위는 **하위가 상위를 덮어쓰기 — Subfolder > Local > Project > User** 순이에요.

### S14. CH.0·P2 CLAUDE.md 작성 예시 ⏱ ~40초
> 실제 예시입니다. Tech Stack, Architecture, Rules 같은 섹션으로 나눠서, "Next.js 16, Drizzle ORM, PostgreSQL을 쓴다", "서버 모듈은 특정 위치에서만 import한다", "TypeScript에서 any 금지", "main 브랜치 직접 push 금지" 같은 **이 프로젝트만의 구체적인 규칙**을 적습니다. 새 프로젝트라면 `/init` 명령으로 구조를 분석한 초안을 자동 생성한 뒤 이렇게 다듬어 쓰면 됩니다. CLAUDE.md 작성법은 챕터 1에서 본격적으로 깊게 다룹니다.

### S15. PART 3 표지 — 시스템 프롬프트 ⏱ ~12초
> 세 번째 파트는 **내장 시스템 프롬프트**예요. 내가 안 썼지만 모델이 매 턴 보는 숨은 지침입니다. 편집할 수 없고, 항상 가장 먼저 로드되며, CLAUDE.md와는 다른 레이어라는 점이 핵심입니다.

### S16. CH.0·P3 모델이 보는 4개 텍스트 레이어 ⏱ ~60초
> 모델이 보는 건 '내가 쓴 것'만이 아닙니다. 네 개 레이어가 있어요. **① 내장 시스템 프롬프트** — 동작 지침·도구 정의·환경 정보가 들어 있고 항상 1순위로 로드, **교체·삭제 불가**입니다. `--append-system-prompt`로 덧붙이거나 Output Styles로 톤만 바꿀 수 있어요. **② 자동 주입 문서** — CLAUDE.md, 200줄 이하 유지 권장. **③ 누적 대화** — 매 턴 추가되고 `/compact`·`/clear`로 관리. **④ 온디맨드 파일** — 접근하는 순간 지연 로드되는, lazy하게 읽는 파일이에요.
> 한 가지 비용 팁. **프롬프트 캐싱** — 시스템 프롬프트는 캐시돼서 첫 턴만 전액, 이후엔 약 1/10 비용입니다. 단, 모델이나 MCP 설정을 자주 바꾸면 캐시가 깨져 다시 전액 청구되니 주의하세요.

### S17. PART 4 표지 — 슬래시 명령어 ⏱ ~12초
> 네 번째 파트, 컨텍스트 관리 명령어입니다. 이 8개 슬래시 명령어가 컨텍스트 운용의 80%를 책임집니다.

### S18. CH.0·P4 명령어 한눈에 보기 ⏱ ~55초
> 여덟 개를 정리하겠습니다. **`/context`** 사용량 시각화, **`/clear`** 대화 히스토리 비우기(CLAUDE.md는 재로드), **`/compact`** 히스토리를 요약본으로 압축, **`/memory`** CLAUDE.md를 편집기로 열기, **`/init`** 프로젝트 분석 후 CLAUDE.md 초안 생성, **`--continue`/`/resume`** 이전 세션 이어가기, **`/status`** 모델·세션 정보, **`/cost`** 누적 토큰·비용 확인입니다. 이 8개만 손에 익히면 컨텍스트 운용은 대부분 해결됩니다.

### S19. CH.0·P4 /clear vs /compact ⭐ ⏱ ~60초
> 자, 오늘의 체크포인트. 가장 헷갈리는 짝입니다. **`/clear`는 완전 삭제**예요. 실행 후 시스템·도구·CLAUDE.md는 자동으로 다시 로드되지만, **대화 히스토리(누적 턴)는 완전히 사라집니다.** 그래서 작업 주제가 완전히 바뀌거나 다른 프로젝트로 전환할 때 씁니다. 반면 **`/compact`는 요약 압축**입니다. 누적된 T1부터 Tn까지를 요약본으로 만들어 **맥락은 유지하면서 공간만 확보**해요. 같은 작업을 계속하면서 70%를 넘겼을 때 씁니다.
> 한 줄로: **주제 바뀌면 `/clear`, 같은 일 계속하면 `/compact`.** 이거 하나는 꼭 가져가세요.

### S20. CH.0·P4 사용률 단계별 대응 ⏱ ~45초
> `/context`가 보여주는 % 값에 따라 행동을 미리 정해 둡시다. **~50%는 여유** — 평소대로 진행. **60~70%는 주의** — 기능 단위를 깔끔히 끊고 `/compact`를 준비. **70~90%는 위험** — 즉시 `/compact` 실행, 안 그러면 응답 품질이 떨어져요. **90% 이상은 임박** — 마무리하고 HANDOFF 작성 후 `/clear`로 새 세션을 시작합니다. 참고로 약 95%에서 auto-compact가 자동 발동하는데, **그 전에 직접 `/compact`를 호출해야 마일스톤을 깔끔하게 끊고 결과 품질도 좋습니다.**

### S21. PART 5 표지 — 보조 문서 ⏱ ~12초

### S22. CH.0·P5 3종 문서 ⏱ ~50초

### S40. CH.1·P4 Progressive Disclosure — 도서관 vs 보따리 ⏱ ~55초
> 'Progressive Disclosure', 우리말로 **점진적 공개**를 비유로 설명할게요. **보따리 안티패턴** — root CLAUDE.md에 API 명세 400줄, DB 스키마 300줄, 컴포넌트 500줄까지 전부 우겨넣으면 합계 1,200줄을 **매 작업마다 통째로 짊어집니다.** 정작 중요한 규칙은 노이즈에 묻혀 무시돼요. **도서관 좋은 패턴** — root는 60줄만 항상 로드하고, 나머지는 `@` 포인터로 서가에 꽂아 둡니다. API 작업할 때만 API 명세를 꺼내 읽는 거죠. 결과적으로 **평소 60줄만 소비하고, 규칙이 선명하게 유지**됩니다. Skill도 같은 원리예요 — frontmatter만 자동 로드되고 본문은 필요할 때만 읽힙니다.

### S43. CH.1·P5 Advisory vs Enforcement ⭐ ⏱ ~55초
> 이 챕터의 본질입니다. **CLAUDE.md는 본질적으로 권고(advisory)예요.** "main 브랜치 직접 push 금지"라고 적어도 **확률적으로 약 80%만 지켜집니다.** 모델이 읽고 참고하지만 무시할 수도 있어요. 비유하면 **'과속 단속 카메라'** — 있어도 과속은 할 수 있죠.
> 반면 **Hook은 강제(enforcement)입니다.** PreToolUse Hook이 `git push origin main`을 감지하면 exit 2로 **100% 차단**해요. 모델이 물리적으로 못 합니다. 비유하면 **'과속 방지턱'** — 무조건 감속해야 넘어갑니다. **확실해야 하는 건 CLAUDE.md가 아니라 Hook으로 코드로 강제하세요.** Hook은 챕터 2에서 다룹니다.

### S45. CH.1·P6 카파시 4원칙 ⏱ ~55초
> 카파시의 CLAUDE.md는 Tech Stack·Commands·Architecture가 **전혀 없습니다.** 대신 행동 원칙 4개로 채워져 있어요. 그래서 User-level(`~/.claude/CLAUDE.md`), 즉 내 모든 프로젝트 공통 기본 행동으로 적합합니다. **① Think Before Coding** — 코딩 전에 생각하고, 모르면 멈춰 질문. **② Simplicity First** — 요청 외 기능 추가 금지, 200줄을 50줄로 줄일 수 있으면 다시 써라. **③ Surgical Changes** — 외과적 변경, 안 깨진 걸 리팩토링하지 말고 모든 변경은 요청에 직접 추적 가능해야. **④ Goal-Driven Execution** — "버그 고쳐"를 "재현 테스트 작성 후 통과"라는 검증 가능한 목표로 변환. **그리고 핵심 질문: 내 CLAUDE.md가 동작하는지 어떻게 측정하나?** 측정 지표를 정의하고, 안 먹히면 규칙을 고치거나 Hook으로 강제하세요.

### S52. CH.2·P1 Command·Skill·Hook 비교 ⏱ ~60초

### S53. CH.2·P1 Advisory vs Enforcement + 2026 통합 추세 ⏱ ~55초
> 다시 본질입니다. **CLAUDE.md·Skill·Command는 권고(advisory)** — 모델 해석을 거치니 확률적이에요. **Hook은 강제(enforcement)** — 셸을 직접 실행하니 결정론적이고 100% 차단 가능합니다.
> 그리고 **2026년 중요한 변화** 하나. **Command가 Skill로 통합됐어요.** `.claude/commands/deploy.md`든 `.claude/skills/deploy/SKILL.md`든 둘 다 `/deploy`를 만듭니다. 둘이 본질적으로 같은 일을 할 수 있게 됐어요. 그래서 **단일 파일이면 commands/, 폴더 구조나 확장 기능이 필요하면 skills/** 를 쓰고, **신규 작성은 Skill을 권장**합니다.

### S55. CH.2·P2 세 자동화의 표준 디렉터리 ⏱ ~55초

### S60. CH.2·P4 12개 이벤트 + Exit code ⏱ ~55초
> Hook 메커니즘입니다. 12개 이벤트 중 핵심 8개를 보면, **PreToolUse**(도구 실행 전, 차단·검증), **PostToolUse**(실행 후, 포맷), **UserPromptSubmit**(프롬프트 제출), **Stop**(응답 1턴 완료), **PreCompact**(/compact 직전), **SessionStart/End**(세션 시작·종료) 등이 있어요.

### S61. CH.2·P4 필수 Hook 7선 + 선택 1 ⏱ ~55초
> 필수 7개와 선택 1개입니다. **① 파괴적 명령 차단**(PreToolUse) — `rm -rf /`, force push, `git reset --hard`. **② 비밀파일 보호**(PreToolUse) — .env, .pem, lockfile, .git/ 편집 금지. **③ 커밋 전 린트·타입체크**(PreToolUse) — 실패 시 차단. **④ 자동 포맷**(PostToolUse) — Write/Edit 직후 prettier/biome 자동 수정. **⑤ Push/PR 전 테스트 강제**(PreToolUse). **⑥ Slack 알림**(Stop) — 응답 완료 시 webhook. **⑦ PreCompact Handoff**(PreCompact+SessionStart) — compact 직전 HANDOFF 저장, 다음 세션 자동 복원. 그리고 **선택 ⑧ 자동 커밋** — 환경변수로 명시적으로 켤 때만.

### S62. CH.2·P4 대표 Hook — 파괴적 명령 차단 ⏱ ~45초 ⏩
> 대표 Hook 하나를 코드로 보겠습니다. `block-dangerous.sh`는 **stdin으로 들어온 JSON에서 명령어를 추출**하고(`jq`로 파싱), `rm -rf /`, `sudo rm`, main 브랜치 force push, `git reset --hard`, `chmod 777`, `dd` 같은 **위험 패턴과 매칭**되면 **exit 2로 차단**합니다. 그러면 그 메시지가 stderr를 통해 클로드에게 전달되죠. 코드 전체는 슬라이드와 튜토리얼에 있으니, 패턴만 보태서 본인 환경에 맞게 쓰시면 됩니다.

### S63. CH.2·P4 가장 흔한 실수 — hook vs hooks ⏱ ~40초
> 여기는 실수 사고가 잦은 곳이라 콕 짚습니다. settings.json에서 **`hook`(단수)으로 쓰면 설정 파일 전체가 무시됩니다.** 반드시 **`hooks`(복수)에 배열로** 써야 해요. 한 글자 차이로 자동화가 통째로 안 먹히니, 안 될 때 제일 먼저 여길 확인하세요. 참고로 2026년 신규로 `if` 필드가 생겨서, `"if": "Bash(git push *)"`처럼 권한 규칙 문법으로 더 좁게 매칭할 수 있습니다.

### S65. CH.2·P5 의사결정 트리 ⏱ ~55초
> 자동화하고 싶은 작업이 생기면 이렇게 질문하세요. **Q1, 사이드이펙트가 있나(deploy·push·send)?** → Command 또는 Skill에 모델 자동 호출 비활성화. **Q2, '절대 안 됨'인가(force push·rm -rf)?** → Hook(PreToolUse + exit 2). **Q3, 자연어로 자동 발동되면 좋은가?** → Skill(description으로 매칭). **Q4, 매 도구 호출마다 자동(포맷·린트)?** → Hook(PostToolUse). **Q5, 세션 시작·종료·compact 시 자동?** → Hook(SessionStart 등). **Q6, 단순 반복 프롬프트 단축?** → Command. 이 흐름이면 헷갈릴 일이 없습니다.

### S66. CH.2·P5 /commit 콤보 ⏱ ~55초
> 안전한 자동화는 **Command와 Hook이 둘 다 있어야 완성**됩니다. `/commit`을 예로 흐름을 보면, ① 사용자가 `/commit` 입력 → ② Command가 `git diff --staged`로 동적 컨텍스트 주입 → ③ 클로드가 Conventional Commits 메시지 생성(여기까진 advisory) → ④ `git commit` 시도 → ⑤ **PreToolUse Hook이 발동해 린트·타입체크 실행(enforcement)** → ⑥ 통과하면 커밋, 실패하면 exit 2로 차단하고 클로드가 stderr 보고 수정 → ⑦ Stop 시점에 Slack 알림. **Skill로 방향을 제시하고 Hook으로 안전망을 깐다 — 한쪽만으론 부족합니다.**

### S67. CH.2 WRAP-UP — 흔한 실수 8가지 ⏱ ~45초 ⏩

## CHAPTER 3 — 토큰의 효율화 (LSP & codesight)

### S68. 챕터 3 표지 ⏱ ~15초
> 챕터 3, 토큰의 효율화입니다. 핵심을 먼저 말씀드리면, **grep은 코드 본문을 통째로 읽지만, LSP는 위치 좌표만, codesight는 압축된 구조 맵만 받습니다.** AI에게 읽힐 입력의 양 자체를 줄이는 게 토큰 절감의 본질이에요.

### S69. 챕터 3 오프닝 — WHY·WHAT·HOW ⏱ ~55초
> **이 챕터에서 배우는 것은** LSP(심볼 단위 정밀 탐색), codesight(구조 맵 사전 주입), 설치·사용법·실측 벤치마크, 그리고 언제 무엇을 쓸지 의사결정입니다.
> **왜** 배우냐면, 새 세션마다 AI가 **grep으로 코드를 헤매며 수만 토큰을 낭비**하거든요. 같은 비용을 매번 다시 치릅니다.
> **어떻게** 배우냐면, grep의 한계를 이해하고 → LSP를 연결하고 → codesight로 구조를 주입하고 → 실측 수치로 검증하고 → 상황별로 조합합니다. 체크포인트는 이거예요. **"토큰 절감은 모델을 바꾸는 게 아니라, AI에게 읽힐 입력의 형태와 양을 바꾸는 데서 나온다. Structure is mechanism."**

### S70. CH.3 한 장 요약 — 결론 먼저 ⏱ ~55초
> 결론부터 한 장으로. 먼저 **토큰(token)**은 'AI가 한 번에 처리하는 글자 묶음'인데, 입력은 AI에게 읽히는 양, 출력은 AI가 쓰는 양이에요. 도서관 비유로 보면, **grep**은 단어가 나온 페이지를 **본문째 복사해 건넵니다**(양 많고 노이즈 포함). **LSP**는 "42쪽 14번째 줄"이라는 **위치 좌표만** 건네요(노이즈 0, 정의 찾기 43배·참조 찾기 2.1배 절감). **codesight**는 책 맨 앞에 끼운 **요약 지도 한 장**이라, 매번 뒤지는 일을 없앱니다(세션당 약 6.8배 절감). **셋은 경쟁이 아니라 상호 보완** — codesight로 지도를 깔고, LSP로 정밀 좌표를 찍습니다.

### S71. PART 1 표지 — LSP ⏱ ~12초
> 첫 파트, LSP입니다. 텍스트 검색을 넘어 IDE 수준의 코드 인텔리전스를 얻는 도구예요. LSP가 뭔지, 언제 효과적인지, 3가지 설치 방식을 봅니다.

### S72. CH.3·P1 LSP란 무엇인가 ⏱ ~60초
> **LSP는 'Language Server Protocol', 편집기와 언어 서버 사이의 통신 표준**입니다. Microsoft가 2016년 오픈 표준으로 공개했고 최신 안정 버전은 3.17이에요. 쉽게는 도서관의 **'색인 카드'** — 본문 대신 "그 단어는 몇 쪽 몇 줄"이라는 위치만 알려줍니다.
> 중요한 건, **LSP 자체는 코드를 분석하지 않아요.** 실제 분석은 각 언어의 언어 서버(TypeScript의 vtsls, Python의 pyright 등)가 하고, LSP는 그 결과를 전달만 합니다. 클로드 코드는 기본적으로 grep으로 코드를 찾는데, 파일이 많아지면 느려지고 주석·문자열 속 텍스트까지 섞여 부정확해져요. LSP를 연결하면 정의로 점프(goToDefinition), 전체 참조 찾기(findReferences), 타입 에러 즉시 확인(getDiagnostics) 같은 기능을 얻습니다. 단, **파일 20개 미만 소규모면 LSP 없이도 충분**합니다.

### S73. CH.3·P1 grep vs LSP ⏱ ~55초
> 같은 질문, "processPayment 정의가 어디야?"를 둘이 어떻게 처리하는지 봅시다. **grep 방식** — 전체 파일을 텍스트 검색해서 847개 매치를 뱉어요(클래스·변수·주석·CSS 다 포함). 그걸 좁히려고 후보 파일들을 통째로 읽고 30~60초간 추론합니다. **LSP 방식** — goToDefinition을 호출하면 언어 서버가 파싱된 심볼 테이블에서 즉시 조회해 **파일명+줄 번호만, 50밀리초에 반환**합니다. 본문을 안 보내요.
> 토큰으로 보면 **약 8,000 대 약 100토큰**. 게다가 getDiagnostics·findReferences·hover는 LSP만 할 수 있는 기능입니다.

### S74. CH.3·P1 설치 — 3가지 방식 ⏱ ~55초
> LSP 연결은 세 가지 방식이 있어요. **방식 1, 공식 빌트인 LSP 플러그인(권장)** — 언어 서버를 설치하고, `/plugin`으로 마켓플레이스에서 설치합니다. **빌트인은 v2.0.74부터 정식 지원, v2.1.50 이상 권장**이고, MCP를 안 거쳐 오버헤드가 없으며 32개 이상 언어를 지원해요. **방식 2, cclsp MCP** — `npx cclsp@latest setup` 위저드로 간단히. **방식 3, Serena MCP** — 심볼 편집·세션 메모리·온보딩까지 포함한 종합 툴킷인데, 빌트인과 중복을 피하려면 `--context` 옵션이 필수입니다. 명령어는 슬라이드를 그대로 따라 하시면 됩니다.

### S75. CH.3·P1 방식 비교 ⏱ ~45초
> 뭘 고를지 비교표입니다. **공식 LSP 플러그인** — 고정 토큰 비용 최소, 설정 쉬움, 대부분 프로젝트에 적합. **cclsp** — 다중·미지원 언어에. **Serena** — LSP에 심볼 편집·메모리·온보딩까지, 대규모 종합 에이전트용이지만 고정 토큰 비용이 높고 Python·uv가 필요해요. **결론: TypeScript 단일 스택이면 공식 플러그인으로 충분**합니다. 심볼 편집·세션 메모리가 필요해지면 그때 Serena를 검토하세요.

### S76. PART 2 표지 — codesight ⏱ ~12초
> 두 번째 파트, codesight입니다. 세션 시작 전에 프로젝트 구조를 한 번에 주입하는 도구예요. 정체·벤치마크·설치·연동 설정을 봅니다.

### S77. CH.3·P2 codesight란 ⏱ ~55초
> **codesight는 범용 AI 컨텍스트 생성기**입니다. 프로젝트 전체를 **AST(추상 구문 트리, 코드를 구조적으로 분석한 결과)**로 파싱해 **구조화된 마크다운 맵**으로 만들어, 세션 시작 시 탐색 비용을 없앱니다. 쉽게는 **책 맨 앞에 끼우는 '요약 지도 한 장'**이에요. TypeScript는 컴파일러 API로 정밀 파싱하고, 그 외 언어는 정규식으로 폴백합니다. Routes·Schema·Components·의존성 그래프 등을 감지하고요. 현재 v1.14.0, 의존성 0, MIT 라이선스입니다. 제작사 공개 벤치마크로는 실제 프로덕션에서 평균 약 11배 절감이 나왔는데, 이건 2차 출처고 독립 실측은 잠시 후 파트 3에서 봅니다.

### S78. CH.3·P2 사용법 ⏱ ~50초 ⏩
> 사용법은 간단해요. 설치 없이 **`npx codesight`로 즉시 실행**하면 `.codesight/`가 생성됩니다. 클로드 코드 연동은 3단계예요. **① `npx codesight --init`으로 최초 1회 설정 파일 생성 → ② `.claude/settings.json`에 MCP 서버로 등록 → ③ `npx codesight --hook`으로 커밋마다 자동 갱신.** 팁 하나, **`--init`은 정적 파일을 1회 주입하고, `--mcp`는 AI가 필요한 것만 실시간으로 쿼리**해서 토큰 효율이 가장 좋습니다. 세션 캐싱으로 첫 호출만 스캔하고요.

### S79. PART 3 표지 — 벤치마크 & 의사결정 ⏱ ~12초
> 마지막 파트. 직접 측정한 수치로 왜·언제 절감되는지 봅니다. grep·LSP·codesight 실측과 절감 메커니즘, 그리고 상황별 의사결정 매트릭스예요.

### S80. CH.3·P3 실측 — grep vs LSP vs codesight ⏱ ~55초
> 실제 오픈소스 저장소(TypeScript·258파일)를 클론해 직접 측정했습니다. **시나리오 A, 정의로 이동** — grep은 47매치(주석·노이즈 포함) 1,122토큰, LSP는 정의 1곳 26토큰, **43배 절감**. **시나리오 B, 모든 참조 찾기(리팩토링)** — grep 2,244토큰, LSP는 좌표만 1,062토큰, **2.1배 절감**. **시나리오 C, 세션 시작 시 구조 파악** — grep/glob으로 헤매면 32,695토큰, codesight는 맵 1개 주입에 4,780토큰, **6.8배 절감**. 토큰은 1토큰≈4글자로 보수적으로 추정했고, 절감 비율은 추정 계수와 무관하게 동일합니다.

### S81. CH.3·P3 왜 절감되는가 — 메커니즘 ⏱ ~45초
> 왜 절감되는지 메커니즘으로 정리하면, **같은 질문이라도 코드베이스가 돌려주는 형태가 다릅니다.** grep은 매칭된 라인 본문 전체(노이즈 포함)를 돌려주고, LSP는 위치 좌표 1곳만(오탐 0), codesight는 압축 구조 맵 1개로 반복 탐색 자체를 없애요. **핵심은, 토큰 절감은 모델이 아니라 입력의 형태와 양을 바꾸는 데서 나온다 — "Structure is mechanism", 구조가 곧 메커니즘**이라는 겁니다.

### S82. CH.3·P3 의사결정 매트릭스 ⏱ ~50초
> 언제 무엇을 쓸지 정리합니다. **파일 20개 미만이면 grep 기본으로 충분.** 그 이상이면 **codesight `--init` + `--mcp` + `--hook`으로 세션 시작 시 구조를 주입**하는 게 기본 셋업이에요. 거기에 **리팩토링·정의 추적이 필요하면 LSP 공식 플러그인**을 더하고, **다중·미지원 언어면 cclsp**, **대규모·심볼 편집·메모리가 필요하면 Serena**를 얹습니다. 다시 강조하지만 **경쟁이 아니라 누적 조합** — codesight로 구조를 깔고 LSP로 정밀 탐색, 둘은 상호 보완입니다.

## EPILOGUE — 엔지니어링의 진화 · SDD · GSD

### S83. 에필로그 표지 ⏱ ~15초
> 마지막, 에필로그입니다. 엔지니어링은 프롬프트 → 컨텍스트 → SDD → 하네스로 진화해 왔는데, 그중 **SDD(Spec-Driven Development)**를 좀 더 깊이 보겠습니다. SDD는 '명세 주도 개발'이라는 뜻이에요.

### S84. 에필로그 진화 4단계 ⏱ ~55초
> 우리가 어디에 있는지 봅시다. 각 단계는 이전을 흡수하며 진화해요. **1단계 Prompt Engineering** — 잘 묻는 법(few-shot, CoT 등). **2단계 Context Engineering** — 모델 시야 통제(CLAUDE.md, /context, /compact). **우리가 오늘 다룬 게 여기까지예요.** **3단계 Spec-Driven Development** — 명세가 진실의 원천이 되는 의도 중심 개발(Spec Kit, GSD). **4단계 Harness Engineering** — 자동화·안전망, 즉 '모델 + 하네스 = AI 에이전트'. 프롬프트만으론 부족해 컨텍스트, 컨텍스트만으론 일관성이 부족해 명세, 명세만으론 자동화가 부족해 하네스로 마무리되는 흡수형 진화입니다.

### S85. SDD 핵심 철학 ⏱ ~55초
> **SDD는 GitHub Spec Kit이 제시한, 'Vibe Coding'을 벗어나 명세가 진실의 원천이 되는 개발 방법론**입니다. 'Vibe Coding'은 "대충 느낌으로 해줘" 식 개발을 말하는데, 이걸 막는 거죠. SDD에는 세 핵심 요소가 있어요. **Constitution(헌법)** — 불변의 법칙으로 AI가 모든 작업에서 최우선 참조(예: TS strict, 테스트 90%). **Spec(명세)** — '무엇'을 만들지, 사용자 관점 요구사항. **Plan(계획)** — '어떻게' 만들지, 기술 구현 설계. 한 문장으로: **"코드가 명세를 따르는 게 아니라, 명세가 코드를 생성한다."**

### S86. SDD 5단계 개발 사이클 ⏱ ~50초
> SDD 사이클은 5단계예요. **① 요구사항 분석**(요구사항 문서) → **② 명세 문서 작성**(spec.md) → **③ 명세 리뷰·확정**(확정 버전) → **④ 명세 기반 구현**(AI에게 명세 전달 → 코드 생성) → **⑤ 명세 대비 검증**(테스트 리포트). 각 단계가 명세 문서를 산출하고, 그게 다음 단계의 입력이 됩니다. Spec Kit으로는 `/speckit.specify`, `/speckit.plan`, `/speckit.tasks`, `/speckit.implement`로 매핑되고요. **핵심: 명세가 바뀌면 코드를 직접 고치지 말고, 명세를 고쳐 코드를 다시 생성**합니다.

### S87. 왜 AI와 SDD가 맞는가 + TDD ⏱ ~50초
> AI는 명확한 지시에 강합니다. **모호한 지시** "TODO API 만들어줘"는 REST인지 GraphQL인지, 필드·에러·인증을 매번 다르게 해석해요. **명세 기반 지시** "API_SPEC.md 읽고 POST /api/tickets 구현해줘"는 모든 게 정의돼 있어 추측 없이 만듭니다. 즉 **명세가 곧 최고의 프롬프트**예요 — 개발자는 리뷰 기준으로, AI는 생성 맥락으로 같은 문서를 읽습니다. 그리고 **SDD + TDD 시너지** — SDD가 '무엇을 만들지' 정의하면, TDD는 '만든 게 동작하는지' 검증합니다. **명세의 각 항목이 곧바로 테스트 케이스로 변환**되죠. TDD는 '테스트 주도 개발'입니다.

### S88. Spec Kit 8개 명령어 ⏱ ~50초 ⏩
> Spec Kit 명령어는 핵심 5개 + 선택 3개입니다. **`/speckit.constitution`**(원칙 정의), **`/speckit.specify`**(요구사항 명세), **`/speckit.plan`**(기술 설계), **`/speckit.tasks`**(태스크 분해), **`/speckit.implement`**(실행)가 핵심이고, 선택으로 **`/speckit.clarify`**(모호함 제거), **`/speckit.analyze`**(정합성 분석), **`/speckit.checklist`**(완전성 검증)가 있어요. 각 명령어가 다음 단계 입력이 되는 명세 파일을 산출합니다. 설치는 `uv tool install`로, 초기화는 `specify init`으로 합니다. 명령어 목록은 슬라이드로 복습하세요.

### S89. 수동 vs Spec Kit ⏱ ~40초
> 언제 뭘 쓰냐. **수동 워크플로** — PRD·TRD·API_SPEC을 직접 작성. 구조를 완전히 통제하고 팀 기존 표준을 그대로 쓸 수 있지만, 작성 시간이 많이 들고 일관성 유지가 어려워요. **Spec Kit** — 슬래시 명령으로 AI와 함께 명세를 생성. 빠르고 표준화되며 자동화되지만, 도구 학습이 필요하고 템플릿이 강제됩니다. **기존 문서 표준이 있으면 수동, 빠르게 시작하려면 Spec Kit** — 이렇게 고르시면 됩니다.

### S90. GSD — Spec Kit의 진화형 ⏱ ~55초
> 마지막으로 **GSD(Get Shit Done)**입니다. Spec Kit과 같은 SDD 계열인데 더 상세하고 무거워서, 복잡한 멀티-Phase 프로젝트에 유리해요. GSD가 푸는 핵심 문제는 **'Context Rot'** — 컨텍스트 윈도우가 차면서 품질이 떨어지는 현상입니다. GSD는 **매 단계를 새로운 200K 컨텍스트에서 격리 실행**해 이걸 해결해요. Spec Kit이 명령어 8개·가벼운 단일 세션이라면, GSD는 50개 이상 명령어·Wave 병렬 실행·단계별 컨텍스트 격리입니다. **새 기능·빠른 시작은 Spec Kit, 복잡한 장기 프로젝트는 GSD** — 단, GSD는 학습 곡선이 높습니다.

### S91. GSD 6단계 워크플로 ⏱ ~50초 ⏩
> GSD 워크플로는 Phase마다 6단계 루프예요. **0 `/gsd-new-project`**(초기화, 최초 1회) → **1 `/gsd-discuss-phase`**(구현 결정 토론) → **2 `/gsd-plan-phase`**(병렬 리서치 + 계획) → **3 `/gsd-execute-phase`**(Wave 병렬 실행, 200K 컨텍스트 격리) → **4 `/gsd-verify-work`**(사용자 수동 UAT) → **5 `/gsd-ship`**(PR 생성). Phase 루프가 끝나면 마일스톤을 감사·완료·다음 버전으로 넘기는 사이클이 또 있고요. 소규모는 `/gsd-quick`, 즉시 실행은 `/gsd-fast`도 있습니다. 자세한 건 튜토리얼 링크를 참고하세요.
