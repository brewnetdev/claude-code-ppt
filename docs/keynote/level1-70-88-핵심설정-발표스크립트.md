## ✅ 검증 노트 (덱 재점검 — 변경 반영)

덱이 직전 버전 대비 **크게 재구성**되어, 본 스크립트를 현재 덱 기준으로 전면 재작성했습니다. 실제 슬라이드 본문을 다시 정독해 대조한 결과는 아래와 같습니다.

1. **슬라이드 순서·번호 체계 변경(반영 완료)** — 제목 번호가 한 단계씩 밀려 **핵심 설정 파일 = `1.5.x`**, **첫 세션 = `1.6.x`** 로 정리됨. 스크립트 슬라이드 라벨을 덱 네비게이션의 실제 면 번호(`S69`~`S88`)에 맞춤. (직전 파일명 `70-88`은 유지)
2. **권한 모드 4종 → 6종(중요·반영 완료)** — `default · acceptEdits · plan · **auto** · **dontAsk** · bypassPermissions` 로 확장됨. `auto`(안전 분류기 검사)와 `dontAsk`(사전 승인 도구만) 두 모드가 신규. 표·내레이션 전면 교체(S76).
3. **`settings.json` 주요 키에 `effortLevel` 추가(중요·반영 완료)** — `low·medium·high·xhigh·max·ultra` 6단계. 기본 모델이 **`claude-opus-4-8`**, 코딩 권장값이 **`xhigh`(Opus 4.8)**. `MAX_THINKING_TOKENS`는 Opus 4.7+에서 무시되며 `effortLevel`로 제어. S73 전면 갱신.
4. **클라우드 세션 슬라이드 삭제 → 참고 자료(References) 추가** — 직전 버전의 "클라우드 세션(`--remote`/teleport)" 면이 이 덱에서 **빠짐**. 대신 마지막에 공식 문서 링크 모음 **참고 자료** 면이 들어옴(S88). 해당 내레이션 제거·신규 작성.
5. **컨텍스트 구조 다이어그램 고도화** — 정적(SYSTEM ~11K · TOOLS ~13K · CLAUDE.md ~5K)/동적(T1~T7 누적, 70%·95% 경고, `/compact`→압축본 ~6K) 흐름도로 상세화됨. S82 내레이션 보강.
6. **`@ · !` 특수 문법** — 제목에서 `#`이 빠졌고(`@ · !`), 본문 표에는 여전히 "**`#`은 지원 안 함** — CLAUDE.md 기록은 클로드에게 직접 요청"으로 유지. 최신 기준 반영(S85).

> ⚠️ **덱 자체에 남은 결함(스크립트 아님 — 덱에서 고쳐야 함):** 제목 번호는 `1.5.x`/`1.6.x`로 맞췄으나 **하단 캡션·page-num이 여전히 stale**합니다. 예) S77 캡션 `1.3.3 …`, S81~S84 캡션 `CH.0 · PART 1 …` + page-num `05/06/07`, S79~S80 캡션 `1.5.1`. 녹화 자막에 캡션을 노출한다면 덱에서 캡션=제목 번호로 통일하고 page-num을 섹션 순번으로 재부여해야 합니다. 원하시면 덱 HTML 일괄 교정을 별도로 진행하겠습니다.

> ⚠️ 덱 HTML 수정은 에디터/웹 뷰어에 자동 반영되지 않습니다. 에디터로 본다면 툴바 **Reset**으로 재파싱, 웹 export면 재빌드, 브라우저로 HTML을 직접 열어 녹화한다면 새로고침하세요.

### S70. 1.5.1 `~/.claude/` 디렉토리 구조 — 전역 vs 프로젝트 ⏱ ~60초
**(화면)** 다이어그램: 홈의 `~/.claude/`(전역) vs 프로젝트의 `.claude/`(프로젝트)

> 설정에는 **두 군데 집**이 있습니다. 이걸 구분하는 게 오늘의 핵심이에요.
> 하나는 **전역(global)** 설정인 `~/.claude/`. 여기서 `~`(틸드)는 '내 홈 폴더'라는 뜻이에요. 이 폴더 안의 설정은 **내 컴퓨터의 모든 프로젝트에 공통**으로 적용됩니다. "나는 어디서 일하든 이 모델, 이 단축키" 같은 개인 취향이 여기 들어가요.
> 다른 하나는 **프로젝트(project)** 설정인, 각 프로젝트 폴더 안의 `.claude/`. 점(`.`)으로 시작하는 숨김 폴더죠. 여기 든 설정은 **그 프로젝트에서만** 적용되고, Git에 같이 올리면 팀원 전체가 똑같은 규칙을 공유하게 됩니다.
> 비유하면, 전역은 '내 책상 서랍'(나만의 기본값), 프로젝트는 '회의실 화이트보드'(그 팀의 공통 규칙)예요. 충돌하면 **더 가까운 쪽, 즉 프로젝트 설정이 전역을 덮어씁니다.**

**용어**
- **전역 / 프로젝트 설정** — 전역 = 내 컴퓨터 전체 공통(`~/.claude/`), 프로젝트 = 해당 폴더 한정(`.claude/`).
- **`~`(틸드, tilde)** — 홈 디렉토리. 맥/리눅스는 `/Users/이름`, WSL은 `/home/이름`.
- **숨김 폴더** — 이름이 `.`으로 시작해 평소엔 안 보임. `ls -a`로 확인.

설정 파일 내부 코드가 좀 복잡하다고 해서 걱정하실게 없어요.
클로드 코드가 다 만들어줄거구요, 다만 뭘 만들어줘야 해 라고 할때 용어는 기본적으로 
알아야 하니까 쉽게 지시를 할 수 있어서 그런거고, 그 키워드들을 알고 계셔야 한다는 의미로 설명을 드리는 겁니다.
훅이 뭔지, 도구 허용 목록이 뭔지, 설정이 어디있는지, 설정 값을 뭘 세팅해야 하는지 
대강은 알아둬야 클로드 한테 지시를 할 수 가 있겠죠?
예를들어 커밋 전에 반드시 테스트 돌려줘 
이런걸 지시하려면 좀 더 분명하기 반드시 매 커밋 전에 훅으로 테스트를 실행하도록 세팅해
이렇게 지시할 수 있겠죠? 


**샘플 코드** — 실제 폴더 구조를 터미널로 보여 주세요.
```bash
# 전역 설정 (내 컴퓨터 전체 공통)
~/.claude/
├── settings.json        # 전역 기본 설정
├── CLAUDE.md            # 전역 메모리(모든 프로젝트에 주입)
└── commands/            # 내 개인 슬래시 커맨드

# 프로젝트 설정 (이 프로젝트에서만, Git 공유)
<프로젝트>/.claude/
├── settings.json        # 팀 공유 설정 (커밋함)
├── settings.local.json  # 내 개인 오버라이드 (.gitignore)
└── CLAUDE.md            # 이 프로젝트 규칙
```

### S71. 1.5.2 `settings.json` — 팀 공유용 설정 ⏱ ~55초
**(화면)** 부제: "Git 커밋해서 팀 전체에 공유하는 프로젝트 설정" / 링크: code.claude.com/docs/en/settings

> 프로젝트 `.claude/` 안의 **`settings.json`**, 이게 팀 설정의 중심입니다. JSON은 'JavaScript Object Notation'의 약자로, **`키: 값` 형태로 설정을 적는 표준 형식**이에요. 사람이 읽기 쉽고 컴퓨터도 바로 이해합니다.
> 핵심 규칙 하나. 이 파일은 **Git에 커밋**해서 팀 전체가 공유합니다. 그래서 "우리 팀은 이 모델을 쓰고, 이런 명령은 금지하고, 이런 훅을 건다"가 코드와 함께 버전 관리돼요. 새 팀원이 와도 클론만 하면 똑같은 환경이 됩니다.
> 그리고 **나만의 개인 설정**은 같은 폴더의 `settings.local.json`에 따로 적고, 이건 `.gitignore`에 넣어 **커밋하지 않습니다.** 둘이 충돌하면 `local`이 이깁니다. 정리하면 — 팀 공통은 `settings.json`, 내 개인 비밀·취향은 `settings.local.json`.

**용어**
- **JSON** — `{ "키": "값" }` 형태의 설정/데이터 표준 형식. 중괄호·따옴표·쉼표 문법이 엄격해서 한 글자만 틀려도 에러.
- **`.gitignore`** — Git이 추적하지 말아야 할 파일 목록. 여기 적힌 파일은 커밋에서 제외됨.

**샘플 코드** — 가장 기본적인 팀 공유 `settings.json` 예시.
```json
{
  "model": "claude-opus-4-8",
  "permissions": {
    "allow": ["Read", "Edit", "Bash(npm run *)"],
    "deny": ["Bash(rm -rf *)"]
  },
  "includeCoAuthoredBy": true
}
```
```bash
# 이 파일은 팀 공유 → 커밋
git add .claude/settings.json
git commit -m "chore: share team Claude Code settings"

# 개인 설정은 커밋하지 않음
echo ".claude/settings.local.json" >> .gitignore
```

### S73. 1.5.4 `settings.json` 주요 키 — `effortLevel` 포함 ⏱ ~55초
**(화면)** 주요 키 표 / 부제: "자주 쓰는 키 · 현재 effort 레벨(low → ultra) 기준"

> `settings.json`에 자주 쓰는 **키(key)**를 한 표로 정리했습니다. 키는 'JSON의 설정 항목 이름'이에요.
> **`model`** — 이 프로젝트의 기본 모델. 지금 기준 권장은 **`claude-opus-4-8`**, 가볍게 갈 땐 `claude-sonnet-4-6`. **`effortLevel`** — 이게 새로 중요해진 키인데, 클로드가 **얼마나 깊이 생각할지(thinking effort)**를 정합니다. `low → medium → high → xhigh → max → ultracode` 여섯 단계예요. **코딩·에이전트 작업이라면 `xhigh`가 권장 기본값**입니다(Opus 4.8 기준). 깊이 생각할수록 똑똑하지만 느리고 비싸니, 작업 무게에 맞춰 고르는 손잡이라고 보시면 돼요.
> 나머지는 앞서 본 것들입니다. **`permissions`** — `defaultMode`(기본 권한 모드)와 allow/deny. **`env`** — 환경 변수를 팀과 공유(다음 장). **`includeCoAuthoredBy`** — 커밋에 'Co-authored-by' 줄을 넣을지.
> 두 가지만 더 짚을게요. **deny는 allow보다 항상 우선**이고, 예전에 쓰던 **`MAX_THINKING_TOKENS`는 Opus 4.7부터 무시**됩니다 — 생각 깊이는 이제 `effortLevel`로 제어하세요.

**용어**
- **키(key) / 값(value)** — JSON에서 `"키": 값`의 왼쪽이 키(항목 이름), 오른쪽이 값.
- **`effortLevel`** — 모델의 사고 깊이 단계. `low·medium·high·xhigh·max·ultra`. 코딩 권장값 `xhigh`(Opus 4.8).
- **`defaultMode`** — 세션을 시작할 때 기본으로 들어가는 권한 모드(예: `acceptEdits`). 권한 모드는 S76에서 설명.
- **`MAX_THINKING_TOKENS`** — (구 설정) Opus 4.7+에서 무시됨. `effortLevel`로 대체.

**샘플 코드** — 주요 키를 한눈에.
| 키 | 설명 | 예시 · 현재 값 |
|---|---|---|
| `model` | 기본 모델 | `"claude-opus-4-8"` · `"claude-sonnet-4-6"` |
| `effortLevel` | thinking effort 수준(낮음→높음) | `low` · `medium` · `high` · `xhigh` · `max` · `ultra` |
| `permissions.defaultMode` | 기본 권한 모드 | `"acceptEdits"` |
| `permissions.allow` | 자동 허용 도구 | `["Read", "Bash(npm run *)"]` |
| `permissions.deny` | 자동 거부 도구 | `["Bash(rm -rf *)"]` |
| `env` | 환경 변수(팀 공유) | `{"CLAUDE_CODE_EFFORT_LEVEL": "xhigh"}` |
| `includeCoAuthoredBy` | 커밋에 Co-authored-by 추가 | `true` |

> 💡 `xhigh` = 코딩·에이전트 작업 권장 기본값(Opus 4.8). `deny > allow`(거부가 항상 우선). `MAX_THINKING_TOKENS`는 Opus 4.7+에서 무시 → `effortLevel`로 제어.
> 📎 공식 문서: https://code.claude.com/docs/en/settings

### S74. 1.5.5 환경 변수 설정 — 3가지 방법 ⏱ ~55초
**(화면)** 부제: "셸 프로파일 · settings.json env · CI/CD 패턴" / 코드 블록 2개

> **환경 변수(environment variable)**는 '프로그램이 시작할 때 읽는 전역 설정값'입니다. 운영체제 차원에서 "이 이름엔 이 값" 하고 미리 정해 두면, 클로드 코드가 켜질 때 그걸 읽어요. 설정하는 방법이 세 가지인데, 적용 범위가 다릅니다.
> **① 셸 프로파일** — `~/.zshrc`나 `~/.bashrc`에 `export`로 적습니다. **내 컴퓨터 전역**, 즉 내가 어떤 프로젝트를 열든 적용돼요. "내 기본 모델은 Sonnet" 같은 개인 취향에 좋습니다.
> **② `settings.json`의 `env` 블록** — 프로젝트 설정에 박아 두면 **팀 전체가 공유**합니다. "우리 프로젝트는 effort high, 생각 토큰 8천" 같은 팀 규칙에 좋아요.
> **③ CI/CD 패턴** — 깃허브 액션 같은 자동화 파이프라인에서는 워크플로 파일이나 secrets에 넣습니다. 사람이 없는 환경이니, API 키 같은 건 여기에 안전하게요.
> 우선순위는 대체로 **더 좁고 가까운 쪽이 이깁니다.** 즉, 임시로 명령에 직접 준 값 → 셸/프로젝트 설정 순으로 덮어써져요.

**용어**
- **환경 변수** — OS가 들고 있다가 프로그램 시작 시 넘겨 주는 `이름=값` 쌍. 코드를 안 고치고 동작을 바꿀 때 씀.
- **`export`** — 셸에서 환경 변수를 '자식 프로그램에도 물려주도록' 등록하는 명령.
- **`~/.zshrc` / `~/.bashrc`** — 터미널(zsh/bash)이 켜질 때마다 자동으로 읽는 개인 설정 파일.
- **CI/CD** — Continuous Integration/Delivery. 코드를 올리면 자동으로 빌드·테스트·배포하는 파이프라인.

**샘플 코드 — ① 셸 프로파일 (개인 전역)**
```bash
# ~/.zshrc 또는 ~/.bashrc
export ANTHROPIC_MODEL="claude-sonnet-4-6"
export CLAUDE_CODE_EFFORT_LEVEL="medium"
# 저장 후 적용: source ~/.zshrc
```
**샘플 코드 — ② `settings.json`의 env 블록 (팀 공유 가능)**
```json
{
  "env": {
    "CLAUDE_CODE_EFFORT_LEVEL": "high",
    "MAX_THINKING_TOKENS": "8000",
    "CLAUDE_STREAM_IDLE_TIMEOUT_MS": "120000"
  }
}
```
**샘플 코드 — ③ CI/CD 패턴 (GitHub Actions 예)**
```yaml
# .github/workflows/claude.yml
env:
  CLAUDE_CODE_EFFORT_LEVEL: high
  ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}  # 키는 반드시 secrets로
```

### S75. 1.5.5+ 전역 `~/.claude/settings.json` — 개인 글로벌 가드 ⏱ ~55초
**(화면)** 전역 설정 파일 전체 예시(코드 블록) / 부제: "모든 프로젝트 공통 개인 기본값 + 범용 안전 deny · 표준 JSON은 주석(//) 불가"

> 방금 환경 변수를 봤으니, 그 설정들이 한 파일에 모이면 어떤 모습인지 보여 드릴게요. 바로 **전역 `~/.claude/settings.json`** 입니다. 앞에서 본 프로젝트 `.claude/settings.json`이 '팀 공유용'이었다면, 이건 **내 컴퓨터의 모든 프로젝트에 공통으로 깔리는 '개인 글로벌 설정'** 이에요. 충돌하면 프로젝트 설정이 이 전역을 덮어씁니다.
> 화면 코드를 위에서부터 읽어 볼게요. **`model`·`effortLevel`** — 내 기본 모델(`opus`)과 기본 추론 강도(`high`). 프로젝트마다 `/model`로 잠깐 바꿔도, 안 정하면 이 값이 깔립니다. **`outputStyle`·`language`** — 답변 스타일과 언어 같은 취향. **`includeCoAuthoredBy: false`** — 커밋에 'Co-authored-by: Claude' 서명을 안 넣겠다는 뜻이에요.
> 그리고 가장 중요한 게 아래쪽 **`permissions`** 입니다. `defaultMode`로 기본 권한 모드를 정하고, **`deny`에 '어느 프로젝트에서든 항상 막을' 범용 안전 가드**를 박아 둡니다. `.env` 같은 **비밀 파일 읽기 차단**(`Read`와 `Bash(cat ...)` 둘 다 막아 우회까지 봉쇄), 그리고 `sudo`·`rm -rf` 같은 **시스템 파괴 명령 차단**. 이 한 블록이 내 모든 작업의 기본 안전망이 돼요.
> 마지막으로 꼭 기억할 함정 하나. 화면 같은 예시 파일에는 설명을 위해 `//` 주석이 달려 있는 경우가 많은데, **실제 `settings.json`은 표준 JSON이라 주석을 허용하지 않습니다.** 주석이 한 줄이라도 남아 있으면 "Invalid or malformed JSON" 오류로 **설정 전체가 무시돼요.** 그래서 주석 달린 예시는 확장자가 `.jsonc`이고, 적용할 땐 **① 모든 `//` 주석 제거 → ② `.json`으로 저장**해야 합니다.

**용어**
- **전역 `~/.claude/settings.json`** — 내 컴퓨터의 모든 프로젝트에 공통 적용되는 개인 설정 파일. 프로젝트 `.claude/settings.json`(팀 공유)과 별개이며, 충돌 시 프로젝트 설정이 우선.
- **JSONC vs JSON** — JSONC는 주석(`//`)을 허용하는 예시용 포맷(`.jsonc`). 실제 적용 파일은 주석 불가인 표준 JSON(`.json`). 주석이 남으면 설정 전체가 무시됨.
- **범용 안전 deny** — 프로젝트와 무관하게 늘 막고 싶은 가드. 비밀 파일(`.env`) 읽기, `sudo`, `rm -rf` 등.

**샘플 코드** — 화면의 전역 설정(필수 요소만). ⚠️ 적용 시 주석 없는 표준 JSON으로 저장.
```json
{
  "model": "opus",
  "effortLevel": "high",
  "outputStyle": "Explanatory",
  "language": "korean",
  "includeCoAuthoredBy": false,
  "permissions": {
    "defaultMode": "acceptEdits",
    "deny": [
      "Read(./.env*)",
      "Bash(cat .env*)",
      "Bash(sudo *)",
      "Bash(rm -rf /*)",
      "Bash(rm -rf ~*)"
    ]
  }
}
```

### S76. 1.5.6 권한 모드 6종 한눈에 비교 ⏱ ~70초
**(화면)** 권한 모드 6종 비교표 / 부제: "안전 ↔ 자율 트레이드오프"

> 클로드 코드의 안전벨트, **권한 모드**입니다. 직전 버전엔 4종이었는데 지금은 **6종**으로 늘었어요. "클로드가 어디까지 내 허락 없이 해도 되는가"를 정하는 단계인데, 대부분은 세션 중에 **Shift+Tab**으로 돌려 가며 바꿉니다.
> **① default(기본)** — **읽기만 자동**, 파일을 고치거나 명령을 실행하기 전에는 **매번 물어봅니다.** 가장 안전해요. 시작·민감한 작업은 여기서.
> **② acceptEdits** — 읽기에 더해 **파일 편집과 기본 파일 명령(mkdir·touch·mv·cp·rm·sed)까지 자동 승인.** 리뷰하면서 코딩을 반복하는 **일상 작업의 균형점**이라 가장 많이 씁니다.
> **③ plan(계획)** — **읽기만 하고 분석·계획만** 합니다. 고치지는 않아요. 손대기 전에 코드베이스를 훑고 설계를 확인할 때.
> **④ auto** — **거의 전부 자동인데, 백그라운드의 '안전 분류기'가 위험한 명령을 걸러 줍니다.** 긴 작업에서 매번 승인하는 피로를 줄일 때 좋아요. Shift+Tab으로는 **plan 다음에 조건부**로 들어갑니다.
> **⑤ dontAsk** — **미리 승인해 둔 도구만 실행하고, 나머지는 자동으로 거부.** 묻지도 않고 막아요. 사람이 개입하지 않는 **잠금형 CI나 스크립트**에 씁니다. 이건 단축키가 아니라 **`--permission-mode dontAsk` 플래그**로만 켜요.
> **⑥ bypassPermissions** — **검사 없이 전부 실행.** 가장 빠르지만 가장 위험해서, **격리된 컨테이너나 VM 전용**입니다. `--dangerously-skip-permissions` 같은 플래그를 거쳐야 들어가요. (이번 강의 환경이 이쪽 결이라, 제가 명령을 확인 없이 막 실행할 수 있는 거예요.)
> 마지막으로 중요한 안전 규칙 — **bypassPermissions를 뺀 모든 모드는, `.git`·`.claude` 같은 보호 경로에 대한 쓰기는 자동 승인하지 않습니다.** 자동 모드라도 정말 중요한 폴더는 한 번 더 손을 들어 물어봐요.

**용어**
- **권한 모드(permission mode)** — 클로드의 행동(편집·명령 실행)을 승인 없이 어디까지 허용할지의 단계. 6종: `default` / `acceptEdits` / `plan` / `auto` / `dontAsk` / `bypassPermissions`.
- **Shift+Tab** — 세션 도중 권한 모드를 순환 전환하는 단축키(`default → acceptEdits → plan`, `auto`·`bypass`는 그 뒤 조건부).
- **안전 분류기** — `auto` 모드에서 백그라운드로 위험 명령을 걸러 주는 검사기.
- **보호 경로** — `.git`·`.claude` 등. bypass 외 모드에선 여기 쓰기를 자동 승인하지 않음.

**샘플 코드** — 비교를 표로 보여 주세요.
| 모드 | 자동 허용(프롬프트 없이) | 적합한 상황 | Shift+Tab |
|---|---|---|---|
| `default` | 읽기만 | 시작 · 민감한 작업 | ○ 기본 순환 |
| `acceptEdits` | 읽기 + 파일 편집 + 기본 파일 명령(`mkdir·touch·mv·cp·rm·sed`) | 리뷰하며 코딩 반복(일상) | ○ |
| `plan` | 읽기만 — 분석·계획만, 편집 안 함 | 손대기 전 코드베이스 탐색 | ○ |
| `auto` | 전부(백그라운드 안전 분류기가 검사) | 긴 작업 · 프롬프트 피로 감소 | △ 조건부(plan 뒤) |
| `dontAsk` | 사전 승인 도구만(나머지 자동 거부) | 잠금형 CI · 스크립트 | ✗ 플래그 전용 |
| `bypassPermissions` | 전부(검사 없음) | 격리 컨테이너 · VM 전용 | △ `--dangerously-skip` 후 |

> 💡 Shift+Tab 순환: `default → acceptEdits → plan` (`auto`·`bypass`는 plan 뒤 조건부, `dontAsk`는 `--permission-mode dontAsk`로만). `bypassPermissions` 외 모든 모드는 보호 경로(`.git`·`.claude` 등) 쓰기를 자동 승인하지 않습니다.
> 📎 공식 문서: https://code.claude.com/docs/en/permission-modes

### S77. 1.5.7 작은 단위 커밋 · git checkpoint — 일상 작업 ⏱ ~45초
**(화면)** 부제: "안전 우선에서 시작 → acceptEdits로 흐름 가속" / 📍 Git checkpoint 후 즉시 복구 코드

> 권한을 풀어 자동 실행을 쓰려면, **되돌릴 안전장치**가 먼저 있어야 합니다. 그게 **git 체크포인트(checkpoint)**예요. 게임에서 어려운 구간 직전에 세이브하는 거랑 똑같습니다.
> 흐름은 네 단계예요. **①** 작업 시작 전에 `git add . && git commit -m "checkpoint"`로 현재 상태를 통째로 저장합니다. **②** 그다음 `acceptEdits`로 클로드를 돌려 자동으로 작업시켜요. **③** 결과가 마음에 들면 그대로 커밋해서 확정합니다. **④** 마음에 안 들면? `git reset --hard HEAD` 한 줄이면 **체크포인트 상태로 즉시 되돌아갑니다.** 클로드가 만진 게 전부 사라지고 깨끗해져요.
> 그래서 핵심 습관은 **'작게 자주 커밋'**입니다. 체크포인트가 촘촘할수록 잃을 게 적습니다.

**용어**
- **체크포인트(checkpoint)** — 작업 직전에 만들어 두는 '되돌아갈 지점' 커밋.
- **`git reset --hard HEAD`** — 마지막 커밋 이후의 모든 변경을 버리고 그 커밋 상태로 복구. (되돌릴 수 없으니 커밋 전 변경에 주의)

**샘플 코드** — 화면의 체크포인트 흐름(전문).
```bash
# 1. 현재 상태 저장 (안전벨트)
git add .
git commit -m "checkpoint"

# 2. acceptEdits로 Claude Code 실행
claude            # Shift+Tab으로 acceptEdits 진입

# 3. 결과가 마음에 들면 → 확정
git add .
git commit -m "feat: Claude가 구현한 기능"

# 4. 결과가 마음에 안 들면 → 즉시 복구
git reset --hard HEAD
```

### S80. 1.6.1 Accept mode 팝업 — 3가지 선택지 (표) ⏱ ~40초
**(화면)** 팝업 3선택지 표

> 팝업의 세 가지 선택지입니다. **1번 Auto-accept** — 모든 작업을 승인 없이 즉시 실행. 빠르지만 그만큼 위험하죠(S76의 bypass와 비슷한 결). **2번 Manually approve edits** — 변경마다 수동 승인. 안전한 기본값입니다. **3번 Plan mode, Keep Planning** — 계획만 세우고 실행은 나중에. 큰 작업 설계용이에요.
> 처음이라면 **2번**으로 시작하시길 권합니다. 클로드가 뭘 하려는지 한눈에 보면서 익숙해진 뒤, 흐름이 손에 붙으면 그때 권한을 풀어도 늦지 않아요.

**샘플 코드** — 화면의 선택지 표.

|---|---|---|
| 1 | Auto-accept | 모든 작업을 승인 없이 즉시 실행 |
| 2 | Manually approve edits | 변경사항마다 수동 승인 (권장 시작점) |
| 3 | Plan mode, Keep Planning | 계획만 수립, 실행 여부는 나중에 선택 |

### S81. 1.6.2 턴 · 세션 · 컨텍스트의 이해 (개념) ⏱ ~60초
**(화면)** 3개 용어 정의+비유 표 / 💡 포함 관계: 턴 ⊂ 세션 ⊂ 컨텍스트

> 클로드 코드를 제대로 쓰려면 이 세 단어를 꼭 구분해야 합니다. 비유와 함께 갑니다.
> **턴(Turn)** — 내 메시지 하나 + 클로드 응답 하나의 **한 쌍**입니다. 중간에 도구를 호출하든 말든, '한 번의 질의응답'이 한 턴이에요. 예를 들어 "테스트 코드 추가해줘"라고 보내고 답이 오면, 그게 한 턴입니다.
> **세션(Session)** — 클로드를 **실행해서 종료할 때까지**의 단위예요. 그 안에 여러 턴이 차곡차곡 쌓이고, 고유한 `session_id`가 붙습니다. IDE를 열어 한참 작업하다 닫는 것, 그 한 덩어리가 세션이에요.
> **컨텍스트(Context)** — 매 추론 순간에 **모델이 실제로 보고 있는 토큰 전체**입니다. 시스템 프롬프트 + 도구 정의 + `CLAUDE.md` + 그동안 쌓인 턴까지 다요. 에디터·터미널·채팅 기록·내부 호출이 전부 누적된 '작업 기억'이라고 보시면 됩니다.
> 화면 아래 💡처럼 **포함 관계가 핵심**입니다. **턴 ⊂ 세션 ⊂ 컨텍스트** — 턴이 모여 세션이 되고, 세션 안에 누적된 턴 + 정적 요소(시스템 프롬프트·CLAUDE.md 등)가 컨텍스트를 이룹니다.

**용어**
- **턴(Turn)** — user 메시지 1 + Claude 응답 1의 쌍(도구 호출 포함).
- **세션(Session)** — `claude` 실행~종료까지. 여러 턴 누적, 고유 `session_id`.
- **컨텍스트(Context)** — 매 추론 시 모델이 보는 토큰 전체(시스템 프롬프트 + 도구 정의 + CLAUDE.md + 누적 턴).
- **토큰(token)** — 모델이 글을 처리하는 최소 단위(대략 단어/글자 조각).

**샘플 코드** — 정의+비유 표.
| 용어 | 정의 | 비유 |
|---|---|---|
| 턴 (Turn) | user 메시지 1 + Claude 응답 1의 쌍(도구 호출 포함) | "테스트 코드 추가해줘" + 응답 = 한 번의 질의응답 |
| 세션 (Session) | 실행~종료 단위. 여러 턴 누적, 고유 session_id | IDE를 열어 개발하고 닫기까지 |
| 컨텍스트 (Context) | 매 추론 시 모델이 보는 토큰 전체 | 에디터+터미널+채팅기록+내부호출의 누적 |

### S82. 1.6.2 컨텍스트의 구조 ⏱ ~70초
**(화면)** 컨텍스트 윈도우 다이어그램 / 부제: "200K(Sonnet·Opus 기본) / 1M(확장) 토큰 윈도우 안에 무엇이 들어가는가"

> 컨텍스트에는 **정해진 크기**가 있습니다. 이걸 **컨텍스트 윈도우(context window)**라고 해요. Sonnet·Opus 기준 기본 **20만(200K) 토큰**, 확장하면 **100만(1M) 토큰**까지 들어갑니다. 책으로 치면 수백 페이지 분량이지만, 무한은 아니에요.
> 이 창 안은 두 부분으로 나뉩니다. 왼쪽 **정적(static)** 부분 — 화면처럼 **SYSTEM(시스템 프롬프트 약 11K, Anthropic 비공개·수정 불가)**, **TOOLS(도구 정의 약 13K, Read·Edit·Bash·MCP 기본 제공)**, 그리고 **`CLAUDE.md`(약 5K, 프로젝트 룰북)**. 이건 **세션 시작 때 한 번 로드**되고 잘 안 변해요. 특히 `CLAUDE.md`는 내가 쓰는 거라 **영향력이 가장 큽니다(★)**.
> 오른쪽 **동적(dynamic)** 부분 — 내가 주고받는 턴이 T1, T2, T3… 쌓이면서 토큰이 계속 늘어요. 화면 숫자처럼 누적되다 **70%에 닿으면 노란불, 95%면 빨간불**입니다.
> 그래서 가득 차기 전에 손을 써요. **`/context`**로 사용률을 그림으로 확인하고, **70%쯤이면 `/compact`로 요약 압축**해서(압축본 약 6K로 줄여) 공간을 비운 뒤 다음 턴을 이어 갑니다. **95%를 넘거나 주제가 완전히 바뀌면 `/clear`로 초기화**하고요. 평소엔 큰 파일은 `head`/`grep`으로 잘라 읽고, 도구 출력은 필요한 부분만 인용하고, 세션을 끝낼 땐 `HANDOFF.md`로 넘기는 습관이 컨텍스트를 오래 건강하게 씁니다. 명령어는 다음 장에서 한 표로 정리할게요.

**용어**
- **컨텍스트 윈도우** — 모델이 한 번에 볼 수 있는 토큰의 최대 한도. Sonnet/Opus 기본 200K, 확장 1M.
- **정적 / 동적 부분** — 정적 = 세션 시작 시 1회 로드(SYSTEM·TOOLS·CLAUDE.md), 동적 = 매 턴 누적(대화·도구 결과).
- **70% / 95% 경고** — 사용률 70%면 압축 신호, 95%+면 초기화/인계 신호.
- **`HANDOFF.md`** — 세션 종료 직전 진행 상황을 적어 다음 세션에 넘기는 인계 문서.

### S83. 1.6.2 컨텍스트 관련 명령어 ⏱ ~55초
**(화면)** 명령어 표(8종)

> 컨텍스트를 다루는 **명령어 8종**입니다. 슬래시(`/`)로 시작하는 클로드 코드 내부 명령이에요.
> **`/context`** — 지금 컨텍스트를 얼마나 썼는지 그림으로 보여 줍니다. 70% 넘으면 정리 신호. **`/clear`** — 대화 기록을 비웁니다(단, `CLAUDE.md`는 다시 로드됨). 작업 주제가 완전히 바뀔 때요. **`/compact`** — 기록을 **요약본으로 압축**해 공간만 확보합니다. 같은 작업을 계속할 때 좋아요. **`/memory`** — `CLAUDE.md`를 편집기로 엽니다. 세션 중 발견한 규칙을 영구 저장할 때. **`/init`** — 프로젝트를 분석해 `CLAUDE.md` 초안을 만듭니다(새 프로젝트에서 딱 한 번). **`--continue`·`/resume`** — 이전 세션을 이어갑니다. **`/status`** — 현재 모델·effort·세션 정보. **`/cost`** — 누적 토큰·비용 확인.
> 다 외울 필욘 없어요. **'`/context`로 보고, 차면 `/compact`로 줄이고, 규칙은 `/memory`로 남긴다'** 이 세 개만 손에 익히면 충분합니다.

**용어**
- **슬래시 커맨드(slash command)** — `/`로 시작하는 클로드 코드 내장 명령.
- **`--continue` / `/resume`** — 종료했던 이전 세션을 다시 이어 여는 방법.

**샘플 코드** — 명령어 표.
| 명령어 | 한 줄 정의 | 언제 쓰는가 |
|---|---|---|
| `/context` | 현재 컨텍스트 사용량 시각화 | 작업 중 수시. 70%+ 도달 시 압축 고려 |
| `/clear` | 대화 히스토리 비움(CLAUDE.md는 재로드) | 작업 주제가 완전히 바뀔 때 |
| `/compact` | 히스토리를 요약본으로 압축 | 같은 작업 계속하며 공간만 확보 |
| `/memory` | CLAUDE.md를 편집기로 열기 | 세션 중 발견한 규칙을 영구 저장 |
| `/init` | 프로젝트 분석 후 CLAUDE.md 초안 생성 | 새 프로젝트 시작 시 1회 |
| `--continue` / `/resume` | 이전 세션 이어가기 | 어제 작업 다시 시작할 때 |
| `/status` | 현재 모델·effort·세션 정보 | 디버깅·점검 |
| `/cost` | 누적 사용 토큰·비용 확인 | 비용 모니터링 |

### S84. 1.6.2 휘발성 — 무엇이 사라지고 무엇이 남는가 ⏱ ~50초
**(화면)** 🔥 휘발 / 📌 보존 / 🔗 이어가기 3블록

> 첫 세션에서 가장 많이 하는 실수가 이거예요. "어제 작업이 왜 사라졌지?" 컨텍스트의 **휘발성(volatility)**을 모르면 꼭 겪습니다.
> **🔥 휘발** — 컨텍스트, 즉 누적된 턴과 도구 결과는 **세션이 끝나면 전부 사라집니다.** 다음 세션으로 자동으로 안 넘어가요. 클로드는 어제 우리가 무슨 얘길 했는지 기본적으로 기억하지 못합니다.
> **📌 보존** — 딱 하나, **`CLAUDE.md`는 다음 세션에서도 자동 재로드**됩니다. 그래서 "어제 정한 규칙"은 반드시 거기 적어 둬야 살아남아요. 휘발성 메모리(대화)와 영구 메모리(CLAUDE.md)를 구분하는 게 핵심입니다.
> **🔗 이어가기** — 그래도 세션을 잇는 방법이 네 가지 있어요. `--continue`, `/resume`, `/compact`, 그리고 진행 상황을 문서로 남기는 **HANDOFF(인계) 문서**. 큰 작업은 끝낼 때 핸드오프를 남겨 두면 다음 세션이 그걸 읽고 이어갑니다.

**용어**
- **휘발성(volatility)** — 세션 종료 시 컨텍스트(대화·도구 결과)가 사라지는 성질.
- **영구 메모리 vs 휘발 메모리** — `CLAUDE.md`(영구, 재로드) vs 대화 컨텍스트(휘발).
- **HANDOFF(핸드오프) 문서** — 진행 상황·결정·다음 할 일을 적어 다음 세션에 넘기는 인계 문서.

### S85. 1.6.3 프롬프트 특수 문법 — `@` · `!` ⏱ ~50초
**(화면)** 부제: "파일 참조 · Bash 직접 실행 · CLAUDE.md 메모리 기록" / 코드 + 표

> 클로드와 대화할 때 쓰는 **특수 문법** 두 개입니다. 입력창에서 바로 먹혀요.
> **`@`(앳) — 파일·디렉토리 참조.** `@`를 치고 경로를 적으면 그 파일이나 폴더 내용을 **컨텍스트에 통째로 넣어** 줍니다. "`@src/utils/auth.js` 이 파일 로직 설명해줘"처럼요. 파일을 넣는 거니 **토큰을 소비**합니다. 입력 중 `@` 뒤에 **Tab**을 누르면 파일 목록이 떠서 고르기 편해요.
> **`!`(느낌표) — Bash 명령 직접 실행.** `!` 뒤에 명령을 적으면 클로드를 거치지 않고 **터미널 명령을 바로 실행**해 결과를 보여 줍니다. 모델 추론을 안 거치니 **토큰을 안 써요.** (이 강의 환경에서도 `! 명령` 형태로 직접 실행할 수 있다고 안내드린 그 문법입니다.)
> **중요 — `#`은 이제 지원하지 않습니다.** 예전엔 `#`으로 메모를 적었는데, 지금은 **`CLAUDE.md`에 기록할 내용은 클로드에게 직접 말로 요청**하세요. "이 내용 CLAUDE.md에 추가해줘"처럼요. (그래서 제목도 `@ · !` 두 개만 남았습니다.)

**용어**
- **`@파일/디렉토리`** — 해당 파일·폴더 내용을 컨텍스트에 포함(토큰 O). 입력 중 Tab으로 경로 자동완성.
- **`!명령`** — Bash 명령을 클로드 추론 없이 즉시 실행(토큰 ❌).
- **`#`** — (옛 메모 문법) **현재 미지원**. CLAUDE.md 기록은 클로드에게 직접 요청.

**샘플 코드** — 화면의 `@`·`!` 예시.
```bash
# 특정 파일을 컨텍스트에 포함
> @src/utils/auth.js 이 파일의 로직을 설명해줘

# 디렉토리 전체 분석
> @src/components/ 이 디렉토리 구조를 분석해줘

# 입력 중 Tab → 파일 목록 자동완성
> @src/

# ! 로 Bash 직접 실행 (추론·토큰 없이)
> !git status
```
| 문법 | 기능 | 토큰 소비 |
|---|---|---|
| `@파일명` | 파일/디렉토리를 컨텍스트에 포함 | O |
| `!명령어` | Bash 직접 실행 | ❌ |
| `#` | **미지원** — CLAUDE.md 기록은 클로드에게 직접 요청 | (해당 없음) |

### S86. 1.6.4 핵심 단축키 9종 ⏱ ~55초
**(화면)** 단축키 9종 / 강조: "Esc 우선 — Ctrl+C는 세션 종료, Esc는 응답 중단"

> 마지막으로 손에 익혀야 할 **단축키**입니다. 딱 하나만 기억하라면 — **멈출 땐 Ctrl+C가 아니라 Esc.**
> 초보자가 가장 많이 하는 실수가, 클로드가 엉뚱한 길로 갈 때 당황해서 **Ctrl+C**를 누르는 건데요. Ctrl+C는 **세션 자체를 종료**시켜 버립니다. 지금까지 쌓은 컨텍스트가 다 날아가요. 대신 **Esc**를 누르면 **지금 응답만 중단**되고 세션은 그대로 살아 있어, 바로 방향을 다시 잡아 줄 수 있습니다.
> 나머지 단축키는 화면을 보며 짚고 가겠습니다. **Shift+Tab**(권한 모드 전환), **Esc 두 번**(이전 입력으로 되돌리기), 위·아래 화살표(명령 히스토리) 정도만 익혀도 작업이 훨씬 매끄러워집니다.

**용어**
- **Esc** — 진행 중인 *응답만* 중단(세션 유지). 방향 수정의 기본기.
- **Ctrl+C** — *세션 종료*(컨텍스트 소실). 멈추려다 잘못 누르기 쉬우니 주의.
- **Shift+Tab** — 권한 모드 순환 전환(S76).
- **Esc Esc** — 직전 입력 시점으로 되돌리기(undo).
