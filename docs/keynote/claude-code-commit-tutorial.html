Claude Code로 커밋 메시지 작성하기

Claude Code는 별도 기능이 아니라 "커밋해줘"라고 시키면 그 시점에 git diff와 git log를 읽고 메시지를 직접 작성·커밋합니다. 이 가이드는 기본 동작부터 CLAUDE.md·슬래시 커맨드로 규칙을 박고 husky + commitlint로 형식을 강제하는 풀 워크플로까지 다룹니다.

한눈에 (결론)

  1. Claude Code는 "커밋해줘" 요청 시 git status·git diff --staged·git log를 읽어 메시지를 스스로 작성하고 실행한다.
  2. 기본값으로 Co-Authored-By 트레일러가 붙는다 → includeCoAuthoredBy: false + CLAUDE.md에 명시로 끈다.
  3. 저장소 규칙(Conventional Commits·prefix·scope)을 따르게 하려면 ① CLAUDE.md + @import, ② .claude/commands/commit.md, 둘 중 하나(또는 둘 다).
  4. 작성은 Claude가, 검증·거부는 commitlint(husky)가 — 생성과 검증은 보완 관계. 둘 다 두면 사람이 쓰든 Claude가 쓰든 형식이 보장된다.

PART 1Claude Code의 자동 커밋 동작

먼저 아무 설정 없이 "커밋해줘"라고 했을 때 Claude Code 내부에서 무엇이 일어나는지를 안다.

3단계 내부 동작

  1. 변경 파악 — git status · git diff --staged로 무엇이 바뀌었는지 읽는다.
  2. 스타일 참고 — git log로 저장소의 기존 메시지 스타일·prefix 분포를 추론한다.
  3. 작성·실행 — 변경 성격에 맞춰 메시지를 작성하고 git commit을 실행한다.

기본 출력 형태 (끄지 않으면 자동 부착)

fix: resolve logout on session timeout

Generated with Claude Code
Co-Authored-By: Claude <noreply@anthropic.com>
함정 — 규칙을 안 알려주면 추론으로 결정됨

저장소 이력에서 본 임의의 스타일로 작성됩니다. 팀이 정한 prefix·scope 규칙을 따르게 하려면 PART 2의 방법 중 하나로 명시해야 합니다.

PART 2규칙을 가르치는 3가지 방법

전제 — 프리픽스 규칙 문서를 저장소 안 고정 경로에 둔다 (예: docs/commit-prefix-guide.md).

방법 A · CLAUDE.md에 규칙 명시 (상시 적용, 권장)

매번 시키지 않아도 자동 적용되어야 할 때 — 팀 표준 강제용.

CLAUDE.md는 세션 시작 시 자동 주입됩니다. 한 번 박으면 모든 세션이 규칙을 들고 시작합니다.

<!-- CLAUDE.md -->
## 커밋 규칙
커밋 메시지는 반드시 @docs/commit-prefix-guide.md 의 프리픽스 표와 형식을 따른다.
- 형식: <type>(<scope>): <subject> (subject는 영문 명령형, 72자 이내, 마침표 없음)
- 허용 type: feat, fix, refactor, perf, docs, test, chore, ci, style
- Co-Authored-By 라인은 추가하지 않는다.

@docs/commit-prefix-guide.md는 해당 파일을 컨텍스트로 불러오는 import 문법입니다. 규칙이 길면 본문에 다 넣지 말고 이렇게 참조만 거는 게 토큰 효율적입니다.

방법 B · 커스텀 /commit 슬래시 커맨드 (필요할 때 호출)

규칙을 상시 주입하기엔 토큰이 아깝거나, 특정 상황에서만 적용하고 싶을 때.

.claude/commands/commit.md 파일을 만들면 채팅에서 /commit으로 실행됩니다.

<!-- .claude/commands/commit.md -->
@docs/commit-prefix-guide.md 의 규칙에 따라 커밋한다.

1. `git status`와 `git diff --staged`로 변경 내용을 파악한다.
2. 변경 성격에 맞는 type을 위 문서의 표에서 고른다.
3. `<type>(<scope>): <subject>` 형식으로 메시지를 작성한다.
   - subject: 영문 명령형·현재형, 72자 이내, 마침표 없음
   - 필요 시 본문(왜 바꿨는지)과 푸터(`Closes #번호`)를 추가
4. Co-Authored-By 라인 없이 `git commit`한다.

사용:

/commit

팀과 공유하려면 .claude/commands/(프로젝트), 나만 쓰려면 ~/.claude/commands/(개인)에 둡니다.

방법 C · 즉석 프롬프트 (1회용)

규칙을 영구화할 필요 없이 한 번만 적용하고 싶을 때.

@docs/commit-prefix-guide.md 규칙대로 스테이징된 변경을 커밋해줘
언제 무엇을 쓰는가

팀 표준이면 A, 본인만의 워크플로면 B, 일회성이면 C. A + B를 함께 쓰는 것도 권장됩니다 — A로 항상 박혀 있고, B로 *명시적으로 부르고 싶을 때* 깔끔하게 호출.

PART 3Conventional Commits — type 9종 풀 표

type은 SemVer(Semantic Versioning) 영향과 직결됩니다 — 배포 자동화 도구가 feat는 Minor, fix는 Patch로 버전을 자동 상향합니다.

type의미예시SemVer
feat새 기능 추가feat(auth): add password reset flowMinor
fix버그 수정fix(editor): resolve text undo state lossPatch
refactor기능 변화 없이 코드 정리refactor(export): extract theme CSS inlinerPatch
perf성능 개선perf(canvas): optimize overlay re-renderPatch
docs문서 추가/수정 (코드 변화 없음)docs(readme): add setup guide없음
test테스트 추가/수정test(generator): assert theme inlining없음
chore빌드·의존성·설정 변경chore(deps): update vite to 5.0없음
ciCI/CD 파이프라인 수정ci(workflow): add pre-push test없음
style포맷·세미콜론 등 코드 스타일만style: remove trailing whitespace없음

BREAKING CHANGE — Major 버전 상향

type과 무관하게 하위 호환을 깬다면 본문 끝에 BREAKING CHANGE: 푸터를 박습니다 → 배포 도구가 Major 버전을 올립니다. PART 5에서 자세히.

좋은 커밋 / 나쁜 커밋 — 5쌍 비교

나쁜 예좋은 예고친 이유
fix bug fix(editor): resolve caret position on mobile type·scope 명시, 구체적인 문제
feat: done feat(export): add PNG per-slide export (1920×1080) "done"은 의미 없음 → 무엇을 한 건지
Update stuff refactor(store): extract slide overlay state to atom 영문 명령형 ("Update" → "extract")
fix(all): lots of fixes fix(canvas): prevent overlay drag outside viewport 무분별한 all scope 회피 → 정확한 scope
docs: korean update docs(guide): translate CLI reference to Korean 객체·목적어 명시 (무엇을 번역했는가)

핵심은 명령형 현재형(imperative mood). "Add X", "Fix Y", "Update Z" 스타일. Git 자체도 "Merge branch 'foo'"처럼 명령형으로 메시지를 만듭니다.

PART 4본문(Body)·푸터(Footer) 작성 가이드

대부분의 커밋은 subject 한 줄로 충분합니다. 다음 3가지 경우엔 본문이 사실상 필수입니다.

  • 버그 수정 — 왜 이 버그가 생겼고 어떻게 고쳤는지
  • 리팩터링 — 기존 방식의 문제점 + 새 방식의 이점
  • 성능 개선 — 병목점 분석 + 개선 전후 메트릭

본문 작성 규칙

  • subject 다음 한 줄 띄우고 시작
  • 72자 이내로 줄바꿈 (git 기본 뷰 80자 컬럼 기준)
  • 무엇을이 아니라 왜에 집중 (무엇은 diff가 말해줌)

실전 예시 — body 있는 버그 수정

fix(editor): handle null overlay data in undo stack

When undoing overlay drag operations without an initial drop,
the overlay data could be null, causing TypeError on subsequent
redo. Now we default to empty position array.

Closes #142

푸터(Footer) — 이슈 추적·BREAKING CHANGE

키워드용법예시
Closes / Fixes / Resolves 이 커밋이 이슈를 해결 (PR 머지 시 자동 close) Closes #142
Refs / See 관련 이슈 참조 (자동 close 안 함) Refs #99, #103
BREAKING CHANGE: 하위 호환 깨짐 — SemVer Major 상향 트리거 BREAKING CHANGE: removed legacy slideFormat API
Co-Authored-By: 공동 작성자 표시 (Claude Code 기본 부착 — 끄는 법은 PART 7) Co-Authored-By: Name <email>

풀 예시 — 모든 요소 포함

refactor(store): redesign slide navigation state

Previous approach used nested reducers causing race conditions
on rapid navigation. Now using atomic Zustand atoms.

Closes #87
Refs #45, #92
BREAKING CHANGE: removed legacyNavigationMode flag from store config

PART 5husky + commitlint — 형식 강제

Claude가 규칙을 어겨도 commitlint가 커밋을 막아주면 형식이 구조적으로 보장됩니다. 작성은 Claude · 검증은 commitlint — 보완 관계.

역할담당
커밋 메시지 작성Claude Code (문서 규칙 참조)
형식 검증·거부commitlint + husky commit-msg 훅

1단계 — 설치

npm install -D @commitlint/config-conventional @commitlint/cli husky

# husky 초기화 (.git이 있다고 가정)
npx husky install

# commit-msg 훅 추가
npx husky add .husky/commit-msg 'npx commitlint --edit "$1"'

2단계 — 설정 파일 commitlint.config.cjs (프로젝트 루트)

module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'refactor', 'perf', 'docs',
      'test', 'chore', 'ci', 'style'
    ]],
    'scope-case':       [2, 'always', 'lower-case'],
    'subject-case':     [2, 'never', ['start-case', 'pascal-case']],
    'subject-full-stop':[2, 'never', '.'],
    'subject-empty':    [2, 'never'],
    'type-case':        [2, 'always', 'lower-case']
  }
};

3단계 — 동작 확인

실패 케이스 — 규칙 위반 시 훅이 막음:

$ git commit -m "add new feature"
husky > commit-msg (node v22.18.0)
✖   input: add new feature
✖   [type-enum] type must be one of [feat, fix, refactor, ...]
✖   1 problems (1 error)

성공 케이스 — 규칙 준수 시 통과:

$ git commit -m "feat(editor): add new overlay snap"
husky > commit-msg (node v22.18.0)
✓   commitlint check passed
[develop abc1234] feat(editor): add new overlay snap
왜 둘 다 두는가

Claude가 규칙을 *대부분* 잘 따르지만 100%는 아닙니다. commitlint는 구조적 안전망으로, Claude가 어겨도·사람이 직접 쓰다 어겨도·외부 기여자가 어겨도 동일하게 막아줍니다. 작성기와 검증기는 분리할 때 가장 강합니다.

PART 6이 저장소의 실 컨벤션 (git log 실측)

이 가이드를 짜는 시점에 git log -100으로 측정한 prefix 분포입니다.

type빈도비고
feat49가장 많음 — 기능·콘텐츠 추가 위주
fix16버그·렌더 수정
docs14발표 스크립트·가이드·README
chore9의존성·하네스·설정
test4테스트 추가
refactor3코드 정리
level1⚠️ 비표준 — 오타/오용
doc1⚠️ 비표준 — 오타 (정답: docs)
반면교사 — 실제로 새어 들어간 오타 2건

이 저장소에 level: ...와 doc: ... 같은 비표준 prefix가 *한 번씩* 새어 들어가 있습니다. 훅(husky + commitlint)이 없거나 --no-verify로 우회됐을 가능성이 큽니다. 이게 정확히 PART 5의 commitlint 셋업이 필요한 이유입니다 — 사람도 Claude도 오타를 냅니다.

이 저장소의 실 패턴 — 권고

  1. subject는 영문 명령형 유지 (Conventional Commits 표준)
  2. scope는 가능한 한 명시 — 특히 feat·fix·refactor에서
  3. 한국어 설명이 필요하면 본문에 — 영문 subject 다음 한 줄 띄우고
  4. 이슈 번호는 footer에 — Closes #N 형식

이 저장소 톤에 맞는 풀 예시

feat(canvas): implement overlay drag-to-reposition

슬라이드 오버레이 이미지를 자유롭게 드래그할 수 있게 구현.
Moveable.js를 활용해 x/y 좌표를 실시간 업데이트.

Closes #156

PART 7Co-Authored-By 끄기 + 트러블슈팅 5선

Co-Authored-By 트레일러 끄기

settings.json에 추가 — 전역(~/.claude/settings.json) 또는 프로젝트(.claude/settings.json):

{
  "includeCoAuthoredBy": false
}
⚠ 함정

이 설정이 간헐적으로 안 먹는 사례가 보고됐습니다 (Bash 도구로 직접 커밋할 때 등). 확실히 막으려면 CLAUDE.md에도 "Co-Authored-By 라인 추가 금지"를 함께 적어두는 것이 안전합니다 (이중 방어).

트러블슈팅 5선

① includeCoAuthoredBy: false가 먹지 않는다

  • 설정이 잘못된 경로에 있음 (글로벌 vs 프로젝트 위치 확인)
  • 업그레이드 후 캐시 미리로드 — 세션 재시작
  • /commit 슬래시 커맨드가 override 지시를 가짐 → 커맨드 본문 확인
  • 이중 방어: CLAUDE.md에도 명시 ("Co-Authored-By 라인 추가 금지")

② commit-msg 훅이 실행되지 않는다

  • .husky/ 디렉터리가 .gitignore로 제외 — clone 후 npx husky install 재실행
  • 훅 파일 실행 권한 없음 — chmod +x .husky/commit-msg
  • 설치 확인: ls -l .husky/commit-msg → -rwxr-xr-x 여야 함

③ git commit --no-verify의 위험

사용 금지가 원칙입니다. 훅을 우회하면 규칙 위반 메시지가 그대로 히스토리에 박힙니다 (PART 6의 level·doc 오타 2건이 정확히 그 결과).

  • 규칙 위반이면 --no-verify가 아니라 메시지를 고쳐서 다시 커밋
  • 정당한 예외가 있으면 .commitlintrc 규칙 자체를 PR로 수정·리뷰

④ 멀티라인 메시지의 따옴표 이스케이프

Bash에서 git commit -m "..."로 본문에 줄바꿈을 넣으면 따옴표 오류가 발생합니다. HEREDOC을 씁니다 (Claude Code 내부도 이 방식):

git commit -m "$(cat <<'EOF'
feat(editor): add new overlay feature

This commit adds support for free-position overlays.
Resolves issue #156.
EOF
)"

또는 git commit 단독 실행으로 에디터가 열리게 두기.

⑤ Claude Code가 git add -A를 무의식으로 쓰는 위험

.env·로컬 설정·큰 바이너리가 실수로 staged돼 커밋되는 사고가 생길 수 있습니다.

  • 커밋 전 항상 git status로 시각 확인
  • .gitignore 철저히 (.env, *.local.json, node_modules)
  • git diff --staged로 시크릿·바이너리 누수 점검
  • CLAUDE.md에 명시: "Use git add <file> for specific files, never git add -A"
CLAUDE.md 보완 스니펫 — Commit Safety

위 함정들을 한 박스로 묶어 CLAUDE.md에 넣어두면 모든 세션이 보호됩니다:

## Commit Safety
- Always run `git status` before commit; verify `git diff --staged`.
- Use `git add <file>` for specific files; never `git add -A` or `git add .`.
- Do not add `Co-Authored-By:` trailer.
- Never use `--no-verify` to bypass commitlint.
- Use HEREDOC for multi-line messages.

마무리워크플로 한 장 정리

  1. 한 번 박는다 — CLAUDE.md에 커밋 규칙 + Commit Safety 스니펫.
  2. 도구를 깐다 — npm i -D husky @commitlint/... + commitlint.config.cjs.
  3. 이후엔 자연어로 — "커밋해줘" 한마디면 Claude가 diff를 읽고 규칙에 맞춰 작성, commitlint가 검증·통과.
  4. 예외는 명시적으로 — 일회성 규칙은 즉석 프롬프트(방법 C), 특수 워크플로는 /commit 슬래시(방법 B).

"작성은 Claude가, 강제는 훅이 한다."
둘을 함께 두면 사람이 쓰든 Claude가 쓰든 외부 기여자가 쓰든 형식이 보장됩니다.
