- **부가설명**: 정적 영역은 SYSTEM(~11K, Anthropic 비공개·수정 불가), TOOLS(~13K, Read·Edit·Bash·MCP 기본 제공), 그리고 우리가 쓰는 CLAUDE.md(~5K, 영향력 ★)로 구성됩니다. 동적 세션은 턴마다 쌓여 T5에서 70% 경고, T6에서 95% 임박에 도달합니다.
- **부가설명**: 그래서 70%에 닿으면 /compact로 압축본(~6K)을 만들어 재개하고, 95%+이거나 주제가 바뀌면 /clear로 초기화하며, /context로 어디가 토큰을 먹는지 % 단위로 실시간 확인합니다.

## p11 · 휘발성 — 무엇이 사라지고 무엇이 남는가
- **핵심**: 세션이 끝나면 누적된 턴과 도구 결과인 컨텍스트는 모두 사라지고, 다음 세션으로 자동으로 넘어가지 않습니다.
- **부가설명**: 단 하나, CLAUDE.md만은 다음 세션에서도 자동 재로드되므로 "어제 정한 규칙"은 반드시 거기에 적어둬야 살아남습니다. "어제 작업이 왜 사라졌냐"는 바로 이 휘발성을 몰라서 생기는 일입니다.
- **부가설명**: 세션을 잇는 방법은 네 가지 — --continue, /resume, /compact, 그리고 HANDOFF 문서입니다.

## p15 · 내장 시스템 프롬프트 — 당신이 안 썼지만 모델이 매 턴 보는 숨은 지침
- **핵심**: PART 3입니다 — 우리가 쓰지 않았지만 모델이 매 턴 맨 앞에서 보는 내장 시스템 프롬프트를 다룹니다. 편집 불가이고 항상 첫 순위로 로드되며, CLAUDE.md와는 다른 레이어입니다.

## p16 · 모델이 매 턴 보는 4개의 텍스트 레이어
- **핵심**: 모델이 보는 텍스트는 ①내장 시스템 프롬프트 ②자동 주입 문서(CLAUDE.md·MEMORY.md) ③누적 대화 ④온디맨드 파일 네 레이어이며, 숨은 시스템 프롬프트가 항상 맨 앞에 붙습니다.
- **부가설명**: 내장 시스템 프롬프트는 교체·삭제 불가 — --append-system-prompt로 덧붙이거나 Output Styles로 톤·역할만 바꿀 수 있습니다. CLAUDE.md는 이와 별개로 첫 user 메시지로 주입되며 200줄 미만 유지가 권장됩니다.
- **부가설명**: 프롬프트 캐싱 덕에 시스템 프롬프트는 첫 턴만 전액 청구되고 이후 턴은 약 1/10 비용으로 재사용되지만, 모델·MCP 설정을 자주 바꾸면 캐시가 깨져 다시 전액이 청구됩니다.

## p17 · 8개 슬래시 명령어 — 컨텍스트 운용의 80%를 책임지는 명령어
- **핵심**: PART 4입니다 — /context /clear /compact /memory /init --continue /status /cost, 이 8개 명령어가 컨텍스트 운용의 80%를 책임집니다.

## p18 · 명령어 한눈에 보기
- **핵심**: 이 8개만 익혀두면 컨텍스트 운용의 80%가 해결됩니다 — 각각 언제 쓰는지 한눈에 정리한 표입니다.
- **부가설명**: /context는 사용량 시각화(70%+ 도달 시 압축 고려), /clear는 주제가 완전히 바뀔 때 히스토리 비움(CLAUDE.md는 재로드), /compact는 같은 작업을 이어가며 공간만 확보할 때, /memory는 발견한 규칙을 영구 저장할 때 씁니다.
- **부가설명**: /init은 새 프로젝트 시작 시 단 1회 초안 생성, --continue·/resume은 어제 작업 재개, /status는 모델·effort·세션 점검, /cost는 누적 토큰·비용 모니터링용입니다.

## p19 · /clear vs /compact

- **핵심**: `/clear`는 대화 히스토리를 완전히 삭제하는 초기화, `/compact`는 누적된 턴(T1..Tn)을 요약본(SUM)으로 압축해 맥락은 살리고 공간만 확보하는 명령입니다.
- **부가설명**: `/clear`는 대화 히스토리를 완전히 비우고 직전 작업 맥락이 사라지지만, SYSTEM·TOOLS는 자동 리로드되고 CLAUDE.md는 다시 주입됩니다. `/compact`는 SYSTEM·TOOLS·CLAUDE.md를 그대로 둔 채 누적 턴만 요약해 약 6K를 확보합니다.
- **부가설명**: 그래서 작업 주제가 완전히 바뀌거나 다른 프로젝트로 전환할 땐 `/clear`, 같은 작업을 이어가면서 컨텍스트가 70%를 넘겨 공간만 필요할 땐 `/compact`를 씁니다.

## p20 · 컨텍스트 사용률 단계별 대응

- **핵심**: `/context` 출력의 % 값에 따라 행동을 미리 정해두는 게 핵심입니다 — 압박을 느끼고 대응하는 게 아니라 임계치별로 자동으로 움직이는 겁니다.
- **부가설명**: 약 50%까지는 여유(SAFE)로 평소대로, 60~70% 주의(CAUTION)에선 마일스톤에서 `/compact`를 고려, 70~90% 위험(WARNING)에선 즉시 `/compact`로 압축 후 작업 재개, 90%+ 임박(CRITICAL)에선 마무리 후 HANDOFF를 쓰고 `/clear`로 새 세션을 엽니다.
- **부가설명**: auto-compact는 약 95%에서 자동 발동하는데, 그 전에 직접 `/compact`를 호출하면 마일스톤을 깔끔하게 끊을 수 있고 결과 품질도 더 좋습니다.

## p21 · CLAUDE.md 그 다음 — HANDOFF · MEMORY

- **핵심**: PART 5로 넘어갑니다. CLAUDE.md 다음으로, 세션을 넘어 작업을 잇거나 장기 결정을 보관하는 보조 문서 패턴 — HANDOFF와 MEMORY를 봅니다.

## p22 · 3종 문서 — 무엇을 어디에 적는가

- **핵심**: 문서 3종은 역할이 다릅니다 — CLAUDE.md는 프로젝트 규칙·컨벤션·금지사항으로 유일하게 자동 주입되고, HANDOFF.md와 MEMORY.md는 자동 주입되지 않습니다.
- **부가설명**: CLAUDE.md는 권장 길이 200줄 이하의 공식 문서, HANDOFF.md는 세션 종료나 `/compact` 직전 다음 세션에 넘길 작업 컨텍스트(수동/Hook/Skill), MEMORY.md는 CLAUDE.md 비대화를 막으려고 장기 의사결정 로그를 따로 뺀 관행 문서입니다.
- **부가설명**: PRD·TRD·DESIGN·AGENTS·REQUIREMENTS 같은 문서는 클로드 코드 마스터 책 내부에서 다루니 이름만 미리 알아두시면 됩니다.

## p23 · HANDOFF.md — 세션 인계 문서 샘플

- **핵심**: HANDOFF.md는 한 세션의 작업을 다음 세션에 그대로 넘기는 인계장입니다 — 날짜·세션 ID와 함께 한 일, 이어서 할 일, 결정 사항, 미해결 이슈를 적습니다.
- **부가설명**: 예시를 보면 "한 일"에 작성한 파일과 추가한 Drizzle 쿼리 3개, "다음에 이어서 할 일"은 체크박스로, "결정된 사항"엔 Result 패턴·soft delete 같은 합의를, "미해결 이슈"엔 categoryId가 null일 때의 정렬 로직 미정 같은 걸 남깁니다.

- **부가설명**: Hook은 PreToolUse에서 git push origin main을 감지해 exit 2로 차단합니다 — 100% 차단, 무시 불가능. 넘어가려 해도 반드시 감속하게 만드는 '과속 방지턱'입니다.

## p63 · ⚠️ 가장 흔한 실수 — hook vs hooks

- **핵심**: settings.json에서 단수 `hook`과 복수 `hooks` 한 글자 차이로 설정 파일 전체가 무시되니 반드시 복수형으로 적어야 합니다.

- **핵심**: 안전한 자동화는 방향을 주는 Skill과 안전망을 거는 Hook이 둘 다 있어야 완성됩니다.
- **부가설명**: /commit 입력으로 Command가 발동하고, 본문에서 `!`git diff --staged`` 로 동적 컨텍스트를 주입해 Claude가 Conventional Commits 메시지를 만듭니다 — 여기까지는 권고(Advisory)입니다.
- **부가설명**: git commit 시도 시 PreToolUse Hook이 Bash(git commit*)에 걸려 린트·타입체크를 강제(Enforcement)하고, 실패하면 exit 2로 차단해 Claude가 stderr를 보고 고칩니다. 통과 후 Stop 시점엔 Slack 알림 Hook이 환경변수 secret으로 webhook을 보냅니다.

## p67 · 자주 발생하는 실수 8가지
- **핵심**: 시연 전에 한 번씩 점검하면 개발 중 사고를 줄일 수 있는 Hook 관련 8대 실수입니다.
- **부가설명**: hook 단수 표기는 Settings Error로 무시되니 hooks 복수 배열로, 에러 메시지는 `>&2`로 stderr에 내보내야 차단이 되고, 차단은 반드시 exit 2(exit 1은 경고만 뜨고 작업은 진행)입니다.
- **부가설명**: webhook URL을 settings.json에 하드코딩하면 커밋 시 secret이 노출되니 환경변수로, 실행 권한 누락은 chmod +x, jq 미설치는 brew/apt install jq, matcher 패턴 오타는 /hooks로 확인하고, PostToolUse에서 차단 시도하면 이미 실행된 뒤라 PreToolUse로 옮겨야 합니다.

## p68 · 엔지니어링의 진화 - Spec-Driven Development
- **핵심**: 에필로그입니다 — 프롬프트에서 컨텍스트, SDD, 하네스로 이어지는 진화 중 SDD를 깊이 다룹니다.
- **부가설명**: 5단계 사이클과 Spec Kit, GSD를 통해 명세 기반 개발이 무엇인지 살펴봅니다.

## p69 · 프롬프트에서 SDD까지 — 우리는 어디에
- **핵심**: 엔지니어링은 4단계로 진화했고, 각 단계는 이전을 흡수합니다 — 지금 우리는 STAGE 3 SDD에 와 있습니다.
- **부가설명**: STAGE 1 Prompt Engineering은 few-shot·CoT·출력형식으로 한 번에 좋은 답을 끌어내는 단계, STAGE 2 Context Engineering은 CLAUDE.md·/context·/compact로 모델 시야를 통제하는 단계입니다.
- **부가설명**: STAGE 3 SDD는 명세를 진실의 원천으로 삼는 의도 중심 개발(Spec Kit·GSD), STAGE 4 Harness Engineering은 모델 외의 자동화·안전망으로 "모델 + 하네스 = AI 에이전트"를 완성합니다.

## p70 · Spec-Driven Development
- **핵심**: SDD는 GitHub Spec Kit이 제시한, Vibe Coding을 벗어나 명세가 진실의 원천이 되는 개발 방법론입니다.
- **부가설명**: "대충 해줘"식 엉뚱한 결과를 막고, 긴 대화에서도 목표·제약을 유지하며, 팀원 누구나 동일 품질을 내고, 의도가 곧 Source of Truth가 됩니다.
- **부가설명**: 핵심 요소는 셋입니다 — Constitution(불변의 법칙, 예: TS strict·UI 라이브러리 고정·테스트 90%), 무엇을 만들지 정하는 Spec, 어떻게 만들지 설계하는 Plan. 철학은 "명세가 코드를 생성한다"입니다.

## p71 · SDD 5단계 개발 사이클
- **핵심**: SDD는 5단계로 돌고, 각 단계가 명세 문서를 산출해 다음 단계의 입력이 됩니다.
- **부가설명**: 1 요구사항 분석(요구사항 문서) → 2 명세 작성(spec.md·API 명세) → 3 명세 리뷰·확정(확정된 명세 버전) → 4 명세 기반 구현(소스 코드) → 5 명세 대비 검증(테스트 결과 리포트) 순입니다.
- **부가설명**: Spec Kit으로는 2가 /speckit.specify, 3·4가 /speckit.plan·/speckit.tasks, 4가 /speckit.implement에 매핑됩니다. 명세가 바뀌면 1단계부터 다시 — 코드를 직접 고치지 않고 명세에서 재생성합니다.

## p72 · 왜 AI와 SDD가 잘 맞는가 + TDD 결합
- **핵심**: AI는 명확한 지시에 강하고, 명세 자체가 AI를 위한 최고의 프롬프트이기 때문입니다.
- **부가설명**: "TODO API 만들어줘"는 REST냐 GraphQL이냐, 필드·에러 형식·인증이 매번 갈리지만, "API_SPEC.md 읽고 POST /api/tickets 구현해줘"는 요청 필드·응답 형식·에러 코드가 다 정의돼 추측 없는 코드가 나옵니다 — 하나의 명세 문서가 개발자에겐 리뷰 기준, AI에겐 생성 맥락이 됩니다.
- **부가설명**: SDD가 "무엇"을 정의하면 TDD가 "동작 검증"을 맡아, 명세의 각 항목이 테스트 케이스로 직접 변환되며 명세 → 테스트 → 구현 흐름이 완성됩니다.

## p73 · Spec Kit 8개 명령어 + 산출물
- **핵심**: Spec Kit은 핵심 5개와 선택 3개, 총 8개 슬래시 명령어로 SDD를 돌리며, 각 명령어가 다음 단계의 입력이 되는 명세 파일을 산출합니다.
- **부가설명**: 핵심 흐름은 constitution(프로젝트 원칙, memory/constitution.md)부터 specify(요구사항 명세, spec.md), plan(기술 설계, plan·research·data-model·quickstart·contracts), tasks(태스크 분해, tasks.md), implement(실제 코드)까지 순서대로 이어집니다.
- **부가설명**: 선택 3개는 clarify(AI가 역질문해 모호함 제거), analyze(정합성·커버리지 분석), checklist(완전성·일관성 검증)이며, 설치는 uv tool로, 초기화는 `specify init my-project --ai claude`로 합니다.

## p74 · 언제 무엇을 쓸 것인가 + 실무 팁
- **핵심**: 팀에 기존 문서 표준이 있으면 PRD·TRD·API_SPEC을 직접 쓰는 수동 워크플로, 빠르게 시작하려면 Spec Kit을 쓰는 게 실무 기준입니다.
- **부가설명**: 수동은 구조·포맷을 완전히 통제하고 팀 기존 표준을 그대로 쓰며 도구 학습도 필요 없지만, 작성 시간이 많이 들고 일관성 유지가 어려우며 AI에 명세를 일일이 전달해야 합니다.
- **부가설명**: Spec Kit은 AI 초안으로 명세를 빠르게 만들고 표준화된 구조로 spec→plan→tasks→impl을 자동화하지만, 도구 설치·학습이 필요하고 템플릿이 강제돼 기존 문서 포맷과는 호환이 어렵습니다.

## p75 · Get Shit Done — 무겁지만 강력한 SDD
- **핵심**: GSD는 Spec Kit과 같은 SDD 계열이지만 더 상세하고 무거워, 복잡한 멀티-Phase 프로젝트에 유리합니다.
- **부가설명**: 해결하려는 핵심 문제는 Context Rot — 컨텍스트 윈도우가 차면서 품질이 떨어지는 현상인데, GSD는 매 단계를 새로운 200K 컨텍스트에서 실행해 이를 막습니다.
- **부가설명**: 명령어는 Spec Kit이 8개인 반면 GSD는 50개 이상이고, 토큰도 약 12K로 무겁습니다(--minimal로 700까지 절감 가능). 학습 곡선이 높은 대신 Wave 병렬 실행·atomic git commit·자동 검증을 제공합니다.

## p76 · GSD 워크플로 사이클 명령어
- **핵심**: GSD는 Phase마다 0번 초기화부터 5번 ship까지 6단계 루프를 돌고, 마일스톤이 끝나면 다음 버전으로 넘어갑니다.
- **부가설명**: 0 new-project(초기화)로 시작해 1 discuss-phase(토론·CONTEXT.md), 2 plan-phase(4 병렬 리서치+계획+검증 3회), 3 execute-phase(Wave 병렬·200K 격리·atomic commit), 4 verify-work(수동 UAT, 실패 시 디버그 에이전트 자동 호출), 5 ship(PR 생성, --draft 가능) 순입니다.
- **부가설명**: 루프가 끝나면 audit-milestone로 갭을 감지하고 complete-milestone로 아카이브·릴리스 태그, new-milestone로 다음 버전을 엽니다. 소규모 단건은 Quick·Fast 모드를 쓰면 됩니다.
