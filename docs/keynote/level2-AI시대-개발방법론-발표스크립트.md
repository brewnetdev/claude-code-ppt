# LEVEL 2 — AI 시대 개발 방법론 · 발표(동영상) 스크립트

> 영상 타이틀: **AI 시대 개발 방법론 ① — 협업 원칙과 프로젝트 설정**
> 대상: 클로드 코드 기본기를 마친 입문~초급 (Level 1 수료 기준)
> 분량: 슬라이드 40장 / 예상 러닝타임 35~45분
> 작성 원칙: **간략·핵심 위주**. 처음 나오는 용어는 `**풀 용어(약어)** = 뜻` 으로 그 자리에서 풀이.
>
> **읽는 법** — `>` 인용 블록이 실제 내레이션, `(화면)` 은 연출 지시, `⏱` 는 목표 시간입니다. 동영상 특성상 장황함을 피하고 핵심만 말합니다.

## ⓿ 오프닝

### S1. 커버 — LEVEL 2: AI 시대 개발 방법론 ⏱ ~30초
> 『클로드 코드 마스터』의 코드빌런입니다. 이번은 **Level 2, 'AI 시대 개발 방법론'** 입니다. Level 1에서 클로드 코드를 설치하고 첫 대화를 나눴다면, 이번엔 **AI와 어떻게 '협업'해야 결과물의 품질이 올라가는가** — 그 원칙과 프로젝트 설정을 배웁니다.

### S2. 인트로 — 왜·무엇을·어떻게 ⏱ ~60초
> **왜** 배우나. AI는 코드를 빠르게 쏟아내지만, 방향이 틀리면 그만큼 빠르게 망가집니다. 그래서 '얼마나 빨리 짜느냐'보다 **'무엇을, 어떻게 시키느냐'** 가 결과를 가릅니다.
> **무엇을** 배우나. 다섯 갈래입니다. ①설계의 원칙 ②작업 쪼개기 ③명확한 지시 ④매 단계 리뷰 ⑤개발 방법론 비교와 프로젝트 구조 설정(CLAUDE.md).
> **어떻게** 배우나. 나쁜 예와 좋은 예를 나란히 비교하며, 바로 실무에 적용할 수 있는 습관으로 정리합니다.
> **체크포인트 한 줄**: "AI에게 '무엇을' 정의해 주는 설계는 개발자의 몫이다."

### S3. 목차 ⏱ ~15초
> 차례입니다. 2.1 협업 원칙 → 2.2 쪼개기 → 2.3 명확한 지시 → 2.4 리뷰 → 2.5 프로젝트 구조 설정. 지금 위치만 잡아 두세요.

## PART 2.1 — AI와 함께하는 개발의 원칙

### S4. 섹션 표지 ⏱ ~10초
> 첫 번째, 협업의 대원칙입니다. 핵심은 하나 — **설계는 사람이, 구현은 AI가.**

### S5. 2.1.1 왜 설계가 더 중요해졌는가 ⏱ ~60초
> AI 시대의 역설입니다. **코드 생성이 쉬워질수록 설계가 더 중요해집니다.**
> AI가 잘하는 건 **How(어떻게)** — 코드 생성·문법·패턴. AI가 모르는 건 **What(무엇을)** — 요구사항·맥락·우선순위. 그 'What'을 정의하는 게 설계이고, 개발자의 몫입니다.
> 구현 전 선행할 3가지: ①**요구사항 분석**(핵심·비기능 요구) ②**아키텍처 결정**(빌드·배포까지 영향) ③**인터페이스 정의**(API 규격·데이터 흐름). `**API(Application Programming Interface)** = 프로그램끼리 주고받는 약속·규격`.

### S6. 설계 없는 개발 — 재작업 사이클 (TODO 앱 사례) ⏱ ~50초
> (화면) 왼쪽은 안티패턴, 오른쪽은 설계 우선.
> 설계 없이 그때그때 시키면 어떻게 되나. 1일차 "TODO 앱 만들어줘" → 3일차 "팀 공유 기능 추가" → 7일차 "SaaS로 전환" → 14일차 "온프레미스도" … 매번 구조를 갈아엎는 **재작업 사이클**에 빠집니다. `**SaaS(Software as a Service)** = 설치 없이 웹으로 쓰는 구독형 소프트웨어`.
> 반대로 구현 전에 요구사항·구조를 먼저 정의하면 **1회 통과**합니다. 요청과 결과가 한 번에 일치하죠.

### S7. 명세 없는 요청 vs 명세 있는 요청 ⏱ ~40초
> 같은 기능도 '명세'가 있느냐로 결과가 갈립니다. 명세가 없으면 AI가 빈칸을 **임의로** 채웁니다.
> 핵심 한 줄: **Garbage In, Garbage Out** — `약어로 GIGO. '쓰레기가 들어가면 쓰레기가 나온다'`, 즉 **입력의 품질이 곧 출력의 품질**이라는 뜻입니다. 모호하게 시키면 모호한 결과가 나옵니다.

### S8~S9. 2.1.2 AI에게 없는 것 — 판단력(Taste) ⏱ ~50초
> AI에게 없는 결정적 능력이 **판단력**, 영어로 **Taste(테이스트) = 무엇이 '좋은' 코드인지 가려내는 안목**입니다.
> AI는 '돌아가는 코드'는 잘 만듭니다. 하지만 가독성·유지보수성·확장성까지 고려한 **'좋은 코드'인지의 판단**은 사람이 해야 합니다.

### S10. Taste — '작동하는 코드' vs '좋은 코드' ⏱ ~40초
> (화면) 작동만 하는 코드 vs 좋은 코드 대비.
> 작동하는 코드는 '지금 통과'가 목표, 좋은 코드는 '6개월 뒤에도 고치기 쉬움'이 목표입니다. 둘의 간극을 메우는 게 개발자의 Taste예요.

### S11. 트레이드오프 결정의 주체는 개발자 ⏱ ~35초
> `**트레이드오프(trade-off)** = 하나를 얻으려면 다른 하나를 포기해야 하는 맞교환` 관계입니다. 속도냐 안정성이냐, 단순함이냐 유연함이냐 — 이런 **선택의 책임은 AI가 아니라 개발자**에게 있습니다. AI에게는 '근거'를 물어보되, 결정은 사람이 내립니다.

### S12. 2.1.3 AI 출력의 비결정적 특성 ⏱ ~45초
> `**비결정성(Non-deterministic)** = 같은 입력을 줘도 매번 똑같은 결과가 나오지 않는 성질`입니다. AI는 확률로 답을 고르기 때문에, 같은 프롬프트도 실행마다 결과가 달라질 수 있어요.
> 그래서 대책은 두 가지 — **작은 단위로 쪼개고, 매번 검증**합니다. 한 번에 크게 맡길수록 비결정성의 위험도 커집니다. (다음 챕터로 자연스럽게 연결)

## PART 2.2 — 작은 단위로 쪼개기의 힘

### S13. 섹션 표지 ⏱ ~10초
> 비결정성을 다스리는 가장 확실한 방법, **쪼개기**입니다.

### S14. 2.2.1 한 번에 하나의 작업만 요청하기 ⏱ ~45초
> 가장 흔한 실수가 **한 번에 너무 많이** 시키는 것입니다. 프롬프트에 **'그리고·또한·추가로'** 가 보이면 → 작업을 쪼개라는 신호입니다.
> 한꺼번에 시키면 ①기능·코드 품질 저하 ②복합 에러로 디버깅 난이도 폭증 ③기능 간 간섭으로 컨텍스트 혼란이 생깁니다. `**디버깅(debugging)** = 코드의 오류(버그)를 찾아 고치는 일`.

### S15. 나쁜 예 vs 좋은 예 — 단계별 분해 ⏱ ~40초
> (화면) "회원가입·로그인·비번변경·이메일인증 전부 만들어줘"(나쁜 예) vs 한 기능씩 순서대로(좋은 예).
> 좋은 예는 한 번에 **하나**만 — 회원가입을 끝내고 검증한 뒤 로그인으로 넘어갑니다. 작은 성공을 쌓는 방식이죠.

### S16. 2.2.2 작업 분해의 기준과 단위 ⏱ ~40초
> 그럼 어디까지 쪼개나. 기준은 **'한 번에 리뷰하고 테스트할 수 있는 크기'** 입니다. 하나의 명확한 입력→출력, 하나의 책임 단위로 자르면 검증이 쉬워집니다.

### S17. 다중 AI 에이전트 활용 워크플로 ⏱ ~45초
> `**에이전트(agent)** = 시키면 스스로 파일을 읽고 고치고 실행하는 AI 작업자`. 클로드 코드를 **여러 개** 띄워 역할을 나눌 수 있습니다.
> 대표 패턴이 **Writer/Reviewer(작성자/검토자)** — 한 에이전트가 코드를 쓰고, 다른 에이전트가 리뷰·테스트합니다. 맥락을 분리하면 한 명이 다 하는 것보다 품질이 올라갑니다.

## PART 2.3 — 명확한 지시의 기술

### S18. 섹션 표지 ⏱ ~10초
> 쪼갰으면, 이제 각 작업을 **명확하게** 지시하는 법입니다.

### S19. 2.3.1 컨텍스트 제한의 원칙 — 명확한 지시의 5요소 ⏱ ~55초
> `**컨텍스트(context)** = AI가 답할 때 참고하는 배경 정보(파일·대화·규칙)`. 컨텍스트는 **넓게가 아니라 정확하게** 줘야 합니다.
> 명확한 지시의 5요소: ①대상 파일·범위 ②해야 할 일 ③하지 말 것(제약) ④완료 조건 ⑤참고 자료. 이 다섯을 갖추면 AI가 헤매지 않습니다.

### S20~S21. 명확한 지시 예시 — 프로필 페이지 수정 ⏱ ~45초
> (화면) 모호한 지시 vs 5요소를 갖춘 지시 비교.
> "프로필 페이지 고쳐줘"는 모호합니다. 대신 "**어떤 파일**의 **무엇을**, **무엇은 건드리지 말고**, **어떤 상태가 되면 완료**인지"를 적어 주면, 결과의 정확도가 확 올라갑니다.

### S22. 2.3.2 검증 가능한 완료 조건 예시 ⏱ ~40초
> 완료 조건은 **측정 가능**해야 합니다. "잘 동작" 같은 표현은 사람마다 기준이 달라 검증이 안 됩니다.

### S23. 나쁜 완료 조건 vs 좋은 완료 조건 ⏱ ~40초
> (화면) 대비 표.
> 나쁜 예: "빠르게 동작해야 함". 좋은 예: "응답 200ms 이내, 테스트 전부 통과, 엣지케이스 처리". `**엣지케이스(edge case)** = 빈 값·최대치·예외 입력처럼 경계에서 터지기 쉬운 상황`. 숫자와 조건으로 못 박아야 AI도 사람도 합격/불합격을 가릴 수 있습니다.

### S24. 2.3.3 부정형 제약 — CLAUDE.md 또는 프롬프트에 명시 ⏱ ~40초
> `**부정형 제약** = "~하지 마라" 형태의 금지 규칙`입니다. "기존 테스트 지우지 마", "이 파일은 건드리지 마" 처럼요. 이런 금지 사항은 매번 말하기보다 **CLAUDE.md(프로젝트 규칙 파일)** 나 프롬프트에 적어 두면 반복 실수를 막습니다.

## PART 2.4 — 매 단계 리뷰하기

### S25. 섹션 표지 ⏱ ~10초
> AI가 만든 코드는 **반드시 리뷰**합니다. 표면은 멀쩡해 보여도 미묘한 버그가 숨어 있거든요.

### S26. 2.4.1 리뷰해야 할 것들 + 회귀 방지 ⏱ ~70초
> 매 단계 세 가지를 봅니다. ①**요구사항 충족**(요청한 게 다 있나, 요청 안 한 게 끼어들지 않았나, 엣지케이스) ②**코드 품질**(가독성·보안 — AI 코드는 취약점이 특히 많음) ③**의도치 않은 변경**(테스트 삭제·설정/의존성 변경).
> 그리고 반복 버그의 핵심, **회귀**입니다. `**회귀(Regression)** = 변경 후, 이전에 잘 되던 기능이 더는 안 되는 현상`. 새 기능·버그 수정 중에 엉뚱한 곳이 깨지는 거죠.
> **CodeRabbit(코드 자동 리뷰 도구) 2025** 연구에 따르면, AI 코드가 포함된 **PR(Pull Request, 변경 합치기 요청)** 은 사람이 쓴 코드보다 심각한 문제가 **1.4~1.7배** 많았습니다. AI 회귀 패턴은 ①기존 코드 무시·재작성 ②관련 없는 파일 수정 ③테스트 삭제·비활성화.
> 회귀 방지 4대 전략: ①**변경 전후 테스트 실행**(수정 전 전부 통과 확인 → 수정 후 재실행) ②**CI/CD 자동 테스트** — `CI/CD = Continuous Integration/Continuous Delivery, 코드를 합칠 때마다 자동으로 빌드·테스트·배포하는 파이프라인` ③**테스트 커버리지 모니터링**(새 코드엔 새 테스트) ④**회귀 테스트 스위트**(핵심 기능 안전망).

### S27. 리뷰에 유용한 슬래시 커맨드 ⏱ ~40초
> `**슬래시 커맨드(slash command)** = 클로드 코드에서 '/'로 시작하는 단축 명령`입니다. 자주 하는 리뷰 작업을 `/명령` 하나로 묶어 반복 호출할 수 있어, 매번 길게 프롬프트를 쓰지 않아도 됩니다.

### S28. 2.4.2 단계별 리뷰 습관과 워크플로 ⏱ ~55초
> 리뷰의 황금률: **50줄짜리 10번이 500줄짜리 1번보다 낫다.** 잘게 자주 봐야 맥락 파악도, 문제 발견도 쉽습니다.
> 습관 4가지: ①작은 작업=작은 리뷰 ②승인 전 직접 실행(문법은 맞아도 논리는 틀릴 수 있음) ③테스트 먼저 ④모르면 AI에게 직접 설명 요청.
> 워크플로: **요청 작성 → AI 코드 생성 → 코드 리뷰 → 테스트 실행 → 커밋.** 리뷰·테스트에서 막히면 수정·되돌리기 후 재시도합니다.

### S29. 되돌리기 전략 — 작업 단위로 자주 커밋 ⏱ ~40초
> `**커밋(commit)** = 작업 내용을 저장하는 세이브 포인트`. 작업 단위로 자주 커밋해 두면, 문제가 생겨도 직전 지점으로 즉시 복귀할 수 있습니다.
> 클로드 코드의 **`/rewind`** 명령은 코드뿐 아니라 **대화·컨텍스트까지 함께 롤백**해 줘서, 잘못된 방향으로 간 세션을 통째로 되돌릴 때 유용합니다.

### S30. 커밋 메시지 prefix 규칙 ⏱ ~45초
> `**prefix(접두어)** = 커밋 메시지 맨 앞에 붙여 '무슨 종류의 작업인지' 표시하는 꼬리표`. 국제 표준 **Conventional Commits** 방식입니다.
> 타입(풀 네이밍)과 실제 표기: **feature → `feat:`**, **fix → `fix:`**, **refactor → `refactor:`**, **database → `db:`**, **API → `api:`**, **UI → `ui:`**, **test → `test:`**, **documentation → `docs:`**. 예) `feat: 사용자 입력 검증 함수 추가`. 변경 의도를 한눈에 남기면 협업·되돌리기가 쉬워집니다.

## PART 2.5 — 개발 방법론 비교와 프로젝트 구조 설정

### S31. 섹션 표지 ⏱ ~10초
> 마지막으로, 지금까지의 원칙을 **파일로 고정**하는 단계입니다. 핵심 주인공은 **CLAUDE.md**.

### S32. 2.5.1 /init 으로 CLAUDE.md 자동 생성 ⏱ ~50초
> `**CLAUDE.md** = 프로젝트의 영구 컨텍스트 파일. 빌드 명령·컨벤션·구조·금지 규칙을 적어 두면 클로드가 매 세션 시작 때 자동으로 읽음`.
> `**/init** = 현재 프로젝트를 분석해 CLAUDE.md 초안을 자동 생성해 주는 명령`입니다. 빈 화면부터 쓰지 말고, `/init`으로 뼈대를 만든 뒤 다듬는 게 빠릅니다.

### S33. CLAUDE.md 섹션별 분석 — 무엇을 넣을까 ⏱ ~55초
> CLAUDE.md는 **길수록 좋은 게 아닙니다.** 토큰을 먹기 때문에 '효과 있는 것만' 넣습니다. `**토큰(token)** = AI가 글을 처리하는 최소 단위. 컨텍스트에 넣을수록 비용·한도를 소모`.
> **필수**: Language Policy(언어 정책), About This Project(프로젝트 개요), Tech Stack(기술 스택), Guardrails(절대 금지 규칙), Investigation Rules(추측 금지·근거 확인).
> **조건부/선택**: Commands, Code Style, Project Structure, Git Workflow.
> **제거 권장**: Working Relationship 같은 일반론, MANDATORY·이모지 강조 — 클로드 기본 동작과 중복돼 토큰만 낭비합니다.

### S34. CLAUDE.md 예시 — TODO App의 TaskFlow API ⏱ ~40초
> (화면) 실제 CLAUDE.md 샘플.
> 좋은 CLAUDE.md는 추상적 다짐이 아니라 **구체적 사실**로 채웁니다 — 빌드/테스트 명령, 도메인 모델, API 규격, "항상 X 하라" 규칙. 새 동료에게 주는 온보딩 문서를 떠올리면 됩니다.

### S35. CLAUDE.md 파일 위치별 공유 범위 ⏱ ~50초
> 위치에 따라 적용 범위가 다릅니다. ①**`~/.claude/CLAUDE.md`** = 내 모든 프로젝트에 적용되는 **개인 전역** 설정 ②**프로젝트 루트 `CLAUDE.md`** = 팀 공유용, **git에 커밋** ③**`CLAUDE.local.md`** = 개인 로컬 설정(개인 DB 주소 등), **`.gitignore`로 제외**.
> 규칙 한 줄: **팀과 공유할 것은 커밋, 개인 환경설정은 제외.**

### S36. .claude/rules/ — 경로 기반 자동 컨텍스트 주입 ⏱ ~50초
> 규칙이 많아지면 CLAUDE.md가 비대해집니다. 그래서 **`.claude/rules/`** 폴더에 규칙을 쪼개 두고, **특정 경로의 파일을 작업할 때만** 해당 규칙이 자동으로 끼어들게 할 수 있습니다.
> 예: `rules/api-routes.md`에 `src/api/**` 작업 시 적용 — API 폴더를 만질 때만 그 규칙이 주입돼, 평소엔 토큰을 아낍니다.

### S37. Glob 패턴 기본 문법 ⏱ ~45초
> 방금 `src/api/**` 같은 표기가 **Glob 패턴**입니다. `**Glob** = 파일 경로를 와일드카드로 매칭하는 패턴`.
> 기본기만: `*` = 한 단계의 모든 문자(`*.ts` → 그 폴더의 모든 .ts), `**` = 하위 폴더까지 전부, `?` = 한 글자. 이걸로 "어떤 파일을 작업할 때 어떤 규칙을 적용할지"를 지정합니다.
