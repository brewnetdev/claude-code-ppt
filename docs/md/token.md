# 토큰 효율화, 왜 "읽힐 양"이 핵심인가

> 대상: 비개발 IT 직무자 포함 전체 수강생 · 작성일 2026-05-31
> 모든 수치는 하단 검증 노트의 출처와 대조됨

---

## 📌 결론 먼저

1. **토큰 = AI가 한 번에 처리하는 글자 묶음.** 입력 토큰은 "AI에게 읽히는 양", 출력 토큰은 "AI가 쓰는 양"이며 **둘 다 돈·속도·정확도에 직접 영향**을 준다.
2. grep·LSP·codesight는 **같은 질문에 답하기 위해 AI에게 얼마나 읽히느냐**가 다르다.
   - **grep** = 단어가 나온 페이지를 본문째 복사해서 건넴 (양 많음, 노이즈 포함)
   - **LSP** = "그 단어는 42쪽 14번째 줄에 있음"이라는 **위치 좌표만** 건넴
   - **codesight** = 책 맨 앞에 붙인 **요약 지도 한 장**으로 매번 뒤지는 일 자체를 없앰
3. **사용자가 던진 검증 질문에 대한 답:** "입력 구조 강제 + 출력 구조 강제" 두 축만으로는 **충분하지 않다.** 이 두 축은 *한 번의 요청 형태*만 다룰 뿐, 실제 비용을 지배하는 **누적·캐시·모델 단가** 세 축이 빠져 있다. (→ Part B)

---

# Part A. grep / LSP / codesight — 비유로 정확히 이해하기

## 1. 먼저, 토큰이 뭔가 (비IT 비유)

AI는 글을 한 글자씩이 아니라 "토큰"이라는 조각 단위로 읽는다. 영어는 대략 4글자 ≈ 1토큰. **AI 서비스 요금은 "읽은 토큰 + 쓴 토큰" 개수로 매겨진다.** 택시 미터기처럼, 읽힌 분량이 많을수록 요금이 오르고 응답도 느려진다.

그리고 한 가지 더 — 읽힌 양이 많다고 똑똑해지지 않는다. 오히려 **관련 없는 내용이 핵심을 가려서** 정확도가 떨어진다(이를 Context Rot이라 부른다, Part B-3 참조). 그래서 "적게, 정확하게 읽히는 것"이 핵심이다.

## 2. 세 가지 방식 — 도서관 비유

질문: **"`scan`이라는 함수가 어디에 정의돼 있지?"** (= "이 책에서 '결재 규정'이 어디 있지?")

| 방식 | 도서관 비유 | AI에게 건네지는 것 | 문제 / 장점 |
|---|---|---|---|
| **grep** | "scan"이 나온 **모든 페이지를 통째로 복사**해서 책상에 쌓아줌 | 단어가 들어간 **줄 본문 전체** | 주석·문자열·우연히 같은 단어까지 다 딸려옴 → **양 많고 노이즈 많음** |
| **LSP** | 도서관 **색인 카드**: "scan = 42쪽 14번째 줄(정의)" 한 줄만 | **위치 좌표(`파일:줄:열`)만**, 본문은 안 줌 | 의미를 아는 검색 → **딱 1곳, 노이즈 0** |
| **codesight** | 책 맨 앞에 끼워둔 **한 장짜리 전체 지도**(목차+구조) | 사전에 1회 생성한 **압축 구조 맵** | 매번 책을 헤매는 반복 탐색 자체를 제거 |

> 한 줄 요약: **grep은 "본문을 읽힌다", LSP는 "위치만 읽힌다", codesight는 "지도만 읽힌다."**
> 토큰 절감의 본질은 더 똑똑한 모델을 쓰는 게 아니라, **읽힐 입력의 형태와 양을 바꾸는 것**이다.

## 3. 실측 예시 (강의에서 그대로 시연 가능)

> 측정 환경: 오픈소스 저장소 `Houseofmvps/codesight`(TypeScript, 258 파일). 토큰 추정 1토큰 ≈ 4글자(보수적).

**예시 ① "scan 함수 정의 위치 찾기"** — 흔한 단어라 grep이 과하게 걸림

| 방식 | 결과 | 토큰 |
|---|---|---|
| grep | 47개 매치 (주석 23 + 노이즈 18 + 실제 6) | **1,122** |
| LSP | 정의 1곳 (`src/core.ts:42:14`) | **26** |

→ **약 43배 절감.** grep이 가져온 것의 절반은 주석·문서 노이즈. AI는 그 노이즈까지 다 "읽는" 비용을 낸다.

**예시 ② "RouteInfo 타입을 쓰는 모든 곳 찾기"** — 리팩토링 상황

| 방식 | 결과 | 토큰 |
|---|---|---|
| grep | 139개 매치 (**코드 줄 본문 전체** 반환) | **2,244** |
| LSP | 136개 참조 (**위치 좌표만**, 오탐 0) | **1,062** |

→ **약 2.1배 절감 + 오탐 0.** grep은 줄 전체를 토큰으로 소비, LSP는 `경로:줄:열`만 반환.

**예시 ③ "세션 시작 시 프로젝트 구조 파악"** — codesight의 영역

| 방식 | 동작 | 토큰 |
|---|---|---|
| grep/glob 반복 탐색 | AI가 파일 트리를 헤매며 읽음 | **32,695** |
| codesight | `npx codesight` → 구조 맵 1장 주입 | **4,780** |

→ **약 6.8배 절감 (세션당 ≈ 27,915토큰 절약).**

> **비IT 직무자용 한 줄 정리:** 같은 질문에 답하려고 AI에게 "300쪽 복사본"을 안기느냐(grep), "포스트잇 한 장"을 안기느냐(LSP), "책 앞 지도 한 장"을 미리 끼워두느냐(codesight)의 차이다. 결과는 같아도 **요금과 정확도가 수십 배 갈린다.**

## 4. 언제 무엇을 쓰나 (실무 판단)

| 상황 | 권장 |
|---|---|
| 파일 20개 미만 소규모 | grep 기본으로 충분 |
| 중·대형 일상 개발 | codesight로 세션 시작 시 구조 주입 |
| 리팩토링·정의/참조 추적 | LSP 추가 (정확한 위치만) |
| 다중/미지원 언어 | cclsp 등 MCP 브릿지 |

세 도구는 경쟁이 아니라 **상호 보완**이다. codesight로 "지도"를 깔고, LSP로 "정밀 좌표"를 찍는 조합이 가장 효율적이다.

# Part B. "입력 구조 강제 + 출력 구조 강제" — 이 두 개로 충분한가?

## 검증 결론: **불충분하다.**

두 축은 **"한 번의 요청을 어떤 형태로 만드느냐(shape)"** 만 다룬다. 하지만 실무에서 토큰 비용은 *형태* 하나로 정해지지 않는다. 비용은 다음 곱으로 결정된다:

```
총 토큰 비용  ≈  (요청 1건의 형태)  ×  (요청 횟수·누적)  ×  (재사용 여부=캐시)  ×  (모델 단가)
                 └ 입력·출력 구조 ─┘    └─ C. 누적 ─┘      └─ D. 캐시 ─┘    └ E. 모델 ┘
                   (사용자가 말한 2축)         (빠진 3축)
```

즉 **입력·출력 구조는 첫 항 하나**일 뿐이고, 나머지 세 항이 비용을 더 크게 좌우하는 경우가 흔하다. 아래가 빠진 세 축이다.

### C. 누적(세션) 축 — "한 번 잘 줄여도 10턴 쌓이면 무의미"

멀티턴 대화에서 **대화 히스토리는 매 턴 input 토큰으로 다시 전송된다.** 입력을 한 번 잘 압축해도, 턴이 쌓이면 같은 내용이 반복 청구된다.

반례(서브에이전트 실측, 20파일 리뷰):

| 방식 | 1턴 | 10턴 누적 |
|---|---|---|
| 인라인(메인에 다 쌓음) | 85k | **850k** |
| 서브에이전트(요약만 반환) | 26k | **260k** |

→ 단발 비용은 서브에이전트가 오히려 +12% 비싸지만, **10턴 누적에선 590k 차이.** 입력 "형태"가 아니라 **누적량**이 비용을 지배한다.

- 대응 도구: `/compact`(히스토리 압축), `/clear`(초기화), HANDOFF(세션 인계), **서브에이전트 컨텍스트 격리**, 40% 룰(컨텍스트 40% 넘으면 압축 시작).

### D. 캐시 축 — "입력을 1토큰도 안 줄여도 90% 절감 가능"

같은 입력이라도 **캐시 히트면 비용이 1/10**이 된다. 입력을 *줄이는* 게 아니라 *재사용*하는, 입·출력 구조와 독립된 축이다.

- 캐시 히트: 기본 input 단가의 **0.1×** (= 90% 절감)
- 5분 캐시 쓰기: 1.25× / 1시간 캐시 쓰기: 2× (1회성 선불)
- **결정적 함정:** Anthropic 캐시는 **prefix(앞부분) 기반**. 접두어 중간이 한 글자라도 바뀌면 **그 뒤 전체가 무효화**된다.
- 대응 패턴: 컨텍스트를 **Stable Prefix**(시스템 프롬프트·도구 정의·장기 요약 — 잘 안 변함)와 **Variable Suffix**(최신 입력·새 도구 출력 — 자주 변함)로 분리. 자주 변하는 걸 뒤로 몰아야 캐시 적중률이 오른다(실측 사례: 7% → 74%).

→ 입·출력 구조를 아무리 다듬어도 **캐시 적중률을 관리하지 않으면** 멀티스텝 에이전트에서 가장 큰 절감 기회를 통째로 놓친다.

### E. 모델·단가 축 — "같은 입출력이어도 단가가 5배 차이"

토큰 *개수*가 같아도 **어떤 모델로 처리하느냐**에 따라 청구액이 달라진다.

| 모델 | 입력 / 출력 (1M 토큰당) |
|---|---|
| Opus 4.7 | $5 / $25 |
| Sonnet 4.6 | $3 / $15 |
| Haiku 4.5 | $1 / $5 |

→ 입력·출력 단가가 모델 간 **최대 5배** 차이. 거기에 **Batch API는 모든 토큰 50% 할인**.

- 대응 패턴: **작업별 모델 라우팅** (Plan=Opus, Code=Sonnet, Lint·요약=Haiku), 지연 허용 작업은 Batch API로 이전.

## 정리 — 토큰 효율화의 전체 지도

| 축 | 핵심 질문 | 주요 수단 | 사용자가 말한 2축에 포함? |
|---|---|---|---|
| **입력 구조** | 무엇을·얼마나 읽히나 | codesight, LSP, JIT 컨텍스트, CLAUDE.md 경량화, `.claudeignore` | ✅ |
| **출력 구조** | 무엇을·얼마나 쓰게 하나 | 출력 형식 제약, `/effort` 추론 강도 조절 | ✅ |
| **C. 누적(세션)** | 같은 내용을 몇 번 다시 보내나 | `/compact`·`/clear`·HANDOFF, 서브에이전트 격리 | ❌ 빠짐 |
| **D. 캐시** | 보낸 걸 재사용하나 | prompt caching, Stable Prefix/Variable Suffix 분리 | ❌ 빠짐 |
| **E. 모델 단가** | 같은 양을 얼마에 처리하나 | 모델 라우팅, Batch API 50% | ❌ 빠짐 |

> **강의 메시지 제안:** "입력·출력 구조 강제는 토큰 효율화의 *출발점*이지 전부가 아니다. 단일 요청의 형태를 다듬는 일(입·출력)에 더해, **반복(누적)을 줄이고 · 재사용(캐시)을 높이고 · 단가(모델)를 낮추는** 세 축을 함께 설계해야 비로소 '효율화'다." 특히 멀티턴 에이전트 환경에서는 **누적·캐시 축의 절감폭이 입·출력 구조보다 큰 경우가 많다.**
