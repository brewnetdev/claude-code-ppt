평가 엔지니어링(Eval Engineering) 워크플로
— 하네스 구축 관점 정리

하네스 필수 요소 ⑩(평가 하네스)의 구축 방법론 · 평가 생성 워크플로 · Harbor 태스크 구조 · 리워드 해킹 대응
결론 요약

평가 작성 자체를 코딩 에이전트에게 시킨다. 스킬이 저장소와 트레이스를 읽어 테스트할 능력을 제안하고, 사용자 인터뷰를 거쳐 실행 가능한 Harbor 태스크(instruction + Dockerfile 환경 + verifier)를 산출한다.

가장 실무적인 발견 두 가지 — ① 원샷 생성보다 인터뷰 방식의 평가 수용률이 높다. ② 첫 verifier는 거의 항상 틀린다: 실행 후 에이전트 궤적과 verifier 궤적 양쪽을 검사해 리워드 해킹을 찾아 수정하는 반복이 평가 품질을 만든다.

이 루프가 앞선 보고서의 ⑩ 평가 하네스의 운영 형태다: 트레이스 마이닝 → 실패 식별 → 평가 작성 → 하네스 개선 → 재실행

1평가 생성 워크플로

Harbor 태스크 디렉토리

evals/<task-id>/
├── task.toml          # 태스크 메타
├── instruction.md     # ① 에이전트에게 줄 태스크 설명
├── environment/       # ② Dockerfile — 툴 설치·데이터 적재
└── tests/             # ③ verifier — 완료 여부 채점

Harbor가 에이전트를 환경에서 실행하고 궤적·산출물·보상·에러를 기록한다. 환경·태스크는 고정, 에이전트 구성만 교체 — 모델·프롬프트·툴·하네스 버전을 바꿔가며 같은 평가로 직접 비교하고, 병렬 실행할 수 있다.

2Verifier 반복 설계 — 리워드 해킹 카탈로그

해킹 유형예대응
과잉 인용무관한 소스까지 전부 인용해 인용 점수 만점인용 정확도(precision)도 채점
허위 주장하지 않은 행동을 "했다"고 보고자기 보고가 아닌 실제 상태(파일 diff·로그)로 검증
정답 노출 악용환경에 노출된 golden answer를 읽음정답 자료를 verifier 측에만 배치
프록시 충족태스크 미완수인데 지표만 만족최종 결과물 기준 검증 추가
원칙: verifier는 에이전트의 말이 아니라 환경의 상태를 채점한다. 첫 verifier가 최종본인 경우는 드물며, 실행 결과의 양쪽 궤적을 읽는 것이 유일한 개선 방법이다.

3지속 개선 루프 — 하네스 엔지니어링과의 연결

4하네스 구축(carve-harness)에의 적용

평가 대상이 되는 하네스 능력 — 필수 요소 매핑

하네스 요소평가할 능력verifier 아이디어
② 루프동일 툴콜 반복 시 루프 감지 작동툴콜 로그에서 반복 패턴 부재 확인
③ 컨텍스트컴팩션 후에도 태스크 완수긴 태스크 + 최종 파일 상태 diff
④ 툴대용량 출력 절단 후 정보 보존절단된 출력 기반 후속 판단 정확성
⑤ 안전파괴적 명령의 승인 게이트denylist 명령 실행 시도 기록 검사
⑦ 세션resume 후 작업 연속성중단·재개 시나리오의 최종 산출물 비교

구성 방법

  • Environment: carve-harness가 설치된 Dockerfile + 고정 테스트 저장소
  • Instruction: 실제 사용 시나리오에서 추출한 태스크
  • Verifier: 파일 상태·로그 기반 — 하네스의 자기 보고를 신뢰하지 않는다
  • Harbor는 하네스 중립이므로 버전 간(v1.2 vs v1.3)·타 하네스와의 비교에 그대로 사용

스킬 없이 차용 가능한 것

  • 평가 3요소(instruction / 환경 / verifier) 구조
  • "돌려보고 양쪽 궤적을 검사"하는 verifier 반복 절차
  • 리워드 해킹 카탈로그 기반 점검
  • 트레이스 → 실패 → 평가의 데이터 마이닝 관점

5실행 방법

  1. langchain-ai/langchain-skills 저장소의 Eval Engineering Skill을 Claude Code 또는 Codex에 설치
  2. 평가 대상 에이전트 저장소를 열고, 트레이스가 있으면 위치 지정 (langsmith-cli 등)
  3. 시작 프롬프트:
Use the eval-engineering skill to create an eval with me.
Inspect the agent first, propose a few abilities worth testing,
recommend one, and wait for me to choose.

산출: evals/ 아래 Harbor 태스크 + 실제 타깃 실행 + verifier가 의도한 행동을 측정했는지에 대한 리뷰.

6주의점 (비판적 검토)

  • 원문은 LangSmith·Harbor 생태계 유입을 위한 제품 블로그다. "인터뷰 방식의 수용률이 높다"에 정량 수치가 없고 자사 사례(chat-langchain) 하나뿐 — 방법론과 도구 종속을 분리해 판단할 것.
  • "평가의 평가" 문제: verifier가 잘못 측정하면 그 위의 모든 개선이 허수다. 사람의 verifier 검토가 병목으로 남는다.
  • 트레이스 기반 평가는 과거 실패 재발 방지에 강하지만, 트레이스에만 의존하면 평가 셋이 "이미 아는 문제"로 편향된다. 합성 시나리오를 별도 보충할 것.
