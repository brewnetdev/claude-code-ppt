범용 에이전트 하네스(Agent Harness)
필수 구성요소와 각 요소가 해결하는 문제

"설치해서 쓰는" 범용 하네스 관점 · 2026년 논문·프레임워크 릴리스·실측 데이터 기반
결론 요약

Agent = Model + Harness. 하네스는 모델이 아닌 모든 것 — 프롬프트, 툴, 실행 환경, 오케스트레이션, 훅, 모델 설정 — 이며, 확률적 추론을 결정론적 행동으로 변환하는 유일한 계층이다.

필수 요소는 10가지로 정리된다. ①~⑥(인터페이스·루프·컨텍스트·툴·안전·환경)이 없으면 하네스가 아니라 챗 래퍼이고, ⑦~⑨(세션·확장·관측성)가 프로덕션 품질을, ⑩(평가 하네스)이 지속 개선 가능성을 결정한다.

실증: 모델 교체 없이 하네스만 개선(검증 루프·컨텍스트 주입·루프 감지)해 Terminal Bench 2.0 순위를 30위에서 5위권으로 끌어올린 사례가 보고됐다.

CLAUDE.md(프로젝트 관례)·AGENTS.md(역할)·MEMORY.md(세션 간 지식)·RULES.md(제약) — 검색이 아닌 "일관된 사전 조건"으로 로드

구성

⑥환경 어댑터OS·셸·인코딩 차이의 흡수
구성
셸 선택(bash/PowerShell), 작업 디렉토리, 타임아웃, 파일 도구의 인코딩·경로 처리
핵심 개념
파일시스템 = 조정 표면: 작업공간·중간 산출물 저장소·다중 에이전트/인간의 협조 표면. Git을 얹으면 버전·롤백·분기가 따라옴
미비 시
플랫폼별 동작 불일치 — "내 머신에선 되는데"의 하네스판

미비 시

설치형 하네스를 직접 만드는 경우, SDK 위에 ①⑤⑥⑦(인터페이스·권한·환경·세션)을 얹는 구조가 표준 경로다 — SDK가 제공하는 것은 주로 ②③④⑧⑨의 프리미티브이기 때문이다.

5설계 체크리스트

  • 루프
  • 최대 턴·토큰 예산이 코드로 강제되는가
  • 동일 툴콜 반복을 감지·차단하는가
  • 정지 조건이 모델의 자율 판단에만 의존하지 않는가
  • 컨텍스트
  • 툴 출력 절단 한도가 있는가
  • 컴팩션 트리거와 보존 우선순위가 정의됐는가
  • 워크스페이스 지침 파일(CLAUDE.md류) 로딩 규약이 있는가
  • 안전
  • 파괴적 행동 분류표와 승인 게이트가 있는가
  • 읽어들인 콘텐츠(파일·웹)를 지시로 취급하지 않는 방어가 있는가
  • 샌드박스 없는 플랫폼에서의 축소 모드가 정의됐는가
  • 운영
  • 세션 resume과 파일 롤백이 되는가
  • 툴콜 단위 트레이스와 비용 회계가 남는가
  • 하네스 변경을 검증할 평가 태스크 셋이 있는가
