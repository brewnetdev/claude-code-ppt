2.1.2 TASTE
2.1.2 AI에게 없는 것 — 판단력(Taste)
문법적으로 '올바른' 코드와 '좋은' 코드를 구분하는 능력
Taste는 기술적 판단력과 미적 감각을 아우르는 개념이다. 여러 선택지 중에서 현재 상황에 가장 적합한 것을 고르는 안목이 Taste다.
AI는 패턴을 인식하지만 왜 그 패턴이 중요한지는 이해하지 못한다. 성능·비용·보안·확장성 사이에서 균형을 잡는 것은 맥락을 아는 개발자만이 할 수 있다.
💡
KISS (Keep It Simpe. Stupid, 단순하게 유지하라) — AI는 엣지케이스를 과도하게 방어하거나 추상화 레이어를 불필요하게 추가하는 경향이 있다. 당장 3개의 케이스만 있는데 팩토리 패턴 + 전략 패턴 + 싱글톤을 조합한 구조를 만들어주는 식이다.
💡
DRY (Don't Repeat Yourself, 반복하지 마라) — 역설적이게도, AI가 DRY를 과도하게 적용하는 경향이 있으므로 반복 여부를 점검해서 3번 이상이면 공통 함수로 작업할 수 있도록 지침을 마련한다. 
💡
YAGNI (You Aren't Gonna Need It, 지금 당장 필요한 것은 만들지 않는다) — AI가 스스로 적용하기 어려운 원칙 AI는 '더 완전한' 코드를 만들려는 경향이 있다. 지금 당장 필요하지 않은 것은 만들지 않는다는 판단은 개발자의 몫이다.
소프트웨어 개발의 원칙은 AI 작업시에도 유효 (YAGNI, KISS, DRY, SRP, SOLID...)

개발 원칙이나 리팩토링 원칙, 컴포넌트 분리 원칙, 코드 컨벤션 가이드 등을 CLAUDE.md에 기본으로 템플릿화 해서 적용하는게 효율적이다. 
## 개발 원칙
- YAGNI: 지금 필요하지 않은 기능은 만들지 마
- KISS: 단순한 해결책을 먼저 시도하고, 복잡한 패턴은 필요할 때만
- DRY: 코드가 3번 반복될 때부터 공통화 고려
- SRP: 함수 설명에 "그리고"가 들어가면 분리 신호
- Fail Fast: 잘못된 입력은 즉시 에러로 처리
- 조기 최적화 금지: 측정 전에는 최적화하지 않음
- 기존 파일 최소 수정: 요청한 것 이외의 리팩터링은 별도 안내

2.1.2 EXAMPLE
Taste — '작동하는 코드' vs '좋은 코드'
AI 생성 — 작동하지만 확장성 낮음
function filterTodos(todos: Todo[], filter: string) {
  if (filter === 'all') {
    return todos;
  } else if (filter === 'active') {
    return todos.filter(todo => !todo.completed);
  } else if (filter === 'completed') {
    return todos.filter(todo => todo.completed);
  } else {
    return todos;
  }
}
개선 — 타입 안전 · 새 필터 추가 용이
type FilterType = 'all' | 'active' | 'completed';

const filterStrategies: Record<FilterType, (todo: Todo) => boolean> = {
  all: () => true,
  active: (todo) => !todo.completed,
  completed: (todo) => todo.completed,
};

function filterTodos(todos: Todo[], filter: FilterType): Todo[] {
  return todos.filter(filterStrategies[filter]);
}
Taste — '작동하는 코드' vs '좋은 코드' 05
- FilterType의 유효값이 3가지이기 때문에 컴파일 타임에 에러를 잡을 수 있다.
- 확장이 함수 밖에서 일어나므로 로직을 건들이지 않아도 된다. 
- 조건에 따른 쓸데없는 else 분기문들이 추가되지 않는다
2.1.2 TRADE-OFF
트레이드오프 결정의 주체는 개발자
AI는 정보를 제공하지만 최종 결정은 맥락을 아는 개발자의 몫 - AI가 생성하는 작동하는 코드를 개발자가 "맥락에 맞는" 코드로 취사 선택 해야 한다. 
  • 오프라인 우선 경험이 중요한가?→ 로컬 스토리지 + 동기화
  • 여러 기기에서 접근해야 하는가?→ 서버 API 필수
  • 빠른 출시가 우선인가?→ 가장 단순한 방법
  • 향후 협업 기능을 추가할 예정인가?→ 처음부터 서버 연동
💡
 타입 설계, 에러 처리, 추상화, 성능 및 보안 점검, DB 설계 등 모든 영역에서 개발자가 무엇이 우리 프로젝트에 더 적합한가? 더 필요한가? 를 판단해서 프로젝트를 진행해 나가야 한다. AI가 진행하는 방향이 틀렸다고 느낄때 바로 되돌릴 수 있는 판단력(=비용)과 방향 제시가 가장 중요하다.

2.1.3 NON-DETERMINISTIC
2.1.3 AI 출력의 비결정적 (Non-deterministic) 특성
Deterministic 프로그래밍과 달리 같은 요청에도 다른 결과
  • 미묘하게 다른 구현정규식 / 조건문 / 외부 라이브러리 — 매번 다른 접근, 같은 입력을 해도 매번 다른 출력이 나온다. 
  • 예상치 못한 변경관련 없는 부분까지 변경, 빌드 성공 주장이 거짓일 수도 있다.
  • 치팅 징후 (가장 위험)테스트를 통과시키기 위해 테스트 자체를 비활성화·삭제 - 테스트를 수정하는게 아니라 Skip/Delete 한 뒤 통과 한 것처럼 행동한다. 
🛡️
테스트 = 결정적 검문소 AI가 어떤 방식으로 구현하든 테스트를 통과하면 요구사항 충족. 결과물의 '동일함'이 아니라 '동작함'이 성공의 기준이다.

TDD 창시자가 제시한 AI 시대의 코딩 방법론 (2025)
테스트 주도 개발(TDD)의 창시자 켄트 벡이 2025년 AI 코딩 도구의 급속한 발전을 지켜보며 제시한 개념이다. BPlusTree3 프로젝트(B+ Tree 라이브러리를 Rust·Python으로 구현)를 통해 이 방법론을 실제로 탐구하고 문서화했다.
💬
Kent Beck - 증강코딩에서는 코드의 품질, 복잡도, 테스트, 커버리지까지 신경 쓴다. 손으로 직접 코드를 작성할 때와 동일한 가치 체계를 유지하면서, 타이핑을 AI가 대신해줄 뿐이다. (https://tidyfirst.substack.com/p/augmented-coding-beyond-the-vibes)

## TDD 필수 규칙

### Red-Green-Refactor 사이클
1. Red: 먼저 실패하는 테스트를 작성한다
   - 테스트가 실제로 실패하는지 확인
2. Green: 테스트를 통과시키는 최소한의 코드 작성
   - 하드코딩도 허용 (다음 테스트가 일반화를 강제)
3. Refactor: 테스트가 통과하는 상태에서 코드 개선
   - 중복 제거, 명확한 이름 사용

### 금지 사항
- ❌ 테스트 없이 구현 코드 작성
- ❌ 테스트와 구현을 동시에 작성
- ❌ 테스트 skip 또는 주석 처리
- ❌ 실패 원인 분석 없이 테스트 수정
