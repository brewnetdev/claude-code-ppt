로봇 데이터 수집·제어 콘솔 — AS-IS / TO-BE 기술스택 구성도

00문제 정의 — 무엇이 무거운가

현행 시스템은 로봇의 비전(카메라)·액팅(액추에이터·관절각·IMU) 데이터를 초당 수십~수천 프레임으로 수집한다. 데이터가 TB급으로 누적되고, 콘솔에서 이를 확인하며 로봇을 제어한다. 현재 이 파이프라인은 C++로 직접 만든 기록기·포맷, 별도로 자작한 프론트 콘솔, 그리고 손으로 배선한 다수의 비동기 스트림으로 구성돼 무겁다.

무거움의 본질은 언어(C++)가 아니라 "로봇의 본질 가치가 아닌 인프라 코드를 자체 유지"하는 데 있다. 기록 포맷의 리더/라이터, 스트림 동기화, 뷰어 UI는 이 도메인 전용으로 이미 성숙한 오픈소스가 있다. TO-BE는 제어 루프(C++)는 그대로 두고, 그 위 인프라 레이어만 기성 표준으로 교체한다.

핵심 관점

로봇 데이터는 multi-rate·multimodal이 본질이다(카메라 30Hz · IMU 1kHz · GPS 1Hz). 범용 Kafka/Parquet/데이터레이크는 "카메라 프레임만, 이 시간 구간만" 같은 선택적 슬라이스에 최적화돼 있지 않아, 이 도메인은 전용 포맷(MCAP·RRD)이 별도로 진화했다.

01AS-IS 구성도 — 현행 스택

붉은 테두리 노드가 자체 개발·유지 부담(무거운 지점)이다. 전송 배선·수집기·기록 포맷·프론트 콘솔이 전부 자작이라 유지 코드가 계속 증식한다.

02TO-BE 구성도 — 경량 대안 스택

청록 노드가 채택한 경량 표준, 황토 노드가 데이터 레이어다. 핵심은 하나의 MCAP 파일이 기록·확인(뷰어)·학습을 공유해, 자작 프론트와 변환 파이프라인이 통째로 사라지는 것이다.

C++를 버리는 게 아니다

실시간 제어 루프·드라이버는 C++ 유지가 맞다. MCAP·Foxglove SDK·Rerun 전부 C++ SDK를 제공하므로, 버리는 것은 C++로 자작한 인프라(포맷·뷰어·전송 배선)뿐이다.

04용어·약어 풀네임과 역할 (상세)

공통 · 데이터 특성

용어풀네임 / 발음역할 상세
C++C plus plus시스템 프로그래밍 언어. 저지연 실시간 제어 루프·드라이버 구현에 씀. TO-BE에서도 제어 루프는 유지 — 버리는 건 C++로 자작한 인프라 코드다.

multi-ratemulti-rate센서마다 샘플링 주파수가 다름(카메라 30Hz · IMU 1kHz · GPS 1Hz). 로봇 데이터 파이프라인 설계의 핵심 제약.
multimodalmulti-modal이미지·포인트클라우드·시계열·관절각 등 이종(異種) 데이터의 혼합.
teleopteleoperation원격 조작. 사람이 로봇을 직접 조종해 imitation 학습용 시연 데이터를 모으는 주 수단.
VLAVision-Language-Action model관측(비전+언어)에서 행동까지 신경망이 사상하는 end-to-end 정책 모델(예: π0 · OpenVLA · RT-2).

TO-BE 스택 용어

용어풀네임 / 발음역할 상세
MCAP"em-cap" (Message Capture)로보틱스 pub/sub 전용 기록 컨테이너 포맷. 직렬화 불가지론(Protobuf·ROS·JSON·FlatBuffers) — ROS 없이도 씀. append-only + per-chunk zstd/lz4 압축 + 시간·토픽·스키마 인덱스. rosbag 대비 부분조회 ~10배. C++/Rust/Python/Go/TS 리더·라이터 기성 제공.
FoxgloveFoxglove Studio / SDK로보틱스 관측·시각화 플랫폼. MCAP를 바로 열고 라이브 스트림도 받음. Foxglove SDK로 C++/Python/Rust에서 이미지·lidar·transform·telemetry를 구조화 스트림으로 발행 → 패널·3D 씬 즉시 탐색하거나 MCAP로 저장. 자작 프론트를 대체.
RerunRerun (rerun.io)"Physical AI 데이터 레이어". 로그→저장→쿼리→시각화→학습을 한 레이어로. RRD 전용 포맷 + 실시간 뷰어. HuggingFace LeRobot가 디버깅 뷰어로 채택. 0.32(2026)에서 쿼리·변환·학습까지 오픈소스 확장. C++/Python/Rust SDK.
RRDRerun Data (recording)Rerun 전용 파일 포맷. block-sparse 컬럼 청크 + entity별 파티션 → "lidar는 안 읽고 카메라+관절각만, 이 시간구간만" 같은 선택적 슬라이스에 최적화.
ZenohEclipse Zenoh (Zero Network Overhead)데이터 중심 프로토콜. pub/sub + query + storage를 한 스택으로 통합. WAN·셀룰러·간헐 연결·NAT 통과·제약 디바이스에 강함. 낮은 discovery 오버헤드.
rmw_zenohRMW for ROS 2 using ZenohZenoh를 하부 전송으로 쓰는 ROS2 RMW. Jazzy(2024)부터 공식 대안. 주의: 2026 기준 기본 아님. 노드 발견을 위해 Zenoh router(gossip discovery)를 전제 — DDS의 완전 P2P와 토폴로지가 다름.
Zenoh-PicoZenoh PicoZenoh의 초경량 구현. MCU·제약 디바이스용(PX4가 채택). 펌웨어단에서 telemetry 발행·제어 수신.
LeRobotHugging Face LeRobot오픈소스 로봇 학습 라이브러리 + 표준 데이터셋 포맷. 파편화된 로봇 데이터를 표준화해 학습에 바로 물림.
LeRobotDataset v3LeRobotDataset version 3.0저장/API 분리 포맷. 고주파 스칼라(관절·액션·타임스탬프)=Parquet, 영상=MP4 샤드, 메타=JSON. 에피소드=parquet 행 범위. Hub 스트리밍·PyTorch DataLoader 직결. (예: Stanford BEHAVIOR 3.27TB / 2만 teleop 데모)
LanceLance columnar format현대 ML용 컬럼형 스토리지 포맷. 고성능 랜덤 액세스·lazy 멀티모달 blob 읽기.
LanceDBLanceDBLance 기반 DB. lerobot-lancedb 패키지로 LeRobot 데이터셋을 frames·episodes·videos 테이블로 노출 → SQL 큐레이션·검색·학습 준비. hf:// URI로 Hub 데이터 직접 오픈.
ParquetApache Parquet컬럼형 파일 포맷. 고주파 스칼라의 배치 데이터로딩에 빠름.
MP4 shardMP4 video shard영상 세그먼트 파일. 프레임당 이미지 대비 압축률이 자릿수 단위로 좋고 랜덤 액세스 유지.
S3Simple Storage Service (AWS)오브젝트 스토리지. TB급 데이터의 사실상 표준 저장소.
MinIOMinIOS3 호환 오브젝트 스토리지. 자체 호스팅(온프레미스) 시 S3 대체.

RedpandaRedpandaKafka 프로토콜 호환 스트리밍 플랫폼. C++ 구현·JVM 없음 → Kafka 드롭인 경량 대안.

05전환 시 주의점

  • 포맷 락인 위험 낮음 — MCAP·RRD·LeRobot은 서로 읽고 상호 변환된다. 기록 포맷 교체는 되돌리기 쉬운 결정.
  • Zenoh는 비가역 결정에 가까움 — 2026 기준 default가 아니고 router 토폴로지를 전제한다. 도입 전 레이턴시·discovery PoC 벤치 필수. 안전필수·LAN 실시간이면 DDS 유지가 나을 수 있다.
  • 실시간 루프 격리 — MCAP 기록 시 write를 큐+별도 스레드로 분리해 제어 루프가 파일 I/O 롱테일에 방해받지 않게 한다.
  • 한 포맷 수렴이 절감의 핵심 — 기록·확인·학습이 같은 파일을 공유해야 자작 프론트/변환 파이프라인이 사라진다. 조각별 최적 툴을 각각 붙이면 통합 비용이 되살아난다.
