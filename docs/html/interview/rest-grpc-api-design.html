00REST와 gRPC란? · 왜난이도 쉬움

REST(Representational State Transfer)는 자원(resource)을 URL로 지목하고 HTTP 메서드(GET/POST/PUT/DELETE)로 조작하는 아키텍처 스타일이다. 텍스트(주로 JSON)로 주고받고, 사람이 읽기 쉽고, 브라우저·캐시·프록시 등 HTTP 생태계를 그대로 쓴다.

gRPC는 Google이 만든 RPC(원격 프로시저 호출) 프레임워크다. .proto IDL로 서비스·메시지를 정의하고, HTTP/2 위에서 protobuf 바이너리로 통신한다. 코드 생성으로 클라이언트 stub을 자동으로 만들고, 4가지 스트리밍을 지원한다.

축RESTgRPC
모델자원 중심(resource) · CRUD 매핑동작 중심(procedure) · 함수 호출
전송/포맷HTTP/1.1 · JSON(텍스트)HTTP/2 · protobuf(바이너리)
계약OpenAPI(선택, 느슨).proto IDL(강제, 엄격)
스트리밍기본 요청-응답(SSE로 단방향 보조)4종(unary·server·client·bidi)
브라우저네이티브 지원직접 불가 → grpc-web 필요
한 줄 정리

외부 공개·브라우저·캐시 친화 = REST. 내부 서비스 간 저지연·강한 계약·스트리밍 = gRPC.

01용어 사전 — 풀네임과 의미난이도 쉬움

API 설계 면접에 쏟아지는 약어를 약어 → 풀 네이밍 → 한 줄 의미로 정리했다.

약어풀 네이밍개념 · 의미
RESTRepresentational State Transfer자원을 URL로 지목, HTTP 메서드로 조작하는 아키텍처 스타일.
RPCRemote Procedure Call원격 함수를 로컬 함수처럼 호출하는 모델. gRPC의 뿌리.
gRPCgRPC Remote Procedure CallsHTTP/2 + protobuf 기반 고성능 RPC 프레임워크(재귀 약어).
HTTPHyperText Transfer Protocol웹 요청/응답 프로토콜. REST의 전송 계층.
HTTP/2(버전 2)멀티플렉싱·헤더 압축·바이너리 프레이밍. gRPC 전송.
protobufProtocol BuffersGoogle의 바이너리 직렬화 포맷 + IDL. 작고 빠르고 스키마 강제.

stub(클라이언트 스텁)protobuf에서 생성된 원격 호출 대리 객체.
deadline / timeout(gRPC 데드라인)호출 마감 시각. 초과 시 취소. 분산 호출 전파에 중요.
grpc-web(브라우저용 gRPC)브라우저 제약 우회를 위한 프록시 기반 gRPC 클라이언트.

02통신 구조 — REST vs gRPC난이도 중간

같은 "클라이언트가 서버를 호출한다"라도 계층·포맷·연결 방식이 다르다.

03설계 원리 / 핵심 요소난이도 중간

REST 핵심

  • Statelessness — 서버가 세션 상태를 안 들고, 요청마다 자기충족. 수평 확장 쉬움.
  • 메서드 · 상태코드 — GET(조회)·POST(생성)·PUT(전체 교체)·PATCH(부분)·DELETE. 2xx 성공·4xx 클라 오류·5xx 서버 오류.
  • 멱등성 — GET/PUT/DELETE는 멱등, POST는 아님. 재시도 안전성의 근거.
  • 캐싱 — ETag·Cache-Control로 HTTP 캐시/CDN 활용(REST의 큰 이점).
  • Richardson 성숙도 — L0 단일 엔드포인트 → L1 자원 → L2 HTTP 메서드/상태코드 → L3 HATEOAS.

gRPC 핵심

  • protobuf IDL — .proto에 서비스·메시지 정의 → 다국어 stub 코드 생성. 계약이 코드로 강제됨.
  • HTTP/2 — 한 연결에 다중 스트림 멀티플렉싱, 헤더 압축(HPACK)으로 저지연·고효율.
  • 4 스트리밍 — unary/server/client/bidirectional. 실시간·대용량에 유리.
  • deadline — 호출 마감을 서버 체인으로 전파해 tail latency·자원 누수 방지.
  • 스키마 진화 — 필드 번호 기반. 필드 추가는 호환, 번호 재사용·타입 변경은 파괴.

04베스트 프랙티스난이도 중간

영역DoDon't
버저닝REST: URL(/v1)·헤더 · gRPC: 패키지 버전 · 하위호환 우선기존 필드 의미 변경 · protobuf 필드 번호 재사용
에러 모델REST: 표준 상태코드 + 문제 상세 · gRPC: status code + details모든 오류를 200으로 감싸기
페이지네이션커서 기반(안정) · 정렬 키 명시대량 데이터에 offset만 · 무제한 목록
계약REST: OpenAPI 문서화 · gRPC: .proto를 단일 출처문서 없이 필드 추가 · 클라와 서버 스키마 불일치
선택외부·브라우저·캐시=REST · 내부 저지연·스트리밍=gRPC · 유연 조회=GraphQL브라우저 직결에 gRPC 강행(grpc-web 없이)

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록 반드시 그릴 그림 · 파고드는 설명 · 정의할 용어 · 아는지 확인하는 체크리스트를 모았다.

① 반드시 그리는 그림 — 선택 기준

② 추가 설명 — 면접관이 파고드는 지점

  • 멱등성이 왜 중요한가 — 네트워크 재시도 시 중복 부작용 방지. POST는 비멱등이라 idempotency key로 보완한다.
  • gRPC가 브라우저에서 왜 안 되나 — 브라우저가 HTTP/2 프레임을 직접 제어 못 해 grpc-web 프록시가 필요하다.
  • HTTP/2 멀티플렉싱 이점 — 한 TCP 연결에 다중 스트림으로 head-of-line 블로킹(HTTP/1.1)을 줄인다.
  • protobuf 스키마 진화 — 필드 번호가 계약. 추가는 호환, 번호 재사용·타입 변경은 파괴적.

③ 반드시 정의할 핵심 용어

용어면접용 한 줄 정의
Statelessness서버가 세션을 안 들어 요청마다 자기충족 → 수평 확장 용이.
Idempotent같은 요청 반복해도 결과 동일. GET/PUT/DELETE 멱등, POST 비멱등.
protobuf바이너리 직렬화 + IDL. 필드 번호가 스키마 계약.
HTTP/2 멀티플렉싱한 연결에 다중 스트림 → 저지연, HOL 블로킹 완화.
gRPC 4 스트리밍unary·server·client·bidirectional.

06예상 질문 & 답변난이도 필수

REST와 gRPC 중 무엇을 언제 쓰나?
외부 공개·브라우저·HTTP 캐시가 중요하면 REST. 내부 마이크로서비스 간 저지연·강한 계약·스트리밍이 중요하면 gRPC. 실무는 엣지(외부)=REST, 내부=gRPC 혼용이 흔하다.
멱등성이 무엇이고 왜 중요한가?
같은 요청을 여러 번 보내도 결과가 같은 성질. GET/PUT/DELETE는 멱등이라 네트워크 재시도가 안전하다. POST는 비멱등이라 결제 같은 곳은 idempotency key로 중복 생성을 막는다.
gRPC가 브라우저에서 바로 안 되는 이유는?
브라우저 fetch/XHR가 HTTP/2 프레이밍을 직접 제어할 수 없어 gRPC 와이어를 그대로 못 쓴다. grpc-web(프록시가 gRPC↔HTTP 변환)을 써야 하며, 이 때문에 순수 브라우저 API는 REST/GraphQL이 흔하다.
HTTP/2가 gRPC에 주는 이점은?
한 TCP 연결에서 다중 스트림 멀티플렉싱으로 HTTP/1.1의 head-of-line 블로킹을 완화하고, 헤더 압축(HPACK)과 바이너리 프레이밍으로 오버헤드를 줄인다. 이 위에서 4가지 스트리밍이 가능하다.
protobuf 스키마를 안전하게 진화시키려면?
필드 번호가 계약이다. 새 필드는 새 번호로 추가(하위호환), 삭제한 번호는 reserved로 잠근다. 번호 재사용·타입 변경·required 강제는 파괴적이라 금지한다.
REST 성숙도(Richardson) 레벨을 설명하라.
L0: 단일 엔드포인트(RPC 흉내), L1: 자원별 URL, L2: HTTP 메서드·상태코드 제대로 사용(대부분의 "RESTful"), L3: HATEOAS(응답에 다음 동작 링크). 실무는 L2가 현실적 목표다.
