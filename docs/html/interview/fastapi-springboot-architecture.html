FastAPI / Spring Boot — 아키텍처와 동시성 모델

00정의 & 왜 — 두 프레임워크의 정체난이도 쉬움

FastAPI는 파이썬 웹 프레임워크다. 자체 웹 서버가 아니라 Starlette(ASGI 웹 툴킷)와 Pydantic v2(데이터 검증) 위에 얹은 얇은 계층으로, 타입 힌트를 기반으로 라우팅·검증·직렬화·문서를 자동화한다. async/await를 1급 시민으로 다루며, 실제 구동은 uvicorn 같은 ASGI 서버가 맡는다. 존재 이유는 "파이썬에서 비동기 I/O 처리량 + 타입 안전 + OpenAPI 문서 자동화를 한 번에" 얻는 것이다.

Spring Boot는 JVM(Java/Kotlin) 프레임워크다. 방대한 Spring Framework 위에 auto-configuration + starter 의존성 + 내장 서버(기본 Tomcat)를 얹어 "설정보다 관례(convention over configuration)"로 프로덕션 앱을 즉시 기동시키는 계층이다. 핵심 엔진은 IoC 컨테이너(Bean 생성·주입)와 그 위의 보안·트랜잭션·데이터 스택이다. 존재 이유는 "성숙한 Spring 생태계를 최소 설정으로, 검증된 기본값과 함께" 쓰는 것이다.

01용어 사전 — 풀네임과 의미난이도 쉬움

두 진영 모두 약어가 쏟아진다. 아래 표는 약어 → 풀 네이밍 → 한 줄 의미로 정리했다. 나머지 섹션을 읽다 막히면 여기로 돌아오면 된다.

WebSocket(약어 아님)단일 연결로 양방향·장수명 통신을 하는 프로토콜. WSGI는 불가, ASGI는 지원한다.

DispatcherServlet(약어 아님)Spring MVC의 Front Controller. 모든 요청을 받아 HandlerMapping으로 알맞은 컨트롤러에 위임한다.
Jackson(약어 아님)Spring이 기본으로 쓰는 Java JSON 직렬화·역직렬화 라이브러리. 응답 객체를 JSON으로 변환한다.
path operation(약어 아님)FastAPI에서 HTTP 메서드+경로에 함수를 매핑한 라우트 단위(데코레이터). Spring의 @RestController 핸들러에 대응.
Depends()(약어 아님)FastAPI의 의존성 주입 장치. DB 세션·인증 등을 함수 시그니처로 주입한다. Spring 컨테이너 DI의 경량판.
라우팅routingHTTP 메서드+URL 경로를 실행할 핸들러 함수에 연결하는 과정. FastAPI는 데코레이터, Spring은 HandlerMapping이 담당한다.
미들웨어middleware요청·응답이 핸들러에 닿기 전후로 통과하는 공통 처리 계층(인증·로깅·CORS 등). FastAPI/Starlette는 미들웨어 체인으로, Spring은 필터·인터셉터로 구현.
Front Controller(디자인 패턴)모든 요청을 단일 진입점이 먼저 받아 알맞은 핸들러로 분배하는 패턴. Spring MVC의 DispatcherServlet이 대표 구현이다.
직렬화serialization메모리 상의 객체를 JSON 등 전송 가능한 형태로 바꾸는 변환(반대는 역직렬화). 응답 생성은 직렬화, 요청 본문 파싱은 역직렬화다.
타입 힌트type hint파이썬 변수·인자·반환값에 예상 타입을 명시하는 문법(PEP 484). FastAPI는 이 힌트를 읽어 검증·직렬화·문서를 자동화한다.

DBDatabase데이터베이스. 두 스택 모두 최종 계층에서 공유하는 영속 저장소(PostgreSQL·MySQL 등).

스레드 풀thread pool미리 만들어 둔 스레드 집합을 재사용해 요청을 처리하는 구조. Tomcat 기본 200개로, 모두 블록되면 이후 요청은 큐에 쌓이거나 거절된다.
커넥션 풀connection poolDB 연결을 미리 열어 두고 재사용하는 풀. virtual threads로 스레드 상한이 사라지면 이 풀 크기가 새 병목이 된다.
캐리어 스레드carrier threadvirtual thread를 실제로 실행하는 OS 플랫폼 스레드. virtual thread가 I/O로 블록되면 캐리어에서 unmount돼 다른 virtual thread가 올라탄다.
리액티브reactive데이터를 스트림으로 보고 논블로킹·이벤트 기반으로 처리하며 backpressure로 흐름을 조절하는 프로그래밍 패러다임. Spring WebFlux가 대표.

02아키텍처 — 계층 + 동시성 모델난이도 중간

아키텍처는 두 층위로 본다. (1) 정적 계층 구조는 두 프레임워크가 거의 같다 — 진입(Router/Controller) → 비즈니스(Service) → 영속성(Repository) → DB. 차이가 나는 지점은 (2) 런타임 동시성 모델 — FastAPI는 단일 event loop + ASGI, Spring MVC는 thread-per-request + Servlet이다.

정적 계층 — 모양은 거의 같다

  • 라우팅 — FastAPI는 데코레이터가 붙은 path operation 함수, Spring은 @RestController의 핸들러 메서드. 둘 다 HTTP를 함수 호출로 매핑한다.
  • 비즈니스 — 도메인 로직. FastAPI는 평범한 함수/클래스, Spring은 @Service Bean으로 컨테이너가 주입한다.
  • 영속성 — FastAPI는 프레임워크에 ORM이 없어 SQLAlchemy 등을 조합, Spring은 Spring Data JPA + Hibernate가 표준 세트다.

런타임 동시성 — 여기서 갈린다

면접 포인트 — 왜 모델이 다른가

FastAPI(ASGI)는 한 스레드가 I/O 대기 시간을 겹쳐 여러 요청을 처리한다(비동기 동시성). Spring MVC(Servlet)는 요청마다 스레드를 배정하고, 그 스레드는 I/O가 끝날 때까지 블록된다. 전자는 async 드라이버가 필수고 CPU 바운드에 약하며, 후자는 코드가 단순하지만 스레드 풀이 상한이다. JVM 진영의 답이 virtual threads와 리액티브(WebFlux)다.

03동작 원리 — 요청 한 건의 여정난이도 중간

FastAPI — ASGI 파이프라인

  1. uvicorn(ASGI 서버)이 소켓에서 HTTP를 받아 ASGI scope/receive/send로 변환한다.
  2. Starlette가 라우팅으로 path operation을 찾고, 미들웨어 체인을 통과시킨다.
  3. Depends() 의존성이 순서대로 해석된다(DB 세션·인증 등). 이것이 FastAPI의 DI다.
  4. Pydantic v2가 요청 본문·쿼리·경로를 타입에 맞춰 검증·파싱한다(실패 시 자동 422).
  5. path operation이 async def면 event loop에서 직접 실행되고, 함수가 await로 I/O를 넘길 때 루프는 다른 요청의 코루틴을 돌린다. 반대로 평범한 def면 외부 스레드풀(AnyIO worker)에서 실행해 루프를 막지 않는다.
  6. 반환값을 response_model 기준으로 직렬화하고, 스키마는 OpenAPI 3.1로 자동 노출된다.
핵심 함정 — async def 안의 블로킹 호출

async def 안에서 동기 블로킹 I/O(예: 동기 DB 드라이버, requests.get, time.sleep)를 호출하면 event loop 스레드 전체가 그 시간 동안 정지한다. 그 순간 다른 모든 요청도 멈춘다. 해결: async 드라이버(asyncpg·httpx)로 await하거나, 정 안 되면 그 함수를 def로 두어 스레드풀에 위임한다.

Spring MVC — Servlet 파이프라인

  1. Tomcat이 커넥션을 받아 스레드 풀(기본 max-threads=200)에서 스레드 하나를 꺼내 요청에 배정한다.
  2. DispatcherServlet이 HandlerMapping으로 컨트롤러 메서드를 찾는다(Front Controller 패턴).
  3. 필요한 Bean(@Service·@Repository)은 기동 시점에 IoC 컨테이너가 이미 생성·주입해 둔 것을 쓴다.
  4. Service가 트랜잭션(@Transactional, AOP 프록시) 경계 안에서 로직을 수행한다.
  5. Repository가 JPA/Hibernate로 블로킹 JDBC 호출을 한다. 이 동안 배정된 스레드는 결과가 올 때까지 대기(블록)한다.
  6. 응답을 직렬화(Jackson)해 반환하면 스레드는 풀로 반납된다.
핵심 함정 — 풀 고갈과 503

느린 I/O가 스레드를 오래 물면, 200개가 모두 대기 상태가 되고 이후 요청은 큐(기본 accept-count=100)에 쌓이다 넘치면 HTTP 503으로 거절된다. 스레드를 무작정 늘려도 코어 수만큼만 실제 병렬이고, 컨텍스트 스위칭·메모리(스레드당 스택)만 커진다. 그래서 나온 게 virtual threads(블로킹 유지 + 값싼 확장)와 WebFlux(논블로킹)다.

virtual threads — 블로킹 모델을 값싸게

JDK 21에서 정식이 된 virtual thread는 JVM이 스케줄링하는 경량 스레드다. Spring Boot 3.2+에서 spring.threads.virtual.enabled=true 한 줄이면, Spring Boot가 Tomcat을 Executors.newVirtualThreadPerTaskExecutor()로 바꿔 요청마다 virtual thread를 만든다. 블로킹 코드는 그대로인데, I/O로 블록되는 순간 JVM이 그 virtual thread를 캐리어(플랫폼) 스레드에서 unmount해 같은 플랫폼 스레드로 다른 요청을 처리한다. 적은 수의 OS 스레드로 수천 동시 요청을 감당한다.

# application.properties — Spring Boot 3.2+ / JDK 21
spring.threads.virtual.enabled=true
virtual threads 두 가지 주의

① pinning — synchronized 블록/네이티브 안에서는 virtual thread가 캐리어에 고정(pin)돼 unmount되지 않는다. 해결은 ReentrantLock으로 교체(라이브러리가 synchronized를 쓰면 그 라이브러리를 갱신). ② 병목 이동 — 스레드 상한이 사라지면 DB 커넥션 풀이 새 병목이 된다. 커넥션 풀 크기를 함께 설계해야 한다.

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록, 반드시 그릴 그림 · 파고드는 추가 설명 · 정의할 핵심 용어 · 아는지 확인하는 체크리스트를 한 곳에 모았다.

① 반드시 그리는 그림

"FastAPI가 왜 동시성에 유리하냐"거나 "Spring이 요청을 어떻게 처리하냐"고 하면 이 그림이다. 세 가지가 핵심 — 한 event loop가 여러 코루틴을 겹치는 쪽 vs 요청마다 스레드가 블록되는 쪽, 그리고 그 한계를 푸는 virtual threads.

② 추가 설명 — 면접관이 파고드는 지점

  • GIL이 있는데 어떻게 동시성이 되나 — async는 병렬이 아니라 동시성이다. 한 스레드가 A의 I/O 대기 시간에 B를 실행할 뿐, 파이썬 바이트코드를 동시에 두 개 돌리지 않는다. 그래서 I/O 바운드엔 강하고 CPU 바운드엔 무익하다.
  • async def에 블로킹 호출을 넣으면 — 그 시간 동안 event loop 전체가 멈춰 모든 요청이 지연된다. async 드라이버로 await하거나 def로 내려 스레드풀에 맡긴다. 면접 단골 함정.
  • thread-per-request의 한계와 virtual threads — 200 풀이 느린 I/O로 고갈되면 503. virtual threads는 블로킹 코드를 유지한 채 I/O 시 unmount로 확장한다. 단 synchronized pinning과 커넥션 풀 병목을 반드시 언급해야 깊이가 산다.
  • Pydantic v2가 v1보다 빠른 이유 — 검증 코어를 Rust(pydantic-core)로 재작성해 순수 파이썬 대비 큰 폭으로 빨라졌다. FastAPI의 요청 검증·직렬화가 여기에 얹힌다.

③ 반드시 정의할 핵심 용어

용어면접용 한 줄 정의

thread-per-request요청당 스레드 1개 · I/O 대기 중 블록 · 풀(기본 200) 고갈 시 503. Spring MVC 기본.

④ 반드시 아는 개념 체크리스트

  1. ASGI vs WSGI — 비동기/WebSocket 지원 여부, FastAPI가 ASGI를 쓰는 이유.
  2. event loop + coroutine — 단일 스레드 동시성, await 지점에서 전환.
  3. 동시성 ≠ 병렬 + GIL — async는 I/O 바운드용, CPU 바운드는 프로세스로 오프로딩.
  4. async def vs def — 전자는 루프, 후자는 스레드풀. 블로킹 호출은 def로.
  5. thread-per-request + Tomcat 풀 — max 200 · accept-count 100 · 고갈 시 503.
  6. virtual threads(Loom, JDK 21) — spring.threads.virtual.enabled=true, unmount, pinning.
  7. WebFlux(리액티브) — Netty event loop, Mono/Flux, virtual threads와의 트레이드오프.
  8. IoC 컨테이너 / Bean / DI — 기동 시 조립, 생성자 주입, AOP(@Transactional).
  9. JPA vs Hibernate — 스펙과 구현, javax→jakarta 네임스페이스 전환.
  10. Pydantic v2 — Rust 코어, 요청/응답 DTO 분리, 자동 422·OpenAPI.

06예상 질문 & 답변난이도 필수

FastAPI가 Spring MVC보다 동시성에서 유리하다는 말은 정확히 무슨 뜻인가?
FastAPI는 단일 event loop가 한 요청의 I/O 대기 시간에 다른 요청의 코루틴을 실행한다. 스레드를 늘리지 않고 대기 시간을 겹쳐 I/O 바운드 처리량을 낸다. Spring MVC는 요청당 스레드를 배정하고 I/O 동안 그 스레드가 블록되므로, 동시 처리량이 스레드 풀 크기(기본 200)에 묶인다. 단 FastAPI의 이점은 async 드라이버로 실제 await할 때만 성립하고, CPU 바운드에는 오히려 불리하다.
async def 핸들러 안에서 동기 블로킹 DB 드라이버를 쓰면 어떻게 되나?
그 호출이 끝날 때까지 event loop 스레드 전체가 정지해 모든 동시 요청이 함께 멈춘다. FastAPI의 장점이 사라지는 대표 실수다. 해결은 두 가지 — ① async 드라이버(asyncpg·httpx)로 바꿔 await하거나, ② 그 핸들러를 평범한 def로 선언해 FastAPI가 외부 스레드풀에서 돌리게 한다(루프 비차단).
GIL이 있는데 파이썬 async가 어떻게 여러 요청을 동시에 처리하나?
async는 병렬이 아니라 동시성이다. GIL은 한 번에 한 스레드만 바이트코드를 실행하게 하지만, 코루틴은 I/O에서 await로 제어를 양보하고 그 틈에 루프가 다른 코루틴을 돌린다. 즉 계산을 병렬로 하는 게 아니라 기다리는 시간을 공유하는 것이라, I/O 바운드에 강하고 CPU 바운드에는 도움이 안 된다(그건 멀티프로세스로 푼다).
Spring의 DI/IoC가 무엇이고 왜 쓰나?
IoC(제어 역전)는 객체 생성·생명주기 제어를 개발자가 아니라 컨테이너(ApplicationContext)가 갖는 것이고, DI(의존성 주입)는 그 구현으로 필요한 Bean을 외부에서 넣어 주는 것이다. 효과는 결합도 감소·교체 용이·테스트 용이(목 주입)다. 실무 권장은 생성자 주입 — 불변이 보장되고 필수 의존성이 명시된다. FastAPI에도 Depends()라는 경량 DI가 있다.
JPA와 Hibernate의 차이는?
JPA(Jakarta Persistence API)는 Java 표준 ORM 스펙(인터페이스 모음)이고, Hibernate는 그 스펙을 구현한 구현체다. 코드는 JPA 인터페이스(EntityManager·@Entity)에 의존하고 런타임에 Hibernate가 실제 SQL을 생성·실행한다. Spring Boot 3부터 패키지가 javax.persistence → jakarta.persistence로 바뀐 점도 자주 묻는다.
virtual threads가 나오면서 이제 리액티브(WebFlux)는 필요 없나?
둘은 목표는 같고 방식이 다르다. virtual threads는 기존 블로킹 코드(명령형)를 그대로 두고 스레드를 값싸게 만들어 확장한다 — 학습·디버깅 비용이 낮다. WebFlux는 논블로킹·백프레셔(backpressure)를 스트림으로 다루어 스트리밍·역압 제어에 강하지만 Mono/Flux 패러다임 학습 비용이 크다. 대부분의 요청/응답 서비스는 virtual threads로 충분하고, 이벤트 스트리밍·정교한 배압이 필요하면 WebFlux를 고른다. 다만 virtual threads는 synchronized pinning과 커넥션 풀 병목을 챙겨야 한다.
Pydantic v2가 v1보다 빠른 이유는? FastAPI와는 어떤 관계인가?
Pydantic v2는 검증·직렬화 코어를 Rust(pydantic-core)로 재작성해 순수 파이썬이던 v1보다 크게 빨라졌다. FastAPI는 요청 본문·쿼리·경로 파라미터를 Pydantic 모델로 검증하고 응답을 response_model로 직렬화하며, 그 스키마로 OpenAPI 문서를 자동 생성한다. 최신 FastAPI는 Pydantic v1 지원을 제거했으므로 v2 API(model_validate·model_dump) 기준으로 답하면 된다.
ASGI와 WSGI는 뭐가 다른가?
WSGI는 동기 요청/응답 한 쌍만 다루는 전통 표준(Flask 등)이라 요청 처리 중 블로킹되고 WebSocket·long-lived 연결을 못 한다. ASGI는 async 핸들러, WebSocket, 서버-보낸 스트림 같은 장수명·양방향 연결을 scope/receive/send 모델로 지원한다. FastAPI/Starlette가 ASGI 위에 서고, uvicorn이 그 ASGI 앱을 구동한다.
