00MLOps란? · 왜 필요한가난이도 쉬움

MLOps(Machine Learning Operations)는 ML + DevOps + Data Engineering을 합친 실무 규율이다. 모델을 한 번 학습해 던지는 게 아니라, 데이터 → 학습 → 평가 → 레지스트리 → 배포 → 모니터링 → (drift 감지) → 재학습을 재현 가능하고 자동화된 파이프라인으로 운영하는 것이 핵심이다.

일반 소프트웨어는 코드만 버전·테스트·배포하면 된다. ML 시스템은 코드 + 데이터 + 모델 세 가지가 모두 산출물이라, DevOps의 CI/CD에 ML 고유의 CT(Continuous Training)가 더 붙는다. 그래서 "코드는 그대로인데 데이터가 바뀌어 결과가 달라지는" 상황을 통제해야 한다.

일반 소프트웨어ML 시스템 (MLOps가 필요한 이유)
버전 대상 = 코드버전 대상 = 코드 + 데이터 + 모델 (3중 버전)
테스트 = 단위/통합 테스트테스트 = 코드 + 데이터 검증 + 모델 검증
배포하면 동작이 고정배포 순간부터 drift로 성능이 열화(data/concept drift)
릴리스 트리거 = 코드 변경릴리스 트리거 = 코드 + 신규 데이터 · 성능 저하 · 스케줄

왜 하는가 — 세 가지 동기

  • 재현성(reproducibility) — "운영 중인 이 모델이 어떤 데이터·코드·하이퍼파라미터로 나왔나"를 추적·복원할 수 있어야 한다. 실험이 재현 안 되면 디버깅·개선이 불가능하다.
  • 거버넌스·감사(governance) — 규제 환경에서 "운영 모델이 어떤 데이터로 학습됐나", "6개월 전 모델을 그대로 재현할 수 있나"에 답해야 하고, 문제 시 롤백할 수 있어야 한다.
  • drift 대응 — 현실 데이터는 시간이 지나면 변한다. 모델은 배포 시점부터 열화하므로 지속 모니터링 + 재학습(CT) 없이는 정확도가 서서히 무너진다.
한 줄 정리

MLOps = "코드·데이터·모델을 함께 버전·테스트·배포하고, 배포 후에도 drift를 감지해 재학습(CT)하는 닫힌 루프로 ML 시스템을 운영하는 것."

01용어 사전 — 풀네임과 의미난이도 쉬움

MLOps 문서·면접에는 약어와 도구명이 쏟아진다. 아래 표는 용어 → (풀 네이밍) → 한 줄 의미로 정리했다. 나머지 섹션을 읽다 막히면 여기로 돌아오면 된다.

자산 · 버전관리

용어풀 네이밍개념 · 의미

lineage(계보 · provenance)운영 모델 → 학습에 쓴 코드 커밋·데이터 버전·하이퍼파라미터로 역추적하는 연결. 감사·재현의 근거.

FeastFeature Store오픈소스 feature store. online/offline 저장을 통일하고 point-in-time 조회로 training-serving skew를 완화.

배포 · 서빙 · 모니터링

용어풀 네이밍개념 · 의미
champion / challenger(운영 · 도전 모델)현재 트래픽을 받는 champion vs 검증 중인 challenger. 좋으면 challenger를 champion으로 승격.
alias(모델 별칭)특정 model version을 가리키는 가변 named 참조(예: @champion). MLflow에서 구 stages를 대체.
canary(카나리 배포)신규 버전에 소량 트래픽(예: 1~10%)부터 보내고 지표를 보며 점진 확대. blast radius 최소화.
shadow(섀도 배포 · dark launch)신규 버전이 동일 입력을 처리하되 응답은 사용자에게 안 주고 로깅만. 무영향으로 실전 검증.
blue-green(블루-그린)old(blue)·new(green)를 동시 유지하고 스위치로 전환. 문제 시 즉시 롤백, 대신 인프라 2배.
servingonline / batchonline=실시간 저지연 요청/응답(REST/gRPC). batch=대량 데이터를 주기적으로 오프라인 예측.
data driftcovariate shift입력 분포가 학습 때와 달라짐. 입력→출력 관계는 그대로. 예: 신규 시장 유입으로 특성 분포 이동.
concept drift(개념 드리프트)입력→출력 관계 자체가 변함. 입력 분포는 같아도 정답이 달라짐. 예: 소비 트렌드 변화로 선호 매핑이 바뀜.
training-serving skew(학습-서빙 편향)학습 때 feature와 서빙 때 feature가 다르게 계산돼 생기는 성능 저하. feature store가 완화.
rollback(롤백)문제 발생 시 이전 model version을 다시 champion으로 재승격. registry 덕에 코드 변경 없이 무중단.
A/B test(A/B 테스트 · split test)트래픽을 두 변형으로 나눠 비즈니스 지표를 통계 검정으로 비교. 어느 모델이 실제 성과가 나은지 판정.
RESTRepresentational State TransferHTTP 기반 API 스타일. online serving에서 요청/응답으로 실시간 예측을 주고받는 대표 인터페이스.
gRPCgRPC Remote Procedure CallHTTP/2 + Protocol Buffers 기반의 고성능 RPC 프레임워크. 저지연·고처리량 online serving에 REST 대안으로 쓰인다.
blast radius(영향 반경)배포 사고가 번지는 범위. canary·shadow는 신규 버전 노출을 좁혀 이 반경을 최소화한다.
PSIPopulation Stability Index학습 분포 대비 현재 입력 분포의 이동량을 하나의 수치로 요약. data drift 탐지 임계값으로 흔히 쓴다(예: 0.2↑ 경고).
KS testKolmogorov–Smirnov test두 표본의 누적분포 최대 격차로 분포 차이를 검정. 연속형 feature의 data drift 탐지에 사용.
DDMDrift Detection Method스트림의 오류율 증가를 감시해 concept drift를 경보하는 온라인 탐지 알고리즘.
ADWINADaptive WINdowing가변 크기 윈도의 통계가 달라지면 오래된 데이터를 잘라내 concept drift를 감지하는 알고리즘.
Evidently(오픈소스 ML 모니터링)데이터 품질·data/concept drift 리포트와 대시보드를 생성하는 오픈소스 도구.

02라이프사이클 — 폐루프 구조난이도 중간

MLOps의 뼈대는 직선이 아니라 닫힌 루프(closed loop)다. 배포는 종착점이 아니라 루프의 한 지점이고, 모니터링이 drift·성능 저하를 감지하면 다시 학습으로 돌아간다(CT). 이 순환을 그림으로 못 그리면 면접에서 바로 드러난다.

단계별 역할

  • ① 데이터 — 수집·검증(스키마·품질), feature 계산. feature store가 학습/서빙 공통 정의를 공급한다.
  • ② 학습(Train) — 실험 추적으로 파라미터·메트릭·아티팩트를 기록. 코드/데이터 버전이 함께 고정된다(CI).
  • ③ 평가(Eval) — 홀드아웃·오프라인 지표로 승격 게이트 통과 여부 판정. 기준 미달이면 여기서 멈춘다.
  • ④ 레지스트리 — 통과 모델을 버전으로 등록하고 alias(예: @challenger)를 부여.
  • ⑤ 배포(Deploy) — canary/shadow/blue-green으로 안전하게 서빙에 반영(CD). 서빙 코드는 alias만 바라본다.
  • ⑥ 모니터 — 성능·데이터 분포·drift를 감시. 이상 신호가 재학습(CT)을 트리거해 루프가 닫힌다.

성숙도 레벨 (Google 기준)

레벨상태특징
L0수동 프로세스노트북에서 손으로 학습·배포. 릴리스가 드물다(연 몇 회). CI/CD 없음.
L1ML 파이프라인 자동화학습 파이프라인 자동화로 CT 달성 — 신규 데이터로 자동 재학습. 데이터·모델 검증, 트리거, 메타데이터 관리 도입.
L2CI/CD 파이프라인 자동화파이프라인 자체를 CI/CD로 빠르게 갱신·배포. 실험→운영 반영이 신속·반복 가능.
면접 포인트 — 폐루프

ML 시스템은 배포로 끝나지 않는다. monitor가 drift·성능 저하를 감지 → CT가 재학습을 트리거 → 새 버전을 registry에 등록 → 다시 배포로 루프가 닫힌다. 이 귀환 경로(CT)가 일반 DevOps와 MLOps를 가르는 핵심이다.

03동작 원리 — 추적 · 승격 · 배포 · 모니터링 · CT난이도 중간

① 실험 추적 · 버전관리

학습마다 파라미터·메트릭·코드 버전·아티팩트를 기록해(experiment tracking) 어떤 실험이 최고였는지 비교하고 재현한다. 재현의 전제는 3중 버전 — 코드(Git) · 데이터(DVC 등) · 모델(registry)을 묶어야 특정 모델을 그대로 되살릴 수 있다. 그 연결이 lineage다. 운영 모델에서 학습 데이터·코드 커밋까지 역추적되어야 감사·디버깅이 가능하다.

② 레지스트리 · 승격

평가를 통과한 모델은 registry에 버전으로 등록된다. 과거 MLflow는 Staging/Production/Archived라는 고정 stage로 상태를 표현했지만, 경직성 탓에 MLflow 2.9에서 stage가 deprecated되고 alias + tag로 대체됐다. 이제 @champion 같은 가변 named 참조를 특정 버전에 붙인다. 한 버전에 여러 alias를 달 수 있어 A/B·점진 롤아웃이 쉽다.

배포 ≠ 재학습 (핵심)

서빙 코드는 항상 models:/MyModel@champion처럼 alias만 바라본다. 새 모델을 밀 때는 challenger를 검증한 뒤 champion alias를 새 버전으로 옮기기만 하면 된다 — 서빙 코드는 그대로. 롤백도 alias를 이전 버전으로 되돌리는 것이라 무중단이다.

③ 배포 전략 — canary · shadow · blue-green

새 모델을 한 번에 전량 교체하면 사고 시 blast radius가 크다. 그래서 위험도·인프라 비용에 따라 세 전략을 쓴다(조합도 가능 — 먼저 shadow로 무영향 검증 후 canary로 점진 확대).

전략사용자 영향인프라언제
Shadow0 (응답 미반영)추가 서빙 1대고위험 변경의 실전 무영향 검증 — 예측만 비교
Canary소수(예: 1~10%)라우팅 필요지표 보며 점진 확대 — 강한 모니터링 전제
Blue-Green전환 순간 전량2배 동시 유지빠른 전환 + 즉시 롤백이 중요할 때
A/B test분할라우팅 + 분석모델 변형의 비즈니스 지표 비교(통계 검정)

④ 모니터링 — 성능과 drift

배포 후에는 두 가지를 본다 — 모델 성능(정답이 지연 도착하면 대리 지표)과 drift. drift는 두 종류로 갈리고, 대응이 다르다.

구분무엇이 변하나예시탐지
Data drift
(covariate shift)		입력 분포만 변함
(입력→출력 관계는 유지)		신규 지역 유입으로 나이·기기 분포 이동입력 분포 통계 비교(PSI, KS test 등)
Concept drift입력→출력 관계가 변함
(입력 분포는 같을 수 있음)		소비 트렌드 변화로 "선호"의 정의가 바뀜성능/오류율 감시(DDM·ADWIN 등)

실무 순서는 넓게 → 좁게 — 먼저 data drift를 감시하고, 성능이 떨어지는데 입력→출력 관계까지 바뀌었는지 확인해 concept drift를 판정한다. data drift가 방치되면 concept drift로 번지기도 한다. 오픈소스로는 Evidently 같은 도구로 데이터 품질·drift 리포트를 만든다.

⑤ CT — 재학습 트리거

모니터링이 잡은 신호가 재학습(CT)을 촉발한다. 트리거는 보통 신규 데이터 도착 · 스케줄(예: 매일) · drift 감지 · 성능 저하다. 재학습된 모델은 다시 평가 게이트 → registry 등록 → 배포를 거쳐 루프가 닫힌다. 이 자동 귀환이 있어야 성숙도 L1 이상이다.

04베스트 프랙티스난이도 중간

영역DoDon't

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록, MLOps에서 반드시 그릴 아키텍처 · 파고드는 추가 설명 · 정의할 핵심 용어 · 아는지 확인하는 체크리스트를 한 곳에 모았다.

① 반드시 그리는 아키텍처

"ML 모델을 운영하는 구조를 그려봐"라고 하면 이 그림이다. 세 축이 핵심 — registry 승격(alias 스왑 = 배포), 모니터링→재학습(CT) 귀환, 3중 버전 재현성 + 롤백.

② 추가 설명 — 면접관이 파고드는 지점

  • 왜 배포가 끝이 아닌가 — 모델은 배포 순간부터 drift로 열화한다. monitor가 성능·분포 변화를 잡아 CT로 재학습을 트리거하는 귀환 경로가 있어야 시스템이 유지된다.
  • 배포와 재학습은 다르다 — 서빙 코드는 @champion alias만 본다. 새 모델 반영은 alias를 새 버전으로 스왑하는 것이지 코드 재배포가 아니다. 롤백도 alias 되돌리기라 무중단.
  • data drift vs concept drift — 전자는 입력 분포만, 후자는 입력→출력 관계가 변한다. 대응이 달라서(재학습 vs 재정의) 구분이 중요하다.
  • training-serving skew — 학습·서빙에서 feature를 따로 계산하면 소리 없이 성능이 무너진다. feature store로 같은 정의를 공유해 막는다.

③ 반드시 정의할 핵심 용어

용어면접용 한 줄 정의
model registry모델 버전 중앙 저장소. alias로 champion/challenger 승격·롤백을 관리(구 stages 대체).
feature store학습/서빙 공통 feature를 online/offline로 서빙. training-serving skew 방지.
CI/CD/CT코드+데이터+모델 검증(CI) · 파이프라인/서비스 배포(CD) · drift 트리거 재학습(CT).
data / concept drift입력 분포 변화 / 입력→출력 관계 변화. 전자는 재학습, 후자는 문제 재정의가 필요.
배포 전략shadow(무영향 검증) · canary(점진 확대) · blue-green(즉시 전환/롤백).
reproducibilitydata+model+code 3중 버전 + lineage로 특정 모델을 그대로 재현.

④ 반드시 아는 개념 체크리스트

  1. MLOps vs DevOps — 코드뿐 아니라 data·model도 버전·테스트·배포, CI/CD에 CT가 추가.
  2. 폐루프 라이프사이클 — data→train→eval→registry→deploy→monitor→(drift)→retrain.
  3. model registry + alias — 승격=alias 스왑, 롤백=이전 버전 재승격. MLflow stages는 2.9부터 deprecated.
  4. 배포 전략 — shadow / canary / blue-green의 사용자 영향·인프라·용도 구분.
  5. data drift vs concept drift — 정의·예시·탐지 방법과 대응 차이.
  6. feature store — online/offline, training-serving skew, point-in-time 누수 차단.
  7. 재현성 3중 버전 — code(Git)·data(DVC)·model(registry) + lineage.
  8. online vs batch serving — 실시간 저지연 vs 대량 오프라인 예측.
  9. 성숙도 L0~L2 — 수동 → 파이프라인 자동화(CT) → CI/CD 자동화.
  10. 대표 도구 — MLflow(추적·registry) · Kubeflow(파이프라인·K8s) · Feast(feature store) · Evidently(drift).

06예상 질문 & 답변난이도 필수

data drift와 concept drift의 차이는?
data drift는 입력 분포가 학습 때와 달라지는 것으로, 입력→출력 관계는 그대로다(예: 신규 지역 유입으로 특성 분포 이동). concept drift는 입력→출력 관계 자체가 변하는 것으로, 입력 분포는 같아도 정답이 달라진다(예: 트렌드 변화로 "선호"의 정의가 바뀜). 전자는 재학습으로 대응하고, 후자는 문제 재정의·레이블 재수집까지 필요해 대응이 다르다. 탐지도 data drift는 입력 분포 통계, concept drift는 성능/오류율 감시가 중심이다.
운영 모델에 문제가 생겼을 때 어떻게 롤백하나?
서빙 코드가 alias(예: @champion)만 참조하도록 설계돼 있으면, champion alias를 직전 정상 버전으로 되돌리기만 하면 된다. 모델 아티팩트는 registry에 버전으로 남아 있으므로 재배포·코드 변경 없이 무중단으로 복구된다. blue-green이라면 스위치를 blue로 되돌리는 것도 동일한 즉시 롤백이다.
shadow와 canary 배포는 어떻게 다르고 언제 쓰나?
shadow는 새 모델이 동일 입력을 처리하되 응답을 사용자에게 주지 않고 로깅·비교만 한다 — 사용자 영향 0이라 고위험 변경의 실전 검증에 쓴다. canary는 새 모델에 소량(예: 1~10%) 실트래픽을 실제로 태우고 지표를 보며 점진 확대한다 — 실제 응답을 주므로 강한 모니터링이 전제다. 실무에선 shadow로 무영향 검증 → canary로 점진 확대 순으로 조합한다.
model registry의 역할은? stages는 왜 없어졌나?
registry는 모델 버전의 중앙 저장소로 버전·배포 상태·승격·롤백을 관리하고, 서빙이 이름으로 모델을 당겨 쓰게 한다. MLflow는 과거 Staging/Production/Archived 고정 stage를 썼지만 경직성·환경 표현 한계로 2.9에서 deprecated됐고, alias + tag로 대체됐다. alias는 한 버전에 여러 개를 붙일 수 있어 champion/challenger·A/B·점진 롤아웃이 유연하다.
모델의 재현성은 어떻게 보장하나?
code·data·model 3중 버전을 함께 고정한다 — 코드는 Git 커밋, 데이터는 DVC 등으로 스냅샷, 모델은 registry 버전. experiment tracking으로 파라미터·메트릭·환경을 자동 로깅하고, 이들을 lineage로 연결해 운영 모델에서 학습 데이터·코드까지 역추적되게 한다. 환경은 컨테이너로 고정하고 랜덤 시드·의존성을 pin해 "같은 입력 → 같은 산출물"을 만든다.
online serving과 batch serving은 어떻게 다르고 언제 쓰나?
online은 요청마다 실시간 저지연 응답(REST/gRPC)이 필요할 때 — 사기 탐지, 추천 등. feature도 online store에서 밀리초로 조회한다. batch는 대량 데이터를 주기적으로 오프라인 예측해 저장할 때 — 야간 스코어링, 리포트 등. 지연 여유가 크고 처리량이 우선이다. 선택 기준은 지연 요구와 신선도 — 즉답이 필요하면 online, 아니면 비용 효율적인 batch.
training-serving skew는 무엇이고 어떻게 막나?
학습 때 계산한 feature와 서빙 때 계산한 feature가 미묘하게 달라 오프라인 성능은 좋은데 운영에서 무너지는 현상이다. 원인은 보통 학습 배치 파이프라인과 서빙 실시간 경로가 다른 코드로 같은 feature를 계산하기 때문이다. feature store로 학습·서빙이 같은 feature 정의를 공유하고, offline store의 point-in-time 조회로 미래 정보 누수까지 막는 것이 표준 대응이다.
CT는 무엇이고 언제 트리거되나? 일반 CI/CD와 뭐가 다른가?
CT(Continuous Training)는 ML 고유의 축으로, 파이프라인이 스스로 재학습·재배포하는 것이다. 트리거는 신규 데이터 도착 · 스케줄 · drift 감지 · 성능 저하다. 일반 CI/CD는 코드 변경에 반응하지만, ML CI/CD는 CI가 데이터·모델 검증까지 포함하고 CD가 학습 파이프라인 자체를 배포하며, 여기에 CT가 더해져 데이터 변화에도 반응한다는 점이 다르다.
