00확장성·가용성이란? · 왜난이도 쉬움

확장성(scalability)은 부하가 늘 때 자원을 더해 성능을 유지하는 능력이다. 한 서버를 키우는 수직(scale-up)과 서버 수를 늘리는 수평(scale-out)이 있고, 현대 시스템은 대개 수평 확장을 지향한다.

가용성(availability)은 시스템이 정상 응답하는 시간 비율(예: 99.9%)이다. 장애는 반드시 나므로, 단일 장애점(SPOF)을 없애고 실패를 격리·복구하는 회복탄력성(resilience) 설계가 핵심이다.

한 줄 정리

"무상태로 만들어 수평 확장하고, SPOF를 없애 복제·failover로 가용성을 지키며, 실패는 circuit breaker로 격리한다."

02확장 아키텍처난이도 중간

전형적 확장·장애대응 아키텍처는 LB → 무상태 서비스 N개 → (캐시) → DB primary + replica 구조다. 상태를 서비스 밖으로 밀어야 수평 확장이 쉽다.

03동작 원리 — CAP/PACELC · resilience난이도 중간

CAP · PACELC

  • CAP — 네트워크 분할(P)이 생기면 일관성(C)과 가용성(A) 중 하나를 포기해야 한다. 분할은 선택이 아니라 현실이므로 실질은 "분할 시 C냐 A냐".
  • PACELC — CAP을 확장: 분할 시 A/C, 평상시(Else)엔 지연(L)과 일관성(C)을 맞바꾼다. 예: Dynamo/Cassandra=PA/EL, HBase/VoltDB=PC/EC, MongoDB=PA/EC(구성 의존).
  • 강일관 vs 결과적 일관 — 강일관은 항상 최신을 보장(비용·지연↑), 결과적 일관은 잠시 낡을 수 있으나 가용·저지연.

Resilience 패턴

  • circuit breaker — 실패율이 임계 초과면 Open으로 전환해 호출을 즉시 실패시켜 연쇄 장애를 막고, 타임아웃 후 Half-Open에서 시험 호출로 회복을 확인한다.
  • retry + backoff + jitter — 일시 오류만 재시도하되 지수 백오프와 지터로 재시도 폭풍(thundering herd)을 막는다.
  • bulkhead — 스레드/커넥션 풀을 기능별로 분리해 한 의존성의 장애가 전체를 삼키지 않게 한다.
  • rate limiting — token bucket 등으로 과부하·남용을 방어한다.
  • idempotency — 재시도·중복 전달에 대비해 쓰기를 멱등키로 안전하게 만든다.

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록 반드시 그릴 그림 · 파고드는 설명 · 정의할 용어 · 체크리스트를 모았다.

① 반드시 그리는 그림 — circuit breaker 상태

② 추가 설명 — 면접관이 파고드는 지점

  • CAP에서 무엇을 포기하나 — 분할은 필연이라 실제 선택은 "분할 시 C냐 A냐". 결제=일관성(CP), 피드·장바구니=가용성(AP)이 흔하다.
  • 수평 확장의 전제 — 서비스가 무상태여야 한다. 세션·상태를 외부(Redis)로 밀지 않으면 아무리 인스턴스를 늘려도 안 는다.
  • circuit breaker가 왜 필요한가 — 느린 의존성에 스레드가 물려 연쇄 장애가 난다. 빨리 실패시키고 fallback을 줘 격리한다.
  • 샤딩 키 선택 — 고르게 분산되고(핫스팟 방지) 조회 패턴에 맞아야 한다. 잘못된 키는 리샤딩이 매우 비싸다.

06예상 질문 & 답변난이도 필수

수직 확장과 수평 확장의 차이와 선택 기준은?
수직은 한 서버를 키우기 — 단순하나 상한·SPOF·비용 급증이 있다. 수평은 서버를 늘리기 — 사실상 무한 확장·가용성이 좋지만 무상태화·데이터 분산이 필요하다. 대개 수평을 지향하되 상태 저장 계층은 신중히 다룬다.
CAP 정리에서 무엇을 포기하나?
네트워크 분할(P)은 피할 수 없으므로 분할이 나면 일관성(C)과 가용성(A) 중 하나를 택한다. 결제·재고는 CP(일관성 우선), 피드·추천은 AP(가용성 우선)로 가는 식이다. PACELC는 평상시 지연(L) vs 일관성(C)까지 본다.
strong consistency와 eventual consistency는?
strong은 쓰기 직후 모든 읽기가 최신을 본다 — 정합성은 좋지만 지연·가용성 비용이 크다. eventual은 잠시 낡은 값을 볼 수 있으나 결국 수렴하며 가용·저지연이다. 도메인의 허용 오차에 맞춰 고른다.
circuit breaker는 왜 쓰고 어떻게 동작하나?
느린/실패하는 의존성에 호출이 쌓여 연쇄 장애가 나는 것을 막는다. Closed에서 실패율이 임계를 넘으면 Open으로 전환해 즉시 실패(+fallback), 일정 시간 뒤 Half-Open에서 시험 호출로 회복을 확인하고 성공하면 Closed로 돌아간다.
샤딩 키는 어떻게 고르나?
고른 분산(핫스팟 방지)과 조회 패턴 적합성이 핵심이다. 예: 사용자 ID 해시로 분산하되, 크로스 샤드 조인이 잦으면 설계를 재고한다. 잘못된 키는 나중에 리샤딩 비용이 크므로 초기 선택이 중요하다.
마이크로서비스의 단점은?
분산 시스템 복잡도(네트워크 지연·부분 실패), 데이터 정합성(분산 트랜잭션·saga), 운영·관측성 부담, 배포·버전 관리가 늘어난다. 경계가 불명확하면 분산 모놀리스가 되어 최악이다. 명확한 도메인 경계와 관측성이 전제다.
