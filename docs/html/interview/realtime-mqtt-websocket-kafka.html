MQTT · WebSocket · Kafka — 실시간 통신 3대 프로토콜

00정의 & 왜 — 무엇을 언제난이도 쉬움

셋 다 "실시간 통신"으로 묶이지만 사는 계층이 다르다. MQTT는 디바이스와 broker 사이의 경량 메시징, WebSocket은 브라우저와 서버 사이의 양방향 채널, Kafka는 서비스와 서비스 사이의 영속 이벤트 로그다. 면접에서 자주 찔리는 첫 질문이 "왜 이걸 골랐냐"이므로, 각각의 존재 이유부터 잡는다.

프로토콜무엇인가언제 쓰나
MQTTOASIS 표준 경량 pub/sub 메시징. broker 경유, TCP 위(기본 1883, TLS 8883). 헤더가 작아 저대역·배터리 디바이스에 맞음.수천~수백만 IoT 디바이스 텔레메트리, 불안정한 무선, 센서·액추에이터 원격 제어.
WebSocketIETF 표준 full-duplex 지속 연결. HTTP 업그레이드로 시작해 하나의 TCP로 양방향 프레임 교환.브라우저↔서버 실시간 푸시 — 채팅, 라이브 대시보드, 협업 편집, 알림, 게임.
Kafka분산 이벤트 스트리밍 플랫폼이자 영속 커밋 로그. 파티션으로 수평 확장, 메시지를 디스크에 보관해 재생(replay) 가능.시스템 간 대용량 이벤트 백본, 로그·클릭스트림 수집, 이벤트 소싱, 스트림 처리 파이프라인.

흔한 오해가 "셋 중 하나면 되지 않나"인데, 실제 시스템은 셋을 이어 붙인다. 예: 현장 센서가 MQTT로 broker에 올리고, broker→백엔드 사이를 Kafka가 받아 영속·분배하고, 최종 사용자 브라우저에는 WebSocket으로 밀어 보여준다.

한 줄 정리

MQTT = 디바이스↔broker 경량 pub/sub · WebSocket = 브라우저↔서버 양방향 채널 · Kafka = 서비스↔서비스 영속 로그 스트리밍. 계층이 다르므로 대체재가 아니라 보완재다.

02아키텍처 — 세 패턴 비교난이도 중간

같은 "실시간"이라도 연결 토폴로지가 다르다. MQTT는 broker를 중심에 둔 N:M pub/sub, WebSocket은 두 끝점을 잇는 1:1 지속 연결, Kafka는 파티션 로그를 여러 소비자가 나눠 읽는 분산 로그 + consumer group이다.

  • MQTT — broker 스타형: 모든 트래픽이 broker를 거친다. publisher와 subscriber는 서로를 모르고 topic으로만 연결(N:M). broker가 retain·LWT·QoS를 책임진다.
  • WebSocket — 점대점 지속 연결: 브라우저와 서버가 하나의 TCP를 열어두고 양방향으로 프레임을 주고받는다. broker도, topic도, 다중 구독 개념도 프로토콜 자체엔 없다(앱이 얹는다).
  • Kafka — 분산 로그: topic이 여러 파티션으로 쪼개져 broker들에 분산되고, 각 파티션은 append-only 로그다. consumer group이 파티션을 나눠 읽어 수평 확장하며, 메시지는 소비 후에도 보존돼 재생된다.
면접 포인트 — 계층 구분

세 프로토콜을 "무엇이 더 낫냐"로 비교하면 함정이다. edge(디바이스) → backbone(서비스 간) → edge(브라우저)로 역할이 다르다고 답하고, 실제 파이프라인에서 MQTT → Kafka → WebSocket으로 이어 붙는 예를 드는 것이 안전하다.

03동작 원리 — QoS · 핸드셰이크 · 파티션난이도 중간

① MQTT — QoS 핸드셰이크 · retain · LWT · keepalive

MQTT의 핵심은 QoS 3단계다. QoS가 올라갈수록 확인 왕복이 늘어 신뢰도는 오르고 처리량은 내려간다. 주의: 전달되는 QoS는 publish QoS와 subscribe QoS의 최솟값이며(broker는 QoS를 올려주지 않음), 이 보장은 client↔broker 한 홉 단위다.

  • retained — publish 시 retain 비트를 세우면 broker가 topic의 마지막 값을 보관한다. 늦게 접속한 구독자도 즉시 현재 상태를 받는다. 빈 페이로드로 retain publish하면 삭제된다.
  • LWT — 접속 시 "내가 비정상 종료되면 device/x/status = offline을 발행하라"를 등록한다. 흔한 패턴: 접속 직후 online(retain) 발행 + LWT로 offline(retain) 예약 → 누구나 정확한 생존 상태를 본다.
  • keepalive — 데이터가 없으면 클라이언트가 PINGREQ를 보내고 broker가 PINGRESP로 답한다. keepalive가 60s면 broker는 약 90s(1.5배)까지 무응답을 허용한 뒤 죽은 연결로 간주한다.

② WebSocket — Upgrade 핸드셰이크 · 프레임 · ping/pong

WebSocket은 HTTP로 시작한다. 클라이언트가 Upgrade 요청을 보내고 서버가 101 Switching Protocols로 수락하면, 그 뒤로는 HTTP가 아니라 프레임 채널이 된다. 서버는 Sec-WebSocket-Key에 매직 GUID를 붙여 SHA-1·base64한 값을 Sec-WebSocket-Accept로 돌려준다.

# Client → Server (HTTP Upgrade 요청)
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

# Server → Client (101 수락)
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

핸드셰이크 후 모든 메시지는 프레임으로 흐른다. 프레임 헤더는 FIN·opcode·MASK·payload length로 구성되고, client→server 프레임은 반드시 마스킹(server→client는 마스킹 금지)한다. opcode로 프레임 종류를 구분한다.

opcode종류용도
0x0continuation이어지는 조각 프레임(fragmentation)
0x1 / 0x2text / binary실제 데이터. text는 UTF-8, binary는 임의 바이트
0x8close닫기 핸드셰이크 시작. 상대도 close로 응답
0x9 / 0xAping / pong생존 확인. ping 수신 시 같은 payload로 pong 회신(≤125바이트)

브라우저 API는 ping/pong을 자동 처리하지만, 서버는 주기적 ping + pong 타임아웃으로 조용한 TCP 절단(모바일·NAT)을 잡는다. WebSocket 자체엔 자동 재연결·pub/sub이 없어 애플리케이션이 얹어야 한다.

③ Kafka — 파티션 · 오프셋 · 복제(ISR) · exactly-once

producer는 메시지를 topic의 한 파티션에 append한다(key 해시 또는 라운드로빈). 각 파티션은 offset 순서가 보장된 로그이고, 순서 보장은 파티션 내부에서만 성립한다. 소비자는 pull 방식으로 읽고, 어디까지 읽었는지를 __consumer_offsets에 커밋한다.

항목동작면접 포인트
consumer group파티션을 그룹 멤버에게 분배(파티션 1개 = 소비자 1명)병렬성 상한 = 파티션 수. 소비자가 파티션보다 많으면 잉여는 논다
replication / ISR파티션을 RF개 복제, leader가 읽기·쓰기, ISR은 leader를 따라잡은 복제본acks=all + min.insync.replicas=2면 ISR 2개 이상 반영돼야 쓰기 성공 → 무손실
log compaction시간·크기 만료 대신 key별 최신 값만 보존"현재 상태" 토픽(예: 사용자 프로필)에 적합. 삭제는 tombstone(null value)
exactly-once멱등 producer(중복 쓰기 제거) + 트랜잭션(원자적 read-process-write)producer 재시도·consumer 재처리에도 중복 없음. transactional.id + isolation.level=read_committed
면접 포인트 — 오해 정리

Kafka의 순서 보장은 파티션 내부만이라 전역 순서가 필요하면 파티션 1개(=병렬성 포기) 또는 key로 순서 단위를 묶어야 한다. acks=all 단독으로는 무손실이 아니며 min.insync.replicas와 함께여야 의미가 있다.

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록, 반드시 그릴 비교 그림 · 파고드는 추가 설명 · 정의할 핵심 용어 · 아는지 확인하는 체크리스트를 한 곳에 모았다.

① 반드시 그리는 비교

"MQTT·WebSocket·Kafka 차이를 그려봐"라고 하면 이 그림이다. 세 축 — 토폴로지 / 전달 보장 / 언제로 한 장에 정리한다.

② 추가 설명 — 면접관이 파고드는 지점

  • MQTT QoS는 end-to-end가 아니다 — 보장은 client↔broker 한 홉 단위이고, 전달 QoS는 publish·subscribe의 최솟값이다. QoS 2도 broker를 두 홉 거치면 각 홉에서 따로 보장된다.
  • WebSocket은 채널만 준다 — 자동 재연결·pub/sub·메시지 보관이 없다. 이런 게 필요하면 SocketIO 같은 라이브러리나 앱 레이어가 얹는다. 조용한 절단은 ping/pong으로만 잡힌다.
  • Kafka 순서·병렬성의 트레이드오프 — 순서는 파티션 내부만, 병렬성 상한은 파티션 수. 전역 순서를 원하면 파티션 1개로 병렬성을 포기하거나 key로 순서 단위를 좁힌다.
  • Kafka 무손실의 조건 — acks=all은 단독으로 부족하고 min.insync.replicas ≥ 2와 결합해야 한다. exactly-once는 멱등 producer + 트랜잭션 + read_committed의 조합이다.

③ 반드시 정의할 핵심 용어

용어면접용 한 줄 정의
QoS 0/1/2at most / at least / exactly once. 2는 PUBREC·PUBREL·PUBCOMP 4-way. 전달 QoS = min(pub, sub).
retained / LWTtopic 마지막 값 보관(늦은 구독자용) / 비정상 절단 시 broker가 발행하는 유언. 둘을 묶어 생존 상태.
Upgrade / 101HTTP로 시작해 101 Switching Protocols 후 프레임 채널로 전환. Accept = SHA-1(Key+GUID).

④ 반드시 아는 개념 체크리스트

  1. 세 프로토콜의 계층 — 디바이스(MQTT) / 브라우저(WebSocket) / 서비스 backbone(Kafka), 대체 아닌 보완.
  2. MQTT QoS 3단계 — 0/1/2 핸드셰이크, 전달 QoS = min(pub, sub), 한 홉 단위.
  3. retained + LWT 패턴 — 생존 상태(online/offline retain) 추적.
  4. keepalive — PINGREQ/PINGRESP, 약 1.5배 유예로 죽은 연결 감지.
  5. WebSocket 핸드셰이크 — Upgrade → 101 → Sec-WebSocket-Accept, 이후 프레임.
  6. 프레임·opcode·마스킹 — text/binary/close/ping/pong, client→server 마스킹 필수.
  7. WebSocket vs 폴링/SSE — 양방향 지속 연결 vs 단방향/반복 요청.
  8. Kafka 파티션·오프셋·consumer group — 순서(파티션 내)·병렬성 상한(파티션 수).
  9. 복제·ISR·acks — RF, ISR, acks=all + min.insync.replicas 무손실.
  10. log compaction·exactly-once — key별 최신 값 보존 / 멱등 producer + 트랜잭션.

06예상 질문 & 답변난이도 필수

MQTT QoS 0·1·2의 차이와 각각 언제 쓰나?
QoS 0은 at most once(확인 없음, 유실 가능) — 고빈도 센서처럼 한두 개 놓쳐도 되는 값. QoS 1은 at least once(PUBACK 확인, 중복 가능) — 대부분의 명령·이벤트. QoS 2는 exactly once(PUBREC/PUBREL/PUBCOMP 4-way) — 중복이 치명적인 결제·과금. 전달 QoS는 publish와 subscribe의 최솟값이고 보장은 client↔broker 한 홉 단위임을 덧붙이면 좋다.
retained message와 LWT는 각각 무엇이고 왜 함께 쓰나?
retained는 broker가 topic의 마지막 값을 보관해 늦게 접속한 구독자에게 즉시 현재 상태를 준다. LWT는 클라이언트가 비정상 절단될 때 broker가 대신 발행하는 유언이다. 둘을 묶는 표준 패턴: 접속 직후 online(retain) 발행 + LWT로 offline(retain) 예약 → 장애로 끊겨도 broker가 offline을 남겨, 누구나(지금·나중 구독자 모두) 정확한 생존 상태를 본다.
WebSocket 연결은 어떻게 수립되나?
HTTP로 시작한다. 클라이언트가 Upgrade: websocket·Connection: Upgrade와 랜덤 Sec-WebSocket-Key를 담아 GET을 보내고, 서버가 101 Switching Protocols로 수락하며 Key에 매직 GUID를 붙여 SHA-1·base64한 Sec-WebSocket-Accept를 회신한다. 이후 같은 TCP가 프레임 채널이 되어 HTTP를 벗어나 양방향으로 통신한다. 같은 포트(80/443)로 HTTP와 공존하도록 설계된 점이 핵심이다.
WebSocket을 HTTP 폴링·SSE 대신 쓰는 이유는?
폴링은 클라이언트가 반복 요청해 지연·오버헤드가 크고, SSE는 서버→클라이언트 단방향이다. WebSocket은 하나의 지속 연결로 full-duplex 저지연 양방향이라 채팅·협업·게임처럼 양쪽이 수시로 밀어야 할 때 유리하다. 단, 자동 재연결·pub/sub·메시지 보관이 없어 앱이 얹어야 하고, 단순 서버 푸시만이면 SSE가 더 가벼울 수 있다.
Kafka에서 partition과 consumer group의 관계, 그리고 병렬성 상한은?
topic은 여러 파티션으로 쪼개지고, 한 파티션은 그룹 내 소비자 1명에게만 배정된다. 따라서 그룹의 소비 병렬성 상한은 파티션 수다. 소비자가 파티션보다 많으면 잉여 소비자는 논다. 순서 보장은 파티션 내부에서만 성립하므로, 순서가 필요한 이벤트는 같은 key로 같은 파티션에 모은다.
Kafka에서 메시지 유실을 막으려면? ISR·acks의 관계는?
각 파티션은 여러 broker에 복제(RF)되고, ISR은 leader를 따라잡은 복제본 집합이다. producer에 acks=all을 주면 ISR에 반영돼야 쓰기가 성공한다. 다만 acks=all 단독으론 부족하고 min.insync.replicas=2와 함께여야, ISR이 부족하면 쓰기를 거부해 무손실을 보장한다. consumer 쪽은 처리 후 offset 커밋으로 유실을 막는다.
Kafka의 exactly-once는 어떻게 달성하나?
두 조각이다. 멱등 producer(enable.idempotence)가 PID+시퀀스로 재시도 시 파티션 내 중복 쓰기를 제거하고, 트랜잭션(transactional.id)이 read-process-write와 offset 커밋을 원자적으로 묶는다. consumer는 isolation.level=read_committed로 커밋된 것만 읽는다. producer 재시도·consumer 재처리에도 중복 없이 정확히 1회가 되는 이유다.
MQTT · WebSocket · Kafka 중 무엇을 언제 고르나?
MQTT는 제약된 디바이스·저대역 무선의 텔레메트리·제어(broker pub/sub). WebSocket은 브라우저↔서버 실시간 양방향(채팅·대시보드). Kafka는 서비스 간 대용량 이벤트 백본·재생·스트림 처리. 대체재가 아니라 계층이 달라, 실제로는 현장 MQTT → 백엔드 Kafka → 사용자 WebSocket처럼 이어 붙는 경우가 많다.
