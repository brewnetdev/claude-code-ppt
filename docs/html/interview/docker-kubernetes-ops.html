00컨테이너·오케스트레이션이란? · 왜난이도 쉬움

Docker는 앱과 의존성을 하나의 이미지로 묶어 어디서나 동일하게 실행하는 컨테이너 도구다. 컨테이너는 VM과 달리 호스트 커널을 공유(namespace·cgroup 격리)해 가볍고 빠르다 — "내 로컬에선 되는데" 문제를 없앤다.

Kubernetes(K8s)는 수많은 컨테이너를 여러 노드에 걸쳐 선언적으로 배치·확장·복구하는 오케스트레이터다. "원하는 상태(desired state)"를 선언하면, 컨트롤러가 실제 상태를 거기에 맞춰 계속 조정(reconcile)한다.

문제DockerKubernetes
일관된 실행 환경이미지로 패키징 → 어디서나 동일—
다수 컨테이너 배치·확장단일 호스트 위주다수 노드 스케줄링·오토스케일
장애 자동 복구재시작 정책 수준desired-state 조정으로 자가 치유
무중단 배포수동롤링 업데이트·롤백 내장
한 줄 정리

Docker = "앱을 이미지로 격리·패키징". Kubernetes = "그 컨테이너들을 선언적으로 배치·확장·치유하는 조정 시스템".

02클러스터 아키텍처난이도 중간

클러스터는 Control Plane(두뇌)과 Worker Node(일꾼)로 나뉜다. 외부 트래픽은 Ingress/Gateway → Service → Pod로 흐른다.

03동작 원리 — 이미지 · 조정 루프난이도 중간

이미지 레이어 & 빌드

  • 레이어 캐시 — Dockerfile 명령마다 레이어. 자주 안 바뀌는 것(의존성 설치)을 위로, 자주 바뀌는 것(소스 복사)을 아래로 배치해 캐시 적중률↑.
  • 멀티스테이지 빌드 — 빌드 스테이지에서 컴파일 후, 런타임 스테이지에 산출물만 복사 → 최종 이미지 슬림·공격면 축소.

선언적 조정 루프(reconcile)

  • 사용자가 desired state(예: replicas=3)를 apiserver에 선언 → etcd에 저장.
  • controller가 actual(현재 Pod 수)과 desired를 비교, 부족하면 생성·초과면 삭제 → 계속 반복.
  • scheduler가 새 Pod를 리소스·제약(affinity·taint)에 맞는 노드에 배치, kubelet이 실행.
  • 노드·Pod 장애 시 조정 루프가 자동으로 재생성 → 자가 치유.

롤링 업데이트 & 헬스체크

Deployment는 새 ReplicaSet을 점진적으로 늘리고 옛 것을 줄인다(maxSurge/maxUnavailable). readiness 프로브가 통과해야 트래픽을 받고, 실패 시 롤아웃이 멈춰 안전하다. 문제 시 kubectl rollout undo로 롤백.

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록 반드시 그릴 그림 · 파고드는 설명 · 정의할 용어 · 체크리스트를 모았다.

① 반드시 그리는 그림 — 조정 루프

② 추가 설명 — 면접관이 파고드는 지점

  • 왜 "선언적"인가 — 명령형(이걸 실행해)이 아니라 "이 상태가 되게 하라". 컨트롤러가 어떤 경로로든 그 상태로 수렴시켜 장애에 강하다.
  • Deployment vs StatefulSet — 무상태·교체 가능(Deployment) vs 안정적 이름·순서·영속볼륨(StatefulSet, DB류).
  • liveness vs readiness — liveness 실패=컨테이너 재시작, readiness 실패=트래픽만 차단(재시작 안 함). 혼동은 롤아웃 사고로 이어진다.
  • requests vs limits — requests=스케줄링·보장, limits=상한(초과 시 CPU throttle·메모리 OOMKill).

③ 반드시 정의할 핵심 용어

06예상 질문 & 답변난이도 필수

컨테이너와 VM의 차이는?
VM은 하이퍼바이저 위에 게스트 OS 전체를 띄워 강하게 격리하지만 무겁다. 컨테이너는 호스트 커널을 공유하고 namespace·cgroup으로 격리해 가볍고 빠르게 뜬다. 대신 커널 수준 격리는 VM이 강하다.
Deployment와 StatefulSet은 언제 각각 쓰나?
교체 가능한 무상태 워크로드(웹 API)는 Deployment. 안정적 네트워크 이름·순서·개별 영속 볼륨이 필요한 상태 워크로드(DB, Kafka)는 StatefulSet. 후자는 Pod 이름이 -0,-1처럼 고정된다.
liveness와 readiness 프로브의 차이는?
liveness 실패 → 컨테이너를 재시작(데드락 복구). readiness 실패 → Service 트래픽에서 제외(재시작 안 함, 준비될 때까지 대기). 초기화가 긴 앱은 startup 프로브로 초기 구간을 분리한다.
requests와 limits를 어떻게 설정하나?
requests는 스케줄러가 노드 배치에 쓰는 보장치, limits는 상한이다. CPU 초과는 throttle, 메모리 초과는 OOMKill. requests를 실측 기반으로 잡고 limits는 여유를 두되, 메모리는 requests≈limits로 안정화하는 편이 흔하다.
무중단 롤링 업데이트와 롤백은 어떻게 되나?
Deployment가 새 ReplicaSet을 maxSurge만큼 늘리고 옛 것을 maxUnavailable만큼 줄이며 교체한다. readiness가 통과해야 트래픽을 받으므로 안전하고, 문제 시 kubectl rollout undo로 이전 리비전으로 되돌린다.
이미지를 작고 안전하게 만들려면?
멀티스테이지 빌드로 빌드 산출물만 슬림 런타임(distroless·alpine)에 복사하고, 레이어 순서를 캐시 친화적으로(의존성 먼저) 배치한다. 비루트 사용자로 실행하고, 태그 대신 digest 고정으로 재현성을 확보한다.
