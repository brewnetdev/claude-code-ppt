00ROS2란? · 왜 만들어졌나난이도 쉬움

ROS2(Robot Operating System 2)는 이름과 달리 운영체제가 아니라, 로봇 소프트웨어를 만드는 미들웨어 + 툴 + 라이브러리 생태계다. 센서 드라이버·인지(perception)·계획(planning)·제어(control)를 각각 독립 프로세스(node)로 짜고, 표준 메시지로 pub/sub·요청/응답을 주고받아 하나의 로봇 시스템으로 엮는 것이 핵심 패러다임이다.

ROS2는 ROS1(2007~, 최종 Noetic)의 점진적 업그레이드가 아니라 바닥부터 재설계(ground-up rearchitecture)다. ROS1이 산업 현장에서 막혔던 지점이 곧 ROS2의 존재 이유다.

ROS1의 한계ROS2의 해법
중앙 rosmaster가 discovery 중개 → 죽으면 전체 마비(SPOF)DDS 분산 discovery — master 제거, node끼리 P2P로 발견
자체 TCPROS/UDPROS → 무선·실시간 제어에 부적합, 품질 제어 불가DDS + QoS — reliable/best-effort 등 통신 품질을 정책으로 튜닝
실시간성·보안 전무(암호화·인증 없음)Executor 정비 + DDS-Security(인증·암호화)
roscpp/rospy 기능 불일치, Linux 전용공통 C 코어 rcl 위 언어 바인딩 · Linux/Win/macOS·micro-ROS
한 줄 정리

ROS2 = "중앙 master를 없애고 DDS를 통신 백본으로 채택해, 분산 discovery·QoS·실시간·보안·멀티플랫폼을 얻은 로봇 미들웨어."

02아키텍처 — 계층 스택난이도 중간

ROS2는 위(내 코드)에서 아래(네트워크)로 내려가는 수직 계층 스택이다. 각 계층이 아래 계층의 세부를 숨기기 때문에, DDS 벤더를 코드 수정 없이 교체할 수 있다.

  • rclcpp / rclpy — 실제 쓰는 언어별 API(create_publisher, spin). executor·parameter·lifecycle·action 래핑이 여기 있다.
  • rcl — node·topic 등 핵심 로직의 언어 독립 C 구현. 각 언어 라이브러리가 로직을 중복 구현하지 않게 해 rclcpp/rclpy를 얇고 일관되게 유지한다.
  • rmw — DDS 벤더 추상화. RMW_IMPLEMENTATION=rmw_cyclonedds_cpp처럼 런타임에 교체(기본 Fast DDS rmw_fastrtps_cpp).
  • DDS / RTPS — 실제 통신 엔진. 같은 호스트면 shared-memory, 아니면 UDP.
면접 포인트 — Discovery

ROS2엔 중앙 master가 없다. 각 프로세스가 DDS Participant가 되어 SPDP(participant 발견)·SEDP(endpoint·topic·QoS 교환)로 서로를 멀티캐스트로 찾아 P2P 연결한다. 발견 범위는 ROS_DOMAIN_ID로 격리한다.

034대 통신 — Node · Topic · Service · Action난이도 중간

Node는 역할 하나에 대응하는 논리적 연산 단위다(/lidar_driver, /motion_controller). node들은 세 가지 통신 방식으로 엮이며, 여기에 설정값을 담는 Parameter가 더해진다.

① Topic — Publish/Subscribe

연속 스트림용. 송신자는 수신 여부를 모르고(fire-and-forget), N:M 가능. 완전 비동기. 예: LiDAR가 /scan을 10~40Hz로 publish → SLAM과 obstacle node가 동시에 subscribe. 속도 명령 /cmd_vel.

② Service — Request/Reply

짧고 즉시 끝나는 질의/명령(RPC). client가 request → server가 응답 1개 반환. 진행 피드백·취소 불필요할 때. 호출은 async 권장(call_async) — 콜백 안에서 같은 executor로 동기 대기하면 deadlock(§05).

③ Action — Goal / Feedback / Result

오래 걸리는(초~분) 작업용. goal 전송 → 진행하며 feedback 스트림 → 완료 시 result, 도중 취소/선점 가능. 내부는 3개 service(send_goal·cancel_goal·get_result) + 2개 topic(feedback·status)로 구성된다(면접 단골). 예: NavigateToPose, MoveIt 팔 궤적, 도킹.

④ Parameters

node가 소유한 typed 설정값. ROS1의 전역 param 서버와 달리 파라미터는 각 node 안에 산다. declare_parameter()로 선언, launch YAML로 주입, add_on_set_parameters_callback으로 검증. 예: max_speed, PID 게인.

결정 트리

연속 스트림·다수 구독자 → Topic / 짧은 즉답 질의·명령 → Service / 장시간·피드백·취소 → Action / 재구성 가능한 설정값 → Parameter.

04QoS — 통신 품질과 mismatch 함정난이도 중간

QoS(Quality of Service)는 각 pub/sub·service의 통신 품질 정책 묶음(profile)이다. DDS 덕분에 ROS2는 TCP만큼 신뢰하게도, UDP만큼 best-effort하게도 튜닝된다.

정책값언제
ReliabilityRELIABLE(재전송 보장) · BEST_EFFORT(유실 허용)명령/설정은 reliable, 고속 센서·무선은 best-effort
DurabilityVOLATILE · TRANSIENT_LOCAL(늦게 온 구독자도 마지막 값 수신)정적 지도·TF static·latched 데이터는 transient_local
History+DepthKEEP_LAST(최근 N) · KEEP_ALL버퍼 크기 — 호환성엔 영향 없음
Deadline / Liveliness / Lifespan주기 감시 · 생존 확인 · 샘플 만료센서 stall 감지, publisher 사망 감지

"QoS mismatch = 조용한 두절" (최다 실전 함정)

매칭은 Requested(subscriber) vs Offered(publisher) 모델이다. subscriber의 요구가 publisher의 제공보다 엄격하면 연결 자체가 안 되고, 에러 없이 데이터가 0이 된다. 실무 90%는 reliability·durability에서 터진다.

제공 ≥ 요구여야 한다(reliable ≥ best_effort, transient_local ≥ volatile). 대표 사고: best-effort로 publish하는 카메라를 기본(reliable) subscriber로 받으면 아무것도 안 들어온다. 진단은 ros2 topic info -v로 양쪽 QoS 대조.

05실행 모델 — Executor · Callback Group · Lifecycle난이도 중간

node 자체는 콜백을 실행하지 않는다. Executor가 spin하며 준비된 콜백을 실행한다. 여기서 스레딩·race·deadlock이 갈린다.

Executor — 콜백 스케줄러

  • SingleThreadedExecutor(기본) — 한 스레드에서 콜백을 하나씩. 결정적이지만 긴 콜백이 나머지를 막는다.
  • MultiThreadedExecutor — 스레드 풀로 콜백 병렬. 단, 병렬 여부는 callback group이 결정한다.

Callback Group — 병렬성·race 제어의 핵심

  • MutuallyExclusive(기본) — 그룹 내 콜백 동시 실행 금지(직렬화).
  • Reentrant — 그룹 내 콜백 병렬 허용(같은 콜백의 중복 인스턴스도 동시 가능).
핵심 함정

node의 모든 콜백이 기본(하나의 mutually-exclusive) 그룹이면, MultiThreadedExecutor를 켜도 직렬 실행된다. 병렬을 원하면 reentrant나 여러 개의 그룹을 명시 배정해야 한다. 공유 자원을 만지는 콜백은 같은 mutually-exclusive 그룹에 묶어 race를 원천 차단한다.

Lifecycle (Managed) Nodes

초기화·종료를 명시적 상태 머신으로 통제한다: Unconfigured → Inactive → Active → Inactive → Finalized. Inactive는 리소스는 잡되 publish·콜백은 비활성 → 결정적 순차 bring-up에 쓴다. 대표적으로 Nav2 Lifecycle Manager가 서버들을 순차 configure→activate하고, 하나라도 실패하면 전체 activate를 막는다.

colcon · Overlay

colcon build --symlink-install로 빌드. 시스템 설치본(underlay, /opt/ros/<distro>) 위에 내 workspace(overlay)를 source install/setup.bash하면 덧씌워지고, 같은 패키지는 overlay 우선이다.

06ROS2 ↔ Backend 연동 구조난이도 중간→필수

로봇은 ROS2/DDS 도메인(LAN·P2P·실시간)에서 돌고, 웹 대시보드·API·DB·fleet 관제는 클라우드 백엔드(인터넷·REST·영속화)에 있다. DDS는 인터넷 너머로 그대로 못 나가므로, 둘 사이에 bridge(브리지)를 둔다.

연동 시 반드시 고려할 것

  • 대역폭 — 이미지·포인트클라우드는 그대로 보내지 말고 압축·다운샘플·해상도 조정. 고빈도 topic은 throttle.
  • QoS ↔ 전송 신뢰성 매핑 — ROS2 reliable을 TCP/WS의 신뢰 전송에, best-effort를 손실 허용 경로에 대응. 브리지가 QoS를 삼키지 않는지 확인.
  • 보안 — 온보드는 DDS-Security(인증·암호화), 브리지 밖 구간은 TLS/mTLS + 인증 토큰. 로봇을 공용 인터넷에 그대로 노출 금지.
  • 시간 동기 — 로봇·백엔드 타임스탬프 정합을 위해 NTP(또는 정밀 제어는 PTP).
  • back-pressure · 오프라인 — 네트워크 단절 시 온보드 버퍼링·재전송 큐, 백엔드 과부하 시 drop 정책.

08면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록, ROS2에서 반드시 그릴 아키텍처 · 파고드는 추가 설명 · 정의할 핵심 용어 · 아는지 확인하는 체크리스트를 한 곳에 모았다.

① 반드시 그리는 아키텍처

"ROS2 노드 둘이 통신하는 걸 그려봐"라고 하면 이 그림이다. 세 가지가 핵심 — master 없는 DDS P2P, 연결마다 붙는 QoS(제공 ≥ 요구), 각 노드 콜백을 도는 Executor + Callback Group.

② 추가 설명 — 면접관이 파고드는 지점

  • 왜 master를 없앴나 — rosmaster는 SPOF였다. DDS 분산 discovery(SPDP/SEDP)로 P2P 연결해 부분 장애 내성·동적 확장을 얻고, QoS·보안까지 표준으로 딸려온다.
  • QoS mismatch는 왜 "조용히" 실패하나 — 매칭 실패 시 예외가 아니라 연결 자체가 안 맺어져 데이터가 0이 된다. best-effort publisher ↔ reliable subscriber가 대표 사고. 규칙은 "제공 ≥ 요구", 진단은 ros2 topic info -v.
  • MultiThreadedExecutor인데 왜 병렬이 안 되나 — 노드 기본 callback group이 mutually-exclusive라 직렬 실행된다. 병렬은 reentrant나 별도 그룹으로 명시 설계해야 하고, 공유 자원 콜백은 같은 mutually-exclusive 그룹에 묶어 race를 막는다.
  • Service를 콜백에서 동기 대기하면 — 같은 executor가 응답 콜백을 못 돌려 deadlock. call_async 또는 호출부/응답부를 다른 callback group에 둔다.

③ 반드시 정의할 핵심 용어

④ 반드시 아는 개념 체크리스트

  1. Topic vs Service vs Action — 스트림 / 즉답 RPC / 장시간+피드백+취소. action = 내부 3 service + 2 topic.
  2. DDS 분산 discovery — master 없음, RTPS SPDP/SEDP, ROS_DOMAIN_ID 격리, P2P.
  3. QoS 호환성 — reliability·durability 매칭(offered ≥ requested), mismatch = 조용한 두절.
  4. Executor + Callback Group — 기본 그룹이 병렬을 막는 함정, race 방지.
  5. ROS1 vs ROS2 — master 제거, TCPROS→DDS, QoS·실시간·보안, 통합 rcl, 멀티플랫폼.
  6. rmw/rcl 계층 — 벤더 스왑(RMW_IMPLEMENTATION), Fast DDS 기본.
  7. Lifecycle nodes — 상태 머신, 순차 bring-up(Nav2 Lifecycle Manager).
  8. ROS2 ↔ Backend 연동 — rosbridge(WS)·mqtt_client·zenoh/DDS Router·gRPC 게이트웨이.
  9. tf2 — 시간버퍼 좌표 변환 트리, REP-105의 map → odom → base_link.
  10. Nav2 / MoveIt2 — 자율주행(costmap·AMCL·NavigateToPose) / 매니퓰레이터 모션 플래닝(IK·충돌회피).

09예상 질문 & 답변난이도 필수

Topic과 Action은 각각 언제 쓰나?
Topic은 연속 스트림 + 다수 구독자 + 응답 불필요(센서 데이터, /cmd_vel). Action은 오래 걸리고(초~분) + 진행 피드백 + 중간 취소/선점이 필요할 때(웨이포인트 이동, 팔 궤적). 짧은 즉답이면 action이 아니라 service.
QoS mismatch로 통신이 안 될 때 원인과 진단은?
subscriber 요구가 publisher 제공보다 엄격하면 매칭 실패로 에러 없이 데이터 0. 최다 케이스는 best-effort publisher(카메라 SensorDataQoS) ↔ reliable subscriber(기본). 규칙은 "제공 ≥ 요구". ros2 topic info -v로 양쪽 실제 QoS를 비교해 맞춘다.
ROS2가 rosmaster를 없앤 이유는?
rosmaster는 모든 discovery를 중개하는 단일 장애점이라 죽으면 새 연결이 전부 불가했다. ROS2는 DDS 분산 discovery(RTPS SPDP/SEDP)로 중앙 조정자 없이 P2P로 서로를 발견한다. 부분 장애 내성·동적 확장·자동 발견을 얻고, ROS_DOMAIN_ID로 논리 격리·QoS·보안까지 표준으로 딸려온다.
MultiThreadedExecutor에서 콜백 race를 어떻게 막나?
callback group으로 통제한다. 공유 상태를 만지는 콜백들을 같은 MutuallyExclusiveCallbackGroup에 배정하면 executor가 그 그룹을 직렬화해 동시 접근이 없다. 병렬이 필요한 독립 콜백만 별도 그룹/reentrant에 둔다. 주의: 기본 그룹은 mutually-exclusive라, 아무 지정 없이 multi-threaded만 켜면 실제로는 직렬 실행된다.
Service를 콜백 안에서 동기로 호출하면 왜 deadlock이 나나?
single-thread(또는 하나의 mutually-exclusive 그룹)에서 콜백 안에서 service를 보내고 같은 executor로 응답을 spin-wait하면, 응답 콜백을 돌릴 스레드가 그 콜백에 붙잡혀 영원히 대기한다. 해결: call_async + future 콜백, 또는 호출부/응답부를 다른 callback group에 두고 multi-threaded 사용.
로봇의 실시간 상태를 웹 대시보드로 어떻게 내보내나?
온보드에 bridge node를 둔다. 웹이면 rosbridge_server(WebSocket+JSON)에 roslibjs로 붙어 topic을 구독하거나, 관측·디버깅은 foxglove_bridge. 고빈도·대용량(카메라)은 다운샘플·압축·best-effort로 대역폭을 줄이고, 브리지 밖 구간은 TLS로 감싼다. 멀티로봇 관제는 zenoh-bridge-ros2dds/DDS Router로 WAN을 잇는다.
Lifecycle node는 일반 node와 뭐가 다르고 언제 쓰나?
일반 node는 생성 즉시 동작하지만, lifecycle node는 Unconfigured→Inactive→Active 상태 머신을 거쳐 리소스 초기화와 동작 시점을 분리한다. Inactive에선 리소스는 잡되 publish·콜백은 비활성이라 결정적 순차 부팅이 된다. 다수 node를 정해진 순서로 켜야 하는 시스템(대표적으로 Nav2)에서 쓴다.
