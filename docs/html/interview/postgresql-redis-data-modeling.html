00PostgreSQL·Redis란? · 왜 함께난이도 쉬움

PostgreSQL은 ACID 트랜잭션·풍부한 타입(jsonb·array)·강력한 인덱스·MVCC를 갖춘 관계형 데이터베이스(RDBMS)다. 데이터의 영속적 단일 진실 소스 역할을 한다.

Redis는 메모리에 자료구조(string·hash·list·set·sorted set·stream 등)를 두는 인메모리 데이터 스토어다. 마이크로초 단위 응답으로 캐시·세션·랭킹·큐·rate limit 등에 쓴다.

강점트랜잭션·조인·복잡 쿼리·정합성초저지연·단순 조회·TTL·원자 연산

역할durable source of truthhot data 캐시 계층
왜 함께 쓰나

읽기 대부분을 Redis가 흡수해 지연·DB 부하를 낮추고, 정합성이 중요한 진실은 PostgreSQL에 둔다. 이 조합의 핵심 패턴이 cache-aside다.

02아키텍처 — cache-aside · 데이터 계층난이도 중간

가장 흔한 조합은 cache-aside다. 앱이 먼저 Redis를 보고, 없으면 PostgreSQL에서 읽어 Redis에 채운 뒤 반환한다.

03동작 원리 — MVCC · persistence난이도 중간

PostgreSQL — MVCC & 격리수준

  • MVCC — 행마다 여러 버전(xmin/xmax). 읽기는 스냅샷을 보고, 읽기가 쓰기를 막지 않는다. 죽은 튜플은 VACUUM이 정리.
  • 격리수준 — Read Committed(기본, 매 문장 최신 커밋 스냅샷) · Repeatable Read(트랜잭션 스냅샷 고정, PG는 phantom도 방지) · Serializable(SSI로 직렬화 이상까지 차단).
  • 이상현상 — dirty read(비커밋 읽기)·non-repeatable read(같은 행 값 바뀜)·phantom(조건 결과 집합 바뀜). 상위 격리일수록 차단↑, 비용↑.
  • 인덱스·플랜 — B-tree 기본, jsonb·배열·전문검색은 GIN, 거대 정렬 데이터는 BRIN. EXPLAIN (ANALYZE)로 실제 플랜 확인.
  • WAL — 변경을 먼저 로그에 써 내구성·crash recovery·스트리밍 복제를 지원.

Redis — 단일 스레드 & persistence

  • 단일 스레드 명령 실행 — 명령이 직렬 처리라 개별 명령이 원자적. 무거운 명령(KEYS *) 하나가 전체를 막으니 금지.
  • persistence — RDB(스냅샷, 콤팩트·빠른 복구, 마지막 스냅 이후 유실) vs AOF(쓰기 로그, 내구성↑·파일 큼). 둘 병행도 가능.
  • eviction — maxmemory 도달 시 정책(allkeys-lru/lfu, volatile-*)으로 축출. 캐시 용도는 LRU/LFU.
  • replication — primary→replica 비동기 복제로 읽기 확장. Sentinel/Cluster로 가용성·샤딩.

05면접 핵심 정리 — 반드시 아는 것난이도 필수

면접 직전 이 섹션만 봐도 되도록 반드시 그릴 그림 · 파고드는 설명 · 정의할 용어 · 체크리스트를 모았다.

① 반드시 그리는 그림 — 격리수준과 이상현상

② 추가 설명 — 면접관이 파고드는 지점

  • MVCC가 왜 좋은가 — 읽기가 쓰기를, 쓰기가 읽기를 막지 않아 동시성이 높다. 대신 죽은 튜플이 쌓여 VACUUM·bloat 관리가 필요하다.
  • 캐시 무효화가 왜 어려운가 — DB와 캐시가 별도라 갱신 순서·실패에 따라 stale이 남는다. 보통 DB 먼저 쓰고 캐시 무효화가 안전하다.
  • cache stampede — 인기 키가 동시에 만료되면 수많은 miss가 DB로 몰린다. TTL 지터·뮤텍스(한 요청만 재계산)로 막는다.
  • RDB vs AOF — 빠른 복구·콤팩트=RDB, 내구성=AOF. 실무는 둘 병행 + AOF everysec가 흔하다.

③ 반드시 정의할 핵심 용어

06예상 질문 & 답변난이도 필수

트랜잭션 격리수준과 각 이상현상을 설명하라.
Read Committed(기본)는 dirty read만 차단, Repeatable Read는 non-repeatable read까지(PostgreSQL은 스냅샷 격리로 phantom도), Serializable은 직렬화 이상까지 차단한다. 위로 갈수록 안전하지만 충돌·재시도·비용이 는다.
MVCC란 무엇이고 장단점은?
행의 여러 버전을 유지해 읽기가 쓰기를 막지 않는 동시성 제어다. 높은 동시성이 장점이고, 죽은 튜플이 쌓여 VACUUM과 테이블 bloat 관리가 필요한 것이 단점이다.
인덱스가 있는데도 쿼리가 느리면?
EXPLAIN (ANALYZE)로 실제 플랜을 본다. 인덱스 미사용(형변환·함수 래핑·저선택도), 통계 낡음(ANALYZE), 정렬/조인 방식, 복합 인덱스 컬럼 순서를 점검한다. 필요하면 부분·표현식·커버링 인덱스를 쓴다.
cache-aside와 write-through의 차이는?
cache-aside는 읽을 때만 캐시를 채우고(lazy), 쓰기는 DB 갱신 후 캐시를 무효화한다 — 단순하고 흔하다. write-through는 쓰기 시 캐시와 DB를 함께 갱신해 정합성이 높지만 쓰기 지연이 늘고, 안 읽힐 데이터도 캐싱될 수 있다.
캐시 무효화·스탬피드는 어떻게 다루나?
쓰기는 DB 먼저 갱신 후 캐시 키 무효화가 안전하다(캐시 먼저 갱신은 실패 시 stale). 인기 키 동시 만료로 인한 stampede는 TTL에 지터를 주고, 재계산은 뮤텍스로 한 요청만 하도록 하며, 계산 중엔 기존 stale 값을 제공한다.
Redis persistence는 RDB와 AOF 중 무엇을 쓰나?
RDB는 스냅샷이라 복구가 빠르고 파일이 작지만 마지막 스냅 이후가 유실될 수 있다. AOF는 쓰기 로그라 내구성이 높지만 파일이 크다. 실무는 둘을 병행하고 AOF fsync를 everysec로 두는 절충이 흔하다.
