PROLOGUE · 왜 디자인 패턴인가
반복되는 강결합을 풀어내는 설계 언어
간결하고 유지보수하기 쉬운 코드 — 확장 가능성과 재사용성을 높이는 약속
자바의 디자인 패턴은 프로그래밍 시 반복되거나 강결합된 코드로 인한 상호 영향을 최소화하기 위한 설계 패턴이다. 주 목적은 간결하고 유지보수하기 쉬운 코드를 작성하는 것이며, 잘 설계된 디자인 패턴은 프로그램을 더욱 확장 가능하고 재사용성을 높여 준다.
🎯
이 챕터에서는 비교적 자주 쓰이는 Factory Method · Singleton을 비롯하여 Strategy · Template Method · Facade · Command 패턴을 소스 코드와 함께 짚어 본다.

패턴분류핵심 의도

PATTERN 01 · CREATIONAL
Factory Method
객체 생성을 추상화한다
구체 클래스에 의존하지 말고 추상에 의존하라.
객체 생성을 하위 클래스에 위임하여 결합도를 낮춘다.
PATTERN 01 · Factory Method SECTION
CH.7-1 · FACTORY METHOD
왜 new 키워드가 결합을 만드는가
DIP — Dependency Inversion Principle 위반
  • new B() 로 직접 생성하면 호출자가 구체 클래스 B에 결합된다B의 시그니처가 바뀌면 호출하는 모든 코드가 영향을 받는다.
  • Factory에 위임하면 클라이언트는 추상에만 의존한다Concrete class를 모르고도 인스턴스를 받을 수 있다.
  • 여러 종류의 인스턴스가 필요할 때 강력하다type 분기를 Factory 한 곳에 가두면 추가/변경 영향이 국소화된다.
💡
"객체 생성에 변경이 잦다"는 신호가 보이면 — Factory를 검토할 시점이다.
Factory Method · 동기와 원칙 03
FACTORY · BEFORE
if-else 분기 — 변경 비용이 누적된다
Car — 단일 클래스
public class Car {
    public static final String SMALL_CAR  = "small";
    public static final String SEDAN_CAR  = "sedan";
    public static final String LUXURY_CAR = "luxury";

    private String productName;

    public Car(String name) {
        this.productName = name;
    }

    @Override
    public String toString() {
        return "Car productName = '" + productName + "'";
    }
}
호출 — new + if/else
private Car createCar(String carName) {
    Car car;
    if (carName.equalsIgnoreCase(Car.SEDAN_CAR)) {
        car = new Car(Car.SEDAN_CAR);
    } else if (carName.equalsIgnoreCase(Car.SMALL_CAR)) {
        car = new Car(Car.SMALL_CAR);
    } else if (carName.equalsIgnoreCase(Car.LUXURY_CAR)) {
        car = new Car(Car.LUXURY_CAR);
    } else {
        car = new Car("");
    }
    return car;
}
⚠️
새 차종이 추가될 때마다 분기가 늘어나고, Car 시그니처 변경 시 호출 영역이 모두 깨진다.
Factory Method · Before — Tight Coupling 04
FACTORY · REFACTOR 1/3
Car를 abstract로 — 책임을 하위로 위임
Abstract Car
public abstract class Car {
    private CarType model;

    public Car(CarType model) {
        this.model = model;
        arrangePart();
    }

    protected void arrangePart() {
        System.out.println("Do one time processing here");
    }

    // 서브 클래스에서 이 메서드를 구현한다.
    protected abstract void construct();
}
CarType (enum)
public enum CarType {
    SMALL, SEDAN, LUXURY
}
Car는 직접 생성될 필요가 없으므로 abstract 로 추상화하고, 하위 클래스가 해야 할 작업의 골격만을 정의한다. 기존 상수 타입은 enum 으로 옮겨 타입 안정성을 얻는다.

Factory Method · Refactor 2 — Concrete Classes 06
FACTORY · FINAL 3/3
CarFactory — 생성을 가두고 클라이언트는 모른다
CarFactory
public class CarFactory {
    public static Car buildCar(CarType model) {
        Car car = null;
        // 실제 추가/변경 되는 부분
        switch (model) {
            case SMALL:
                car = new SmallCar();
                break;
            case SEDAN:
                car = new SedanCar();
                break;
            case LUXURY:
                car = new LuxuryCar();
                break;
            default:
                break;
        }
        return car;
    }
}
TestFactoryPattern
public class TestFactoryPattern {
    public static void main(String[] args) {
        System.out.println(CarFactory.buildCar(CarType.SMALL));
        System.out.println(CarFactory.buildCar(CarType.SEDAN));
        System.out.println(CarFactory.buildCar(CarType.LUXURY));
    }
}

PATTERN 02 · CREATIONAL
Singleton
단 하나의 인스턴스
어디서든 접근 가능한 단일 인스턴스를 보장한다.
환경설정 · 커넥션풀 같이 종료까지 살아있는 자원에 적합하다.
PATTERN 02 · Singleton SECTION
CH.7-2 · SINGLETON
전역 변수 없이 단일 인스턴스를 만든다
  • 단 하나의 인스턴스만 만들고 어디서든 접근하도록 보장불필요한 객체 생성을 방지하고 객체 간 결합도를 낮춘다.
  • 전역 변수 없이 클래스 단위 단일성을 확보전역 상태가 가지는 부작용 — 추적 곤란, 의존성 폭발 — 을 일부 회피.
  • 공용 자원의 수명 — 환경설정, 커넥션풀, 캐시프로그램 종료 시점까지 단 하나의 인스턴스만 생성·재사용·관리.
⚠️
멀티스레드 환경에서의 안전한 구현이 핵심이다 — 다음 슬라이드부터 4단계 진화를 살펴본다.
Singleton · Use Cases & Concerns 08
SINGLETON · STAGE 1
고전적 lazy — 멀티스레드에 깨진다
public class Singleton {
    private Singleton() {}
    private static Singleton uniqueInstance;

    public static Singleton getInstance() {
        if (uniqueInstance == null) {
            uniqueInstance = new Singleton();
        }
        return uniqueInstance;
    }
}
왜 위험한가
1번 스레드가 if (uniqueInstance == null) 을 통과한 직후 2번 스레드가 같은 분기에 진입하면 두 인스턴스가 동시에 생성된다.
🧨
synchronized 로 메서드 자체를 동기화하면 안전하지만, 호출마다 락이 걸려 오버헤드가 누적된다.
Singleton · Stage 1 — Race Condition 09
SINGLETON · STAGE 2-3
Eager 초기화 vs Double-Checked Locking
Eager — 클래스 로딩 시 생성
public class Singleton {
    private static Singleton uniqueInstance = new Singleton();
    private Singleton() {}

    public static Singleton getInstance() {
        return uniqueInstance;
    }
}
간단하지만 인스턴스 생성 비용이 비쌀 때 시작 지연이 발생한다.
DCL — Double Checked Locking
public class Singleton {
    private volatile static Singleton uniqueInstance;
    private Singleton() {}

    public static Singleton getInstance() {
        if (uniqueInstance == null) {
            synchronized (Singleton.class) {
                if (uniqueInstance == null) {
                    uniqueInstance = new Singleton();
                }
            }
        }
        return uniqueInstance;
    }
}
volatile 로 가시성을 확보하고 첫 진입에서만 락을 잡는다.
Singleton · Eager & DCL 10
SINGLETON · STAGE 4 ★
SingletonHolder — JVM이 보장하는 lazy + thread-safe
public class Singleton {
    private Singleton() {}

    private static class SingletonHolder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return SingletonHolder.INSTANCE;
    }
}
// 호출
Singleton singleton = Singleton.getInstance();
왜 안전한가
  • final한 번 초기화되면 재할당 불가.
  • static + Inner classSingletonHolder.INSTANCE 가 처음 참조될 때 비로소 클래스가 로드된다.
  • JVM Class Loader클래스 초기화 자체가 thread-safe — 별도 동기화 불필요.

Strategy
알고리즘을 갈아끼운다
기능을 인터페이스로 분리하고 클라이언트에서 호출 시점에 선택한다.

PATTERN 03 · Strategy SECTION
CH.7-3 · STRATEGY
런타임에 알고리즘을 교체한다
조건 분기로 흩어진 로직을 클래스로 응집
  • 알고리즘을 쉽게 교체조건 분기 대신 객체 주입 — 동일한 인터페이스로 다른 구현을 호출.
  • 클라이언트와 알고리즘 분리호출자는 어떤 구현이 동작하는지 알 필요가 없다.
  • 코드 재사용성새 전략을 추가해도 기존 코드는 변하지 않는다(OCP).
Strategy · Intent & Benefits 12
STRATEGY · INTERFACE + IMPL
CalculationStrategy + 4가지 사칙연산
Interface
public interface CalculationStrategy {
    public int calculate(int num1, int num2);
}
덧셈 / 뺄셈
public class AddOperation implements CalculationStrategy {
    @Override public int calculate(int a, int b) { return a + b; }
}

public class SubstractOperation implements CalculationStrategy {
    @Override public int calculate(int a, int b) { return a - b; }
}
곱셈 / 나눗셈
public class MultiplyOperation implements CalculationStrategy {
    @Override public int calculate(int a, int b) { return a * b; }
}

public class DivideOperation implements CalculationStrategy {
    @Override public int calculate(int a, int b) { return a / b; }
}
🧩
동일한 calculate(a,b) 시그니처가 4개 구현 클래스에 분산되어 있다.
Strategy · 4 Concrete Strategies 13
STRATEGY · CONTEXT + DEMO
호출 시점에 전략을 주입한다
Calculation (Context)
public class Calculation {
    private CalculationStrategy strategy;

    public Calculation(CalculationStrategy strategy) {
        this.strategy = strategy;
    }

    public int executeStrategy(int num1, int num2) {
        return strategy.calculate(num1, num2);
    }
}
Demo
public class CalculationDemo {
    public static void main(String[] args) {
        Calculation add = new Calculation(new AddOperation());
        System.out.println("8 + 4 = " + add.executeStrategy(8, 4));

        Calculation div = new Calculation(new DivideOperation());
        System.out.println("8 / 4 = " + div.executeStrategy(8, 4));
    }
}
출력
8 + 4 = 12
8 - 4 = 4
8 * 4 = 32
8 / 4 = 2
🎯
전략을 추가/수정/삭제해도 다른 클래스에는 영향이 없다 — OCP가 자연스럽게 지켜진다.

PATTERN 04 · BEHAVIORAL
Template Method
변하는 것과 변하지 않는 것
상위 클래스가 흐름을 정의하고, 하위 클래스가 빈자리를 채운다.

PATTERN 04 · Template Method SECTION
CH.7-4 · TEMPLATE METHOD
알고리즘 골격은 상위가, 구현은 하위가
  • 변하지 않는 것 — 상위 클래스의 final 템플릿 메서드알고리즘의 실행 순서·조건은 한 곳에 응집한다.
  • 변하는 것 — abstract 메서드로 하위에 위임하위 클래스가 채워야 할 빈자리를 강제한다.
  • Hollywood Principle — "Don't call us, we'll call you"하위가 상위를 부르는 게 아니라, 상위 템플릿이 하위 메서드를 부른다.
🧱
대부분의 프레임워크 — JUnit, Spring, Servlet — 이 이 패턴을 활용해 사용자 코드를 끼워 맞춘다.
Template Method · Inversion of Control 15
TEMPLATE · ABSTRACT BASE
HouseTemplate — 집을 짓는 순서를 고정한다
@Slf4j
public abstract class HouseTemplate {
  private Logger logger = LoggerFactory.getLogger(this.getClass());

  // template method, final so subclasses can't override
  public final void buildHouse() {
    buildFoundation(); // 건물 기초를 짓는다.
    buildPillars();    // 건물 기둥을 짓는다.
    buildWalls();      // 건물 벽을 짓는다.
    buildWindows();    // 건물 창호를 만든다.

    logger.info("집이 완공되었다.");
  }

  // default implementation
  private void buildWindows() {
    logger.info("창호가 완성 되었다.");
  }

  // methods to be implemented by subclasses
  public abstract void buildWalls();
  public abstract void buildPillars();

  private void buildFoundation() {
    logger.info("건물의 기초 공사가 시작 되었다.");
  }
}
🏗️
buildHouse() 가 템플릿 메서드 — final 키워드로 하위가 흐름을 변경하지 못하게 막는다.

PATTERN 05 · STRUCTURAL
Facade
단일 진입점으로 감싼다
복잡한 하위 시스템을 상위 인터페이스로 래핑한다.
클라이언트는 건물의 정면(facade)만 본다.
PATTERN 05 · Facade SECTION
CH.7-5 · FACADE
강결합을 와퍼로 끊는다
Oracle DB와 MySQL DB에 데이터가 따로 저장돼 있고, 각각 PDF/HTML 리포트를 만들어야 한다고 하자. 추가로 Excel·JSON 같은 포맷이 늘어날수록 호출 영역의 if/else 분기가 폭발한다.
🏛️
Facade 는 기존 인터페이스 위에 와퍼 인터페이스를 두어, 클라이언트가 하위 시스템의 복잡성을 알 필요 없이 단일 창구로 호출하게 만든다.
  • 단일 진입점클라이언트는 Facade 메서드 시그니처만 안다.
  • 느슨한 결합하위 비즈니스 로직 변경이 클라이언트에 닿지 않는다.
  • JDBC 드라이버대표적 활용 사례 — 다양한 DB 벤더를 동일 API로 감싼다.

HelperFacade — 두 번의 switch 를 한 곳에 가둔다
public class HelperFacade {
  public static void generateReport(
      DBTypes dbType,
      ReportTypes reportType,
      String tableName) {
    Connection con = null;
    switch (dbType) {
      case MYSQL:
        con = MySqlHelper.getMySqlDBConnection();
        MySqlHelper my = new MySqlHelper();
        switch (reportType) {
          case HTML: my.generateMySqlHTMLReport(tableName, con); break;
          case PDF:  my.generateMySqlPDFReport(tableName, con);  break;
        }
        break;
      case ORACLE:
        con = OracleHelper.getOracleDBConnection();
        OracleHelper or = new OracleHelper();
        switch (reportType) {
          case HTML: or.generateOracleHTMLReport(tableName, con); break;
          case PDF:  or.generateOraclePDFReport(tableName, con);  break;
        }
        break;
    }
  }

  public static enum DBTypes     { MYSQL, ORACLE; }
  public static enum ReportTypes { HTML, PDF; }
}

PATTERN 06 · Command SECTION
CH.7-6 · COMMAND
호출자와 실행자를 갈라놓는다
커맨드 패턴은 객체 간 결합도를 낮추기 위한 패턴 중 하나다. 요청을 객체로 캡슐화하고, 이를 나중에 실행할 수 있도록 만든다 — 실행을 다른 객체에 전달하고 캡슐화하여 내부적으로 처리한다.
  • 로깅로그 메시지 생성과 실행을 분리해 다양한 레벨/대상으로 확장.
  • UI 이벤트버튼 클릭과 동작 객체를 분리해 매크로/되돌리기 구현.
  • Undo / Redo실행 객체에 역연산을 함께 두면 자연스럽게 히스토리가 만들어진다.

COMMAND · CONTRACT + RECEIVER
Command 인터페이스와 Logger / LogCommand
Command 인터페이스
public interface Command {
    void execute();
}
Logger (Receiver)
public class Logger {
    public void log(String message) {
        System.out.println("Logging: " + message);
    }
}
LogCommand
public class LogCommand implements Command {
    private Logger logger;
    private String message;

    public LogCommand(Logger logger, String message) {
        this.logger = logger;
        this.message = message;
    }

    @Override
    public void execute() {
        logger.log(message);
    }
}
📦
LogCommand는 Logger 객체와 메시지를 받아 execute()로 실행한다.
