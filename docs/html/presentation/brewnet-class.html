CH.5 · 방법론 비교
Kent Beck의 증강코딩 5원칙
TDD 창시자 · XP 창시자 · 애자일 선언문 공동 저자 — kentbeck.com · tidyfirst.substack.com
  • Constrain Context — 컨텍스트 제한 AI에게 다음 단계에 필요한 것만 알려줄 것. 전체 코드베이스를 한 번에 던지지 말 것.
  • Preserve Optionality — 씨앗 옥수수를 먹지 마라 나중에 바꿀 수 없는 설계 결정은 인간이 직접. AI가 미래 선택지를 닫게 두지 말 것.
  • Balance Expansion & Contraction — 확장-수축 균형 AI는 팽창만 하려는 경향. 기능 개발과 리팩토링 사이클을 반드시 유지.
  • Maintain Human Judgment — 인간 판단 유지 AI 출력을 주니어 코드처럼 리뷰. 아키텍처 결정은 반드시 인간이.
  • Warning Signs — 경고 신호 인식 불필요한 루프 / 스펙 외 기능 추가 / 테스트 삭제 통과 시도 → 즉시 중단·개입

CH.6 · 실전 워크플로우
증강코딩 프롬프트 패턴
❌ Vibe Coding (위험)
# 이런 프롬프트는 위험합니다
"B+ Tree 구현해줘"
# → AI 출력 Accept All 반복
# → 아무도 이해 못하는 1,000줄
# → 기술 부채 / 유지보수 불가
✅ 증강코딩 (Kent Beck 방식)
CLAUDE PROMPT 
# plan.md의 지침을 항상 따를 것
"go" 라고 하면:
  plan.md에서 미완료 테스트 1개를 찾아서
  테스트 구현 후 그 테스트만 통과하는
  최소한의 코드만 작성할 것

경고 조건 (즉시 중단):
  함수 30줄 초과 → 분리 제안
  스펙 외 기능 추가 → 중단
  테스트 삭제 시도 → 즉시 거절
TDD Red-Green-Refactor
# 1. RED — 테스트 먼저 작성
$ claude "calculateDiscount
  테스트 작성 (Jest, TypeScript)"

# 2. 실패 확인
$ npx jest discount.test.ts
FAIL — not a function ✗

# 3. GREEN — 최소 구현
$ claude "테스트 통과하는
  최소한의 구현만 추가"

# 4. REFACTOR
$ claude "테스트 유지하며
  가독성 개선"
PASS ✓  (3 tests)
💡
핵심: 한 번에 하나씩. 테스트가 곧 명세다.
