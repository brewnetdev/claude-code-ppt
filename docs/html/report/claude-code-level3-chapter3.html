긴 컨텍스트는 단순한 "기억 용량"이 아니다. 모델은 입력이 길어질수록 정보를 균일하게 처리하지 못하고 정확도가 떨어진다.
발생 원인 4가지
  • 주의 분산 — 토큰이 많을수록 핵심 신호가 희석됨
  • 위치 편향 — 중간에 놓인 정보를 잘 못 찾음 (Lost in the Middle)
  • 오래된 사실 우선 — 초기 컨텍스트의 결정이 후속 정보를 덮어씀
  • 모순 누적 — 수정·번복이 쌓이면 일관된 의도 추론 실패

# Drizzle Conventions

## 컬럼 명명
- 테이블: snake_case 단수 (`user`, `document`).
- timestamp: `created_at`, `updated_at` 필수.
- soft delete: `deleted_at` (nullable timestamp).

## 인덱스
- 외래키는 항상 인덱스 — `index('idx_doc_user_id').on(t.userId)`
- 검색용 컬럼은 `index().using('gin', ...)`로 GIN.

## 관계
- 1:N — `references()` + `relations()` 양쪽 정의.
- N:M — 별도 join 테이블 `*_to_*` 명명.

0성공 — 그대로 진행stdout은 transcript 모드에서 사용자에게 표시
1비차단 오류 — 실행 계속stderr는 사용자에게 표시되지만 도구 호출 막지 않음
2차단 — 도구 호출 거부stderr가 모델에게 전달되어 다시 시도하게 만듦
기타비차단 오류로 처리1과 동일 동작
⚠️
정책 강제는 exit 2 "이 파일은 수정 금지" 같은 가드는 exit 2로 stderr에 사유를 적어야 차단되고 모델이 재시도한다. exit 1은 경고만 띄우고 작업은 계속된다.
💡
JSON output 고급 stdout으로 {"decision": "block", "reason": "..."} 형태 JSON을 내보내면 exit code보다 세밀한 제어 가능 (PreToolUse 한정).
