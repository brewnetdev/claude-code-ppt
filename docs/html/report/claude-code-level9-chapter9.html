💡
실패 모드 1 — 컨텍스트 불안 (Context Anxiety) 컨텍스트 창이 채워질수록 품질 저하. 일부 모델은 한계에 다가가면 작업을 조기 마무리 — 기능이 stub 상태로 남거나 핵심 로직 빠진 채 완료 선언. 해결: **컨텍스트 리셋** + 구조화된 핸드오프.
💡
실패 모드 2 — 자기평가 불능 (Self-Evaluation Blindspot) 에이전트에게 자기 작업을 평가하면 자신의 결과물을 과하게 칭찬. '이 API가 동작합니까?' → '네, 잘 됩니다.' (실제로는 에러 처리 없음, 항상 200 반환, stub 상태). 해결: **Generator**-**Evaluator** 분리.
📘
Anthropic 팀의 해결 원칙 "Separating the agent doing the work from the agent judging it proves to be a strong lever to address this issue." — 만드는 에이전트와 평가하는 에이전트를 분리하는 것이 강력한 레버.

7.2.3 OpenHarness 분석 II — Agent Loop 의사코드와 세 기둥 매핑
도구 실행의 6단 파이프라인: PreHook → Schema Validation → Permission → Execute → Output Offload → PostHook
  • Stop reason driven — 모델의 stop_reason이 루프 제어하네스는 '도구가 있는지', '권한이 있는지'만 판정
  • Streaming first — 텍스트는 토큰 단위 스트림, 사용자에게 즉시 노출
  • Parallel tool calls — N개 호출은 asyncio.gather로 병렬 처리모든 tool_use에 대응하는 tool_result가 있어야 API 다음 턴 수신
  • Self-healing context — prompt too long 감지 시 reactive compactionauto_compact → reactive compact 2단계. 통과 못 하면 ErrorEvent
  • Hook-permission-execute 3단 — 모든 도구 실행은 동일한 3단. 우회 경로 없음
