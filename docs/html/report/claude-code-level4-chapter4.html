스킬 · 커맨드 · Hook · MCP — Claude Code 확장과 자동화
기본 사용을 넘어 Claude Code를 나만의 도구로 확장한다 — 반복은 커맨드/스킬로 캡슐화, 안전은 훅으로 강제, 외부 연결은 MCP로.
🎯WHY
왜 배우나

AI를 기본값으로만 쓰면 매번 같은 지시를 반복하고 안전장치 없이 위험에 노출된다. 커맨드·스킬·훅·MCP는 반복을 제거하고 결정적 가드를 거는 나만의 하네스(Level 8)의 토대다.

📚WHAT
무엇을 배우나 — 5개 파트
  • ▸ 4.1 Slash Commands — 반복 작업을 단축 명령으로
  • ▸ 4.2 Agent Skills — 재사용 가능한 메뉴얼 지침 
  • ▸ 4.3 Hooks — 라이프사이클 결정적 가드레일 
  • ▸ 4.4 MCP — 외부 도구·데이터 연결
  • ▸ 4.5 실습 — 나만의 커맨드·스킬·훅 만들기
🛠️HOW
어떻게 배우나

각 기능을 정의 파일로 직접 작성 → 동작 확인 → 조합. 나쁜 예 vs 좋은 예로 트리거·권한 설계를 체감하고, Level 5 프로젝트에 붙여 적용을 확인 해본다.

4.1.3 CUSTOM
커스텀 커맨드 — .claude/commands/
반복되는 프롬프트를 파일로 박아두면 슬래시 한 줄로 호출, YAML frontmatter 사용 + description 필드 + allowed-tools 
# .claude/commands/code-review.md
---
description: 현재 git diff 기준 코드 리뷰
allowed-tools: ["Bash(git:*)", "Read", "Grep"]
---

다음 절차로 코드 리뷰를 수행해줘:

1. `git diff origin/main...HEAD` 결과를 읽는다.
2. 변경된 파일별로 ① 의도 ② 위험 ③ 누락 테스트를 정리.
3. 보안·성능·접근성 관점 위반이 있으면 명시.
4. 최종 표 형태로 출력 — 파일 / 위험도 / 권고.
📘
스코프 Project: .claude/commands/ — 팀 공유 / User: ~/.claude/commands/ — 모든 프로젝트 / 둘 다 있으면 Project 우선.

$ARGUMENTS — 커맨드에 인자 받기
# .claude/commands/fix-issue.md
---
description: GitHub 이슈 번호로 수정안 작성 + PR 초안 
argument-hint: [issue-number]
allowed-tools: Bash(gh:*), Read, Grep, Glob
---
## Issue Context !`gh issue view $ARGUMENTS`## Issue Comments !`gh issue view $ARGUMENTS --comments`
## Task  
이슈 #$ARGUMENTS를 처리하라:
1. 분석: 이슈 본문과 코멘트에서 핵심 요구사항·재현 시나리오 파악
2. 탐색: Grep/Glob으로 관련 파일 식별
3. 최소 변경안 제안: - 어떤 파일을 수정할지
   - 변경 전후 코드 (diff 형태)   - 영향 범위
4. PR 본문 초안 작성:요약 / 변경 내용 / 테스트 방법 / 관련 이슈(Closes #$ARGUMENTS)
# .claude/commands/add-feature.md
---
description: 새 기능 스캐폴딩(Next.js+Drizzle+Vitest)
argument-hint: [feature-name]allowed-tools: Write, Read, Glob, Bash(mkdir:*), Bash(pnpm:*), Bash(ls:*)---
# Add Feature 
## 인자 검증 `$ARGUMENTS`가 다음 조건을 모두 만족하는지 확인:
- 정규식 `^[a-z][a-z0-9-]*$` 일치 (영문 소문자·숫자·하이픈)
- 길이 2~40자
## 컨벤션 적용 준비
CLAUDE.md를 확인하고 다음 영역 적용:
- 파일 명명: kebab-case
- 함수: 50줄 이하 (Cyclomatic 10 이하)
- 입력 검증: zod 스키마 (서버 액션 진입점만)

## 기존 패턴 참고
!`ls src/features/ 2>/dev/null | head -5`

## 스캐폴딩 작업
1. `mkdir -p src/features/$ARGUMENTS`
2. 컴포넌트·서버액션·Drizzle 스키마·Vitest 테스트 파일 생성
3. 생성 결과 요약 출력

4.1.5 PATTERNS
실전 명령어 조합 패턴 
① TDD 사이클
테스트를 먼저 — 모델이 Red·Green·Refactor 단계를 섞지 못하게 강제
/tdd-red실패하는 테스트만 작성 (구현 X)
↓
/tdd-green그 테스트를 통과할 최소 구현만
↓
/tdd-refactor동작 유지한 채 구조만 정리
결과 단계가 분리돼 검증 가능한 사이클 — Red 없이 Green 불가
② 자동화 파이프라인
네 점검을 래퍼 커맨드 /pre-commit 하나에 순서대로 담는다
/lint-fix린터로 포맷·스타일 자동 수정
↓
/typecheck타입 오류 검사 (tsc --noEmit)
↓
/test-affected변경에 영향받은 테스트만 실행
↓
/commit-msg변경 diff 기반 커밋 메시지 생성
자동화 방법 .claude/commands/pre-commit.md 본문에 위 4단계를 순서대로 기술 — /pre-commit 한 줄이 전체 실행, 실패한 단계에서 멈추고 수정 후 이어간다
③ 컨텍스트 오염 방지
긴 세션에 쌓인 시행착오·노이즈를 끊고 깨끗하게 재개
/checkpoint핵심 결정·상태를 파일로 저장(dump)
↓
/clear대화 컨텍스트 완전 초기화 (내장)
↓
/restore-checkpoint저장한 결정만 다시 로드
결과 누적 오류 없이 깨끗한 컨텍스트로 작업 재개
공통 원칙한 작업 = 한 커맨드 — 각 커맨드가 자기 책임만 갖고 행위를 명확히 끊는다. ※ /clear 외에는 .claude/commands/ 에 직접 만드는 커스텀 예시 커맨드

4.2.1 CONCEPT
Agent Skill — 모델이 자가 호출하는 절차 묶음
AI 에이전트가 새로운 기능과 전문성을 부여하는 표준화된 방식
구성 : SKILL.md 한 파일 = 이름 · 설명 · 허용 도구 · 본문 절차
관점Slash Command(User-invoke)Skill(Model-invoke)
호출사용자가 /명령 입력모델이 description 매칭으로 자동 호출
발견사용자가 알아야 함, 단일 작업 발동기 모델이 컨텍스트 보고 판단, 재사용 가능한 작업 모듈 
본문프롬프트 텍스트마크다운 + frontmatter (allowed-tools 등)
📘
통합 메커니즘 현재 Slash Command와 Skill은 동일 lazy-load 메커니즘 위에서 동작한다. 신규는 SKILL.md로 작성 권장 — 모델 자가 발견·호출이 가능해서 사용자 부담이 줄어든다. 
Command = 단축키 VS Skill = 자가호출 컨텍스트 메뉴 모음 (필요할 때 읽는 메뉴얼) 
⚠️
Description이 전부 모델은 description만 보고 호출 여부를 결정한다. "언제 써야 하는지" 명확히 적지 않으면 절대 호출되지 않는다.
👉  https://agentskills.io/

# 코드 리뷰 가이드
## 이 Skill을 사용할 때 
- PR 리뷰 요청, 코드 품질 분석
- 리팩터링 전 검토

## Instructions
### 적용 범위 
이 Skill은 프로젝트 전체 코드에 적용됩니다: 
- `frontend/src/**/*.{ts,tsx}` 

## 리뷰 관점 
### 1. 코드 품질 
- 타입 안정성 (any 사용 여부), 함수 길이 (30줄 이하 권장), 중복 코드 여부 
### 2. 컨벤션 준수 
- 네이밍 규칙, 파일/폴더 구조 준수 여부 
## 제한 사항 
이 Skill은 읽기 전용입니다. 코드를 직접 수정하지 않고 피드백만 제공합니다.

4.2.2 Skill이 수행할 수 있는 작업 유형 
Skill이 수행할 수 있는 작업 유형
  • 파일 시스템 접근
    파일 읽기·쓰기·수정에 대한 구체적인 규칙과 패턴을 정의할 수 있다.
  • 명령어 실행 특정 상황에서 실행해야 할 셸 명령어나 스크립트를 정의할 수 있다. 
  • 웹 검색 및 문서 참조 
    최신 정보나 공식 문서를 참조해야 하는 상황에 대한 지침을 제공할 수 있다. 
  • 권한 제어나 보안 
    allowed-tools 필드를 통해 활성화된 스킬의 도구를 제한할 수 있다.

두 가지 Skill 유형 — Reference vs Action

# Coding Standards

## 네이밍 규칙
- 함수: `camelCase` 동사 시작 (`getUser`, `saveDraft`)
- 컴포넌트: `PascalCase` (`UserCard`, `SearchBar`)
- 상수: `UPPER_SNAKE_CASE` (`MAX_RETRY`, `API_BASE_URL`)
- 불리언: `is`/`has` 접두사 (`isLoading`, `hasError`)
## 가독성 (중요)
**함수는 한 가지 일만 한다.** 20줄이 넘으면 분리를 먼저 검토.
"X 작업할 때 이 규칙을 참조해" — 지식·컨벤션만 제공, 도구 호출이 거의 없는 형태

4.3.4 작업 완료 시 알림 받기 - NOTIFICATION
Slack Webhook 개설·연동 — URL 발급부터 환경변수 등록까지
Webhook = 이벤트 발생 시 미리 등록한 URL로 데이터를 자동으로 쏘는 방식 — Slack이 발급한 URL 하나에 메시지를 POST하면 지정 채널에 글이 뜬다. 이 장은 그 URL을 발급받아 환경변수로 등록하는 절차다.
STEP · 개설 4단계
①
Create App — api.slack.com/apps → "From scratch", 앱 이름·워크스페이스 선택
②
Incoming Webhooks — 앱 설정 좌측 메뉴에서 상단 토글을 On
③
Add to Workspace — 알림 받을 채널(#dev-alerts) 선택 → 허용
④
Copy URL — hooks.slack.com/services/T.../B.../... 복사
# macOS · Linux — 셸 프로필에 export
export SLACK_WEBHOOK_URL="https://hooks.slack.com/..."
source ~/.zshrc        # 또는 새 터미널

# Windows — setx (새 터미널부터 적용)
setx SLACK_WEBHOOK_URL "https://hooks.slack.com/..."

# WSL — Windows 값 미전달 → WSL ~/.bashrc 에 export

# 연동 테스트 (macOS·Linux·WSL)
curl -X POST -H 'Content-type: application/json' \
  --data '{"text":"연동 테스트"}' $SLACK_WEBHOOK_URL
보안이 URL 자체가 비밀번호다 — 아는 사람은 누구나 그 채널에 글을 쓸 수 있다. 코드·settings.json에 하드코딩 금지, 환경변수로만. 실수로 커밋했다면 Slack 앱에서 revoke 후 재발급.

#!/usr/bin/env bash
# .claude/hooks/notify-slack.sh — Slack Webhook 알림

payload=$(cat)

# Webhook URL 미설정 시 silent exit
if [ -z "$SLACK_WEBHOOK_URL" ]; then
  exit 0
fi

msg=$(echo "$payload" | jq -r '.message // "Done"')

# jq로 JSON 페이로드 안전하게 빌드 (따옴표 escape 자동)
json=$(jq -n --arg text "[Claude] $msg" '{text: $text}')

curl -s -X POST "$SLACK_WEBHOOK_URL" \
  -H "Content-Type: application/json" \
  -d "$json" > /dev/null || true

exit 0

#!/usr/bin/env bash
# .claude/hooks/lint-verify.sh — PostToolUse, 위반 발견 시 Claude에게 피드백

payload=$(cat)
file=$(echo "$payload" | jq -r '.tool_input.file_path // ""')

# 빈 경로는 통과 셍략 
case "$file" in
  *.ts|*.tsx)
    LINT_OUT=$(pnpm exec eslint "$file" --max-warnings 0 2>&1)
    if [ $? -ne 0 ]; then
      echo "ESLint 위반 in $file:" >&2
      echo "$LINT_OUT" >&2
      echo "위 위반 사항을 수정해주세요." >&2
      exit 2
    fi
    ;;
esac

exit 0

#!/usr/bin/env bash
# .claude/hooks/typecheck-on-stop.sh — Stop, 응답 종료 시 1회

# Stop은 file_path 받지 않음. 전체 검사로 의미 있음.
TSC_OUT=$(pnpm exec tsc --noEmit 2>&1)
if [ $? -ne 0 ]; then
  echo "TypeScript 컴파일 에러 발견:" >&2
  echo "$TSC_OUT" >&2
  exit 2   # Stop의 exit 2는 "계속 작업" 의미

4.3.7 CAVEATS
주의사항·디버깅
  • 훅은 사용자 권한으로 실행시스템·DB·자격증명에 영향을 주는 명령을 무심코 적지 말 것 — 훅은 자동 실행
  • jq 미설치 환경 주의CI·서버에서 brew·apt로 jq 설치 필수, 없으면 훅이 조용히 실패
  • 비차단(exit 1) 메시지는 사용자만 봄모델에게 사유를 전달하려면 exit 2 + stderr 필수
  • 매처 누락 = 모든 도구PreToolUse에 matcher 안 적으면 Read·Grep까지 가로챔 → 성능 저하
  • 디버깅claude --debug로 훅 발화·exit code·stderr 추적. 훅 자체에 set -x + 로그 파일 저장 권장
✅
설계 원칙 훅은 "1초 안에 끝나야" — 느린 훅은 모든 도구 호출을 지연시킨다. 무거운 검증은 PostToolUse + 비동기로.

# GitHub — 원격 HTTP (구 server-github은 2025-04 지원 종료)
claude mcp add --transport http github https://api.githubcopilot.com/mcp/ --header "Authorization: Bearer YOUR_PAT"

claude mcp add postgres --env DATABASE_URI=postgresql://localhost:5432/mydb -- uvx postgres-mcp --access-mode=restricted
# Brave Search — Brave 공식 v2 (구 server-brave-search deprecated)
claude mcp add brave-search --env BRAVE_API_KEY=YOUR_KEY -- npx -y @brave/brave-search-mcp-server
# Context7 — 라이브러리 최신 문서 주입 (원격 HTTP)
claude mcp add --transport http context7 https://mcp.context7.com/mcp --header "CONTEXT7_API_KEY: YOUR_KEY"
# Filesystem — 인자로 넘긴 디렉토리만 접근 허용
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem ~/projects
# Playwright — 브라우저 자동화 (Microsoft 공식)
claude mcp add playwright npx @playwright/mcp@latest
# Sequential Thinking — 단계별 구조화 추론 (레퍼런스 서버)
claude mcp add sequential-thinking -- npx -y @modelcontextprotocol/server-sequential-thinking

.mcp.json — 프로젝트 공유 구문
# <프로젝트루트>/.mcp.json — project scope · git 커밋으로 팀 전체 공유
{
  "mcpServers": {
    "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/",
                "headers": { "Authorization": "Bearer ${GITHUB_PAT}" } },
    "context7": { "type": "http", "url": "https://mcp.context7.com/mcp",
                  "headers": { "CONTEXT7_API_KEY": "${CONTEXT7_API_KEY}" } },
    "postgres": { "command": "uvx", "args": ["postgres-mcp", "--access-mode=restricted"],
                  "env": { "DATABASE_URI": "${DATABASE_URI}" } },
    "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] }
  }
}
# 원격 서버 = type + url + headers · 로컬 stdio = command + args + env
💡
${ENV} 확장 .mcp.json은 git에 커밋되는 파일 — API 키 리터럴 금지. ${GITHUB_PAT}처럼 쓰면 Claude Code가 로드 시 환경변수로 치환한다.

4.4.3 SETUP
API 키는 어디에 저장하나 — ${ENV} OS별 세팅
${VAR}는 claude를 실행한 셸의 환경변수에서 치환된다 — .mcp.json은 git에 커밋되는 파일, 키는 셸 프로필에만 둔다
# macOS — ~/.zshrc 에 export 로 저장
export GITHUB_PAT="ghp_..."
export CONTEXT7_API_KEY="ctx7_..."
export DATABASE_URI="postgresql://..."

source ~/.zshrc  # 적용 후 claude 재시작

# 보안 강화 — Keychain에 넣고 꺼내 쓰기
export GITHUB_PAT="$(security find-generic-password -s github-pat -w)"
# Windows — setx 로 사용자 환경 변수 등록
setx GITHUB_PAT "ghp_..."
setx CONTEXT7_API_KEY "ctx7_..."
setx DATABASE_URI "postgresql://..."

# ⚠ 새 터미널부터 적용 — 열려 있던 창엔 반영 안 됨
echo $env:GITHUB_PAT  # PowerShell 확인

# GUI 경로: 시스템 속성 → 고급 → 환경 변수
WSL 주의WSL에서 claude를 돌리면 Windows 환경변수는 전달되지 않는다 — WSL 쪽 ~/.bashrc에 export 한다. 그리고 어떤 OS든 .mcp.json에 키 리터럴을 직접 쓰지 말 것 — 커밋 한 번이면 유출이다.
