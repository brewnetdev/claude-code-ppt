적절한 작업 크기의 4가지 기준
1
단일 책임 원칙
하나의 요청은 하나의 책임만 — '사용자 생성'과 '이메일 발송'은 별개
2
테스트 가능한 단위
예/아니오로 답할 수 있는 크기
3
10분 내 검증 가능한 크기
AI 출력 → 코드 훑기 → 테스트가 10분 넘으면 더 쪼갠다
4
파일·기능 단위 경계
회원가입·수정과 로그인·로그아웃은 별도 파일로 분리
점진적 설계 — 처음부터 완벽할 필요 없다
1차 결정은 바꾸기 어렵고 · 2~3차는 구현 중 조정 가능
1차 설계
프로젝트 시작 전 — 필수
배포 형태: 온프레미스  ·  기술 스택: Node.js + PostgreSQL  ·  인증 방식: Google OAuth + 자체 인증
2차 설계
1주차 — 구현 중
API 엔드포인트 상세 정의  ·  데이터베이스 스키마 확정  ·  에러 코드 체계 정의
3차 설계
2주차 — 통합 전
프런트엔드 컴포넌트 구조  ·  상태 관리 전략  ·  API 응답 형식 표준화
