### PPT 슬라이드 구성안 (Claude Code 기준)

**프롬프트 엔지니어링 vs 컨텍스트 엔지니어링**

부제: "말을 잘 하는 것"에서 "일하는 환경을 설계하는 것"으로

[스크린샷 영역: 빈 표지 — 좌측 상단 라벨 `LEVEL 3 · 프롬프트와 컨텍스트`, 하단 강의 브랜드]




**오늘의 결론**

- **프롬프트 엔지니어링** = 완벽한 메시지 한 통을 쓰는 기술 → *한 번의 입력 최적화*
- **컨텍스트 엔지니어링** = AI 책상 위에 어떤 자료가 올라가 있는가를 설계하는 기술 → *추론 환경 전체 설계*

> Anthropic은 컨텍스트 엔지니어링을 "프롬프트 엔지니어링의 자연스러운 다음 단계"로 정의한다.
> 대체가 아니라 **확장**이다.

출처: Anthropic, *Effective context engineering for AI agents* (2025.09.29)
https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### 질문이 다르다

**두 기술은 던지는 질문이 다르다**

| 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|
| "어떻게 **물을까**?" | "AI가 추론할 때 **무엇을 보게 할까**?" |
| 단일 턴, 메시지 1개 | 다중 턴, 세션 전체 |

비유:

- 프롬프트 = 완벽한 이메일 **한 통** 쓰기
- 컨텍스트 = 전문가가 일할 **사무실**을 차려주기

### 예시 도입

**같은 작업을 두 방식으로 시켜본다**

요청: `markflow 프로젝트에 회원가입 API 만들어줘`

- 상황 A — 프롬프트만 잘 쓰는 경우 (빈 세션)
- 상황 B — 환경을 미리 설계한 경우

> 핵심: **요청은 같다. 환경이 다르다.**

[흐름도: 동일한 요청 "회원가입 API" → 분기 → (A) 긴 프롬프트 한 통 / (B) 짧은 프롬프트 + 미리 깔린 환경]

---

### 상황 A: 프롬프트만

**상황 A — 메시지 하나에 전부 욱여넣기**

CLAUDE.md 없음 · 관련 파일 미로드 → 품질이 메시지 하나에 전부 달림

```text
markflow에 회원가입 API를 만들어줘.

[기술 스택] Next.js 16 App Router, Drizzle ORM, PostgreSQL 16, TypeScript
[규칙] Result<T, E> 패턴 / parameterized statement만 / any 금지
[검증-Zod] email 필수·형식 / password 필수·8자 이상
[에러 형식 예시]
return NextResponse.json(
  { error: '이미 가입된 이메일입니다.' },
  { status: 409 }
);

위 패턴을 그대로 따라서 POST /api/auth/signup 구현해줘.
```

> 잘 쓴 프롬프트(제로샷+퓨샷+제약)지만 **매번 반복**해야 한다.
> 다음 작업("로그인 만들어줘")에서도 스택·규칙·에러형식을 또 붙여야 함.

---

### 상황 B: 환경 설계

**상황 B — AI 책상 위에 자료를 미리 깔아둔다**

① CLAUDE.md (세션마다 자동 로드 = 영구 컨텍스트)

```markdown
## Tech Stack
- Next.js 16 App Router · Drizzle ORM · PostgreSQL 16 · TypeScript
## Rules
- API 응답은 Result<T, E> 패턴
- DB 쿼리는 parameterized statement만
- any 금지 / 검증은 Zod / 에러는 한국어 메시지 + 상태 코드
```

② 관련 파일만 적시 로드 → `@app/api/auth/login/route.ts`

③ 그러면 프롬프트가 이렇게 짧아진다

```text
로그인이랑 같은 패턴으로 POST /api/auth/signup 만들어줘.
```

> 한 줄 요청인데 같은 품질. 환경이 정보를 **대신 들고 있기** 때문.
> 이 환경은 다음 작업에도 재사용된다.

---

### 왜 환경을 설계하나 (Context Rot)

**"더 많이 넣으면 좋다"는 틀렸다 — Context Rot**

- LLM도 사람처럼 주의력이 유한하다
- 정보가 많아질수록 핵심 집중·정확한 회수 능력이 **떨어진다**
- 컨텍스트 윈도우를 늘린다고 성능이 좋아지지 않는다

| 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|
| "더 많이, 더 자세히" | "필요한 것만, 적시에" |

원인: Transformer Self-Attention의 O(n²) 비용 — 토큰 2배 → 처리 관계 4배

출처: Anthropic 위 블로그 / Chroma Research, *Context Rot*
https://www.trychroma.com/research/context-rot

### Claude Code 도구 매핑

**Claude Code의 도구는 컨텍스트 엔지니어링을 위해 존재한다**

| 도구 | 역할 |
|---|---|
| `CLAUDE.md` | 규칙을 영구 컨텍스트로 고정 (매 세션 1회 주입) |
| `@멘션` | 전체가 아닌 **필요한 파일만** 올림 |
| `/context` | 책상 위 토큰 사용량 시각화 |
| `/compact` | 같은 작업 유지하며 히스토리 압축 |
| `/clear` | 주제 전환 시 잔해 비우기 |

### 한 장 비교표

**정리 — 한눈에 보는 차이**

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|---|
| 질문 | 어떻게 물을까? | 무엇을 보게 할까? |
| 범위 | 단일 턴 / 메시지 1개 | 다중 턴 / 세션 전체 |
| 관리 대상 | 지시문·예시·제약 | 시스템 프롬프트 + CLAUDE.md + 도구 + 파일 + 히스토리 |
| 비유 | 완벽한 이메일 한 통 | 일할 사무실 차려주기 |
| 수단 | 제로샷·퓨샷·CoT | CLAUDE.md · @멘션 · /context · /compact |
| 시점 | 요청 전 한 번 | 매 추론마다 반복 |

## Slide 10 — 다음으로

**포함 관계로 이해하기**

```
프롬프트  ⊂  컨텍스트  ⊂  하네스
```

- 각 단계는 이전을 **부정하지 않고 포함**한다
- 좋은 컨텍스트는 여전히 좋은 프롬프트를 요구한다
- 다음 단계 **하네스 엔지니어링**은 LEVEL 7에서 다룬다

[흐름도: 2022–2024 프롬프트 → 2025 컨텍스트 → 2026 하네스, 우상향 계단형 + 각 칸에 핵심 질문]
